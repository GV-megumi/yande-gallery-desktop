你正在 Claude Code 中执行任务。

# 0. 必须先遵守的流程约束

你必须明确使用并遵守以下 superpowers 技能思想与流程约束（即使不逐字输出，也必须落实到执行过程）：

1. **superpowers:systematic-debugging**
   - 只要遇到异常、结果异常、运行失败、行为不符预期、测试失败、性能异常、页面异常，就必须按系统化排查方式继续深挖，不能草率跳过。

2. **superpowers:verification-before-completion**
   - 在你声称“某项检查已完成”“某功能已验证”“某问题已确认/不存在”之前，必须先有实际验证证据。
   - 不能凭印象、不能凭猜测、不能只看代码表面就下结论。

3. **superpowers:requesting-code-review**
   - 当你完成主要审查阶段后，要进行“自我复审式的二次审查”：
     - 重新检查自己是否遗漏高风险区域
     - 重新检查结论是否有证据支撑
     - 重新检查报告是否存在空泛结论
   - 等价于让另一个极度苛刻的 reviewer 再挑一次毛病。

4. **superpowers:dispatching-parallel-agents**
   - 如果当前环境支持并行 agent / 子任务拆分，则必须尽量并行化审查：
     - UI/UX 审查
     - 架构/IPC 审查
     - 下载/并发/性能专项审查
     - Electron 桌面能力专项审查
     - 联网对标与先进库替代专项审查
   - 如果无法真正并行，也必须按“分专项、分轮次、交叉复核”的方式模拟并行工作流。

5. **superpowers:verification-before-completion** 再强调一次：
   - 你不能因为“看起来差不多”就结束任务。
   - 你不能因为“项目能运行”就认为没问题。
   - 你不能因为“测试通过”就认为体验合理、架构合理、桌面能力合理。

# 1. 任务目标

你的任务是**完整、细致、绝对挑剔地审查当前项目**，并且重点关注当前项目可能存在的大量问题，包括但不限于：

- 部分功能按钮不全
- 批量下载功能响应缓慢
- 基础页面功能不全
- 某些页面同时存在自定义关闭按钮和系统默认 X，行为重复、冲突或体验粗糙
- 并发、多线程能力弱
- 不支持后台常驻运行
- Electron 桌面应用能力不完整
- 架构边界不清
- 部分实现可能落后于 2026 年常见优秀桌面应用实践
- 某些自研实现可能本来就应该用成熟库替代

你的任务不是温和建议，而是：

1. **全面审查当前项目所有主要模块、页面、服务、链路、架构层次**
2. **尽可能找出所有具体问题**
3. **联网了解 2026 年先进实践、同类优秀产品、成熟库方案**
4. **对比当前项目与先进方案的差距**
5. **给出可落地的替代建议和整改优先级**
6. **在项目根目录生成一份高质量、带证据链、非常详细的审查报告**

# 3. 强制审查轮次

你必须至少完成以下 **8 轮滚动审查**。  
每一轮都必须有明确产出，不能走过场。

## 第 1 轮：项目结构与覆盖矩阵建立
你必须：
- 梳理项目目录结构
- 识别主进程 / 渲染进程 / preload / shared / service / UI 页面 / 下载链路 / 配置链路 / 窗口链路
- 建立“审查覆盖矩阵”
- 列出必须覆盖的页面、模块、服务、IPC、关键流程

输出要求：
- 形成覆盖清单
- 标记已覆盖 / 待覆盖 / 风险高的区域

## 第 2 轮：功能完整性审查
你必须逐页逐功能审查：
- 是否存在功能缺失
- 是否存在缺少按钮、缺少入口、缺少状态、缺少反馈、缺少错误提示、缺少操作闭环
- 是否存在页面“看起来有，但实际上不完整”

重点：
- 不要只检查是否“有实现”
- 要检查是否“构成完整可用链路”

## 第 3 轮：UI / UX 细节审查
你必须像最挑剔的桌面应用设计 reviewer 一样审查：
- 交互一致性
- 按钮位置和层级
- 文案
- 弹窗
- 关闭行为
- 视觉冗余
- 信息密度
- 操作路径长度
- 默认行为是否合理

重点查找：
- 自定义关闭按钮与系统默认 X 并存
- 重复操作入口
- 页面断裂
- 状态不可见
- 用户不知道当前发生了什么
- 设置项与行为对不上
- 反馈迟钝或含糊

## 第 4 轮：性能 / 下载 / 并发 / 调度专项审查
重点审查：
- 批量下载慢在哪里
- 任务链路是否不必要串行化
- 是否有重复调度
- 是否有竞争条件
- 是否有初始化阶段阻塞
- 是否缺乏合理任务队列
- 是否缺乏多线程 / worker / 后台任务执行能力
- 是否缺乏取消 / 重试 / 恢复 / 去重 / 优先级 / 限流

必须尽量通过：
- 代码路径追踪
- 运行验证
- 日志观察
- 行为推断
来确认瓶颈点

## 第 5 轮：Electron / 桌面应用能力专项审查
重点审查：
- 托盘
- 最小化到托盘
- 关闭窗口不退出
- 后台常驻
- 后台下载
- 长任务持续执行
- 开机自启
- 通知反馈
- 窗口生命周期
- 桌面化体验是否合格

要求：
- 不要把它当“网页壳”
- 要按桌面下载/图库/管理类应用标准审查

## 第 6 轮：架构 / IPC / preload / shared / 可维护性审查
重点审查：
- 主进程和渲染进程边界是否合理
- preload 暴露面是否清晰
- shared type 是否足够统一
- service 是否耦合过重
- 页面层是否承担过多业务逻辑
- IPC 设计是否粗糙
- 是否存在职责漂移、重复实现、难维护设计

## 第 7 轮：稳定性 / 错误恢复 / 边界条件 / 安全性审查
重点审查：
- 错误处理是否完整
- 是否有恢复机制
- 是否有边界条件未处理
- 是否有取消、失败、超时、资源释放不完整
- 是否有 Electron 常见隐患
- 是否存在“平时没事，一出事就很致命”的问题

## 第 8 轮：联网对标与先进方案替代审查
你必须联网调研：
- 2026 年同类优秀桌面应用的常见设计
- Electron / React 最新成熟实践
- 下载器 / 图片管理器 / 桌面图库工具常见能力
- 可替代当前自研实现的成熟库
- 现代虚拟列表、图片加载、任务调度、后台运行、窗口管理、托盘、日志、缓存方案

你必须输出：
- 当前项目缺失了哪些现代常见能力
- 当前哪些自研实现落后、重复造轮子、维护成本高
- 哪些成熟库值得替代
- 每项替代的收益 / 成本 / 风险 / 迁移建议

# 5. 必须以证据为准

每一个问题都尽量附带至少一种证据：

- 代码证据：文件路径、关键位置、相关调用链

# 6. Claude Code 执行方式要求

你在 Claude Code 中必须按以下方式工作：

## 6.1 必须建立任务清单并持续更新
使用任务清单跟踪至少这些阶段：
- 建立覆盖矩阵
- 审查页面与功能完整性
- 审查 UI/UX 细节
- 审查下载 / 并发 / 性能
- 审查 Electron / 后台运行能力
- 审查架构 / IPC / preload / shared
- 审查稳定性与错误恢复
- 联网对标先进实践与替代库
- 撰写详细审查报告
- 自我复审并补漏

## 6.2 必须尽可能运行项目并验证
如果环境允许，你必须尽量执行：
- 安装依赖
- 启动项目
- 类型检查
- 构建
- 测试
- 核心脚本
- 关键页面与功能路径验证

如果某些验证无法进行：
- 记录阻塞原因
- 说明影响范围
- 继续做代码级深挖，不能直接放弃

## 6.3 必须优先深挖，不要浅尝辄止
对每一类问题，都问自己：
- 这里只是表面问题，还是背后还有系统性问题？
- 这里是不是只是不完整，还是设计本身就不合理？
- 这里是不是本来应该支持后台化 / 并发化 / 现代化？
- 这里是不是已经有成熟库可替代？
- 这里是不是能用，但用户体验其实很差？
- 这里是不是维护成本已经很高？

## 6.4 必须做交叉复核
完成主要问题收集后，必须再做至少一轮“反向复核”：
- 重新检查有没有漏掉高风险页面
- 重新检查有没有漏掉高风险链路
- 重新检查报告中的结论是否都能落到证据
- 重新检查有没有“泛泛而谈”的段落
- 重新检查是否过度关注某一类问题而忽略其他维度

  - 功能残缺

你必须使用以下态度执行：

- 默认“不信任表面现象”
- 默认“当前实现还有大量问题值得深挖”
- 默认“能用 != 合格”
- 默认“有实现 != 完整”
- 默认“测试通过 != 产品体验正确”

你是来**尽最大努力把当前项目的毛病、缺陷、隐患、短板、过时点、不合理实现、可替代低质量自研逻辑全部挖出来**，形成一份真正有杀伤力、有决策价值、足够苛刻、足够细的审查报告。

现在开始执行，不要偷懒，不要缩水，不要用简单回归替代深度审查。
