# 安卓本机相册（手机相册 tab）与复制/移动体系设计（v0.8.0）

> 状态：📝 设计完成，待实施。承接 v0.7.0 图片镜像层（`2026-07-13-android-image-mirror-design.md`，其 §9/D8 明确「导出到系统相册等本机相册功能落地时一并提供」——即本文）。
> 目标版本：安卓 v0.8.0（versionCode 9）；**桌面端零改动**（复制/移动到桌面相册全部复用既有手机面写接口）。

## 0. 背景与决策快照

### 0.1 动机

App 目前是「桌面图库的局域网伴侣」，只认识桌面域数据；手机本机的系统相册（相机、截图、微信等 MediaStore bucket）完全不可见。目标：底部导航新增「手机相册」tab，把本机相册浏览/管理补成一等公民，并借势把「加入相册」升级为跨域的「复制到 / 移动到」体系。

### 0.2 用户已定决策

| # | 决策点 | 结论 |
|---|---|---|
| D1 | 定位档次 | **B 档：完整本地图库**——浏览 + 分享 + 删除本地照片，朝「可替代系统相册的日常浏览」走；不含手机→桌面上传（那是 C 档，另立项） |
| D2 | 视频 | **图片 + 视频**：网格显示视频缩略图 + 时长角标，点开**调系统播放器**（app 内不做播放器）；删除/分享/复制/移动与图片同权 |
| D3 | 删除语义 | **永久删除**（`createDeleteRequest` 系统确认弹窗），不用 `createTrashRequest`——国产 ROM（MIUI 系）系统回收站入口碎片化，照片会「消失进找不到的地方」 |
| D4 | 桌面→手机导出 | **做**，语义为「复制到」：复制目标支持桌面相册（走桌面同步写接口）与手机相册（下载原图落 MediaStore 目标相册） |
| D5 | 桌面域移动 | 「移动到」仅支持桌面相册（桌面同步）；**禁止移动到手机相册**——不允许以移动之名删除桌面端图片 |
| D6 | 手机侧复制/移动 | **做**：手机图片可在手机相册之间复制（MediaStore 拷贝新行）与移动（`createWriteRequest` 系统授权后物理移动文件） |
| D7 | 新建手机相册 | **做**：目标选择器内命名即建（主路径）+ 相册列表页「+」新建 0 张占位相册（本地待落地记录），目录固定 `Pictures/<名称>` |
| D8 | 低版本策略 | 不为旧 API 养已退役代码：26–29 的 `RecoverableSecurityException` 删除链路（v0.7.0 刚删）**不加回**，低版本按 §7 门控矩阵隐藏对应入口 |
| D9 | 域隔离 | 手机相册域与桌面镜像域（照片/相册）**完全分离**：不混排、不互通、不进 Room；`Pictures/YandeGallery/` 历史下载文件夹被扫到就当普通手机相册展示，零特殊处理 |

### 0.3 现状锚点（实施前以代码为准）

- 底导两 tab + 路由表：`ui/AppNav.kt`（`Routes` 常量、`bottomTabs`、`MiuiNavBar`、`AppScaffold`）；多选态底栏替换逻辑在 `AppScaffold.bottomBar`。
- 「加入相册」现有入口：多选底栏 `ui/common/SelectionBars.kt`（`onAddToGallery`）、大图页更多菜单 `ui/viewer/ViewerActionBar.kt`；选择弹窗 `ui/common/GalleryPickerDialog.kt`（列 Room 镜像相册、`excludeIds` 防自指）。
- 桌面域写接口：`WriteRepository`（乐观改镜像 → 请求 → 失败回滚，404 视为成功，写成功后冗余对账）已含相册成员增/删——「复制/移动到桌面相册」的全部原料。
- 镜像层：`ImageMirrorStore.ensure(serverId, imageId, tier)` 是唯一落盘入口；「下载原图」= `ensure(ORIGINAL)`（下载后删同图 HQ 副本，D7 语义：已有原图始终保留）。
- MediaStore 写链路 v0.7.0 已整体退役：manifest 无任何媒体读写权限（仅 INTERNET/CAMERA/前台服务/通知/网络状态）；`MediaStoreGateway`、`downloads` 表、`WRITE_EXTERNAL_STORAGE`、legacy 权限门卫均已删除。本文的 MediaStore 能力是**全新读写面**，不是复活旧代码。
- 网格/大图复用件：`PinchStepState`（3/4/5 列捏合）、`SelectionState`/`SelectionActions`（多选）、`ZoomableImage` + Pager（大图）、`MiuiDialog`/`MiuiMoreMenu`/`MiuiTopBars`（壳）、`RetryableAsyncImage`（加载重试）。
- 工程：minSdk 26 / targetSdk 35 / versionCode 8（v0.7.0）；测试为纯 JVM Robolectric，依赖注入缝在 `di/AppGraph.kt`。

### 1.2 本轮不做（排除项）

- **手机→桌面上传**：桌面端无上传 API，属独立立项（C 档），本文所有选择器不出现该方向。
- **app 内回收站**：删除即永久删（D3），不做已删列表。
- **手机域搜索、置顶/收纳偏好、排序面板、日期分组头**：v1 手机相册列表固定排序（§4.3）、网格固定时间倒序平铺；桌面域照片 tab 的这些能力不搬。
- **app 内视频播放器**：一律外抛系统播放器。
- **26–29 删除、26–28 复制、26–29 移动**：见 §7 门控矩阵，入口直接隐藏。
- **手机域图片编辑（裁剪/旋转）、设为壁纸**。
- **多选跨域混选**：多选发生在单一网格内，天然单域，不做跨域聚合选择。

## 2. 信息架构与导航

### 2.1 底部导航

三 tab：**照片 / 相册 / 手机相册**（`Icons.*.PhoneAndroid` 线框/实心对，testTag `tab_device_albums`）。`AppScaffold.showBottomBar` 与多选态底栏替换逻辑扩到新路由。手机相册 tab **不依赖服务器配对**：未配对/离线时功能完整——tab 内全部动作都是纯本机操作，无任何跨域动作（跨域只有桌面域侧的「复制到手机相册」一个方向，其离线置灰见 §6.1）。

### 2.2 路由

```
Routes.DeviceAlbums       = "device_albums"                    // 相册列表页（tab 根）
Routes.DeviceAlbumDetail  = "device_albums/{bucketKey}"        // 相册网格页
Routes.DeviceViewer       = "device_viewer/{mediaId}?bucketKey={bucketKey}"  // 本机大图页
```

`bucketKey` 为字符串，承载三种上下文：`b<BUCKET_ID>`（真实 bucket）/ `all`（全部照片聚合）/ `p<URL编码名称>`（待落地相册，无内容，进入即空态）。

### 2.3 页面结构

- **相册列表页**：大标题「手机相册」+ 右上「+」（新建；与复制同门 29+ 显示——26–28 无法复制文件进去，占位相册永远无法落地；见 §5.5）。置顶一张**全部照片**聚合卡（全库图片+视频总数、最新一张做封面），下方 bucket 卡网格（复用相册 tab 的 Adaptive 卡片视觉：封面 12dp 圆角 + 名称 + 张数）。
- **相册网格页**：居中双行顶栏（相册名 + 张数）；平铺网格复用照片 tab 网格视觉（3dp 缝 + 3dp 圆角），默认 4 列、捏合 3/4/5（`PinchStepState` 复用，档位不持久化——手机域无偏好记录，YAGNI）；长按进多选。
- **本机大图页**（`DeviceViewer`，独立于桌面域 Viewer 的轻量件）：HorizontalPager + `ZoomableImage` 复用，单击沉浸、下滑关闭沿用；视频显示海报帧 + 中央播放键，点击 `ACTION_VIEW` 外抛系统播放器。底部操作栏：**分享 / 删除 / 复制到 / 移动到 / 详情**。详情面板：文件名、完整路径、大小、分辨率、拍摄/修改时间、（视频）时长——纯本机元数据，无标签、无所属相册区块（那是桌面域概念）。

### 2.4 多选（手机域）

长按进入，视觉与桌面域一致（空心圈/蓝底白勾/角标/全选）。底部动作栏：**分享 / 删除 / 复制到 / 移动到**。动作按 §7 门控隐藏，跨域动作不存在（手机图片没有「加入桌面相册」）。

## 3. 权限模型

| API | 清单声明 | 说明 |
|---|---|---|
| 33+ | `READ_MEDIA_IMAGES` + `READ_MEDIA_VIDEO` | 运行时申请，一次弹双权限 |
| 34+ | 追加 `READ_MEDIA_VISUAL_USER_SELECTED` | 「部分照片」模式：tab 内顶部常驻横幅「仅可访问部分照片」+「管理」按钮重拉系统选择器 |
| 26–32 | `READ_EXTERNAL_STORAGE`（`maxSdkVersion="32"`） | 覆盖图片+视频读取 |

- 首次进入手机相册 tab 才申请（不在启动时打扰）；未授权时 tab 内显示引导页（说明文案 + 「授权」按钮），永久拒绝后按钮变「去设置」跳 app 详情页。
- **不声明任何 WRITE 权限**：29+ 写自有新文件免权限，改/删他人文件走 30+ 系统弹窗（`createWriteRequest`/`createDeleteRequest`），与 v0.7.0「无媒体写权限」的架构决策一致。

## 4. 数据层（`data/device/`，不进 Room）

### 4.1 原则

手机域是**纯 MediaStore 实时读**：系统媒体库本身就是数据库，再镜像一份进 Room 只会制造同步问题。新包 `data/device/` 提供查询与写操作网关，接口缝 + Android 实现（照 `AppGraph` 注入惯例，测试给 fake）。代码命名前缀 **`Device*`**（`local` 一词已被镜像域占用——`localFile`/本地镜像）。

### 4.2 模型（内存态，非 Entity）

```kotlin
DeviceAlbum(bucketKey, name, relativePath, count, coverUri, isPending)  // isPending=待落地空相册
DeviceMedia(mediaId, uri, isVideo, width, height, sizeBytes, takenAt, durationMs?)
```

### 4.3 查询

- **相册列表**：按 `BUCKET_ID` 聚合图片+视频（`MediaStore.Files` 过滤 `MEDIA_TYPE IN (IMAGE, VIDEO)`），排序：**相机（DCIM/Camera）→ 截图（Screenshots）置顶，其余按张数降序**；合并 DataStore 待落地相册（0 张，排最后）。
- **网格**：单 bucket（或全库）按 `DATE_TAKEN ?: DATE_MODIFIED` 倒序，Paging 3 自定义 `PagingSource` 分页加载。
- **刷新**：`ContentObserver` 监听 images/videos URI，变更即 invalidate 分页流 + 相册列表流——app 内外的增删改（包括本 app 自己的复制/删除落地）统一走这一条刷新链，操作代码不手工刷列表。

### 4.4 图片加载

`content://` URI 走 Coil 默认数据源（无需鉴权 OkHttp、无需自定义 Fetcher）；视频海报帧加 `coil-video` 的 `VideoFrameDecoder`。与镜像域两只 loader 互不干扰。

## 5. 手机域操作

### 5.1 删除（30+）

单张与批量统一 `MediaStore.createDeleteRequest(uris)` → 系统确认弹窗（批量一次弹窗）→ 永久删除。结果回调后无需手工刷新（ContentObserver 兜底）。26–29 隐藏删除入口（D8：29 的 `RecoverableSecurityException` 逐张链路是 v0.7.0 刚删的代码，你的真机全 30+，不为零台设备养尸体）。

### 5.2 分享

MediaStore `content://` URI 直接 `ACTION_SEND` / `ACTION_SEND_MULTIPLE`（系统内容提供者，无需 FileProvider）。视频同权。

### 5.3 复制到手机相册（29+，含手机→手机与桌面→手机两个来源）

- **手机→手机**：读源文件字节 → `MediaStore.insert`（`RELATIVE_PATH` = 目标相册相对路径，`IS_PENDING=1` 挂起写入 → 落完置 0）→ 新行归本 app 所有。同名冲突交给 MediaStore 自动改名（`xxx (1).jpg`），不做覆盖询问。
- **桌面→手机**：见 §6.1。
- 复制目标相册限 **`DCIM/` 与 `Pictures/` 下的 bucket**（系统对三方图片写入的目录约束），选择器只列合法目标 + 「新建相册」。

### 5.4 移动到手机相册（30+）

`MediaStore.createWriteRequest(uris)` → 系统授权弹窗（批量一次）→ 逐条更新 `RELATIVE_PATH` 为目标路径（系统物理移动文件）。源相册消失、目标相册出现，符合移动直觉。**手机域移动在「全部照片」聚合里同样可用**（每个文件有唯一属地，不需要「当前相册」上下文）——这是与桌面域移动（§6.2 仅相册详情）的刻意不对称。

### 5.5 新建手机相册（D7）

MediaStore 的硬约束：bucket 由文件派生，系统无「空相册」概念。两条路：

1. **选择器内新建**（主路径）：复制/移动的目标选择器带「新建相册」项，命名后作为本次操作目标，文件落进去即真实存在。目录固定 `Pictures/<名称>`。
2. **列表页「+」新建**：写一条**待落地相册**记录（DataStore：名称 + 相对路径），列表合并显示为 0 张相册；首个文件复制进来后真实 bucket 自然接管（按相对路径去重，删占位记录）；对 0 张占位相册提供「删除」（仅删记录）。

名称合法性：过滤路径分隔符等非法字符，重名（对既有 bucket 或待落地记录）拒绝。

### 5.6 详情

底部弹层（复用 MIUI 卡片视觉）：文件名 / 相对路径 / 大小 / 分辨率 / 拍摄时间（缺省用修改时间标注）/ 视频时长。只读，v1 不做重命名。

## 6. 桌面域「复制到 / 移动到」改造

### 6.1 复制到（吸收替换「加入相册」）

多选栏与大图页的「加入相册」入口改名「复制到」，选择器升级为**两节**（复用并扩展 `GalleryPickerDialog` → `CopyTargetPicker`）：

- **桌面相册节**：行为完全等于现「加入相册」——`WriteRepository` 成员添加（乐观镜像 + 失败回滚 + 事后对账），桌面端同步可见。`excludeIds` 防自指语义保留。
- **手机相册节**（29+ 显示；离线置灰整节）：目标为合法手机相册（§5.3 约束）或新建。执行 = 对每张图 `ImageMirrorStore.ensure(serverId, imageId, ORIGINAL)` 取得原图 → 字节拷贝 `MediaStore.insert` 落目标相册。**副作用明示**：这就是「下载原图 + 导出」的组合——镜像层该图升为原图档（同 D7 语义：已有原图始终保留、删同图 HQ 副本），不额外发明「绕过镜像的临时下载」路径。
- 批量走 WorkManager（唯一工作名去重，串行逐张——复用 ensure 的磁盘满/404 分流），**计数进度通知**「正在复制到手机相册 x/y」，失败项汇总提示（404 跳过、磁盘满中止重试，沿用镜像同步的错误分类文案）。

### 6.2 移动到（仅桌面相册）

- 入口**仅在相册详情上下文**出现（多选栏 + 相册上下文的大图页）——时间轴里的图没有「当前相册」可移出；「移出当前相册」保留为独立动作不变。
- 选择器**只有桌面相册节**（复用同一选择器组件的单节配置），**永不出现手机相册**（D5：禁止以移动之名删桌面图）。
- 执行：目标相册成员添加成功 → 当前相册成员移除；移除失败则补偿回滚（撤销刚才的添加），两步全程乐观镜像 + 事后冗余对账兜底（沿用 `WriteRepository` 惯例）。

### 6.3 镜像层规则重申

除 §6.1 手机相册节这条**显式**导出出口外，任何链路不写系统相册：「下载原图」仍只落私有镜像、对账级联不碰 MediaStore、分享仍走 FileProvider。

## 7. 版本门控矩阵

| API 区间 | 手机相册 tab | 复制（→手机相册，两来源同门） | 删除（手机域） | 移动（手机域） |
|---|---|---|---|---|
| 26–28 | 浏览 + 分享 + 详情 | ✗（insert 需 WRITE 权限，已架构性移除） | ✗ | ✗ |
| 29 | 同上 | ✓（`IS_PENDING` 挂起写入，自有新文件免权限） | ✗ | ✗ |
| 30+ | 同上 | ✓ | ✓ `createDeleteRequest` | ✓ `createWriteRequest` |

不可用 = 入口隐藏（不是置灰），杜绝「点了弹失败」。桌面域复制/移动到桌面相册不受此矩阵影响（纯网络写，26+ 全可用）。

## 8. 测试策略

- **接口缝 + fake**：`DeviceMediaGateway`（查询/insert/写请求封装）走 AppGraph 注入，Robolectric 测 VM/UI 全逻辑（相册聚合、门控矩阵、选择器两节显隐、待落地相册生命周期、移动回滚补偿）。
- **系统弹窗、真实 MediaStore 行为、系统播放器唤起、34+ 部分授权**无法无头覆盖——列入 `docs/superpowers/plans/2026-07-05-M3实机联调计划.md` 新增 §L 用例（MuMu API32 测 29–32 路径 + 红魔 API34 测部分授权与写弹窗）。
- 桌面域改造回归：既有「加入相册」相关用例改名迁移，移动/回滚补新用例；`npm run test` 桌面 gate 照常（虽零桌面改动，防漂移）。

## 9. 非目标与未来方向

- **手机→桌面上传**（C 档）：需桌面新增上传 API + 去重管线，独立立项；本设计的选择器结构（两节目标）为它预留了形态。
- **手机域偏好**（排序/置顶/密度记忆）、**回收站**、**编辑**：按需求追加。
- 26–29 写操作补齐：如未来有真实设备诉求再评估，优先级极低。

## 10. 领域术语（同步落根级 `CONTEXT.md`）

| 术语 | 含义 | 避免 |
|---|---|---|
| 相册 | 桌面图库的相册（Room 镜像的 gallery） | 图集 |
| 手机相册 | 本机 MediaStore 的一个 bucket（文件夹派生） | 本地相册、系统相册 |
| 照片 | 桌面域时间轴 tab（全部桌面图片） | — |
| 本机媒体 | MediaStore 里的一张图片或一条视频 | 本地图片 |
| 图片镜像 | app 私有目录里的桌面图片副本（HQ/原图档） | 缓存、下载 |
| 复制到 | 源不动、目标出现：桌面相册=成员添加；手机相册=文件复制落盘 | 加入相册（旧称）、导出、保存到相册 |
| 移动到 | 源消失、目标出现：桌面域=成员转移（仅桌面目标）；手机域=文件物理移动 | — |
| 待落地相册 | 已命名但尚无文件的手机相册占位（本地记录） | 空相册 |
| 全部照片 | 手机相册 tab 的全库聚合视图 | — |
