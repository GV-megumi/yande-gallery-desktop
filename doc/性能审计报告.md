# 性能审计报告

> 审计日期: 2026-03-06
> 审计范围: 磁盘 I/O、并发调度、内存占用、数据库、React 渲染

---

## 一、严重程度说明

| 级别 | 含义 |
|------|------|
| **CRITICAL** | 会导致 UI 冻结、数据损坏或系统不可用 |
| **HIGH** | 显著影响性能，用户可感知的延迟或资源浪费 |
| **MEDIUM** | 有优化空间，影响中等 |
| **LOW** | 小幅优化，锦上添花 |

---

## 二、磁盘 I/O 问题

### P1 [CRITICAL] 下载管理器使用同步 fs 操作阻塞事件循环

**文件**: `src/main/services/downloadManager.ts`
**行号**: 179, 372-373, 433-435, 450-452

```typescript
// 阻塞整个 Node.js 事件循环
if (fs.existsSync(targetPath)) {
  fs.unlinkSync(targetPath);
}
fs.mkdirSync(targetDir, { recursive: true });
```

**影响**: 下载期间 UI 卡顿，所有 IPC 通信被阻塞。
**修复**: 替换为 `fs/promises` 的 `access()`, `unlink()`, `mkdir()`。

---

### P2 [HIGH] 批量标签写入缺少事务

**文件**: `src/main/services/booruService.ts` (行 839-857)
**文件**: `src/main/services/imageService.ts` (行 292-317)

```typescript
// booruService: 每个标签一次 DB 操作
for (const tag of tags) {
  await run(db, `INSERT OR REPLACE INTO booru_tags...`);
}

// imageService: 每个标签 3-4 次 DB 操作
for (const tagName of tagNames) {
  let tag = await get(...);     // 查询
  if (!tag) await run(...);     // 插入
  await run(...);               // 关联
}
```

**影响**: 1000 个标签 = 1000+ 次独立数据库操作，比事务包裹慢 ~100x。
**修复**: 用 `BEGIN TRANSACTION ... COMMIT` 包裹批量操作。

---

### P3 [HIGH] 图集扫描冗余文件系统操作

**文件**: `src/main/services/galleryService.ts` (行 371-447)

扫描流程对同一目录执行 3 次重复读取：
1. `scanSubfoldersAndCreateGalleries()` → readdir 获取子文件夹
2. `checkFolderHasImages()` → 对每个子文件夹再次 readdir
3. `scanAndImportFolder()` → 第三次 readdir + 逐文件 `fs.stat()`

**影响**: 100 个子文件夹 = 300+ readdir 调用 + 100+ DB 查询。
**修复**: 单次扫描收集所有信息，传递给后续步骤。

---

### P4 [MEDIUM] 下载进度频繁写数据库

**文件**: `src/main/services/downloadManager.ts` (行 396-411)

虽然已节流到 500ms 一次，但每次仍写数据库。
**修复**: 进度仅更新内存，完成/失败时才写库；或增大节流间隔到 2-3 秒。

---

### P5 [MEDIUM] 文件导入无分批处理

**文件**: `src/main/services/imageService.ts` (行 505-571)

`scanAndImportFolder()` 逐个处理所有文件，无分批。10000 个文件全部顺序处理。
**修复**: 分批 100-500 个文件处理，每批用事务包裹。

---

### P6 [MEDIUM] 逐文件存在性检查（N+1 查询）

**文件**: `src/main/services/imageService.ts` (行 519-525)

```typescript
for (const file of files) {
  const existing = await get<{ id: number }>(db, 'SELECT id FROM images WHERE filepath = ?', [file]);
}
```

**影响**: 1000 个文件 = 1000 次 SELECT。
**修复**: 批量 `SELECT filepath FROM images WHERE filepath IN (?, ?, ...)` 一次查询。

## 三、并发调度问题

### P7 [HIGH] downloadManager processQueue 竞态条件

**文件**: `src/main/services/downloadManager.ts` (行 308-342)

```typescript
if (this.activeDownloads.size >= this.maxConcurrent) return;
if (this.isProcessing) return;
this.isProcessing = true; // flag 在 size 检查之后设置
```

多次快速调用 `processQueue()` 可能导致下载数超过 `maxConcurrent` 限制。
**修复**: 引入 `p-queue` 或自实现信号量，替代简单 flag。

---

### P8 [HIGH] 批量下载 200ms 轮询循环

**文件**: `src/main/services/bulkDownloadService.ts` (行 1116-1170)

主下载循环每 200ms 轮询一次，浪费 CPU。
**修复**: 改为事件驱动模型 — 下载完成/失败时触发下一个任务。

---

### P9 [HIGH] IPC 处理器中的阻塞式顺序数据库写入

**文件**: `src/main/ipc/handlers.ts` (行 388-584)

```typescript
for (const post of posts) {
  const dbId = await booruService.saveBooruPost({...}); // 逐个保存
  savedPostIds.push(dbId);
}
```

20 个 post 顺序保存，阻塞 IPC 响应。
**修复**: `Promise.all()` 并发保存，或用事务批量插入。

---

### P10 [HIGH] 批量下载并发控制松散

**文件**: `src/main/services/bulkDownloadService.ts` (行 1079-1182)

用简单计数器 `activeCount++` 控制并发，无互斥锁，存在 TOCTOU 竞态。
**修复**: 使用 `p-queue` 或 `p-limit` 库。

---

### P11 [MEDIUM] 下载流无背压控制

**文件**: `src/main/services/downloadManager.ts` (行 394-412)

响应流的 `data` 事件没有检查写入流的 `isPaused()` 状态。大文件可能导致内存暴涨。
**修复**: 使用 `pipeline()` 或手动 `pause()`/`resume()` 处理背压。

---

### P12 [MEDIUM] Moebooru 客户端无请求限流

**文件**: `src/main/services/moebooruClient.ts`

无速率限制、无请求队列。批量操作可能触发服务端限流。
**修复**: 添加令牌桶算法或使用 `bottleneck` 库。

---

### P13 [MEDIUM] 标签查询 N+1 问题

**文件**: `src/main/services/moebooruClient.ts` (行 336-403)

对每个未知标签单独发 API 请求。50 个未知标签 = 50 次网络请求。
**修复**: 批量查询或合并请求。

---

### P14 [MEDIUM] Fire-and-forget 异步操作无追踪

**文件**: `src/main/ipc/handlers.ts` (行 471-501)
**文件**: `src/renderer/pages/BooruPostDetailsPage.tsx` (行 186-211)

标签获取和图片预加载使用 `.forEach(async ...)` 或立即执行异步函数，无取消机制。
用户快速切换时，旧请求仍在执行。
**修复**: 使用 `AbortController` 或追踪 pending 请求。

---

## 四、内存占用问题

### P15 [HIGH] 缩略图状态无界增长

**文件**: `src/renderer/components/ImageGrid.tsx` (行 59, 96)
**文件**: `src/renderer/pages/GalleryPage.tsx` (行 631)

```typescript
const [thumbnailPaths, setThumbnailPaths] = useState<Record<number, string | null>>({});
// 每批加载只追加，从不清理
setThumbnailPaths((prev) => ({ ...prev, ...thumbnails }));
```

2000 张图片 = 2000 条记录永驻内存，切换标签页后仍不释放。
**修复**: 切换标签页/图集时重置 `thumbnailPaths`；或引入 LRU 缓存。

---

### P16 [HIGH] Image.PreviewGroup 包含 2000+ 张图片

**文件**: `src/renderer/components/ImageGrid.tsx` (行 318-359)

Ant Design `Image.PreviewGroup` 会为所有包裹的图片创建隐藏预览上下文，2000 张图片导致巨大内存开销。
**修复**: 仅对当前可见的图片使用 PreviewGroup，或自定义预览逻辑。

---

### P17 [HIGH] 图片预加载无并发控制

**文件**: `src/renderer/pages/BooruPostDetailsPage.tsx` (行 157-222)

前后各 3 张图片同时下载缓存，用户快速翻页时不取消旧请求，请求不断累积。
**修复**: 限制预加载并发数（如 2 个），翻页时取消旧的预加载任务。

---

### P18 [HIGH] 缓存服务驱逐策略粗暴

**文件**: `src/main/services/imageCacheService.ts` (行 121-132)

缓存超限时删除最老的一半，且无单文件大小校验。单个 1GB 文件可突破 500MB 限制。
**修复**: 下载前校验文件大小；改用渐进式 LRU 驱逐而非一次删一半。

---

### P19 [MEDIUM] 多页面各自重复排序 posts 数组

**文件**: `src/renderer/pages/BooruPage.tsx` (行 354-356)
**文件**: `src/renderer/components/BooruGridLayout.tsx` (行 39-41)

同一数组在页面和组件中分别排序两次。
**修复**: 只在页面层排序，组件接收已排序数据。

---

### P20 [MEDIUM] rating 过滤未 memoize

**文件**: `src/renderer/pages/BooruPage.tsx` (行 485)
**文件**: `src/renderer/pages/BooruFavoritesPage.tsx`、`BooruTagSearchPage.tsx`

```typescript
posts.filter(post => ratingFilter === 'all' || post.rating === ratingFilter)
```

每次渲染都创建新数组。
**修复**: 用 `useMemo` 包裹，依赖 `[posts, ratingFilter]`。

### P21 [MEDIUM] 大量内联 style 对象每次渲染重建

**文件**: `src/renderer/pages/GalleryPage.tsx` (行 820-831)
**文件**: `src/renderer/pages/BooruPostDetailsPage.tsx` (行 341-348, 428-435)
**文件**: `src/renderer/components/BooruImageCard.tsx` (行 130-137)

固定样式应提取为常量或 `useMemo`。
**修复**: 静态样式提取为模块级常量。

### P22 [MEDIUM] GalleryPage 级联 setState

**文件**: `src/renderer/pages/GalleryPage.tsx` (行 517-550)

切换 tab 时 5-6 个 `setState` 调用触发多次重渲染。

### P23 [MEDIUM] setInterval 在页面隐藏时仍运行

**文件**: `src/renderer/pages/BooruSettingsPage.tsx` (行 53-54) — 每 5 秒刷新缓存统计

## 五、数据库问题

### P24 [HIGH] 缺少关键索引

**文件**: `src/main/services/database.ts`

| 缺失索引 | 影响查询 |
|----------|---------|
| `booru_posts(siteId, createdAt DESC)` | 分页浏览、排序 |
| `booru_favorites(postId)` | 收藏状态查询 |
| `bulk_download_records(sessionId, status)` | 批量下载进度统计 |
| `booru_favorite_tags(siteId, tagName)` | 收藏标签查询 |
| `booru_posts(localImageId)` | JOIN 查询 |

**修复**: 在数据库初始化中添加 `CREATE INDEX IF NOT EXISTS`。

---

### P25 [MEDIUM] GROUP_CONCAT + LOWER(LIKE) 查询无法命中索引

**文件**: `src/main/services/imageService.ts` (行 35-49)

```sql
SELECT i.*, GROUP_CONCAT(t.name) as tags
FROM images i
LEFT JOIN image_tags it ON i.id = it.imageId
LEFT JOIN tags t ON it.tagId = t.id
WHERE LOWER(i.filename) LIKE ? OR LOWER(t.name) LIKE ?
GROUP BY i.id
```

`LOWER()` 函数导致索引失效，三表 JOIN + GROUP BY 开销大。
**修复**: 使用 `COLLATE NOCASE` 建索引，或存储小写副本字段。

---

### P26 [MEDIUM] 唯一名生成循环查询

**文件**: `src/main/services/galleryService.ts` (行 497-515)

```typescript
while (true) {
  const existing = await get(db, 'SELECT id FROM galleries WHERE name = ?', [name]);
  if (!existing) return name;
  name = `${baseName} (${suffix})`;
  suffix++;
}
```

**修复**: `SELECT name FROM galleries WHERE name LIKE ?` 一次取出所有相关名称。

## 七、总结统计

| 级别 | 数量 |
|------|------|
| CRITICAL | 1 |
| HIGH | 10 |
| MEDIUM | 16 |
| LOW | 3 |

**最影响用户体验的 Top 5:**
1. 同步 fs 阻塞事件循环 (P1) — UI 冻结
2. 批量标签无事务 (P2) — 操作缓慢
3. 缩略图状态无界增长 (P15) — 内存泄漏
4. Image.PreviewGroup 2000+ 图片 (P16) — 内存暴涨
5. IPC 顺序写数据库 (P9) — 响应延迟
