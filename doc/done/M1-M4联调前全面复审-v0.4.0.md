## 三、联调前必修 · Critical（2）

### C1 · WorkManager 前台服务未声明 `foregroundServiceType` → API 29+ 真机首次下载即崩溃 ✅

`android/app/src/main/AndroidManifest.xml:12` · M4 · 两个 finder 独立命中（AndroidManifest:12 与 :13），同一缺陷。

> **处置：联调前必修。** 一行清单即可修复，不修则 J.1/J.5/J.7（下载 / 分享 / 通知）在几乎所有真机上第一步就崩，联调无法推进。

- **问题**：清单声明了 `FOREGROUND_SERVICE_DATA_SYNC` **权限**（AndroidManifest 12–13），但从未用 `tools:node="merge"` 给 WorkManager 的 `androidx.work.impl.foreground.SystemForegroundService` 补 `android:foregroundServiceType` **属性**；`DownloadNotifier.foregroundInfo`（`DownloadNotifier.kt:45`）在 SDK≥29 返回带 `FOREGROUND_SERVICE_TYPE_DATA_SYNC` 的 `ForegroundInfo`。Android 自 API 29 起强制 `startForeground` 的 type 必须是清单声明 type 的子集——清单 type=0 时，29–33 抛 `IllegalArgumentException`，34+（本 app targetSdk 35）抛 `MissingForegroundServiceTypeException`。异常抛在 `SystemForegroundService.onStartCommand`（主线程、service 组件内），`DownloadWorker.kt:68-71` 的 `runCatching{ setForeground() }` 只包住「发 intent 那步」，**根本捕不到**；WorkManager 2.11.2 自身 `Api29Impl` 无 catch、`Api31Impl` 只 catch `ForegroundServiceStartNotAllowedException`+`SecurityException`，两处都接不住。minSdk 26，仅 26–28 老机走无 type 的两参 `startForeground` 幸免。
- **控制器亲验 ✅**：`android/app/build/intermediates/merged_manifest/debug/.../AndroidManifest.xml:94` 存在 `SystemForegroundService` 元素，但全文件 `grep foregroundServiceType` **零命中** → 合并产物确实缺该属性。`DownloadNotifier.kt:43-45`、`AndroidManifest.xml:12-13`、`build.gradle.kts:15-16`（minSdk 26 / targetSdk 35）逐一核对属实。
- **建议**：`<application>` 内加 `<service android:name="androidx.work.impl.foreground.SystemForegroundService" android:foregroundServiceType="dataSync" tools:node="merge" />`（清单根已声明 `xmlns:tools`，WorkManager 长时 worker 官方范式）。并修正 `DownloadWorker.kt:20-22` 关于「`runCatching` 可优雅降级」的注释——该异常在 service 侧抛出，worker 侧 catch 不可达。

### C2 · 批删本地镜像 `deleteByIds` 未分块 → API<31 设备 ≥1000 选中触发 `too many SQL variables` 崩溃 ✅

`android/app/src/main/java/com/bluskysoftware/yandegallery/domain/write/WriteRepository.kt:126` · M4 · 与 major「批删本地镜像 deleteByIds 未按 900 分块」为同一缺陷。

> **处置：联调前必修。** 快滚+全选轻松超 1000；崩溃沿 suspend 链冒泡到 Screen 的 `scope.launch`（无 try/catch）→ 未捕获协程异常 → 应用崩溃。

- **问题**：M4-T14/D12B 的 900 分块只做在了 batch **HTTP** 调用上；乐观本地删除 `db.imageDao().deleteByIds(unique)`（`WriteRepository.kt:126`）把整个去重后 id 列表一次性传入 `DELETE FROM images WHERE id IN (:ids)`（`ImageDao.kt:38`），每 id 一个绑定变量。Android 12 以下（API 26–30，本 app minSdk 26）系统 SQLite < 3.32，`SQLITE_MAX_VARIABLE_NUMBER` 默认 999——≥1000 个 id 在 prepare 阶段抛 `too many SQL variables`。
- **控制器亲验 ✅**：`WriteRepository.kt:126` 确为未分块整表传入，`:128` 的 `chunked(BATCH_CHUNK)` 只作用于其后的 HTTP 循环；而 `RoomMirrorStore.kt:80-81` 对**完全相同的 DAO 方法**在对账路径特意 `ids.chunked(DELETE_CHUNK).forEach{ deleteByIds(it) }`——同仓库已知要分块，写路径漏了。同型隐患：`removeFromGallery` 的 `deleteGalleryLinks`（:117）、`addToGallery` 回滚（:112）同样整表传入。现有 901 边界测试（`WriteRepositoryTest`）在 Robolectric（新版 SQLite 限额 32766）下**测不出**。
- **建议**：本地删除同样按 900 分块并可包进 `db.withTransaction`：`unique.chunked(BATCH_CHUNK).forEach{ db.imageDao().deleteByIds(it) }`；`removeFromGallery`/`addToGallery` 回滚的 `deleteGalleryLinks` 一并分块。补 ≥1000 id 回归测试（至少断言 DAO 收到多次 ≤900 的调用形状）。

## 四、Major · 数据完整性与竞态（分簇）

以下 major 呈**三条系统性根因**，分簇陈列。逐条的 file:line 与验证均已附。

### 簇 A · 乐观写回滚只恢复「直接行」，丢失 CASCADE/pre-existing 关联（自愈前提不成立）✅

**共性根因**：写操作失败/回滚时只 `upsert` 被直接删的那一行，却遗漏了 Room `ON DELETE CASCADE` 连带删掉、或本已存在的 tag/图集成员关联；且代码注释一律假设「靠下一轮增量同步对账收敛」——但增量同步以桌面端 `changeSeq` 变化为准，**写失败意味着桌面端未变更、无 `changeSeq` bump**，故这些本地丢失的关联行**增量同步永不自愈**，只有 `dataVersion` 全量重建才恢复。控制器已亲读 `WriteRepository.kt` 全文 + `ImageDao` 冲突策略核实。

**deleteImage/batchDeleteImages 回滚只恢复 image 行，丢失 CASCADE 连带删掉的 tags/图集成员，且增量同步永不自愈**  
`android/app/src/main/java/com/bluskysoftware/yandegallery/domain/write/WriteRepository.kt:52` · M3 · 验证：3 视角均未驳（code/intent/impact）
- **问题**：deleteImage 的乐观删除 `db.imageDao().deleteByIds(listOf(imageId))`（batchDeleteImages 同理，line 126）是一条普通 `DELETE FROM images WHERE id IN(...)`（ImageDao line 38），Room 默认开启 `PRAGMA foreign_keys=ON`，gallery_images/image_tags 对 images 建的是 ForeignKey.CASCADE（Entities.kt line 36/55），故删 image 行会连带删掉该图的全部标签链接与图集成员链接。但 rollback 只做 `upsertAll(listOf(snapshot))`（line 52；batchDelete 的 line 139/147 亦然），snapshot 是纯 ImageEntity，不含任何子表链接——回滚后 image 行回来了，image_tags/gallery_images 却是空的。关键：当服务端删除因瞬时网络故障失败（LAN app 桌面短暂不可达 → ApiException 非 404 → 走 rollback 分支），桌面端该图根本没被删、changeSeq 保持不变；而增量同步是 `WHERE changeSeq > cursor`（syncService.ts line 117，客户端 cursor 已越过该图旧 changeSeq），对账步骤又确认该图仍在 remoteIds 里不删本地，replaceTags/replaceGalleries 只换『列表』不重建成员链接。三条恢复路径全部不覆盖该图 → 该图在镜像里永久呈现『无任何标签、不属于任何图集』，只有换服/dataVersion bump 触发全量重建才修复。
- **证据**：deleteByIds=`DELETE FROM images WHERE id IN(:ids)`(ImageDao.kt:38) 触发 CASCADE 删 image_tags/gallery_images(Entities.kt:36,55)；rollback=`upsertAll(listOf(snapshot))`(WriteRepository.kt:52) 只回 ImageEntity 无子表；增量同步 `WHERE changeSeq > ?`(syncService.ts:117) + 失败的桌面删除不 bump changeSeq → 该图不再被重发。这与已知清单里 deleteGallery 那条不同：此处是最高频的『删照片』路径，且丢的是被删图自身的 tags 与图集归属两类数据。
- **建议**：删除类回滚必须同时快照并恢复子表链接：deleteImage/batchDeleteImages 在乐观删前 `tagNamesOf/galleryIdsOf`（或直接查 image_tags/gallery_images）快照，rollback 时把 ImageEntity 与其 image_tags/gallery_images 链接一并 upsert 回；或改为把删除包进 `withTransaction` 并在失败时整体不提交（乐观删本就无需先落库）。

**批删多块场景：前置块的 failedIds 在后续块抛异常时永不恢复，镜像行与 downloads 映射双重丢失**  
`android/app/src/main/java/com/bluskysoftware/yandegallery/domain/write/WriteRepository.kt:138` · M4 · 验证：3 视角均未驳（code/intent/impact）
- **问题**：batchDeleteImages 先乐观全删（:126），逐块调用后把「真失败」id 累进 failedIds（:133），但这些行的恢复统一推迟到循环结束后的 :147。若块 i 的响应里有 failedIds（服务端删除失败、图片仍在桌面端），而块 j>i 抛异常，catch 分支的回滚集只含 `chunks.drop(index)`（当前块+未发块，:138），随即 return（:141）——块 0..i 的 failedIds 行既不在回滚集、也到不了 :147，且该路径不调 requestSync。后果：这些图片在服务端仍存在但镜像行已删。由于增量同步按 changeSeq 游标只拉「服务端变过」的行（SyncEngine.kt:37-50），而服务端删除失败通常不触碰该行，ids 对账又只删不补（SyncEngine.kt:53-57），这些行永远不会回到镜像（直到桌面端恰好改动该图或全量重建）。紧接着 SelectionActions.batchDelete 的清理循环（SelectionActions.kt:99）按 `byId(id)==null` 判定「确实已删」，会把这些仍在服务端的图片的本服 downloads 行也删掉——它们的 MediaStore 映射被错误销毁，未来真删除时无法级联本机副本。现有测试（WriteRepositoryTest.kt:301『某块失败——回滚该块与未发块』）未覆盖「前置块含 failedIds + 后续块抛异常」的组合。
- **证据**：WriteRepository.kt:133 `failedIds += results.filter { !it.success && it.error != "NOT_FOUND" }...`（只记账不恢复）→ :138 `val rollback = (chunks.drop(index).flatten()).mapNotNull { snapshots[it] }`（不含已发块的 failedIds）→ :141 提前 return，:147 的 failedIds 恢复不可达。SelectionActions.kt:99 `existing.forEach { id -> if (imageDao.byId(id) == null) downloadDao.delete(serverId, id) }` 将未恢复的 failedIds 一并清映射。
- **建议**：catch 分支的回滚集改为 `chunks.drop(index).flatten() + failedIds`，并在该失败路径也调用 requestSync() 兜底；补组合测试：块 0 部分失败 + 块 1 抛 500，断言块 0 失败项镜像行已恢复且 downloads 行保留。

**回滚/取消残留的「下一轮 sync 收敛」前提不成立：失败的 deleteGallery 使图集成员关系永久丢失**  
`android/app/src/main/java/com/bluskysoftware/yandegallery/domain/write/WriteRepository.kt:105` · cross · 验证：3 视角均未驳（code/intent/impact）
- **问题**：多处回滚设计（及台账记债）依赖「镜像靠下一轮同步对账收敛」，但同步协议无法收敛『本地已改、服务端未变』的残留：增量同步按 changeSeq 键集只重拉服务端变更过的行（SyncEngine.kt:37-50）；每图的 tag/gallery 链行仅随该图被重拉时 replaceTagLinks/replaceGalleryLinks 重建（RoomMirrorStore.kt:53-61）；ids 对账只删不补（SyncEngine.kt:53-57）；桌面端 images.updatedAt/changeSeq 只在写操作真正落库时由触发器触碰（galleryService.ts:1765、database.ts:1283-1314）。写操作失败即服务端零变更 → 下一轮 sync 什么也不重放。最确定性的路径：deleteGallery 乐观执行 `deleteById + clearMembership`（:104），一次网络超时/500 后回滚只 `insertOne(old)` 恢复图集行，注释称「成员行靠下一轮 sync 恢复」（:105）——实际 replaceGalleries 只覆盖 galleries 表（RoomMirrorStore.kt:94-95），成员图片在服务端未变更、永不重拉，gallery_images 成员行永久为空：相册卡片显示服务端 imageCount（非零），点进 AlbumDetail 网格却空白，直到桌面端恰好改动那些图或换服/dataVersion 全量重建。同理：deleteImage/batchDelete 失败回滚只 upsert 主行，CASCADE 掉的 tag/gallery 链行永不回来；取消路径（Screen 用 rememberCoroutineScope 起批删协程，确认后立刻切 tab/返回即取消，PhotosScreen.kt:366 / AlbumDetailScreen.kt:243）若服务端未处理，整批镜像行永久消失。台账记录的是『主行级回滚不恢复 CASCADE 链行（sync 收敛）』『取消路径乐观变更滞留至下一轮 sync』——两条债的收敛前提均不成立，实际风险等级与记债时的假设不同。
- **证据**：WriteRepository.kt:104-105 `optimisticApply = { deleteById; clearMembership }` / `rollback = { insertOne(old) }  // 成员行靠下一轮 sync 恢复`；SyncEngine.kt:35-57 增量=cursor 键集拉取 + `stale = local.filter { it !in remoteIds }` 只删；RoomMirrorStore.kt:53-61 链行仅随被拉取的图重建、:94-95 replaceGalleries 不含成员；galleryService.ts:1765 「images.updatedAt 由 gallery_images 触发器触碰」——仅在服务端实际写入时发生。
- **建议**：① 回滚做全快照：deleteGallery 前快照该图集 gallery_images 行、deleteImage/batchDelete 前连同 tag/gallery 链行一起快照，回滚时一并恢复；② 写操作移入 app 级 scope（graph 常驻 scope）执行、UI 只 await，导航离开不再取消写事务本身；③ 若保留取消不回滚语义，需给同步协议加『本地缺失但远端存在』的修复通道（如对账时对 remote-有-local-无 的 id 触发按 id 拉取或强制全量重建）。

**（补验提升，原因会话额度中断未验）** addToGallery / addTags 乐观插入用 `OnConflictStrategy.IGNORE`、回滚却无条件 `delete` ✅  
`WriteRepository.kt:111-112 / 57-64` · cross · 控制器亲验
- **问题**：`insertGalleryLinks`/`insertTagLinks` 均 `OnConflictStrategy.IGNORE`（`ImageDao.kt:50,57`）——对**已存在**的成员/标签是空操作；但 `addToGallery` 回滚（`:112`）无条件 `deleteGalleryLinks(galleryId, imageIds)`、`addTags` 回滚（`:64`）无条件 `deleteTagLinks`。当用户对「部分已在该图集/已有该标签」的集合执行添加且服务端调用失败 → 回滚把**本已存在、非本次新增**的关联一并删除，且同样无法经增量同步自愈。

### 簇 B · 切换服务器中途无「锁定目标服务器」不变量 → 域污染 / 撕裂 / 错图

**共性根因**：多处异步流程（一次 sync、飞行下载、api() 缓存、选择态、镜像/缓存重建）在执行过程中读取「当前激活服务器」，若用户在流程中途切服，前半段属服务器 A、后半段属服务器 B，混装导致镜像污染、内容错域或永久缺失。

## 八、跨问题模式观察（供修复排期参考）

1. **乐观写回滚的「级联/pre-existing 盲区」是系统性的**（簇 A，五处同型：deleteImage / batchDelete / deleteGallery / addToGallery / addTags）。根因是「回滚 = 还原直接行 + 依赖下一轮 sync」这一范式对「桌面端未变更的行」失效。**建议一次性引入统一的 pre-image 快照/精确还原机制**，而非逐点打补丁。
2. **「切服中途无锁定目标服务器」是第二条系统性根因**（簇 B，五处同型）。建议给一次 sync / 一次下载 / 一次批量写在入口处**捕获并贯穿 serverId**，完成落库前校验仍匹配（M4-T9 已在单图下载做了飞行门，扩展到 sync/批写/缓存重建即可）。
3. **两条 critical 与多条 major 都是「JVM/Robolectric 全绿、真机才炸」型**（FGS 类型、SQLite 变量上限、service 组件异常、`startForeground` 校验）。**联调计划应把「首次下载不崩」「≥1000 全选删除不崩」列为 J 节前置断言**，因为宿主机测试结构上测不到。
4. **写权限默认关（簇 C）的失败链条无人接管**：配对测试、连接横幅、写结果三处各自为政，合起来使「配对成功但没开写权限」这一必然初始态体验崩坏。属**产品完成度**而非纯 bug，值得联调前统一收口。
