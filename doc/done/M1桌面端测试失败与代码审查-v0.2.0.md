## 失败范围

失败文件：

- `tests/main/services/galleryRelocateService.caseCanonical.test.ts`

相关服务：

- `src/main/services/galleryRelocateService.ts`
- `src/main/services/config.ts`

错误摘要：

```text
applyRelocateRoot 失败: 配置尚未加载，请先调用 loadConfig()
```

日志中可见 `applyRelocateRoot` 已完成数据库路径改写，并准备广播 `gallery:paths-relocated`，随后在同步数据版本号时失败。

## 初步根因

远端 M1 合入后，`applyRelocateRoot` 在数据库路径重写后增加了 `bumpSyncDataVersion()` 调用，用于在图库路径迁移这类写操作后提升同步数据版本。

`galleryRelocateService.caseCanonical.test.ts` 是大小写归一化相关的隔离测试，当前只 mock 了数据库和 `appEventPublisher`，没有 mock 或加载配置服务。真实的 `bumpSyncDataVersion()` 会读取 `getConfig().sync.dataVersion`，而该测试使用隔离内存数据库时没有调用 `loadConfig()`，因此在路径改写后抛出“配置尚未加载”错误。

已有相近测试 `tests/main/services/galleryRelocateService.apply.test.ts` 已通过 mock `bumpSyncDataVersion` 隔离这个副作用。

## 后续处理建议

- 在 `tests/main/services/galleryRelocateService.caseCanonical.test.ts` 中按现有 apply 测试模式 mock `bumpSyncDataVersion()`。
- 保持生产逻辑不变：真实写操作仍应 bump 同步数据版本。
- 修复后至少运行：
  - `npx vitest run --config vitest.config.ts tests/main/services/galleryRelocateService.caseCanonical.test.ts`
  - `npm run test`
  - `npm run build`

## M1 桌面端代码审查记录

审查范围：`v0.2.0..1a20106`，覆盖同步远端后引入的安卓相册 M1 桌面端改动（预览档、二进制响应补强、sync 五接口、图片/图集写操作、同步版本号、二维码配对）以及本记录文档。该范围不包含 `origin/master` 上后续 M2 安卓骨架提交。独立 reviewer 未发现可确认的 Critical 问题；以下按 Important / Minor 记录。

### Issue 1：增量同步游标依赖 `updatedAt` 墙钟时间，存在同毫秒漏同步风险

- 严重级别：Important
- 涉及位置：
  - `src/main/services/syncService.ts:79-81`
  - `src/main/services/syncService.ts:93-100`
  - `src/main/services/database.ts:1212-1226`
- 问题：`listSyncImages()` 以 `(updatedAt, id)` 做键集分页，触发器在标签/图集归属变化时用 `strftime('%Y-%m-%dT%H:%M:%fZ','now')` 写 `images.updatedAt`。如果客户端已经拿到某个 `(updatedAt, id)` 游标，随后同一毫秒内较小 `id` 的图片被触发器更新到相同 `updatedAt`，下一页谓词 `updatedAt > ? OR (updatedAt = ? AND id > ?)` 会跳过该较小 `id`。
- 影响：移动端可能漏掉标签或图集归属变化。普通写操作不会递增 `dataVersion`，漏项可能一直保留到下一次全量重建。
- 建议：新增单调递增的同步序列字段（如 `syncVersion` / `changeSeq`）并用它做游标；或维护全局变更序列，避免用低精度墙钟时间承担“全序变更日志”的职责。

### Issue 2：合并后测试套件被 `caseCanonical` 隔离测试阻断

- 严重级别：Important
- 涉及位置：
  - `src/main/services/galleryRelocateService.ts:461-466`
  - `tests/main/services/galleryRelocateService.caseCanonical.test.ts:32-43`
- 问题：`applyRelocateRoot` 在真实写入后调用 `bumpSyncDataVersion()`，但 `galleryRelocateService.caseCanonical.test.ts` 只 mock 了数据库和事件发布器，没有 mock `config.js`，也没有加载配置。
- 影响：`npm run test` 当前失败，不能把同步后的 `master` 作为通过测试的发布基线。
- 建议：按 `tests/main/services/galleryRelocateService.apply.test.ts` 的方式 mock `bumpSyncDataVersion()`，让大小写归一化测试继续只验证重定位逻辑。

### Issue 3：`sync.serverId` 首次懒生成存在并发返回非持久值的风险

- 严重级别：Important
- 涉及位置：
  - `src/main/services/config.ts:1237-1318`
  - `src/main/services/config.ts:1338-1348`
  - `src/main/services/syncService.ts:52-60`
  - `tests/main/services/config.sync.test.ts:47-65`
- 问题：`ensureSyncServerId()` 在 `sync.serverId` 为空时执行“读当前值 -> 生成 UUID -> saveConfig -> 返回本次 UUID”。`saveConfig()` 内部会让旧保存等待最新保存结果，但不会让旧调用改用最终落盘的 `serverId`。如果首次 `/api/v1/sync/meta` 被并发请求，两次调用都可能在内存 `config.sync.serverId` 仍为空时生成不同 UUID；较早调用最终可能返回一个没有持久化下来的 UUID。
- 影响：移动端首次配对或多客户端同时连接时，可能短暂记录错误 `serverId`，随后下一次 `meta` 又看到另一个持久值，触发不必要的全量重建或服务器身份变化处理。
- 建议：给 `ensureSyncServerId()` 增加模块级 pending promise / mutex；或者保存成功后返回 `getConfig().sync.serverId` 的最终值，并补一个并发用例锁定“多次并发首次调用返回同一 serverId”。

### Issue 4：`decodeSyncCursor` 对游标内容校验偏宽

- 严重级别：Minor
- 涉及位置：
  - `src/main/services/syncService.ts:38-44`
  - `src/main/api/routes/syncRoutes.ts:36-42`
- 问题：`decodeSyncCursor()` 只校验 `u` 是字符串、`i` 是整数，没有要求 `i > 0`，也没有校验 `u` 是否符合项目内 ISO 时间戳格式。
- 影响：外部客户端可以传入畸形但形状“合法”的 cursor，服务端不会返回 422，可能造成重复页、异常窗口或难排查的同步状态。
- 建议：要求 `i` 为正整数，并校验 `u` 至少符合 `YYYY-MM-DDTHH:mm:ss.sssZ` 一类项目时间戳格式。

### Issue 5：sync 大 JSON 响应使用同步 gzip，可能阻塞 Electron 主进程

- 严重级别：Minor
- 涉及位置：
  - `src/main/api/response.ts:31-47`
  - `src/main/api/routes/syncRoutes.ts:48`
  - `src/main/api/routes/syncRoutes.ts:56`
  - `src/main/api/routes/syncRoutes.ts:64`
  - `src/main/api/routes/syncRoutes.ts:72`
- 问题：`sendSuccessMaybeGzip()` 先 `JSON.stringify` 完整响应，再用 `gzipSync()` 同步压缩。`/sync/images` 单页上限 5000，`/sync/tags`、`/sync/galleries`、`/sync/image-ids` 是全量接口，大库或多客户端同步时会在主进程同步 CPU 压缩。
- 影响：同步期间可能卡住 Electron 主进程事件循环，造成 UI、IPC、其他 API 请求短时间无响应。当前测试覆盖了 gzip 结果，但没有覆盖大载荷下的阻塞行为。
- 建议：改为异步 `zlib.gzip` / `createGzip` 流式压缩，或把大载荷压缩移出同步主线程；同时增加一个大载荷响应的性能/不阻塞回归测试或压缩函数单测。
