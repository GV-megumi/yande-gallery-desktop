## BUG-01 · Critical · 「查看原图/原图下载」点击即闪退（前台服务类型缺声明）

- **现象（用户可见）**：大图页点「查看原图」→ App 立即闪退回桌面。100% 复现。连带批量下载、带下载的分享（J.5）、下载前台通知（J.1）全部走 DownloadWorker 前台服务，同样崩溃。
- **受影响档位**：**全部 API 31+**（Android 12+）。本次三台联调机（API 32/33/34）全中。
- **运行时证据**：
  - **MuMu API 32（确定日志）**：
    ```
    FATAL EXCEPTION: main  Process: com.bluskysoftware.yandegallery
    java.lang.RuntimeException: Unable to start service androidx.work.impl.foreground.SystemForegroundService@... with Intent { act=ACTION_START_FOREGROUND ... }:
    java.lang.IllegalArgumentException: foregroundServiceType 0x00000001 is not a subset of foregroundServiceType attribute 0x00000000 in service element of manifest file
        at androidx.work.impl.foreground.SystemForegroundService.startForeground(SystemForegroundService.java:141)
        at androidx.work.impl.foreground.SystemForegroundDispatcher.handleNotify(SystemForegroundDispatcher.java:332)
    ```
    进程 PID 5569 崩溃 → 系统重启新进程。
  - **NX769J API 34（进程死亡）**：force-stop 后启动 PID=18475 → 点「查看原图」→ PID 消失（进程死亡）+ 前台回到 `com.zte.mifavor.launcher`。（ZTE 限制 logcat，无栈但进程死亡确证。）
  - **小米 API 33**：PIN 锁屏无法驱动 UI；同一 APK/manifest，根因为 manifest 级 → 必然同样中招。
- **根因（代码）**：
  - `data/media/../DownloadNotifier.kt:45`：SDK≥29 时 `ForegroundInfo(id, notification, ServiceInfo.FOREGROUND_SERVICE_TYPE_DATA_SYNC)`（类型 = 0x1 dataSync）。
  - `AndroidManifest.xml`：声明了 `FOREGROUND_SERVICE` + `FOREGROUND_SERVICE_DATA_SYNC` **权限**（line 12-13），但**没有**为 WorkManager 的 `androidx.work.impl.foreground.SystemForegroundService` 加 `android:foregroundServiceType="dataSync"` 的 `<service tools:node="merge">` 覆盖。
  - 合并后该 service 的 `foregroundServiceType` 属性 = 0x0。Android 12+ 要求 `startForeground(type)` 的 type 必须是 manifest service 元素 `foregroundServiceType` 的子集；0x1 ⊄ 0x0 → 抛 `IllegalArgumentException` → 崩溃。DownloadNotifier.kt:44 注释「targetSdk 35 强制与清单 FOREGROUND_SERVICE_DATA_SYNC 匹配」误以为**声明权限即够**，遗漏了 service 元素的类型声明。
- **修复方向（不在本次改）**：在 manifest `<application>` 内补
  ```xml
  <service android:name="androidx.work.impl.foreground.SystemForegroundService"
           android:foregroundServiceType="dataSync" tools:node="merge" />
  ```
  （API 34 另需确认运行时具备 `FOREGROUND_SERVICE_DATA_SYNC` 使用条件。）

## 代码分析发现（opus 对抗验证 CONFIRMED）

> 均为 opus 子智能体「证伪优先」复核后仍成立者。`runtimeConfirmable` 标注能否在 API32 模拟器运行时复现。多数写路径缺陷未在本次做运行时复现（避免污染用户真实 5202 图库），以代码+调用链坐实为准。

### Important（7）

**BUG-02 · 写操作应用级错误(403/422/500)被误判为「离线」** — `WriteRepository.kt:37` + `ConnectionMonitor.kt:44`
- 现象：配对+首同步成功后（在线、写按钮可用），任何写操作被服务端**非 404** 拒绝（最典型：桌面端 imageWrite/galleryWrite **默认关**→403 PERMISSION_DENIED），除正确弹「XX 失败」外，全局 `online` 被翻 false → 顶部误报「未连接到 <服务器>」离线横幅（网络完全正常），Albums「+」FAB/长按菜单/多选底栏/大图页删除·更多/详情「编辑」**全部置灰**，刚失败的按钮已灰无法原地重试。
- 根因：`ConnectionMonitor.reportFailure` 只对 `code=="UNAUTHORIZED"` 特判，其余非 404 一律置 `online=false`——把「服务器已应答但拒绝」与「连不上」混为一谈；且失败分支不 `requestSync()`。直接违反 README §3.3 / :97「权限未开只弹失败、按钮不隐藏」。
- 影响：**这是 README 反复标注的头号联调坑**（写权限默认关），命中率极高，且把用户导向错误补救（去管理服务器而非去桌面开权限）。可经回前台/下拉刷新自愈。修法：`reportFailure` 只对真连接失败（IOException/超时）翻 online，403/422/500 只弹 snackbar。

**BUG-03 · 删除失败回滚只恢复 ImageEntity，丢失被 CASCADE 删的标签链/相册成员链** — `WriteRepository.kt:52`
- 现象：在相册详情删一张带标签的图，若服务端返回 403（imageWrite 默认关）/瞬时 500 → 弹「删除失败」，但该图**从当前相册消失、详情标签清空、从标签搜索掉出**（时间轴里仍在）。用户观感：删除报错了、照片却离开相册还丢了标签。
- 根因：`gallery_images`/`image_tags` 对 `imageId` 声明 `ON DELETE CASCADE`（Entities.kt:36,55），乐观 `deleteByIds` 已级联删链，回滚只 `upsertAll(images 行)` 不重建链；且**例行增量同步不恢复**（该图 changeSeq 未变、不会被重拉，`replaceGalleryLinks/replaceTagLinks` 只覆盖被拉到的图）——持续到全量重建（dataVersion 变/切服）。爆炸半径 = 该图全部相册归属 + 全部标签。

**BUG-04 · 批量「加入相册」失败回滚误删选中图原有的相册成员关系** — `WriteRepository.kt:112`
- 现象：多选 [X, A]（X 已在相册 G），点「加入相册」选 G，若失败（galleryWrite 默认关→403），**X 被从 G 悄悄删掉**（`insertGalleryLinks` 用 IGNORE 对已存在链无操作，但回滚 `deleteGalleryLinks(G,[X,A])` 无条件全删）。只弹「加入相册失败」，用户以为无副作用；例行同步不恢复。单图「更多→加入相册」路径同样中招。
- 根因：apply 与 rollback 不对称（过度回滚），与 BUG-03/BUG-14 同类。测试只 seed 无前置成员链的场景，故套件全绿而 bug 潜伏。

**BUG-05 · SseClient 共享监听器无 stale-source 守卫 → 累积泄漏孤儿 SSE 长连（eventsSubscribe 开启时为 Critical）** — `SseClient.kt:121`
- 现象：**仅当桌面开启 eventsSubscribe（SSE 走 200）时**——每次 `restart()`（回前台/网络恢复/切服都会触发）遗留一条孤儿 SSE 连接：仍流式接收、仍不断 `requestSync("sse")`、退后台也不断开。因 `sseHttpClient` 与 Coil/Retrofit 共享同一 OkHttp Dispatcher（maxRequestsPerHost=5），孤儿 `readTimeout=0` 永不释放槽位——约 5 条孤儿打满后**所有发往桌面的缩略图/同步/下载请求永久排队 → App 假死**，进程不重启不自愈。
- 根因：单一 listener 复用给所有 EventSource，`cancel()` 触发的异步 `onFailure(canceled)` 无守卫地 `eventSource=null`（覆盖新连接引用）+ `scheduleReconnect()` → 30s 后再建重复连接，旧的成孤儿。修法：三回调首行加 `if(source!==eventSource)return` + SSE 用独立 Dispatcher。
- 定级：验证后由 Critical **下调 Important**——因桌面 eventsSubscribe **默认关**，默认路径下 S2 也收 403 走降级、短命自愈、三台默认联调机不假死；**一旦用户开 eventsSubscribe 即为 Critical**（稳定累积、无自愈、静默假死难定位）。

**BUG-06 · 大图页定位期间底部操作栏对第 0 页「错图」生效** — `ViewerScreen.kt:534`
- 现象：库 >360 张时，从网格深处（如第 1000 位）点开一张图，定位需多轮 append（数百 ms~1s，其间 `located=false`）。黑色占位层只盖住图片、**没盖操作栏**（`if(!immersive)` 未受 `located` 门控），此窗口内操作栏可见可点，而 `pagerState.currentPage` 恒为 0 → `currentImage` = **时间轴最新一张**（createdAt DESC）。此时点「分享」/「查看原图」**无二次确认、静默对最新图**分享/下载入系统相册；点「删除」确认框显示的是最新图文件名，误确认即**删掉最新图**（数据丢失路径）。
- 根因：防错图占位层只挡了画面像素、没挡同屏可交互的图片相关操作。修法：操作栏一并以 `located` 门控。

**BUG-07 · API 26-28 原图下载全链路失败：WRITE_EXTERNAL_STORAGE 声明却从不运行时申请** — `AndroidMediaStoreGateway.kt:35`
- 现象：在 API 26/27/28 设备上，查看原图/批量下载/带下载分享**一律静默失败**（弹「下载失败」），图片永不进系统相册。
- 根因：<29 走 legacy `resolver.insert(EXTERNAL_CONTENT_URI)` 需运行时 WRITE_EXTERNAL_STORAGE；manifest 以 maxSdkVersion=28 声明了权限，但全 App 只申请 CAMERA/POST_NOTIFICATIONS，**从无 requestPermission(WRITE_EXTERNAL_STORAGE)**（README §7 列为待验证项，实现缺失）。
- 影响：**本次三台联调机 API 31+ 全走 scoped-storage 分支、不受影响**（runtimeConfirmable=false，需 API26-28 设备）；面向 minSdk 26 的潜伏功能级失效。

**BUG-08 · 「测试连接」不做 scheme 归一化，裸 IP 误报连接失败** — `ServersViewModel.kt:48`
- 现象：添加服务器时输入不带 `http://` 的地址（如 `192.168.1.5:38947`，正是 M4-T14/BaseUrlValidationTest 明确支持的形态），点「测试连接」→ 弹**英文** IllegalArgumentException「连接失败：Expected URL scheme 'http' or 'https'...」；而同一输入点「保存并激活」（走 normalizeBaseUrl 补 scheme）却成功。测试判失败、保存判成功，判定相反 → 卡住 A.2「连接测试通过」验收、误导用户放弃配对。
- 根因：testConnection 跳过 normalizeBaseUrl，把裸地址直喂 Retrofit.baseUrl() 在构建期（网络请求前）抛异常，且 UI 原样透传英文 message。runtimeConfirmable=true（API32 离线即可复现）。

### PLAUSIBLE（3，机制成立但症状弱/难复现）

- **AppGraph 主线程 runBlocking 读 DataStore 构建 Coil loader**（`AppGraph.kt:112`）：反模式属实（主线程 park 等 DataStore 首读），但危害仅一次性几 ms、快存储下几乎观察不到，「贡献 ANR」夸大；且候选称「StrictMode 会标记」不成立（DataStore 真实读在自有 IO 线程、且 App 未装 StrictMode）。
- **离线批量分享 stale 副本挂起**（`PhotosScreen.kt:140`）：离线守卫用「只数 downloads 行、不校验文件」的口径，手删过系统相册副本的项被当已下载滑过、分享协程等一个离线永不完成的下载；「**永久**挂起」取决于 WorkManager 是否已 prune 历史终态——重启后可达，非每次必现（否则误报「下载失败」）。
- **ScanScreen 相机权限缺 rememberSaveable 守卫**（`ScanScreen.kt:80`）：旋转确会重跑 `launcher.launch`，但**框架 `mHasCurrentPermissionsRequest` 跨旋转去重挡住二次弹框**——真实症状是罕见的伪「denied」翻转，而非候选宣称的「重复弹框」。修法（对齐 NotificationPermissionEffect 的 rememberSaveable）仍值得做。

### REFUTED（1，假阳性）

- **Coil 默认策略缓存 404 永久破图**（`ImageLoaders.kt:34`）：候选对 Coil 3.5 内部反编译准确（404 属 CACHEABLE_STATUS_CODES、read 不重校验），**但漏了本仓前置**——两个 ImageLoader 用的是带错误拦截器的同一 OkHttp，拦截器对任何非 2xx **先抛 ApiException(IOException)**，404 以异常抵达 Coil、Response 从不返回，故**根本不写盘**；文件恢复后点重试即恢复。`RetryableAsyncImage` 注释「Coil 不缓存失败结果」恰因该拦截器成立。→ 非缺陷。

## 自动化局限（非 App 缺陷）

- `uiautomator dump` 在 MuMu 返回 0 节点 → 只能截图估坐标点击，小控件/手势脆弱。
- adb `input swipe` 原地无法触发 Compose `combinedClickable(onLongClick)` → **多选长按**无法经 adb 复现（交由代码分析）。
- 无多点触控原语 → 捏合缩放（B.4）、双指切档（J.2）不可自动化。
- 物理机：NX769J OEM 限制 logcat + 无 sqlite3 CLI；小米 MIUI 反侧载（`/data/local/tmp` 与 Download 推送的 APK 被内容级秒删，最终用户手动装）+ PIN 锁屏。
- 单档 API 32 为主 → 计划设备矩阵的 26-28 legacy 档、QR 扫码配对、微信等外部 app 分享、填满磁盘等未覆盖。
- **未做会改动用户真实图库的写测试**（删除/新建相册/标签），避免污染 5202 图的真实库；写路径缺陷以代码分析为准。
