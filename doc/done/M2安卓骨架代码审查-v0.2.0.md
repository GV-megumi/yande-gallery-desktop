### Issue 1：新增或切换服务器后不会自动触发同步

- 严重级别：Important
- 涉及位置：
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/YandeGalleryApp.kt:19-21`
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/ui/servers/ServersViewModel.kt:37-44`
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/ui/photos/PhotosViewModel.kt:65-67`
- 问题：前台同步只在进程 `onStart` 触发；`add()` 和 `activate()` 只写入/切换激活服务器，没有调用 `syncScheduler.requestSync()`。如果用户首次打开 app 时还没有服务器，`onStart` 的同步会因未配置服务器失败或无效；随后扫码/手输保存服务器不会自动开始首次全量同步。切换服务器时也只更新 active 行，Room 镜像仍保留旧服务器数据，直到用户手动下拉或重新进前台。
- 影响：README 写明“配对即激活；随后自动首次全量同步”，但真实流程会停留在空库或旧库状态。多服务器切换时，UI 会立即用新 `baseUrl` 加载缩略图，却仍用旧服务器的本地 image id，可能出现错误缩略图、404 对账或误导性的旧内容。
- 建议：在新增/激活服务器成功后触发同步，并在切换 active server 时重置连接状态、重连 SSE；必要时在同步开始前显示“正在为新服务器建立镜像”的明确状态，避免旧镜像被当作新服务器内容展示。

### Issue 2：SSE 生命周期没有绑定激活服务器变化

- 严重级别：Important
- 涉及位置：
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/domain/sync/SseClient.kt:37-55`
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/domain/sync/SseClient.kt:85-91`
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/di/AppGraph.kt:51-53`
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/di/AppGraph.kt:116-124`
- 问题：`SseClient.start()` 只在前台时调用一次。如果当时没有激活服务器，`connect()` 因 `urlProvider() == null` 直接返回，但 `started` 已经是 `true`，后续新增服务器不会再次连接。切换激活服务器时，已有 `EventSource` 不会取消并按新 baseUrl 重连。另一个问题是 403 会设置全局 `permanentlyDisabled = true`，即使用户后来在桌面端开启 `eventsSubscribe`，或切换到另一个已开启权限的服务器，本次 app 进程内也不会恢复 SSE。
- 影响：M2 的“桌面端有 gallery 事件时自动触发一次对账同步”在常见路径下会失效，只剩前台/下拉刷新兜底。切换服务器后还可能继续监听旧服务器事件，用旧事件触发新服务器同步。
- 建议：让 SSE 订阅以激活服务器 flow 为驱动：active server 变化时取消旧连接、清理当前服务器的 403 降级状态并重连。403 降级应至少按服务器或当前凭据隔离，而不是 app 进程全局永久关闭。

### Issue 3：同步调度器“合并”请求实际会丢弃运行期间的新触发

- 严重级别：Important
- 涉及位置：
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/domain/sync/SyncScheduler.kt:29-32`
  - `android/app/src/test/java/com/bluskysoftware/yandegallery/domain/sync/SyncSchedulerTest.kt:27-40`
- 问题：`requestSync()` 发现已有任务运行时直接 `return`，测试也锁定了“第二次请求被忽略”。这不是串行合并，而是丢弃。前台同步、下拉刷新、SSE 事件和二进制 404 对账都会走同一个入口；如果图库变更事件或 404 发生在一次长同步过程中，结束后不会补跑一轮同步。
- 影响：长时间首次同步或大库对账期间发生的变更可能要等下一次手动刷新/重新进前台才会出现。对二进制 404 来说，原本期望立即触发 `image-ids` 对账，实际可能被正在运行的同步吞掉。
- 建议：增加 `pending` 标记或队列语义：运行中收到请求时记一笔 pending，当前同步完成后若 pending 为真再串行跑一轮；仍保持同一时间只有一个同步任务。

### Issue 4：缩略图缓存键只按 imageId 命名，跨服务器会污染缓存

- 严重级别：Important
- 涉及位置：
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/data/image/ImageLoaders.kt:15-16`
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/data/image/ImageLoaders.kt:32-36`
  - `android/app/src/test/java/com/bluskysoftware/yandegallery/data/image/ImageLoadersTest.kt:28-30`
- 问题：`thumbnailCacheKey(imageId)` 固定为 `thumb:<id>`，不包含激活服务器、本地 server id 或远端 `serverId`。M2 支持保存多个服务器，同时本地 Room 的图片 id 来自各桌面端数据库；不同服务器完全可能存在相同 image id 但对应不同图片。
- 影响：切换服务器后，Coil 内存/磁盘缓存可能直接命中上一台服务器的同 id 缩略图，显示错误图片。当前测试还把“与 baseUrl 无关”作为期望锁定，无法发现多服务器污染。
- 建议：缓存键至少包含稳定服务器命名空间。若要保留“同一图库换 IP 后不作废缓存”的优点，可在首次同步拿到远端 `serverId` 后用它命名；同步前可临时用本地 server row id，拿到远端身份后做迁移或接受重新缓存。

### Issue 5：图集卡片封面兜底存在 N+1 查询

- 严重级别：Minor
- 涉及位置：
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/ui/albums/AlbumsViewModel.kt:35-41`
  - `android/app/src/main/java/com/bluskysoftware/yandegallery/data/db/GalleryDao.kt:27-34`
- 问题：`AlbumsViewModel.albums` 在每次 `observeAll()` 发射后，对每个 `coverImageId == null` 的图集逐个调用 `coverFallback()`。图集数量较多或同步频繁覆盖 `galleries` 时，会在 Room 查询线程上形成 N+1 查询。
- 影响：大图集库下相册 tab 首屏和同步后的重算可能变慢。M2 当前可用，但这个路径后续容易成为列表卡顿点。
- 建议：新增 DAO 查询一次性返回图集和兜底封面 id，例如用 left join / correlated subquery 生成 `AlbumCard` 投影，避免 ViewModel 中逐项查库。

### Issue 6：M2 diff 的空白检查会被 `gradlew.bat` 拦截

- 严重级别：Minor
- 涉及位置：
  - `android/gradlew.bat:1-82`
- 问题：`git diff --check 5be35a6..78dd711` 将新增的 `android/gradlew.bat` 每一行都报告为 trailing whitespace。
- 影响：如果后续 CI 或发布流程启用 `git diff --check`，M2 范围会被直接拦截。当前不影响 Android 工程本身的 Linux/macOS `gradlew`，但会影响仓库级质量门禁。
- 建议：规范化 `android/gradlew.bat` 的行尾/空白处理，或为 Gradle wrapper 批处理文件增加明确的 `.gitattributes` 规则并确认 `git diff --check` 不再误报。
