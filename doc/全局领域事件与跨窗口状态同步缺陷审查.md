审查范围：`src/shared/types.ts`、`src/main/services/`、`src/main/ipc/handlers/`、`src/main/api/`、`src/preload/`、`src/renderer/pages/`、`src/renderer/components/`、`src/renderer/hooks/`，以及 `doc/` 中已有事件相关说明。

审查目标：找出当前项目中“主进程权威状态已经变化，但其他窗口、固定缓存页面、子窗口或 API SSE 订阅者无法收到统一通知”的缺陷，并给出应该补哪些全局领域事件、如何实现和如何验证。

## 结论摘要

项目已经有一条可用的跨窗口事件总线：主进程服务发布 `RendererAppEvent`，`rendererEventBus` 广播到所有 `BrowserWindow`，同时桥接到 API SSE；renderer 通过 `window.electronAPI.system.onAppEvent` 订阅。

但当前事件模型明显过窄：`RendererAppEvent` 只覆盖批量下载会话、收藏标签、图库导入、图库列表变化和缩略图生成。大量真正会影响多窗口一致性的主进程状态仍停留在“谁调用谁自己刷新”的模式，例如 Booru 帖子收藏、服务端收藏、黑名单、站点认证状态、保存搜索、收藏分组、图片删除、无效图片、相册封面、配置变更、备份恢复等。

因此，当前核心缺陷不是某个页面漏了一次 `reload()`，而是缺少一套稳定的领域事件规范：哪些状态变化必须由 service 层发事件、payload 怎么定义、renderer 如何统一订阅、API SSE 如何暴露、哪些局部 UI 状态不应该进入全局事件。

## 判断标准

参考 React 状态管理的分层原则，本项目可以把状态分成三类。

第一类是局部 UI 状态，不应该做全局领域事件。例如弹窗开关、loading、表单草稿、当前 hover、图片查看器缩放、页面内展开折叠、临时排序控件、搜索输入框草稿。这些状态只属于当前组件或当前窗口。

第二类是页面查询状态，通常不直接做领域事件。例如当前页码、当前搜索关键词、当前筛选条件、当前 tab。它们可以持久化到 pagePreferences，但不代表业务事实本身发生了变化。若要多窗口同步，应走配置同步或窗口级偏好策略，而不是混入 Booru / Gallery 领域事件。

第三类是主进程权威状态，应该做全局领域事件。判断条件是：

- 数据已经写入数据库、配置文件、主进程内存服务或外部站点状态。
- 可能由多个入口修改，例如主窗口、子窗口、设置页、IPC、API 路由、后台任务。
- 多个页面或窗口依赖它展示按钮状态、过滤结果、数量、列表、卡片标记或功能开关。
- 旧状态会造成用户可见错误，例如收藏红心错、黑名单过滤不生效、站点认证状态错、图库还显示已删除图片。
- 事件 payload 可以保持轻量，只作为“失效 / 增量更新提示”，renderer 收到后可局部 patch 或防抖重新拉取。

## 当前事件架构现状

### 已有 RendererAppEvent

当前 `RendererAppEvent` 定义在 `src/shared/types.ts:536` 起，事件 union 在 `src/shared/types.ts:652` 起，实际只有以下类型：

- `bulk-download:sessions-changed`
- `favorite-tag-download:created`
- `favorite-tags:changed`
- `gallery:images-imported`
- `gallery:galleries-changed`
- `thumbnail:generated`

这些事件的公共字段包括 `type`、`version`、`occurredAt`、`source`、`payload`。

### 发布链路

主进程服务调用 `emitBuiltRendererAppEvent`，由 `src/main/services/rendererEventBus.ts` 补齐公共字段，再广播到所有窗口的 `system:app-event`。同一条事件也会桥接到 API SSE。

关键位置：

- `src/main/services/rendererEventBus.ts:22`：构造事件。
- `src/main/services/rendererEventBus.ts:32`：根据事件类型映射 API SSE channel。
- `src/main/services/rendererEventBus.ts:66`：发布到 API hub。
- `src/main/services/rendererEventBus.ts:81`：广播到所有 `BrowserWindow`。
- `src/preload/shared/createSystemApi.ts:46`：renderer 订阅 `system.onAppEvent`。
- `src/main/api/routes/eventRoutes.ts:20`：`GET /api/v1/events/:channel` SSE 订阅。

### 当前有效消费点

真正直接消费 `system.onAppEvent` 的 renderer 位置不多：

- `src/renderer/pages/BooruBulkDownloadPage.tsx:152`：批量下载会话、收藏标签下载创建。
- `src/renderer/pages/FavoriteTagsPage.tsx:334`：收藏标签、收藏标签下载、批量下载会话。
- `src/renderer/pages/GalleryPage.tsx:435`：图库导入、图库变化、缩略图生成。
- `src/renderer/components/ImageGrid.tsx:330`：缩略图生成。

也就是说，Booru 绝大多数页面、设置页、图库无效图片页、保存搜索页、收藏分组页并没有统一消费全局领域事件。

### 已有但不统一的旧链路

项目还存在几条绕过 `RendererAppEvent` 的旧链路：

- `config:changed`：`src/main/ipc/handlers/configHandlers.ts:49` 直接向窗口发送，preload 暴露了 `config.onConfigChanged`，但没有进入 API SSE，也没有统一纳入 `RendererAppEvent`。
- 单张 Booru 下载进度 / 状态：`src/main/services/downloadManager.ts:331`、`src/main/services/downloadManager.ts:634`、`src/main/services/downloadManager.ts:648` 通过 `BOORU_DOWNLOAD_*` IPC 直发。
- API log：API 服务内部发 `api-logs` SSE，但不是 renderer 领域事件。
- `system:navigate`：这是导航命令，不是状态同步事件。

这些旧链路不一定都要废掉，但需要明确边界：高频进度可以保留专用通道，业务事实变更应进入统一领域事件。

- 列表、分页、聚合数量用防抖 refetch。

## 主进程实现规范

### 事件必须从 service 层发出

同一个状态可能由 IPC、API route、后台任务、备份恢复触发。如果只在 IPC handler 里发事件，API 或后台路径会漏。

应该在写入权威状态的 service 方法成功后发事件，例如：

- `booruService.addToFavorites`
- `booruService.removeFromFavorites`
- `booruService.setPostLiked`
- `booruService.addBlacklistedTag`
- `booruService.toggleBlacklistedTag`
- `galleryService.setGalleryCover`
- `imageService.deleteImage`
- `invalidImageService.reportInvalidImage`
- `bulkDownloadService.createBulkDownloadTask`

### 先成功写入，再发送事件

事件必须代表已经发生的事实，而不是尝试动作。对当前会吞掉错误的方法要先整理错误语义。例如 `setPostLiked` 现在 catch DB 错误后只打印日志，后续如果要发事件，需要能判断 DB 更新是否成功。

### 批量操作只发聚合事件

批量导入黑名单、备份恢复、清空无效图片、批量删除记录，不要对每条数据发几十上百个窗口事件。应发一条聚合事件：

- `affectedCount`
- `siteId` / `galleryId`，如果能确定
- `action: 'imported' | 'cleared' | 'restored'`

renderer 收到后按当前页面需要重新查询。

### 高频进度与领域事实分开

下载进度可以继续使用专用进度通道，避免把 `RendererAppEvent` 变成高频流。但状态跨阶段变化应有领域事件，例如 queued、completed、failed、removed、markedDownloaded。

# 第二次完整复扫补充（2026-06-08）

## 复扫方法与总体结论

第二次复扫把审查范围拆成 14 个子系统并行扫描（Booru 收藏/服务端收藏、黑名单、站点认证、保存搜索/收藏分组/搜索历史、投票/帖子动作、单图下载、批量下载、图库图片/封面、无效图片/忽略文件夹/元数据、配置/备份/更新/API 服务/通知、事件总线基础设施、跨窗口/子窗口机制、renderer 消费方普查、完整性批判），并对每条 P0/P1 新发现做了对抗式核验。

总体结论与首轮一致：**传输层（`rendererEventBus` 经 `BrowserWindow.getAllWindows()` 广播 + 子窗口 preload 也暴露 `system.onAppEvent` + 桥接 API SSE）是健全的，缺陷全部集中在「发布端不发事件」与「消费端不订阅」两端**。但复扫又暴露了三类首轮低估或漏掉的结构性问题：

1. **API HTTP 路由是与 IPC 对等的第二写入面**，首轮只把收藏/喜欢缺口挂在 `useFavorite`/`booruHandlers` 上，漏了 `booruRoutes`。修复必须落在 **service 层**才能同时覆盖两条入口。
2. **轻量子窗口「能订阅却不订阅」**：tag-search/artist/character 三类子窗口渲染的页面没有任何 `onAppEvent` 调用，等于所有领域事件对整类子窗口失效。
3. **「发了事件也没用」**：Gallery 的 recent 消费链路只做 after-cursor 增量、`all` 子 tab 零订阅、`ImageGrid` 只处理缩略图，即便补了 `gallery:images-changed(deleted)` 也无法从已渲染列表移除死图——必须同步改消费逻辑。

对抗式核验结果：**8 条 P0/P1 新发现全部 `isReal=true`（high 置信），无一被推翻**。

`booru` SSE 频道当前完全空转：`resolveApiEventChannel` 的 `booru:` 分支、`eventHub`/`eventRoutes`/`permissions` 的 `booru` 频道都已就绪，但 `RendererAppEvent` union（`src/shared/types.ts:652-658`）没有任何 `booru:*` 成员，`booruService`/`booruHandlers`/`booruRoutes` 也从不发 `booru:*` 事件。所有 Booru 领域事件的落地只差「补 union 类型 + 在 service 写入点 emit」。

## 三、重点结论深化

### 1. 事件必须从 service 层发出（API 路由是对等写入面）

`booru` 收藏/喜欢有三条写入入口共享同一组无事件的 service 方法：IPC（`booruHandlers.ts:783/:1694/:1719`）、HTTP API（`booruRoutes.ts:555/:583`）、后台修复（`booruHandlers.ts:749` 经专用旁路通道）。若只在 IPC handler 补事件，API 与后台修复路径会漏。**正确修复点是 `addToFavorites`/`removeFromFavorites`/`setPostLiked` 这三个 service 方法成功写库后 emit**，三条入口与 booru SSE 一并受益。同理黑名单 6 个写入点（含遗漏的 `2833`/`3107`）、站点写入点（`addBooruSite`/`updateBooruSite`/`deleteBooruSite`/`setActiveBooruSite`）都应在 service 层发。

注意：`setPostLiked` 现在 `catch` DB 错误后只打印日志（`booruService.ts:1126`），要发事件需先能判断写库是否成功——「先成功写入，再发送事件」。另外复扫发现 API 的 like 接口只翻本地 `isLiked` 不调外部站点，与 IPC 行为不一致，属另一个语义一致性问题，建议单独核查。

### 2. 子窗口「传输通、消费断」

`rendererEventBus` 经不过滤的 `getAllWindows()` 广播，子窗口 preload（`subwindow-index.ts`）也经 `createSystemApi` 暴露 `onAppEvent`，`tests/preload/subwindow-exposure.test.ts` 已断言其存在——**传输层对子窗口是通的**。问题在于三类轻量子窗口渲染的页面（`BooruTagSearchPage`/`BooruArtistPage`/`BooruCharacterPage`）从不订阅。补任何 booru 领域事件时，**务必同步在这三个页面接入订阅**，否则「补了事件子窗口仍不刷新」。（secondary-menu 子窗口用主 preload 且渲染 `GalleryPage`/`BooruPage` 等，不属此问题。）

### 3. keep-alive 退订与无重放 → 消费 hook 应「常驻订阅 + 标脏」

保活页隐藏时仍挂载，但 `GalleryPage`/`BooruBulkDownloadPage`/`FavoriteTagsPage` 的订阅 effect 在 `suspended/!active` 时直接 early-return，隐藏期错过的事件无重放。建议未来的 `useRendererAppEvent`/`useBooruDomainEvents`：**订阅本身常驻、收到事件先标脏，激活时再消费**，只把昂贵的全量刷新 gate 住；`ImageGrid:330` 的无条件订阅是现成正面范例。

### 4. 「发了事件也没用」——消费链路要能真正移除数据

Gallery 删除/无效场景最隐蔽：即便补了 `gallery:images-changed(deleted)`，当前 recent 仅做 after-cursor 增量、`all` 零订阅、`ImageGrid` 只更新缩略图，没有任何一处会按 id 从本地数组移除已删/无效图。**补事件与改消费逻辑必须成对落地**，否则死图仍可见。

### 5. booru SSE 频道空转——基础设施已就绪，只差契约与 emit

`resolveApiEventChannel` 的 `booru:` 分支、`eventHub`/`eventRoutes`/`permissions` 的 `booru` 频道都在，但 union 无 `booru:*` 成员、无任何 emit。新增任意 `booru:` 前缀事件会自动落到 `booru` SSE 频道，**无需改 SSE 层**——落地成本极低。

## 四、对「最小修复清单」的补充

在首轮 5 步基础上补充以下约束：

1. **事件一律在 service 层发**：booru 收藏/喜欢/黑名单/站点的 emit 点放在 `booruService` 方法内（覆盖 IPC + API + 后台修复三入口）；黑名单务必含遗漏的 `addBlacklistedTags(2833)`、`updateBlacklistedTag(3107)`。
2. **消费 hook 接入子窗口 + 常驻订阅**：`useRendererAppEvent`/`useBooruDomainEvents` 必须在 `BooruTagSearchPage`/`BooruArtistPage`/`BooruCharacterPage` 接入，并采用「常驻订阅 + 标脏」规避 keep-alive 退订丢事件（NF6/NF7）。
3. **Gallery 删除/无效要改消费逻辑**：`reportInvalidImage` 优先复用 `gallery:galleries-changed(statsUpdated)`（零新增类型让图库页封面/计数失效），同时为 recent/all/detail/ImageGrid 增加按 id 剔除（NF4/NF5）。
4. **批量下载 record 终态进总线**：NF2/NF3 的 record completed/failed/pending-reset 至少发聚合失效提示，避免对成百上千条记录逐条发窗口/SSE 事件。
5. **扩 `RendererAppEventSource`**（`configService`/`backupService`/`apiService`）+ 清理死成员 `'ipc'`，作为接入 config/backup/api-service/`app:data-restored` 事件的前置（NF13/NF17）。
6. **`config:changed` 至少补消费**：把 `config:changed` 纳入统一总线，或先让 `App`/`SettingsPage`/`BooruSettingsPage` 订阅 `config.onConfigChanged`（当前 renderer 零消费，NF 更正表第 1 项）；`CONFIG_UPDATE_GALLERY_FOLDERS` 补 `broadcastConfigChanged(['galleries'])`。
