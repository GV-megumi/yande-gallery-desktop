# 性能优化任务规划

> 基于 performance-audit.md 审计报告制定
> 按优先级分阶段执行，每阶段内按依赖关系排序

---

## 第一阶段：紧急修复（影响可用性）

预计改动文件 3 个，改动量小，风险低。

### Task 1.1 — 替换 downloadManager 中的同步 fs 调用
- **对应问题**: P1 (CRITICAL)
- **文件**: `src/main/services/downloadManager.ts`
- **改动**:
  - `fs.existsSync()` → `await fs.access().catch()`
  - `fs.unlinkSync()` → `await fs.unlink()`
  - `fs.mkdirSync()` → `await fs.mkdir()`
  - 涉及行: 179, 372-373, 433-435, 450-452
- **验证**: 下载文件时主进程不再阻塞 IPC 响应

### Task 1.2 — 为批量标签操作添加事务
- **对应问题**: P2 (HIGH)
- **文件**: `src/main/services/booruService.ts`, `src/main/services/imageService.ts`
- **改动**:
  - `saveBooruTags()` 用 `BEGIN TRANSACTION / COMMIT` 包裹循环
  - `addTagsToImage()` 同上
  - 添加 `runInTransaction()` 工具函数到 `database.ts`
- **验证**: 批量保存 500 个标签耗时从秒级降到毫秒级

### Task 1.3 — 添加缺失的数据库索引
- **对应问题**: P24 (HIGH)
- **文件**: `src/main/services/database.ts`
- **改动**: 在 `initDatabase()` 中追加：
  ```sql
  CREATE INDEX IF NOT EXISTS idx_booru_posts_site_created ON booru_posts(siteId, createdAt DESC);
  CREATE INDEX IF NOT EXISTS idx_booru_favorites_postId ON booru_favorites(postId);
  CREATE INDEX IF NOT EXISTS idx_bulk_records_session_status ON bulk_download_records(sessionId, status);
  CREATE INDEX IF NOT EXISTS idx_favorite_tags_site_tag ON booru_favorite_tags(siteId, tagName);
  CREATE INDEX IF NOT EXISTS idx_booru_posts_localImageId ON booru_posts(localImageId);
  ```
- **验证**: 收藏查询和批量下载进度统计查询提速

## 第二阶段：内存优化（影响长时间使用稳定性）

预计改动文件 4-5 个，改动量中等。

### Task 2.1 — 修复缩略图状态无界增长
- **对应问题**: P15 (HIGH)
- **文件**: `src/renderer/components/ImageGrid.tsx`, `src/renderer/pages/GalleryPage.tsx`
- **改动**:
  - 切换标签页/图集时重置 `thumbnailPaths` 为 `{}`
  - 可选：引入简单 LRU Map，限制最大缓存条目数（如 500）
- **验证**: 反复切换标签页后内存不持续增长

### Task 2.2 — 优化 Image.PreviewGroup 大列表问题
- **对应问题**: P16 (HIGH)
- **文件**: `src/renderer/components/ImageGrid.tsx`
- **改动**:
  - 方案 A：仅对当前渲染批次（如 visibleCount）的图片包裹 PreviewGroup
  - 方案 B：自定义预览逻辑，点击时单独打开预览，不预创建所有预览节点
- **验证**: 加载 2000 张图片时渲染进程内存下降 30%+

### Task 2.3 — 图片预加载添加并发控制和取消
- **对应问题**: P17 (HIGH)
- **文件**: `src/renderer/pages/BooruPostDetailsPage.tsx`
- **改动**:
  - 引入 `AbortController`，翻页时取消旧的预加载
  - 用 `p-limit(2)` 限制同时预加载数
  - 将 `.forEach(async ...)` 改为受控的并发执行
- **验证**: 快速翻页不再累积大量并发下载

### Task 2.4 — 改善缓存驱逐策略
- **对应问题**: P18 (HIGH)
- **文件**: `src/main/services/imageCacheService.ts`
- **改动**:
  - 下载前检查文件大小（HEAD 请求或 Content-Length），拒绝超大文件
  - 改为 LRU 逐个删除直到低于阈值，而非删一半
  - 添加单文件最大尺寸配置（如 200MB）
- **验证**: 缓存不再突破配置限制

## 第三阶段：并发调度优化

预计改动文件 4-5 个，改动量中-大。

### Task 3.1 — downloadManager 引入正式并发控制
- **对应问题**: P7 (HIGH)
- **文件**: `src/main/services/downloadManager.ts`
- **改动**:
  - 引入 `p-queue` 库或自实现信号量
  - 替换 `isProcessing` flag + `activeDownloads` Map 模式
  - 添加流背压处理（P11），使用 `stream.pipeline()`
- **验证**: 并发下载数严格不超过配置值

### Task 3.2 — 批量下载改为事件驱动
- **对应问题**: P8 (HIGH), P10 (HIGH)
- **文件**: `src/main/services/bulkDownloadService.ts`
- **改动**:
  - 去掉 200ms 轮询循环
  - 下载完成/失败时通过回调或 EventEmitter 触发下一个任务
  - 用 `p-queue` 替代 `activeCount` 计数器
- **验证**: CPU 空转消失，任务调度更及时

### Task 3.3 — IPC 处理器批量数据库操作
- **对应问题**: P9 (HIGH)
- **文件**: `src/main/ipc/handlers.ts`
- **改动**:
  - `BOORU_GET_POSTS` handler 中的 `saveBooruPost()` 循环改为事务批量插入
  - 标签获取改为可取消的后台任务
- **验证**: 获取 20 个 post 的 IPC 响应时间从秒级降到百毫秒级

### Task 3.4 — Moebooru 客户端添加请求限流
- **对应问题**: P12 (MEDIUM), P13 (MEDIUM)
- **文件**: `src/main/services/moebooruClient.ts`
- **改动**:
  - 添加 `bottleneck` 或简单令牌桶
  - 标签批量查询改为 `name=tag1,tag2,tag3` 格式（如 API 支持）
- **验证**: 高频请求不再触发服务端 429

## 依赖与风险

| 任务 | 依赖 | 风险 |
|------|------|------|
| 1.1 同步 fs → async | 无 | 低 — 纯 API 替换 |
| 1.2 事务包裹 | 1.2 需先实现 `runInTransaction` | 低 |
| 1.3 索引 | 无 | 极低 — 只加不改 |
| 2.1-2.2 内存 | 无 | 中 — 需测试图片预览功能不受影响 |
| 3.1-3.2 并发 | 可能需新增 npm 依赖 (p-queue) | 中 — 核心下载逻辑重构 |
| 4.x React | 无 | 低 — 纯优化，不改功能 |

## 新增依赖建议

| 包名 | 用途 | 大小 |
|------|------|------|
| `p-queue` | 并发队列控制 | ~5KB |
| `p-limit` | 简单并发限制 | ~1KB |
| `bottleneck` | API 请求限流 | ~15KB |

以上均为可选，也可自实现简易版本。
