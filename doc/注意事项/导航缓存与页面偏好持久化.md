# 导航缓存与页面偏好持久化

## 文档定位

约束 App 壳层导航、二级菜单缓存、子窗口打开、页面偏好（`pagePreferences`）持久化这几件事的配合方式，避免"切一级菜单就重新加载"、"返回后又被自动恢复"、"子窗口污染主窗口记忆"这类交叉问题。

## 模型

### 1. 渲染层次

App 壳层当前有三种内容层并存：

- **基础页**（base）：由 `selectedKey` + `selectedSubKey` / `selectedBooruSubKey` / `selectedGoogleSubKey` 派生的当前 section 唯一可见内容。
- **固定标签缓存层**（pinned）：`mountedPinnedIds` 集合内的所有页面常驻挂载，通过 `activePinnedId` 和 `display:none` 切换显示。
- **导航栈叠加层**（overlay）：tag-search / artist / wiki 等临时子页面，`navigationStack` 栈顶显示，其余 `display:none` 暂存。

**核心约束**：任何页面在同一时刻都应该只有**一个激活实例**。"base 层 + pinned 层"渲染同一 (section, subKey) 时会出现双实例，必须避免。

### 2. 页面偏好持久化

`pagePreferences.*` 存在 `config.yaml`，负责跨会话记住"上次停留的子页面 / 搜索 / 过滤"。但要遵守以下边界：

- **存静态偏好**：搜索词、排序、分页、是否展开等。
- **慎存活动状态**：像"当前打开的图集 id"这种和导航语义强耦合的字段，会制造"返回后仍被自动还原"等反直觉 Bug。要存也需要产品明确语义。
- **主窗口和子窗口共享同一份偏好文件**，子窗口写回会影响主窗口的记忆，必须显式关闭写回。

## 约定

### 3. 一级菜单切换时要恢复当前应激活的层

一级菜单 onSelect 不能一刀切 `setActivePinnedId(null)`。应检查目标 section 的当前子 key：

- 若该子 key 命中 `pinnedItems` → 调用 `handlePinnedClick` 恢复到固定层缓存实例。
- 否则 → 走基础层。

缺这一步就会出现"cached 实例在树里但被隐藏了，前台看到的是基础层的新鲜重载"。

### 4. 外层容器 key 不要包含 `selectedKey`

外层包裹 div 若使用 `` key={`${selectedKey}-...`} ``，一级菜单切换时 key 变 → React 卸载整棵基础层子树 → 所有未 pin 的二级页面都会丢状态。

正确做法：外层 key 只包含**不会在一级菜单切换时变的**字段（如 subKey），或者按"每个 (section, subKey) 独立常驻"改造成多层叠加（参考 `mountedPinnedIds` 那套结构）。

### 5. 用户主动动作要同步落盘，不要依赖防抖 effect

页面偏好的保存 effect 通常带 250ms 防抖，目的是避免"每次打字都写盘"。但防抖会在 `subTab` 变化、组件卸载时被 `clearTimeout` 取消。

**反模式**：

```
用户点返回 → setSelectedGalleryId(null)
→ 防抖 effect 安排 250ms 后落盘 selectedGalleryId=undefined
→ 用户 < 250ms 内切换二级菜单
→ cleanup 函数 clearTimeout → 落盘被取消
→ 磁盘上残留旧 id → 下次进入被自动恢复
```

**正确做法**：代表"明确意图"的用户动作（返回、关闭、删除、取消）直接 `await persistPreferences(...)`，绕过防抖。写字段时显式写 `null` 或省略字段，要与主进程合并策略对齐（字段级合并需要主进程支持 `null` = 删除）。

### 6. 子窗口与主窗口共享偏好时要显式隔离

通过 `openSecondaryMenu(section, key, tab?, extra?)` 打开的子窗口会加载相同的页面组件，默认会读写同一份 `pagePreferences`。这会导致：

- 子窗口打开时被主窗口的偏好水合，显示的不是"显式传入的 id"。
- 子窗口里的操作会反向污染主窗口记忆。

解决方式：

- 显式参数传入子窗口要用 URL query / prop，不要借用 `pagePreferences`。
- 在子窗口模式下（通过 prop 判断），**禁用**相关字段的写回（新增一个 `disablePreferencesPersistence` 或等价 prop）。
- 子窗口里"返回"按钮的语义应当等价于 `window.close()`，不是 App 内 pop。

### 7. 自动恢复 vs 主动关闭要分清

"下次进入时自动恢复到上次打开的详情"这类记忆行为，必须有明确口径区分两种离开方式：

- **主动返回/关闭**：用户显式不想再看 → 下次不恢复 → 偏好字段应被清。
- **未主动离开就切走**：应用关闭、切到别的 section → 下次还能回到原位。

产品上可以只保留其中一种，但不能让行为在两种语义之间摇摆。

## 实施自查清单

- [ ] 一级菜单切换时，如果目标 subKey 在 `pinnedItems` 中，我恢复了 `activePinnedId`，而不是一刀切 null。
- [ ] 外层容器 key 不包含会在一级菜单切换时变化的字段，或已按多层叠加结构重构。
- [ ] "返回 / 关闭 / 删除"这类主动动作直接同步写盘，没有依赖防抖 effect。
- [ ] 子窗口模式下，不会写回可能污染主窗口记忆的偏好字段。
- [ ] "自动恢复"字段的清空时机有明确产品口径，不会被半吊子生命周期留下幽灵状态。
- [ ] 新增 `pagePreferences` 字段时，对应的水合 + 保存 + 清空三处都写到位。
