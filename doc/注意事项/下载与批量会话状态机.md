# 下载与批量会话状态机

## 文档定位

约束下载、批量下载、收藏标签下载等"长任务"的状态语义，避免"任务已存在"被误触发、"暂停被打成失败"、"并发无上限"这一类反复出现的状态机问题。所有长任务本质上都有**任务模板**和**运行时会话**两层概念，二者要严格区分。

## 两层模型

- **任务模板**（template / config）：`bulk_download_tasks` 这类"配置行"。同一 `path + tags` 只需要一份，去重是为了避免插入重复配置。
- **运行时会话**（session / run）：`bulk_download_sessions` 这类"本次执行"的行。一个任务模板可以对应多次会话（历史、重试、重新下载）。

两层的去重和生命周期**完全不同**，写判定逻辑时必须分清楚。

## 约定

### 1. 去重粒度不能合并

- "任务模板已存在" ≠ "下载已经在跑"。拿到 `deduplicated: true` 只代表"没插入新的配置行"，**不代表**要阻止新会话创建。
- 正确判定"是否拒绝重启"的条件是：该任务**是否存在存活会话**（`pending` / `queued` / `dryRun` / `running` / `paused`）。
- 没有活跃会话就应该允许复用模板、创建新会话并启动。

### 2. 存活状态、槽位状态和终态要分清

下载 / 批量下载 / 收藏标签下载至少有三类状态语义：

```
pending | queued | dryRun | running | paused  — 存活态 / 同 taskId 互斥
dryRun | running                            — 实际占用批量下载并发槽位
completed | failed | cancelled | allSkipped  — 终态
suspended                            — 系统级暂停（开关 / 关机前）
```

前端过滤（如 `BooruBulkDownloadPage.tsx` 的 `activeSessions`）、后端同 taskId 互斥判定、并发槽位计数、通知触发集合不是同一个问题，不能混用字符串列表。当前代码中 `countActiveSessions()` 只数 `dryRun | running`，而 `createBulkDownloadSession()` / `ensureCanEnterRunning()` 的互斥会把 `queued` 也视为存活态。

### 3. 并发上限必须闭环

如果需要并发上限（例如批量下载会话最多 3 个），**不能只在入口加计数**，要五处同时落地：

1. 新增 `queued` 状态到 `BulkDownloadSessionStatus`。
2. `StatusTag` 映射 `queued → "等待中（排队）"` 类似文案。
3. 前端活跃会话过滤集合要包含 `queued`。
4. 后端 `startSession` 入口查 `countActiveSessions`，超限则写 `queued` 直接返回。
5. 所有"离开活跃态"的出口（完成 / 暂停 / 取消 / 错误 catch）都要调 `promoteNextQueued` 推进队首；进程重启恢复逻辑也要走同一闸门。

缺任一步就会留"死排队"会话——DB 里是 `queued` 但没有调度器会把它推起来。

### 4. 进入 running 前要有统一看门

`startSession`、`retryAllFailed`、`retryFailedRecord`、启动恢复都可能把会话推进到运行态。它们必须复用同一个“进入 running 前的看门”：

- 同 taskId 已有存活会话时，不能再启动第二个运行时会话。
- 历史会话重试遇到存活会话时，允许返回 `merged: true` / 软删历史会话，而不是制造重复会话。
- 启动恢复遇到同 taskId 多条异常 running 记录时，应只保留 keeper，其余回落到 `paused` 或软删。

这类逻辑的当前入口是 `ensureCanEnterRunning()`，不要在某个调用点重新写一套“查状态然后启动”。

### 5. 用户主动中断的标志是权威，不要做二次校验

当代码里已经存在"用户主动暂停 / 取消"的显式标志（例如 `downloadManager.userInterruptedStatuses`），错误处理分支就应该**完全信任它**，不要再用错误消息字符串匹配（`isAbortError` 之类）做二次判定。

反模式：

```ts
// ❌ 字符串匹配太窄，'canceled' / 'socket hang up' / 'ECONNRESET' 都可能 fallthrough 到 failed
if (interruptedStatus && isAbortError(errorMessage)) { ... }
else { updateDownloadStatus(queueId, 'failed', errorMessage); }
```

正确：

```ts
if (interruptedStatus) {
  // 用户主动中断，状态已由 pause/cancel 入口写好，不要覆盖
  this.userInterruptedStatuses.delete(queueId);
  return;
}
```

### 6. 不要让长任务阻塞 UI 反馈

`startSession` 这类"内部含大段 dryRun / 扫描"的调用，会阻塞几秒甚至几十秒。前端不要等它 `await` 完成再刷新列表：

- `createSession` 成功后**立刻** `loadSessions()` 一次，让 `pending` 卡片先出现。
- 把 `startSession` 放后台 IIFE，或监听后端的会话状态变化事件，让 UI 自己流动。
- 收藏标签一键下载同样遵守这个规则：`startFavoriteTagBulkDownload()` 只等任务模板和会话创建完成就返回，并广播 `favorite-tag-download:created`；后台再启动会话和扫描。

用户看到"新会话卡片出现"和"卡片从 pending 变 running"这两步反馈，比"点完按钮十秒后突然冒出一条 running"要可预期得多。

### 7. 新增状态 / 新增开关必须做能力闭环

从 `Notification` 的例子（TP-10）来看，底层能力有了但 UI 入口没兜起来，会被当成"伪能力"。任何"新增开关、状态、动作"都需要以下五件事：

1. **Source of truth**：产品定义、默认值、触发时机写明。
2. **Service 层**：提供具体行为 / 判断函数。
3. **IPC + preload**：跨进程通道 + 类型声明。
4. **UI 入口**：Settings 或对应页面里看得见、可调。
5. **验收用例**：至少一条测试或人机观察记录。

遗漏任一步就是"只做一半"。

## 实施自查清单

改下载 / 批量下载 / 收藏标签下载相关逻辑时：

- [ ] 区分了"任务模板已存在"和"会话正在跑"两种语义。
- [ ] 区分了存活态、运行槽位态和终态，没有把三类状态混成同一个字符串列表。
- [ ] 如果引入并发上限，五处闭环都落了。
- [ ] 错误处理分支信任显式中断标志，没有用脆弱字符串匹配做二次校验。
- [ ] 长后端调用前后有"立刻一次" + "完成后一次"两次 `loadSessions`。
- [ ] 跨页面刷新通过 `system.onAppEvent` 等广播事件消费，避免一个页面直接改另一个页面的状态。
- [ ] 新增状态 / 开关同时覆盖 service、IPC、UI、测试四层。
