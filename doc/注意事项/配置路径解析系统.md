# 配置路径解析系统

## 核心原则

路径解析不依赖编译输出目录，也不默认依赖项目根目录，而是围绕配置根目录和数据目录两层展开。

## 解析链路

1. `.env` 的 `CONFIG_DIR` 决定配置根目录。
2. `config.yaml` 从配置根目录读取。
3. `config.yaml` 的 `dataPath` 决定运行数据目录。
4. 数据库、缩略图、缓存、日志再分别解析。

## 两类路径函数

- `resolveConfigPath()`：适合数据库等与 `config.yaml` 同级的文件。
- `resolveDataPath()`：适合缩略图、缓存、日志等运行数据。

## 缓存目录归属

不要把所有“缓存”混成一个业务概念。当前至少有三类缓存，清理入口和影响范围不同：

- **Booru 原图缓存**：由 `imageCacheService` 管理，路径来自 `getCachePath()`，通常位于运行数据目录下的 `cache`。`booru.getCacheStats()` / `booru.clearCache()` 只作用于这一类缓存。
- **本地图库缩略图缓存**：由 `thumbnailService` 管理，路径来自 `getThumbnailsPath()`，通常位于运行数据目录下的 `thumbnails`。它不在 Booru 原图缓存统计和清理范围内。
- **Booru 标签缓存**：存储在数据库表（如 `booru_tags`）中，统计和清理由 `getTagCacheStats()` / `cleanExpiredTags()` 这类接口负责，不属于磁盘原图缓存。

因此 UI 文案应明确写成“Booru 原图缓存”或“Booru 图片缓存”。如果放在全局设置里写“缓存管理”，用户会误以为会清理缩略图、标签缓存或其它应用缓存。

## 这样设计的原因

- 避免 `__dirname` 在打包后进入只读或不可预期位置。
- 避免 `process.cwd()` 在不同启动方式下不稳定。
- 允许用户把配置和运行数据迁移到自定义目录。
