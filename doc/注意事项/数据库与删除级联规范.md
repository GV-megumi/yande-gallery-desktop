# 数据库与删除级联规范

## 文档定位

约束数据库查询与删除动作的一致性，避免"SELECT 不存在的列"、"删完留垃圾"、"FK 级联不到位"这类问题。schema 源头以 `src/main/services/database.ts` 为准。

## 核心原则

### 1. SELECT 的列必须存在于当前 schema

`images`、`galleries`、`invalid_images`、`booru_posts` 等表字段不完全一致，**跨表复制查询很容易带错列**。

- 写或改 `SELECT` 前，先回到 [database.ts](../../src/main/services/database.ts) 找到对应表的 `CREATE TABLE`，逐列核对。
- 典型易错点：
  - `images` **没有** `thumbnailPath` 列——它只在 `invalid_images` 里。普通 images 的缩略图路径由 `thumbnailService.getThumbnailPath(imagePath)` 按哈希计算得到，不落库。
  - `galleries` 没有直接的 `images` 外键，图集与图片的关联是**隐式的**（`images.filepath` 前缀匹配 `galleries.folderPath`）。
- 写动态补列迁移时，同步更新所有相关 SELECT；不要让"schema 已经加了字段但查询还按旧版"。

### 2. 删除操作必须级联清理关联数据

**隐式关系不会自动级联**。当一个实体被删除，所有依赖它的数据都应显式清理：

- DB 行：查出归属范围后 `DELETE FROM ...`，不要寄希望于"过一阵自己会被清"。
- 磁盘文件：原图不动，但**缩略图、临时文件、下载 part 文件**等缓存要一并删。
- 间接字段：`booru_posts.downloaded/localPath`、`favorite_tag_download_bindings.lastStatus` 这类"跨表软引用"要手动复位。
- 页面偏好里残留的 id（例如 `pagePreferences.gallery.galleries.selectedGalleryId`）也要一起清，否则用户下次进入还会被自动还原。

### 3. 磁盘清理走统一工具函数，不要直接 `fs.unlink`

- **缩略图清理**：用 [thumbnailService.deleteThumbnail(imagePath)](../../src/main/services/thumbnailService.ts)。它会按 hash 反推路径并吞 ENOENT，调用方不必再 try/catch。
- **下载临时文件**：用下载服务里已有的 `buildDownloadTempPath` + unlink 模板。
- 自己拼路径 + `fs.unlink` 的写法要警惕：一旦 hash 规则或路径布局改了，清理链立刻断。

### 4. 不要假设字段命名跨表一致

- `invalid_images.thumbnailPath`、`booru_posts.fileUrl`、`images.filepath`——同一概念在不同表里名字不一样。
- 写跨表查询 / 迁移脚本时显式起别名；不要在业务代码里做"反正都叫 xxx"这种推断。

### 5. 删除动作与扫描/恢复逻辑要成对设计

删除不是孤立事件，常伴随一个对应的"再发现/恢复"路径：

- 删图集 → 扫描根文件夹时会立刻把它再扫出来重建 → 需要"忽略名单"配合（`gallery_ignored_folders`）。
- 删下载会话 → 启动时 init 阶段会遍历 pending/running 会话恢复 → 需要显式把被删的会话置 `deletedAt` 或状态置 `cancelled`。
- 做删除功能时，同步检查：**是否有定时任务、扫描、启动恢复逻辑会把它再拉回来**；有就一并堵住。

## 实施自查清单

改动涉及 DB 删除 / schema / 磁盘缓存时，逐条过：

- [ ] 我引用的所有列都在当前 `database.ts` 的对应 CREATE TABLE 里。
- [ ] 删除主实体时，与它关联的 DB 行（FK `ON DELETE SET NULL` 的表也要判断残余是否要清）、磁盘缓存、偏好 id 都已处理。
- [ ] 磁盘清理走的是封装好的工具函数，不是裸 `fs.unlink`。
- [ ] 确认过"扫描/启动恢复"这类周期逻辑不会把被删的数据再拉回来。
- [ ] 删除流程在失败时有日志、不会半残（比如 DB 删了但磁盘还在）；如果无法原子化，至少做成"先 DB 再磁盘"顺序，保证下次启动能接着清。
