# 性能约束与反模式

## 文档定位

本文件是 `doc/性能审计报告.md` 的摘要入口，用来列出当前最值得避免的几类问题。

## 高风险反模式

- 主进程中使用同步文件系统 API，阻塞事件循环。
- 批量数据库写入不包事务。
- 下载、预加载、批量任务使用松散计数器而不是稳定并发控制。
- 页面层无限累积缩略图、预览上下文或未取消的异步任务。
- 重复排序、大量内联样式、缺少 memo / callback 导致无意义重渲染。
- 缓存页恢复时把列表完整清空再重拉，导致瀑布流旧块重新排版、缩略图重复加载、滚动位置丢失。

## 图库列表性能边界

- 最近图片页完整进入时可以重载 `getRecentImages(2000)`；从 `suspended` 恢复时只能做游标增量查询。
- 最近图片旧数据按 200 张历史块冻结，新增图片按 20 张增量块冻结。新增图片不能合并进旧历史块。
- 图集详情页按 200 张可见窗口增量渲染；加载更多只追加新批次，旧批次不能因为新增图片重新参与瀑布流排版。
- 图集详情切换排序字段或升降序时，应回到顶部并把可见窗口重置为 200，不能对已经加载出的 600 / 1000 张图片一次性重排。
- `ImageGrid` 的批次策略要按排序类型区分：时间排序先全局排序再切批次；文件名排序先切批次再批内排序，以稳定旧批次成员。
- `ImageGrid` 缩略图缓存应以可见图片 id 为边界做增量加载；列表增加新块时不得清空旧图片的缩略图映射。
- 用户显式点击刷新按钮时可以完整重载并重排；自动恢复、菜单切回、固定项切回不属于显式刷新。
- 批量下载页的手动刷新按钮只表示用户点击刷新。事件驱动刷新、轮询刷新和任务创建后的后台同步不能驱动该按钮 loading，否则会造成“按钮自动闪烁”的误导。
