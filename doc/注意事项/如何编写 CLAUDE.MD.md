# 如何编写 `CLAUDE.MD`

## 1. 文档定位

`CLAUDE.MD` 是仓库根目录下提供给 Claude / AI coding agent 的项目级入口文档。

它的主要职责是：

1. 用最短篇幅说明项目是什么；
2. 提供跨任务适用的开发规则；
3. 给出最常用的开发命令；
4. 提供关键目录和模块的高层导航；
5. 指向更详细、可持续维护的正式文档。

因此，`CLAUDE.MD` 应被视为：

- 入口文档；
- 约束文档；
- 导航文档。

它不应被视为：

- 功能总手册；
- 数据库参考手册；
- Renderer API 手册；
- 历史实现记录；
- 路线图或 TODO 汇总。

## 2. 核心原则

### 2.1 只保留长期稳定的信息

`CLAUDE.MD` 只应收纳那些跨多数任务都成立、且短期内不易变化的信息，例如：

- 项目定位；
- 主技术栈；
- 核心开发命令；
- 项目级开发规范；
- 关键目录职责；
- 文档导航；
- source of truth 说明。

如果某类信息会随着功能迭代频繁变化，就不应展开写在 `CLAUDE.MD` 中。

### 2.2 只写 Claude 不能稳定推断的信息

如果某条内容 Claude 通过阅读代码、标准工具链约定或已有文档即可较稳定地推断出来，则不应在 `CLAUDE.MD` 中重复描述。

应优先写入的，是以下内容：

- 本仓库特有的开发约束；
- 不阅读专题文档就容易踩坑的规则；
- 某类任务的实际 source of truth；
- 某类问题应优先查看的文档入口。

### 2.3 以导航替代复制

当某个主题已经在 `doc/` 中有完整说明时，`CLAUDE.MD` 应只保留简短原则和链接，不应复制其正文内容。

推荐写法是：

1. 用一句话说明规则；
2. 明确相关文档路径；
3. 把细节维护在 `doc/` 中。

这样可以降低重复维护和信息漂移的风险。

### 2.4 以信息密度为目标，而不是以篇幅堆积为目标

`CLAUDE.MD` 会成为进入仓库时的默认上下文之一。文件越大，越容易稀释真正重要的规则。

因此，判断某段内容是否应该写入 `CLAUDE.MD`，建议优先问两个问题：

1. 这条信息是否跨多数任务都适用？
2. 缺少它是否会显著增加误改、误判或重复探索的概率？

若不能同时满足，通常不应放入根级 `CLAUDE.MD`。

## 3. 推荐结构

建议 `CLAUDE.MD` 长期保持以下结构：

### 3.1 项目简介

用一小段说明：

- 项目是什么；
- 核心技术栈是什么；
- 当前详细说明以 `doc/` 为准。

这一节只建立最小上下文，不展开功能细节。

### 3.2 核心命令

只保留最常用、最稳定的命令，例如：

- `npm install`
- `npm run dev`
- `npm run test`
- `npm run build`
- `npm run rebuild`
- `npm run dist`

如果某类问题有固定排查入口，可以在这一节附上对应文档。

### 3.3 架构速览

用短篇幅描述：

- `src/main/` 的职责；
- `src/renderer/` 的职责；
- `src/preload/` 的职责；
- `src/shared/` 的职责。

如果要提到 API 暴露面，只写主要分域，不写全量接口清单。

### 3.4 开发规范

这是 `CLAUDE.MD` 的核心部分，只保留项目级、跨任务、长期复用的规则。例如：

- 外部网络请求必须走主进程；
- 修改前必须先查看定义、调用和相关类型；
- 共享类型优先放在共享层；
- 可复用 UI 能力优先抽象；
- 路径与数据目录不能凭经验假设；
- 数据库 schema 以实际代码为准；
- Renderer API 以 preload 为准；
- Booru 相关修改必须先确认站点差异。

每条规则都应尽量满足以下要求：

- 可独立理解；
- 可直接执行；
- 含义明确；
- 最好附带相关文档入口。

### 3.5 按任务查文档

这一节建议长期保留。

建议写成“任务类型 -> 文档入口”的形式，例如：

- 改数据库 -> `doc/数据库结构文档.md` + `src/main/services/database.ts`
- 改 Renderer API -> `doc/Renderer API 文档.md` + `src/preload/index.ts`
- 改 Booru -> `doc/Booru功能实现文档.md` + 对应注意事项文档

它的主要价值是减少无效探索，并把 Claude 导向真正的 source of truth。

### 3.6 文档导航

用于说明 `doc/` 的整体分层，通常包括：

- 当前主文档；
- 专题注意事项；
- 历史归档。

它应是总览入口，而不是详细目录。更细的索引由 `doc/README.md` 负责。

### 3.7 冲突优先级说明

建议在文末明确声明：

- `CLAUDE.MD` 与 `doc/` 冲突时，优先以 `doc/` 和实际代码为准；
- `doc/` 与代码冲突时，优先以代码为准，并在修改后同步更新文档；
- `CLAUDE.MD` 应保持精简，不承载详细手册型内容。

## 4. 适合放入 `CLAUDE.MD` 的内容

以下内容通常适合写入：

1. 项目一句话定位；
2. 主技术栈；
3. 最常用命令；
4. 高层目录职责；
5. 跨模块开发规范；
6. 高价值、非显然的仓库约束；
7. 各类任务对应的文档入口；
8. source of truth 的明确声明。

这些内容的共同特点是：

- 不容易过时；
- 对多数任务有帮助；
- 无需大段上下文即可理解；
- 可以直接减少误操作。

## 5. 不适合放入 `CLAUDE.MD` 的内容

### 5.1 详细功能说明

例如：

- 全量功能列表；
- 页面能力逐项枚举；
- 复杂交互行为说明；
- 某个子系统的完整能力矩阵。

这些内容应放在 `README.md`、`doc/功能总览.md` 或子系统专题文档中。

### 5.2 详细数据库与 API 参考

例如：

- 字段表格；
- 外键、索引、约束细节；
- 全量 `window.electronAPI` 方法清单；
- IPC 通道清单。

这些内容应放在 `doc/数据库结构文档.md`、`doc/Renderer API 文档.md` 以及对应代码中。

### 5.3 容易漂移的实现细节

例如：

- 完整目录树；
- 页面内部组件清单；
- service 的逐函数介绍；
- 某阶段的临时实现说明。

这类信息变化频繁，不适合作为根级常驻说明。

### 5.4 历史记录、路线图与未来计划

例如：

- 已完成任务流水账；
- 旧方案讨论；
- TODO 细项；
- 未来功能规划。

这些内容应放在 `TODO.md`、`doc/done/` 或专门的计划文档中。

### 5.5 教程型内容与大段示例

例如：

- 配置大样例；
- 用户操作教程；
- 安装排障教程；
- 大段代码示例。

`CLAUDE.MD` 不应承担教程文档职责。

## 6. 与其他文档的分工

### 6.1 与 `README.md` 的分工

- `README.md` 面向项目读者、开发者和使用者；
- `CLAUDE.MD` 面向 Claude / AI coding agent，同时也帮助开发者快速建立开发上下文。

通常 `README.md` 可以承担更多项目介绍和能力概览，而 `CLAUDE.MD` 应更偏向规则和导航。

### 6.2 与 `doc/` 的分工

- `doc/` 是详细说明的正式位置；
- `CLAUDE.MD` 只负责引导到这些说明。

可以把 `CLAUDE.MD` 理解为 `doc/` 的入口层，而不是替代层。

### 6.3 与代码的分工

当某类信息已经由代码天然表达且代码才是最终事实来源时，`CLAUDE.MD` 只需要指出“去哪里看”，不应复制实现细节。

例如：

- 数据库 schema -> `src/main/services/database.ts`
- Renderer API -> `src/preload/index.ts`
- IPC 通道 -> `src/main/ipc/channels.ts`

## 7. 维护要求

### 7.1 控制篇幅

`CLAUDE.MD` 应尽量控制在约 200 行以内。

超过这一范围时，应优先考虑：

1. 是否有内容本应归入 `doc/`；
2. 是否有内容只是重复已有文档；
3. 是否可以改成一句规则加一个链接；
4. 是否可以压缩同类导航项。

### 7.2 优先删除重复，而不是继续堆叠

当某条内容已在 `doc/` 中有更准确版本时，更新 `CLAUDE.MD` 的优先动作不是复制过来，而是：

- 删除旧描述；
- 改成简短原则；
- 添加文档入口。

### 7.3 冲突处理优先级

维护时建议遵循以下优先级：

1. 代码；
2. 当前 `doc/` 主文档；
3. `CLAUDE.MD`。

### 7.4 仅在项目级规则变化时更新

不要因为局部功能新增、页面微调、某个 API 新增一个方法，就立即往 `CLAUDE.MD` 堆信息。

通常只有以下情况才适合更新它：

- 项目级开发规范改变；
- 常用命令改变；
- 关键目录职责改变；
- 文档结构改变；
- 某个跨任务都会踩的坑需要升级为根级规则。

## 8. 质量标准

一份合格的 `CLAUDE.MD` 应满足：

### 8.1 可快速阅读

第一次进入仓库时，应能在几分钟内读完并建立正确心智模型。

### 8.2 可直接执行

其中的规则应能直接转化为行为，而不是口号式描述。

例如：

- “数据库结构以 `src/main/services/database.ts` 为准”是可执行的；
- “注意数据库问题”则不可执行。

### 8.3 不与正式文档争夺事实来源

它必须明确承认：详细说明在 `doc/`，实现真相在代码。

### 8.4 低漂移风险

如果某段内容很可能在未来几周或几个月里频繁变化，就不应写入 `CLAUDE.MD`。

### 8.5 对当前仓库有效

不要写泛泛的 AI 提示词。`CLAUDE.MD` 必须服务于当前仓库的真实结构、真实约束和真实工作流。

## 9. 当前仓库的建议边界

对于本仓库，`CLAUDE.MD` 建议长期只保留：

- 项目简介；
- 核心命令；
- 架构速览；
- 开发规范；
- 按任务查文档；
- 文档导航；
- 冲突处理说明。

以下内容不应再放回 `CLAUDE.MD`：

- 大段功能总览；
- 长目录树；
- 字段级数据库说明；
- 详细 Renderer API 清单；
- 详细配置示例；
- 历史实现记录；
- TODO / 路线图 / 未来规划。

## 10. 修改检查清单

修改 `CLAUDE.MD` 前后，至少检查以下问题：

1. 新增内容是否属于长期稳定的信息？
2. 新增内容是否属于跨任务通用的信息？
3. 该内容是否已经在 `doc/` 中有更详细版本？
4. 是否可以改写为“一句规则 + 一个文档链接”？
5. 修改后是否仍保持结构清晰、导航完整？
6. 是否引入了新的重复、漂移风险或历史包袱？
7. 如果与代码不一致，是否已经同步修正？

如果上述问题中有多项答案不理想，应优先继续压缩内容或将其迁移到 `doc/`。

---

本说明文件的目标不是约束写作措辞本身，而是确保根目录下的 `CLAUDE.MD` 长期保持：

- 精简；
- 稳定；
- 可维护；
- 以规则和导航为主；
- 不与 `doc/` 和代码争夺事实来源。
