# 图库懒加载设计

## 核心原则

图库列表先展示元数据，图片扫描和导入按需触发，不在应用启动时把所有目录一次性扫完。

## 这样设计的原因

- 大图库启动扫描会明显拖慢应用进入速度。
- 图库页更适合先给用户可浏览的结构，再逐步加载内容。
- 封面和缩略图优先展示，可以减少初次渲染成本。

## 设计要点

- `galleries` 表存目录元数据和缓存统计。
- 真正的图片导入在用户进入图库或触发扫描时进行。
- 图集封面优先使用缩略图，而不是原图。

## 最近图片页：缓存、增量与分块

最近图片页同时承担“快速回到原位置”和“及时发现新图”两个目标，所以必须区分完整加载和缓存恢复。

### 触发条件

- **完整加载**：首次进入最近图片页，或最近图片页已被图库二级菜单切换释放后再次进入。
  - 行为：调用 `getRecentImages(2000)`，清空新增块和待查看新增队列，重置可见数量为 200。
- **缓存恢复**：最近图片页仍在 `mountedPageIds` 缓存层中，只是从 `suspended=true` 恢复到可见。
  - 行为：不调用 `getRecentImages(2000)`，不清空旧缩略图，不重置 `recentVisibleCount`，只调用 `getRecentImagesAfter(updatedAt, id, limit)` 查询新增图片。

### 游标规则

- 游标取当前页面顶部第一张图片：
  - 如果已经有新增块，取第一个新增块的第一张。
  - 否则取原始最近列表的第一张。
- 游标字段是 `updatedAt + id`，主进程按 `(updatedAt > cursor.updatedAt) OR (updatedAt = cursor.updatedAt AND id > cursor.id)` 查询。
- 排序必须保持 `updatedAt DESC, id DESC`，避免同一时间戳下的新增图片顺序不稳定。
- 增量查询返回数量等于 `limit` 时，必须用该页最后一张图片作为 `beforeUpdatedAt + beforeId` 继续查询下一页，直到返回少于 `limit`。
- 如果最近图片页缓存恢复时没有顶部游标，且当前没有历史块、新增块和待查看队列，说明这是空缓存页；此时允许回退到一次完整 `getRecentImages(2000)`。
- 渲染层插入前要按 `image.id` 去重，去重范围包括原始最近列表、已有新增块和待查看队列。

### 分块规则

最近图片页有两类块：

- **历史块**：完整加载得到的最近图片，按 200 张一批冻结。下拉继续显示 201-400、401-600 时，只追加新的历史块，不重排旧块。
- **新增块**：缓存恢复后查询到的新图片，放在历史块上方，按最多 20 张一块。

新增块的冻结规则：

- 顶部最多有一个“活动新增块”，数量小于 20 时可以继续接收新图片并局部重排。
- 活动块达到 20 张后冻结，再来的新图片会形成新的顶部活动块。
- 冻结后的新增块和历史块一样，不再因为后续新增而重新排版。

### 滚动位置规则

- 如果用户位于页面顶部附近，新增图片可以自动插入顶部新增块。
- 如果用户已滚动到较深位置，不立即插入，先显示“新增 N 张，点击查看”提示。
- 用户点击提示后再插入新增块并滚回顶部。

这样可以避免“用户正在看旧图片时，顶部插入新图导致当前视口跳动”。

### 明确禁止的反模式

- 缓存恢复时重新调用 `getRecentImages(2000)`。
- 为了插入新增图片而重建完整 `recentImages` 数组并让历史瀑布流重新分组。
- 把新增图片合并进旧日期组，导致旧日期块重新排版。
- 二级菜单已释放最近页缓存后仍强行套用增量逻辑。
- 只按 `updatedAt` 做游标，不用 `id` 处理同一时间戳的稳定顺序。
- 增量查询只取第一页就推进顶部游标，导致缓存期间新增超过一页时漏掉后续新增图片。
- 空缓存页恢复时因为没有游标而直接返回，导致导入第一批图片后页面仍然保持空白。
