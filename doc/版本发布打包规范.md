# 版本发布打包规范

本文件定义 Yande Gallery Desktop 的版本号、构建产物、发布流程和命名约定。所有发布动作都应按此执行。

## 适用范围

- 通过 `electron-builder` 生成的桌面安装包（Windows / macOS / Linux）
- 通过 GitHub Release 对外分发

不适用：

- 开发态 `npm run dev` / `npm run build`（这些只做本地运行或 Renderer 产物编译，不落盘到 `dist-v*`）
- 内部调试用的 `--dir` 目录（`dist/win-unpacked/` 等）

## 版本号约定

- 使用 semver 三段式 `major.minor.patch`
- `0.x.y` 阶段视为 prerelease，GitHub Release 勾选 `--prerelease`
- `1.0.0` 之后再移除 prerelease 标记
- 版本号唯一来源是 `package.json` 的 `version` 字段，`git tag`、构建目录、安装包名都从这里派生

## 构建输出目录

**目录命名：** `dist-v<版本号>`

```text
dist-v0.0.2/
  yande_gallery_win_x64_v0.0.2_setup.exe
  release-notes.md
  ...
```

**规则：**

- 每次发布构建都输出到独立的 `dist-v<版本号>` 目录，不复用 `dist/`
- `dist/` 留给日常 `electron-builder --dir` 调试使用，不进发布流程
- `dist-v*` 已在 `.gitignore` 中忽略，不随源码提交
- 构建命令通过 `-c.directories.output=dist-v<版本号>` 覆盖默认输出目录，避免占用仍被打开应用锁住的 `dist/`

**标准构建命令（Windows）：**

```bash
npx electron-builder --win -c.directories.output=dist-v0.0.2 --publish=never
```

（把 `0.0.2` 替换成 `package.json` 里当前的版本号）

发布前要先 `npm run build` 产出 `build/main`、`build/preload`、`build/renderer`，再跑上面的打包命令。`npm run dist:win` 等脚本用于非发布构建（输出到 `dist/`），不要直接用来做发布。

## 安装包命名约定

**格式：** `yande_gallery_<平台>_<架构>_v<版本>_setup.<扩展名>`

| 平台 | 架构 | 扩展名 | 示例 |
|------|------|--------|------|
| `win` | `x64` / `ia32` / `arm64` | `exe` | `yande_gallery_win_x64_v0.0.2_setup.exe` |
| `mac` | `x64` / `arm64` | `dmg` | `yande_gallery_mac_arm64_v0.0.2_setup.dmg` |
| `linux` | `x64` / `arm64` | `AppImage` | `yande_gallery_linux_x64_v0.0.2_setup.AppImage` |

**落实方式：** `package.json` 的 `build.win / build.mac / build.linux` 下各自配置 `artifactName` 模板：

```json
"artifactName": "yande_gallery_win_${arch}_v${version}_setup.${ext}"
```

修改版本后不需要手动改这个模板；`${version}` 和 `${arch}` 会由 electron-builder 注入。

## 发布上传内容

**只上传两样东西：**

1. 对应平台的安装包（如 `yande_gallery_win_x64_v0.0.2_setup.exe`）
2. 源码（GitHub Release 自动附带的 `Source code (zip)` / `Source code (tar.gz)`，不需要手动上传）

**显式不上传：**

- `*.exe.blockmap` / `*.dmg.blockmap`（electron-builder 生成的增量更新元数据）
- `latest.yml` / `latest-mac.yml` / `latest-linux.yml`（auto-updater 通道文件）
- `dist-v<版本>/win-unpacked/` / `mac/` / `linux-unpacked/` 等未打包产物
- 构建缓存、调试日志、`release-notes.md` 原文件（正文已经写进 release body，不需要额外传）

理由：当前没有接 auto-updater，blockmap / latest.yml 只会把发布页变成一堆无人用的附件；未打包目录体积巨大且不应当作分发物。

## 发布流程

按顺序执行：

1. **预检**
   - 工作区干净：`git status` 无未提交变更
   - 当前在 `master` 分支且与远端一致
   - 相关文档已经跟新版本对齐（Renderer API、Booru、功能总览等）

2. **版本号升级**
   ```bash
   # 修改 package.json 里的 version 字段，然后：
   git add package.json
   git commit -m "chore: 版本号升到 0.0.2"
   ```

3. **写发布说明**
   - 文件路径：`dist-v<版本号>/release-notes.md`
   - 语言：中文
   - 结构参考历史 release：更新内容（按功能分组）→ 安装 → 已知问题
   - 只是 release body 的源文件，不作为附件上传

4. **构建安装包**
   ```bash
   npm run build
   npx electron-builder --win -c.directories.output=dist-v0.0.2 --publish=never
   ```
   - 构建前先关闭仍在运行的旧版应用实例，避免 `app.asar` 被锁
   - 构建完成后确认 `dist-v0.0.2/yande_gallery_win_x64_v0.0.2_setup.exe` 存在、可双击安装

5. **打标签并推送**
   ```bash
   git tag v0.0.2
   git push origin master
   git push origin v0.0.2
   ```

6. **创建 GitHub Release**
   ```bash
   gh release create v0.0.2 \
     dist-v0.0.2/yande_gallery_win_x64_v0.0.2_setup.exe \
     --title "Yande Gallery Desktop v0.0.2" \
     --notes-file dist-v0.0.2/release-notes.md \
     --prerelease
   ```
   - 标题格式：`Yande Gallery Desktop v<版本号>`
   - `< 1.0.0` 的版本加 `--prerelease`
   - 只上传安装包这一个附件
   - `gh` 会把文件名里的下划线转成点号显示，但实际下载名保留原样，这点不必处理

7. **发布后核对**
   - 在 GitHub Release 页面确认附件只有 1 个安装包
   - 确认 release body 渲染正常
   - 确认 tag 指向刚刚 bump 版本号的 commit

## 跨平台补充

当前默认只出 Windows 安装包。如果需要 mac / linux：

- `mac`：需要在 macOS 宿主机执行 `npx electron-builder --mac -c.directories.output=dist-v<版本>`；产出 `yande_gallery_mac_<arch>_v<版本>_setup.dmg`
- `linux`：宿主机或 Docker 里执行 `npx electron-builder --linux -c.directories.output=dist-v<版本>`；产出 `yande_gallery_linux_<arch>_v<版本>_setup.AppImage`
- 发布到同一个 `v<版本>` tag 即可，`gh release upload v<版本> <文件>` 追加上传

目前 CI 里没有 mac / linux 构建任务，这两个平台的产物靠有需要时手动补传。

## 清理策略

- 每次发布完成后，`dist-v<老版本>` 可以保留到本地作备份，但不强制
- `dist/` 是日常调试产物，随时可以删除或覆盖
- 不要把 `dist-v*` 里的任何东西加入 git 追踪

## 常见误区

1. **用 `npm run dist:win` 当作发布命令**
   - 该脚本把产物落到 `dist/`，会和正在运行的旧版应用互锁，而且命名是默认的 `Yande Gallery Setup <版本>.exe`，不符合命名规范。发布一律走 `-c.directories.output=dist-v<版本>` + `artifactName` 模板。

2. **连同 blockmap / latest.yml 一起传上去**
   - 这些是 auto-updater 需要的文件，当前项目没有接入 auto-updater，上传只会增加噪音。见"发布上传内容"章节。

3. **amend 已经发出的 tag commit**
   - 版本号 bump 的 commit 一旦打了 tag 并推送，就不要再 amend。发现问题直接 bump patch 版本重来。

4. **忘记先 `npm run build`**
   - `electron-builder` 只会把 `build/**` 和 `node_modules/**` 打进 asar；没跑 `npm run build` 的话打出来的是老代码。
