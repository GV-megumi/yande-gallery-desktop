# 架构总览

## 项目定位

Yande Gallery Desktop 是一个基于 Electron + React + TypeScript 的桌面应用，核心目标是同时服务两类场景：

1. 本地图库管理：扫描、检索、分组、缩略图、预览与标签管理。
2. Booru 浏览与下载：统一接入 Moebooru、Danbooru、Gelbooru 等站点，支持搜索、收藏、下载、缓存和批量任务。

## 整体分层

### 主进程 `src/main/`

- `services/`：数据库、配置、图库扫描、缩略图、下载、缓存、Booru 客户端等服务。
- `ipc/`：对外暴露统一 IPC 通道，承接渲染进程请求。
- `window.ts`：主窗口与子窗口创建、运行时配置、子窗口 IPC 处理（含 `window:open-secondary-menu`）。

### 渲染进程 `src/renderer/`

- `App.tsx`：主导航、固定页面、主题、快捷键、页面切换；基础页和固定页统一由 `mountedPageIds` 页面缓存层承载。
- `SubWindowApp.tsx`：子窗口应用入口，通过 URL hash 路由决定渲染页面（`tag-search` / `artist` / `character` / `secondary-menu`），不包含侧边栏。
- `pages/`：图库、Booru、Google、设置等页面。其中 `BooruDownloadHubPage` 和 `BooruTagManagementPage` 是通过 Segmented 组件实现页内子导航的 Hub 页面。
- `components/`：网格、详情区块、表单、浮层、排序菜单、`IconOnlyButton`（统一纯图标按钮 + tooltip）等复用组件。
- `utils/`：查询拼装、多选逻辑、标签关系、查看器控制等纯函数。

### 预加载层 `src/preload/`

- 使用 `contextBridge.exposeInMainWorld` 暴露受控 API。
- 主入口 [src/preload/index.ts](../src/preload/index.ts) 统一承接 `db.*`、`gallery.*`、`config.*`、`booruPreferences.*`、`pagePreferences.*`、`image.*`、`booru.*`、`bulkDownload.*`、`window.*`、`system.*` 等接口。
- 轻量子窗口（`tag-search` / `artist` / `character`）加载独立的精简 preload [src/preload/subwindow-index.ts](../src/preload/subwindow-index.ts)，只暴露 `window` / `booru` / `booruPreferences` / `system` 四个域；由 `src/main/window.ts` 按 URL hash 前缀在创建时选择不同 preload 脚本。二级菜单子窗口仍使用主 preload。
- 主 / 子窗口共享的 API 工厂集中在 [src/preload/shared/](../src/preload/shared/)（`createBooruApi` / `createBooruPreferencesApi` / `createWindowApi` / `createSystemApi`）。

#### 预加载 API 分域

- `db`：数据库初始化、本地图片查询
- `gallery`：图库 CRUD、封面、按目录读取、扫描导入、图集文件夹同步、无效图片管理、图集忽略名单
- `config`：配置读取、保存、重载、配置变更监听；包含 `notifications` / `desktop` 分域 getter/setter
- `booruPreferences`：Booru 外观偏好读取 + 订阅
- `pagePreferences`：按页面维度的持久化偏好（favoriteTags / blacklistedTags / gallery / appShell）
- `image`：扫描、缩略图生成 / 获取 / 删除
- `booru`：站点、帖子、标签、收藏、下载、缓存、Wiki、论坛、用户、评论、Pool、注释、版本历史、分组、保存搜索等
- `bulkDownload`：任务、会话、记录、重试、恢复
- `window`：子窗口管理，包括打开标签 / 艺术家 / 角色专用子窗口，以及在独立窗口中打开任意二级菜单页面（支持附加 `extra` query）
- `system`：文件夹选择、外链、显示文件、备份导入导出、网络测试、批量下载事件监听、系统导航事件监听
- Google Drive、Photos、Gemini 页面通过 webview 嵌入对应 Google 网站，不通过 preload API 接入

#### 预加载中的事件订阅

- `config.onConfigChanged`（主进程只广播 `ConfigChangedSummary` 摘要，preload 自动回调去敏 config）
- `booruPreferences.appearance.onChanged`（同样基于 `config:changed` 摘要事件，preload 自动拉取最新 appearance DTO）
- `booru.onFavoritesRepairDone`
- `booru.onDownloadProgress`
- `booru.onDownloadStatus`
- `booru.onQueueStatus`
- `system.onBulkDownloadRecordProgress`
- `system.onBulkDownloadRecordStatus`
- `system.onSystemNavigate`

也就是说，`window.electronAPI` 不只是 invoke 风格的方法集合，还承担了一部分主进程到渲染进程的事件桥接。

### 共享层 `src/shared/`

- 保存本地图库、Booru、批量下载、标签体系等共享类型定义。

## 当前模块版图

### 本地图库

- `imageService.ts`：图片导入、搜索、标签关联。
- `galleryService.ts`：图库 CRUD、封面、按目录扫描、图集文件夹同步、无效图片管理。
- `thumbnailService.ts`：缩略图生成与缓存。

### Booru 子系统

- `booruClientInterface.ts`：统一客户端接口。
- `moebooruClient.ts`、`danbooruClient.ts`、`gelbooruClient.ts`：站点适配层。
- `booruClientFactory.ts`：按站点类型分发客户端。
- `booruService.ts`：本地持久化、收藏、标签、保存的搜索、分组、黑名单等。
- `downloadManager.ts`、`bulkDownloadService.ts`：单任务与批量任务下载调度。
- `imageCacheService.ts`：Booru 图片缓存与统计。
- `backupService.ts`：应用数据导出与恢复。
- `imageMetadataService.ts`：查看器元数据读取。
- `networkScheduler.ts`：站点请求调度相关逻辑。
- `updateService.ts`：GitHub Releases 版本检查，供设置页"关于" Tab 的"检查更新"使用。
- `notificationService.ts`：基于 `Notification` 的桌面通知服务。消费 `notifications` 配置的三级开关（全局 enabled / 按会话终态 / 任务级 `notifications`）和 `clickAction`，被批量下载会话终态、单图下载完成等入口调用。

### 辅助集成

- 渲染层包含 `GoogleDrivePage.tsx`、`GooglePhotosPage.tsx`、`GeminiPage.tsx` 等页面入口，均通过 webview 嵌入对应 Google 网站，不依赖 Google API 或 OAuth 认证。

### 初始化流程

- `src/main/services/init.ts` 负责应用启动初始化。
- 启动顺序是：初始化路径 -> 加载配置 -> 确保目录存在 -> 初始化数据库 -> 从配置创建初始图库 -> 后台恢复下载和清理标签缓存。
- 后台恢复逻辑包括：恢复普通下载队列、恢复批量下载会话、清理过期标签缓存。

### 主进程启动行为

- `src/main/index.ts` 负责 Electron 主进程启动编排。
- 启动前会先注册 `app://` 自定义协议 scheme。
- 应用启用单实例锁；第二个实例会把焦点切回已有窗口。
- 硬件加速默认被禁用，用于规避部分渲染问题。
- `app.whenReady()` 后的执行顺序是：移除默认菜单 -> 注册受控 `app://` 文件协议 -> 创建主窗口和托盘 -> 注册 IPC / 窗口 IPC -> 调用 `initializeApp()` 做配置、数据库和图库初始化 -> 应用 `desktop.autoLaunch` / `startMinimized`。
- 开发环境下还会尝试安装 React DevTools。

### 桌面行为与系统托盘

- 由 `desktop` 配置决定：
  - `closeAction`：主窗口点 X 时是 `hide-to-tray`（缩到托盘）、`quit`（直接退出）、还是 `ask`（弹确认）
  - `autoLaunch`：是否开机自启（`app.setLoginItemSettings`）
  - `startMinimized`：自启时是否直接隐藏到托盘
- 托盘支持"显示主窗口 / 退出 / 暂停下载"等常用入口。
- `before-quit` / `will-quit` / `render-process-gone` / `uncaughtException` / `unhandledRejection` 已接入主进程，保证关闭和异常退出路径一致。
- 这些行为和通知开关的 source of truth 都在 `notifications` / `desktop` 配置段，参见 `doc/开发与配置指南.md` 和 `doc/Renderer API 文档.md`。

### IPC 说明

- `src/main/ipc/channels.ts` 是 IPC 通道常量清单。
- `src/preload/index.ts` 和 `src/preload/shared/*.ts` 是 Renderer 可见 API 的实际暴露面。
- `src/main/ipc/handlers.ts` 是多数通道的主进程处理入口；窗口类通道还要同时看 `src/main/window.ts`。
- 因此核对 IPC 能力时，应同时查看 channels、preload 暴露和 handler 实现，不要只看其中一个文件。

## 数据与存储

### 路径体系

- 配置根目录由 `.env` 中的 `CONFIG_DIR` 决定；未指定时默认落到用户目录下的 `.yandegallery`。
- `config.yaml` 位于配置根目录。
- `dataPath` 决定运行数据目录，数据库、缩略图、缓存、日志围绕它解析。

相关说明见：

- `doc/开发与配置指南.md`
- `doc/注意事项/配置路径解析系统.md`
- `doc/注意事项/数据迁移规则.md`

### 数据库

- SQLite 是主数据存储。
- `src/main/services/database.ts` 是当前 schema 的实际来源。
- 本地图库、Booru 记录、批量下载、收藏标签、黑名单、保存的搜索等都持久化在同一个数据库中。

详情见 `doc/数据库结构文档.md`。

## 关键设计原则

### 1. 外部网络请求走主进程

渲染进程不直接请求外部站点，统一通过 IPC 交给主进程执行，以规避 CORS、集中处理代理和错误。

### 2. 组件抽象优先于页面复制

图片网格、列表包装、Booru 详情区块、批量下载卡片等能力尽量以可复用组件沉淀。

### 3. 本地图库和 Booru 记录并行存在

本地 `images` 与 Booru `booru_posts` 分开建模，但通过 `localImageId` 等字段建立关联，兼顾本地管理与远端元数据。

### 4. 配置与运行数据分离

路径解析不再依赖编译目录，减少打包后路径失效和不可写问题。
