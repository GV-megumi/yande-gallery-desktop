# 00-重构文档总览与使用说明

## 1. 文档目的

本目录下的文档用于把 `审查报告.md` 中已经确认的问题、优先级、证据链和路线图，重组为一套可以直接支撑后续实施的重构说明体系。

这套文档要解决的不是“怎么把报告拆开更整齐”，而是以下四个执行问题：

1. 后续到底应该先修什么、后修什么
2. 哪些问题可以并行推进，哪些必须共享前置改造
3. 把任务交给不同智能体时，每个智能体应该看哪份文档开工
4. 多轮复核之后，如何证明拆分没有丢内容、没有失真、没有变成机械摘抄

## 2. 适用范围

本套文档覆盖以下问题域：

- 下载并发与调度
- Electron 桌面能力
- IPC / preload / shared / renderer 边界
- 页面功能完整性
- UI 交互一致性
- 稳定性 / 错误恢复 / 安全性
- 成熟库替代与长期现代化升级

不覆盖：

- 重新发明一份新的审查报告
- 直接修改代码后的实现细节
- 与本轮审查无关的额外功能设计

## 4. 这套文档怎么用

### 4.1 负责人 / 协调者

推荐顺序：
1. 读 [01-现状问题与目标架构设计.md](./01-现状问题与目标架构设计.md)
2. 读 [02-优先级与分阶段整改路线图.md](./02-优先级与分阶段整改路线图.md)
3. 读 [03-任务拆分总控.md](./03-任务拆分总控.md)
4. 按需进入对应专项文档
5. 用 [04-覆盖矩阵与原报告映射.md](./04-覆盖矩阵与原报告映射.md) 防止排期时漏项

适合回答的问题：
- 先修什么
- 哪些可以并行
- 哪些任务包共享前置依赖
- 当前阶段验收应该看什么

### 4.2 子智能体 / 具体实施者

推荐顺序：
1. 读 [03-任务拆分总控.md](./03-任务拆分总控.md) 找到被分配的任务包
2. 读对应专项文档（`10` 到 `16`）
3. 如任务跨专题，再补读 [01-现状问题与目标架构设计.md](./01-现状问题与目标架构设计.md)
4. 实施前用 [04-覆盖矩阵与原报告映射.md](./04-覆盖矩阵与原报告映射.md) 确认没有漏掉同组问题

适合回答的问题：
- 我这一包要改什么
- 不该顺手改什么
- 要回归哪些页面 / 模块
- 验收口径是什么

### 4.3 独立复核智能体 / 复核者

推荐顺序：
1. 读 [02-优先级与分阶段整改路线图.md](./02-优先级与分阶段整改路线图.md)
2. 读 [04-覆盖矩阵与原报告映射.md](./04-覆盖矩阵与原报告映射.md)
3. 针对特定专题读对应专项文档
4. 把复核意见合并到 [05-复核记录与整合结论.md](./05-复核记录与整合结论.md)

适合回答的问题：
- 文档是否漏掉原报告重要结论
- 总控与专项是否边界重叠
- 任务包是否真的能直接分发执行
- 术语、优先级、标题风格是否一致

## 5. 边界规则

### 5.1 一个问题只能有一个主责任文档

例如：
- 批量领取非原子 → 主责任在 `10`
- 协议单一来源 → 主责任在 `12`
- `app://` 路径映射 → 主责任在 `15`
- `pageMode` 伪设置 → 主责任在 `13`
- 一级导航只切左栏 → 主责任在 `14`

其他文档只允许保留：
- 依赖关系说明
- 引用链接
- 对执行顺序的影响

不允许在多个文档中无序复制同一问题的完整说明。

### 5.2 总控文档不吞掉专项文档

- `01` 负责目标结构和演进原则
- `02` 负责阶段顺序和依赖关系
- `03` 负责任务分发
- `04` 负责追踪映射
- `05` 负责复核沉淀

它们都不应该替代 `10` 到 `16` 的具体整改说明。

### 5.3 专项文档不替代总体架构文档

专项文档负责“这类问题怎么改”，但不负责重新定义全局分层目标。凡是涉及：
- main / preload / renderer / shared 目标边界
- 下载体系、桌面运行形态、安全边界的总体结构
- 什么必须先做，什么可以后做

都应回到 `01` 或 `02`。

## 6. 使用建议

### 6.1 如果要开始实施
- 不要直接从 `审查报告.md` 开工
- 先从 `03` 找任务包，再读对应专项文档
- 如任务有跨专题前置依赖，必须补看 `01` 与 `02`

### 6.2 如果要做任务分发
- 优先按 `03` 中的任务包分发
- 不要按“原报告第几章”分发
- 不要把共享前置改造拆散给多个任务同时改

### 6.3 如果要做阶段验收
- 先用 `02` 确定当前阶段的完成条件
- 再用 `03` 看任务包验收标准
- 最后用 `04` 检查是否有遗漏项未被承接

## 7. 与 `审查报告.md` 的关系

- `审查报告.md` 提供证据、级别、章节化审查结果
- 本目录文档把这些结果重组为可实施的文档系统
- 若两者出现细节冲突，应先回到 `审查报告.md` 和代码证据重新核对
- 若后续实施改变了判断，应更新本目录文档和 `05-复核记录与整合结论.md`，而不是悄悄偏离原报告
