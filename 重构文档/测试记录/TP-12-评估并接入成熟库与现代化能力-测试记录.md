## 4. 候选库评估结论

### 4.1 `p-queue` 执行池
- 当前状态：**只评估未接入**
- 替代对象：自研 `downloadManager.activeDownloads` + `bulkDownloadService`
  worker 池逻辑
- **不**替代对象：批量调度的领取 / 分页 / 断点恢复 / 会话生命周期（
  这些归 TP-01 / TP-02）
- 前置条件：
  1. TP-01 下载状态机语义稳定（已达成）
  2. TP-02 原子领取稳定（已达成）
  3. `activeDownloadSessions` 的生命周期边界已覆盖（已达成）
- 不接入条件：
  - 当前自研 pool 已通过 abort / notifications / resume / tp02 全量测试
  - 在没有性能瓶颈或 worker 数动态调整需求之前，引入 `p-queue`
    只会新增依赖面但不带来可验证增益
- 许可证：MIT（satisfactory）
- 维护活跃度：积极（每季度有更新）
- Electron/Node 兼容性：ESM only 从 v7 开始；当前项目 `"type": "module"`
  本身兼容，但需要先确认 Electron 39 的 main 进程 ESM bundle 形态
- 升级策略：引入前先做 1 个会话规模的 pilot；若 30 天稳定再铺开
- 回退策略：保留当前自研 pool 的代码路径至少 2 个版本

### 4.2 `Bottleneck` 限流器
- 当前状态：**只评估未接入**
- 替代对象：`networkScheduler` 中按站点速率的 token bucket 近似实现
- **不**替代对象：与 `p-queue` 的并发池职责完全分开（`Bottleneck`
  只负责速率，`p-queue` 只负责并发；除非先定义职责边界，否则不得
  同时引入）
- 前置条件：
  1. `networkScheduler.integration` 测试覆盖 rate-limit 语义（已达成）
  2. 站点级策略稳定（已达成）
- 不接入条件：
  - 当前自研 scheduler 在集成测试中 rate-limit 行为正确
  - `Bottleneck` + `p-queue` 混用会引入职责重叠风险；应当等 TP-01/02
    全部稳定、且出现自研方案无法扩展的具体需求后再接入
- 许可证：MIT
- 维护活跃度：稳定但更新频率较低（~6 个月一次）
- 兼容性：CJS/ESM 双发，兼容性良好
- 升级策略：先做单站点 pilot
- 回退策略：保留自研 scheduler

### 4.3 `react-virtuoso` / `@tanstack/react-virtual` 虚拟列表
- 当前状态：**只评估未接入**
- 评估选型结论：若未来接入，首选 **`@tanstack/react-virtual`**
  - 理由 1：hook 风格，可与 `BooruGridLayout` 既有布局协同，不要求
    重写 DOM 树
  - 理由 2：TanStack 维护活跃，API 稳定
  - 理由 3：Bundle 体积更小
- 替代对象：`BooruGridLayout` / `ImageGrid` 等大列表在高条目数下的全量
  渲染
- **不**替代对象：列表上的筛选 / 排序 / 空错态 / 详情回跳逻辑（这些
  归 TP-04 / TP-09）
- 前置条件：
  1. TP-04 页面伪能力闭环（已达成）
  2. TP-09 空错态与隐藏页副作用规则（主要完成）
  3. TP-13 页面偏好持久化（已达成前四刀）
  4. 需要新增：滚动定位恢复 + 选择态保持 + 懒加载兼容 的端到端测试
- 不接入条件：
  - 当前每页默认 ≤ 200 条帖子，未达到虚拟化收益临界点
  - 在典型硬件上未观察到滚动卡顿、首屏长时间白屏等可测症状
- 许可证：MIT（两者）
- 维护活跃度：两者都活跃
- 兼容性：两者都与 React 18 ^ antd 5 兼容
- 升级策略：选 1 个页面（建议 BooruTagSearchPage）做 pilot，持续

### 4.4 `electron-log` 日志落盘
- 当前状态：**只评估未接入**
- 替代对象：当前 `console.log` / `console.error` 混合的日志输出
- **不**替代对象：业务层的模块前缀打印（不被 `electron-log` 覆盖的
  业务语义部分保持不变）
- 前置条件：
  1. 敏感字段清单：必须在接入前完成以下字段脱敏声明
     - `config.network.proxy.username/password`
     - `config.google.clientSecret`
     - `booru_sites.salt/apiKey/passwordHash`
     - Auth URL 中的 password / token 查询参数
     - 下载路径中的用户名目录（Windows `C:\Users\<name>`）
  2. 日志分级策略：error / warn / info / debug 的边界
  3. 日志轮转策略：单文件 ≤ 10MB，保留 5 份
- 不接入条件：
  - 在没有脱敏规则作为强约束时直接接入 `electron-log` 会把敏感字段
    直接写进滚动日志文件
  - 当前 `console` 输出虽然粗糙但没有落盘持久化，反而是可接受的
    零泄漏状态
- 许可证：MIT
- 维护活跃度：活跃
- 兼容性：支持 Electron 20+
- 升级策略：先做主进程 error 通道接入 pilot
- 回退策略：保留 `console.*` 调用，`electron-log` 作为叠加层引入

### 4.5 `@sentry/electron` 崩溃监控
- 当前状态：**只评估未接入**
- 替代对象：无（当前无崩溃监控）
- **不**替代对象：主进程的 `uncaughtException` / `unhandledRejection`
  兜底（这些由 TP-08 负责）
- 前置条件：
  1. 必须先完成 4.4 的脱敏字段清单
  2. 必须先决定上报范围：是否上报路径、URL、站点配置

- 许可证：MIT
- 维护活跃度：非常活跃
- 兼容性：支持 Electron 28+
- 升级策略：先在开发模式下做脱敏能力 pilot，再在用户可见版本接入
- 回退策略：Sentry 接入本身就是可选项，不接入不影响主链路
