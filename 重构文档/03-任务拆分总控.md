# 03-任务拆分总控

## 1. 文档目的

本文件把后续改造拆成一组**中粒度任务包**，用于：

- 直接给不同智能体分发任务
- 让人工按包推进，而不是按零散问题推进
- 让阶段验收可以按任务包检查

每个任务包都尽量覆盖一个相对完整的问题域，避免过碎，也避免过大。

## 2. 任务包使用规则

1. 先读 [02-优先级与分阶段整改路线图.md](./02-优先级与分阶段整改路线图.md) 决定当前阶段
2. 再从本文件挑选对应任务包
3. 进入任务包指向的专项文档查看详细说明
4. 开工前确认前置依赖是否已满足
5. 结束后按验收标准和回归重点完成检查

## 4. 任务包详情

### TP-01 修下载状态机与暂停/恢复/取消语义

**目标**
- 统一普通下载与批量下载的暂停、取消、失败、恢复语义
- 让用户主动操作不再污染为 `failed`
- 让 `pending / paused / failed / running / completed` 成为可信状态

**范围**
- 普通下载
- 批量下载记录级状态
- 启动恢复与暂停语义的第一轮收敛

**关联问题编号**
- P0-06
- P1-01
- P1-02
- P1-21

**涉及模块 / 当前主要落点文件**
- `src/main/services/downloadManager.ts`
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/init.ts`
- `src/renderer/pages/BooruDownloadPage.tsx`
- `src/renderer/pages/BooruBulkDownloadPage.tsx`

**前置依赖**
- 无

**并行关系**
- 可与 TP-04 并行
- 不建议与 TP-10 并行深入开发，因为桌面后台能力依赖稳定状态语义

**风险点**
- 普通下载与批量下载现有状态并不对称
- 恢复逻辑、UI 状态表达、统计口径会被连带影响

**验收标准**
- 用户主动暂停/取消不再落入 `failed`
- `paused` 不会在重启后被无条件恢复
- 普通下载 `pauseAll()` 不再回写 `pending`
- 页面能区分真正运行态与等待态

**建议回归**
- Download 页面
- Bulk 页面
- 应用重启后的恢复链路

### TP-02 修批量调度生命周期与原子领取

**目标**
- 让批量下载会话真正拥有单一活跃执行体
- 防止重复调度、重复领取、取消失效

**范围**
- 会话生命周期
- 活跃会话标记
- 记录 claim 原子性
- 调度器监听与解绑

**关联问题编号**
- P0-01
- P0-02
- P1-07

**涉及模块 / 当前主要落点文件**
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/networkScheduler.ts`
- `src/main/services/database.ts`
- `src/main/services/init.ts`

**前置依赖**
- 最好先完成 TP-01 的状态语义收敛

**并行关系**
- 可与 TP-04 并行
- 可与 TP-08 中的运行入口修复部分并行

**风险点**
- DB 状态与内存活跃会话的边界要重写
- 监听解绑与退出清理会影响恢复链路

**验收标准**
- 同一记录不会被并发领取两次
- 会话活跃标记与真实下载循环一致
- 暂停/取消能正确找到 controller / 活跃会话
- `networkScheduler` 监听具备生命周期对称性

**建议回归**

- 重复 start / 恢复时的幂等性

### TP-03 收紧下载文件协议与完整性校验

**目标**
- 让文件落盘、校验、取消清理和恢复策略具备可解释性

**范围**
- 普通下载和批量下载文件协议
- `.part + rename`
- 完整性校验
- 失败/取消清理策略
- 队列表数据库约束

**关联问题编号**
- P1-06
- P1-08
- P1-09
- P1-24

**涉及模块 / 当前主要落点文件**
- `src/main/services/downloadManager.ts`
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/database.ts`
- `src/main/services/booruService.ts`

**前置依赖**
- TP-01
- TP-02

**并行关系**
- 不建议与下载状态机重写同时大范围进行
- 可在状态机稳定后作为独立包推进

**风险点**
- 文件系统边界与恢复策略容易互相牵连
- 取消逻辑中的误删风险需要谨慎收口

**验收标准**
- 下载先写临时文件，再校验通过后 rename
- 普通下载补完成后二次校验
- cancel/failed 不再无条件删除最终目标文件
- 队列表具备更可靠的数据库约束或幂等写入语义

**建议回归**
- 下载成功、失败、取消、异常退出后恢复
- 重复下载与文件已存在场景

**范围**

**前置依赖**

- 若协议和状态边界还没稳定，facade 只会包住旧问题

**验收标准**
- 多页面重复业务流程明显减少
- 页面逻辑更聚焦展示和交互
- hooks / facade 边界清晰，不继续把复杂度塞回页面

**建议回归**
- Booru 多页面群的收藏、喜欢、下载、搜索与状态回填

### TP-13 建立页面偏好持久化与配置边界分层

**目标**
- 为页面级排序 / 筛选 / 分页状态建立统一持久化层
- 明确 config、数据库、localStorage 三者边界，阻止配置容器继续膨胀

**范围**
- 高频管理页页面偏好持久化
- config DTO 收窄
- page preferences 存储模型
- 已写入 config 的 UI 行为状态清理与迁移

**关联问题编号**
- P1-25
- P1-27

**涉及模块 / 当前主要落点文件**
- `src/main/services/config.ts`
- `src/main/services/database.ts`
- `src/main/ipc/handlers.ts`
- `src/preload/index.ts`
- `src/shared/types.ts`
- `src/renderer/App.tsx`
- `src/renderer/pages/FavoriteTagsPage.tsx`
- `src/renderer/pages/GalleryPage.tsx`
- `src/renderer/pages/BlacklistedTagsPage.tsx`

**前置依赖**
- TP-05 建议优先完成
- 与 TP-07 的 config DTO 收紧强相关

**并行关系**
- 可与 TP-04 的局部页面闭环修复协同
- 不宜与大范围 preload / shared 改造完全脱节，应共享同一 DTO 设计

**风险点**
- 若边界没定义清楚，容易把更多 UI 状态继续写进 config
- 若直接各页各写 localStorage，会制造新的跨窗口不一致

**验收标准**
- FavoriteTags / Gallery / BlacklistedTags 等高频管理页的状态持久化有统一存储层
- config 对 renderer 的暴露面收窄为 DTO，而不是整包对象
- 主题、语言等轻量 renderer 偏好与产品级工作上下文明确分层

**建议回归**
- FavoriteTags、Gallery、BlacklistedTags
- 应用重启后的状态恢复
- 多窗口 / 多页面对同一偏好状态的一致性

### TP-12 评估并接入成熟库与现代化能力

**目标**
- 在边界稳定后，把最容易出结构性问题的基础设施交给成熟库接管

**范围**
- `p-queue`
- `Bottleneck`
- `react-virtuoso` / `@tanstack/react-virtual`
- `electron-log`
- `@sentry/electron`
- 构建分块策略

**关联问题编号**
- P2-01
- P2-02
- 第 9 章全部相关内容
- 第 10 第三、第四优先级相关内容

**涉及模块 / 当前主要落点文件**
- `src/main/services/downloadManager.ts`
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/networkScheduler.ts`
- `src/renderer/App.tsx`
- 各图片密集型列表页
- 主进程入口与日志链路

**前置依赖**
- TP-01
- TP-05
- TP-08

**并行关系**
- 虚拟列表评估、日志方案评估、执行池评估可并行
- 实际接入时应分专题逐步落地

**风险点**
- 过早接入新库会把旧边界问题一起包进去
- 列表虚拟化可能与当前布局策略冲突

**验收标准**
- 明确哪些库要引入、替代什么、何时引入、何时不引入
- 接入顺序与前置条件明确
- 现代化升级不与边界收口相互打架

**建议回归**
- 下载性能与稳定性
- 列表滚动性能
- 日志、崩溃、构建产物变化

## 5. 任务分发建议

- 单个智能体优先拿一个任务包，而不是同时横跨多个高耦合任务包
- 若要并行，优先挑选：
  - TP-01 / TP-04
  - TP-07 / TP-08
  - TP-09 中的不同子线
- 不要让多个智能体同时修改协议单一来源、preload、shared、handler 的同一批文件
