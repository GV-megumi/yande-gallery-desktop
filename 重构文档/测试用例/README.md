# 测试用例

## 1. 文档目的

本目录用于承接整套重构工作的测试规范与任务包测试说明，目标不是替代实现文档，而是把“怎么测、测什么、按什么顺序测、怎么做阶段验收”统一起来。

这套测试文档的作用主要有四个：

1. 为人工测试提供稳定、可执行的检查清单
2. 为自动化补测提供统一的补充入口
3. 为阶段验收提供一致的判定口径
4. 为跨任务包回归提供可追踪的测试路径

本目录与上层重构文档配合使用：
- [02-优先级与分阶段整改路线图.md](../02-优先级与分阶段整改路线图.md) 用来判断当前处在哪个阶段
- [03-任务拆分总控.md](../03-任务拆分总控.md) 用来确认当前任务包对应哪份测试文档

同时需要说明的是：本目录当前仍处于测试文档骨架搭建阶段。现在已经创建的是目录入口 `README.md` 和总则文档 `00-测试规范总则.md`，其余 TP 测试文档大多尚未在仓库中落地。因此，阅读和使用本目录时，应把它视为测试体系的导航骨架，而不是已全部完成的测试资产清单。

## 2. 阅读顺序

建议按下面顺序阅读：

1. 优先阅读已创建的 [00-测试规范总则.md](./00-测试规范总则.md)，先统一测试范围、证据、验收与回归口径
2. 根据任务包编号进入对应的 `TP-xx` 测试文档
3. 如需做阶段验收，先看当前阶段涉及的所有 TP 文档，再回到 [02-优先级与分阶段整改路线图.md](../02-优先级与分阶段整改路线图.md) 对照完成标志
4. 如需做跨任务包回归，优先按依赖链顺序阅读相关 TP 文档，而不是只看单个任务包

如果某个 TP 测试文档尚未创建，请先回看 [03-任务拆分总控.md](../03-任务拆分总控.md) 与对应专项文档，按任务包目标、范围、验收标准和建议回归点组织当前测试，不要默认该 TP 文档已经存在。

## 5. 使用规则

1. **TP 文档必须遵循已创建的 [00-测试规范总则.md](./00-测试规范总则.md)**
   - `00` 是当前目录下已经落地的总规则文档，应作为其他 TP 文档的统一约束来源
   - 若 TP 文档中出现特殊测试要求，应在不违背总则的前提下补充说明

2. **TP 文档以人工测试为主**
   - 每个任务包首先应给出可手工执行的验证步骤
   - 自动化测试只作为补充，不应取代人工验收

3. **TP 文档要服务于阶段验收**
   - 任务包测试结果必须能回收为阶段完成条件判断
   - 阶段验收时，应按 TP 文档逐项核对关键路径、边界条件与回归点

4. **TP 文档要支持跨回归**
   - 一个任务包可能影响多个页面、多个进程边界或多个运行时状态
   - 测试时不得只验证单点，应沿依赖链补测相邻任务包和共享边界

5. **TP 文档只负责测试，不负责改造方案本身**
   - 需求解释、实现策略、架构调整请回到 [02-优先级与分阶段整改路线图.md](../02-优先级与分阶段整改路线图.md)、[03-任务拆分总控.md](../03-任务拆分总控.md) 和对应专项文档
   - 本目录重点是把“怎么验证”说清楚

## 6. 目录使用建议

- 只想快速开始测试：先读已创建的 [00-测试规范总则.md](./00-测试规范总则.md)，再找对应 `TP-xx` 规划文档；若目标 TP 文档尚未创建，再回看上层文档
- 需要做阶段验收：先看当前阶段涉及的所有 TP 文档，再对照 [02-优先级与分阶段整改路线图.md](../02-优先级与分阶段整改路线图.md)
- 需要做跨任务包回归：优先从依赖较早的 TP 文档开始，避免遗漏前置状态问题
- 需要补自动化：先按 TP 文档确认人工步骤，再决定哪些步骤适合转成自动化用例
- 如发现目标 TP 文档尚未创建：先回到 [03-任务拆分总控.md](../03-任务拆分总控.md)，再进入对应专项文档组织当前测试

## 7. 说明

本目录是整套重构工作的测试入口。当前它仍处于测试文档骨架搭建阶段：目录入口与总则文档已经建立，但大部分 TP 测试文档仍待后续补齐。它的目标是让测试可分发、可复核、可补测、可回归，而不是把尚未落地的计划写成已经全部完成的事实。
