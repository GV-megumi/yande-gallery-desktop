# TP-02 修批量调度生命周期与原子领取

## 1. 文档目标

本 TP 面向批量下载调度可信性问题，验证批量会话是否只会存在一个活跃执行体、记录领取是否具备原子性、监听注册与解绑是否可验证、重复 start / resume 是否幂等。

本任务的测试目标是证明批量调度链路不会出现以下问题：

1. 同一会话被重复启动成多个执行体
2. 同一条待下载记录被多个调度循环重复领取
3. 调度器监听注册后不解绑，导致重复触发或内存泄漏
4. 暂停、取消、恢复、退出后的生命周期不闭合

## 3. 测试范围

### 3.1 必测范围
- 批量下载会话从启动到结束的单一活跃性
- 记录领取的原子性与去重结果
- start / resume / pause / cancel 的幂等性
- 调度器监听的注册、注销、重复注册防护
- 启动恢复与退出清理对生命周期的影响

### 3.2 不在本次范围内
- 普通下载状态语义收敛，属于 TP-01
- 文件落盘 `.part + rename` 与完整性校验，属于 TP-03
- 页面伪能力与交互闭环，属于 TP-04

## 4. 风险摘要

1. 调度生命周期与状态机耦合较深，修复时可能带出暂停、恢复、取消分支的回归
2. 原子领取问题往往只有并发触发时才暴露，手工低频验证不能替代并发验证
3. 监听解绑如果遗漏，短期不一定报错，但会在重复启动、重启恢复后逐渐放大
4. 如果只看任务列表状态，不看执行体数量，容易误判为正常

## 5. 测试前置条件

### 5.1 环境前置
- 应用可启动并可访问批量下载页面
- 必须准备至少一个包含多条记录的批量下载任务
- 必须具备查看日志、调试输出或任务状态变更的能力
- 若要验证监听治理，必须能够重复执行 start / stop / restart 场景；否则相关用例只能记为待补测

### 5.2 数据前置
- 批量任务中至少包含多条 `pending` 记录
- 必须准备可持续运行数秒以上的下载目标，便于观察执行体与领取行为
- 若要观察原子领取，必须有多个任务项可参与领取；否则原子领取类用例只能记为环境阻塞

### 5.3 操作前置
- 必须具备手工触发多次 start / resume / cancel 的条件
- 必须具备录屏工具，记录重复点击与状态变化
- 如要验证退出清理，必须可快速重启应用并对比日志

### 5.4 执行体观察前置
- 必须通过以下至少一种证据证明“同一时刻只有一个执行体”：
  - 日志中的会话启动标识
  - 调度循环实例标识
  - listener 计数器
  - 等价调试输出
- 不能仅凭页面看起来没有重复下载、没有重复刷新或没有重复提示，就判定“只有一个执行体”
- 若当前环境无法提供上述任一证据，则单一执行体相关用例只能记为待补测，不得判通过

## 6. 术语与判定口径表

| 术语 | 定义 | 判定口径 | 不允许混用为 |
|---|---|---|---|
| 会话 | 一次批量下载任务的业务实体，拥有唯一会话标识与一组待处理记录 | 会话是否存在，以会话 ID 和会话状态为准；一个会话可以跨暂停、恢复、重启延续 | 执行体、监听 |
| 执行体 | 实际运行下载循环的活跃运行单元，可对应一个正在处理会话的主进程循环或任务体 | 同一会话在任一时刻最多只能对应一个活跃执行体；若日志或行为显示同时存在两个处理同一会话的活跃循环，则判失败 | 会话、领取 |
| 领取（claim） | 将某条 `pending` 记录从可领取状态切换为当前执行体占有状态的动作 | 同一记录 ID 在同一轮处理中只允许成功领取一次；若同一记录被两个执行体处理，则判失败 | 开始、恢复 |
| 监听 | 调度器或相关事件源的订阅关系，用于接收网络或调度信号 | 若实现为按会话注册监听，则每次注册必须存在对应解绑；若实现为进程级单例监听，则整个测试过程中监听总数不得增长，且不能重复注册同一监听 | 会话、执行体 |

补充口径：
- 文中若出现“执行循环”“运行单元”等表述，统一归入“执行体”。
- 文中若出现“实例”但语境用于描述实际运行处理单元，也统一按“执行体”判定。
- 监听对称性不能只看“次数差不多”。必须按实现方式判断：
  - 若为按会话注册/解绑：每创建一次会话级监听，就必须在会话暂停、取消、完成或销毁时解绑一次。
  - 若为进程级单例监听：测试前后监听总数必须一致，且重复 start / resume / restart 不得增加监听总数。

## 7. 六层测试口径与验证重点

### 7.1 配置与数据层
重点看：
- 会话活跃标记是否与真实执行体一致
- 记录状态是否因重复领取而被二次覆盖
- 退出后数据库状态是否仍可恢复
- 无 `pending` 记录时是否仍错误写入活跃状态

### 7.2 主进程服务层
重点看：
- 同一会话是否只允许一个活跃执行体
- `claim` 是否具备原子性和幂等性
- 调度器监听治理是否符合术语表中的判定口径
- start / pause / resume / cancel 是否真正驱动相应生命周期分支

### 7.3 IPC / preload / shared 边界层
不适用原因：
- 本 TP 的核心是批量调度生命周期、执行体唯一性、claim 原子性与监听治理，默认基于既有下载接口执行，不直接新增或修改 IPC 通道、preload 暴露面或 shared 类型。
- 若实际执行中发现页面触发 start / resume 后主进程行为缺失或参数不一致，应记录为跨层问题并转交 TP-05 / TP-06，而不是在本 TP 内假定协议变更。

### 7.4 渲染层交互层
重点看：
- 连续点击开始、暂停、恢复时 UI 是否没有出现重复启动迹象
- 批量状态数是否与实际执行一致
- 无 `pending` 记录时 start 的页面反馈是否正确

### 7.5 生命周期与异常层
重点看：
- 退出、重启、恢复时是否产生额外执行体
- 长时间运行后是否出现重复通知、重复轮询、重复回调
- 取消后再 start 是否会继承旧活跃状态

### 7.6 非功能层
重点看：
- 稳定性：高频 start / pause / resume / cancel 后系统不得进入半活跃状态
- 资源治理：监听不得持续增长，执行体不得泄漏
- 幂等性：重复 start、重复 resume、无 `pending` 时 start 都不得产生额外副作用
- 可观测性：必须通过日志中的会话启动标识、调度循环实例标识、listener 计数器或等价调试输出来证明执行体数量与监听治理结果
- 统计一致性：混合操作风暴后，完成数、失败数、取消数、剩余待处理数必须与真实记录集合一致

## 8. 测试设计文档与测试记录承接说明

本文档负责定义测试设计、覆盖范围、用例步骤、预期结果与判定口径，不直接承接某一次执行结果。

实际执行时，每条用例必须由对应测试记录补充以下最低字段：
- 实际结果

## 9. 详细测试用例

### TP-02-TC-001 同一批量会话只能存在一个活跃执行体
- 前置条件：
  - 有一个正在运行的批量下载会话
  - 已满足 5.4 中的执行体观察前置
- 场景/步骤：
  1. 启动一个批量会话
  2. 在会话执行过程中观察日志与状态
  3. 尝试再次触发同一会话 start
- 预期结果：
  - 只存在一个活跃执行体
  - 重复 start 不应再创建新的执行体
  - UI 不应出现双倍推进或双倍完成
- 关键观察点：日志中的启动次数、调度循环实例标识、listener 计数、完成计数

### TP-02-TC-002 同一记录的 claim 必须具备原子性
- 前置条件：
  - 批量任务中存在多个待领取记录
  - 具备可观察记录 ID 与领取时间的日志
- 场景/步骤：
  1. 触发批量调度
  2. 通过日志或状态观察记录领取过程
  3. 在 1 秒内连续触发两次 start 或 resume，或使用自动化并发脚本模拟两个入口同时争抢
- 预期结果：
  - 每条记录只会被成功领取一次
  - 不应出现同一记录同时处于两个执行体上下文中
- 关键观察点：记录 ID、claim 时间、claim 成功次数、执行归属

### TP-02-TC-003 暂停后恢复只能恢复原会话且只对应一个执行体
- 前置条件：
  - 已有运行中的批量会话
  - 已满足 5.4 中的执行体观察前置
- 场景/步骤：
  1. 暂停会话
  2. 点击恢复
  3. 观察恢复后的会话 ID 与执行体数量
- 预期结果：
  - 恢复后继续处理原会话 ID
  - 同一时刻只存在一个活跃执行体
  - 恢复后仍保持原有进度与记录归属
- 关键观察点：会话 ID、已领取记录、恢复日志、调度循环实例标识
- 判定补充：若当前实现恢复后会新建会话 ID，则该用例只能在产品明确定义“恢复创建新会话”的前提下重新编写；在该定义缺失前，不得判通过

### TP-02-TC-004 取消后调度器应停止领取并清理活跃标记
- 前置条件：批量会话正在执行
- 场景/步骤：
  1. 点击取消整个会话
  2. 观察后续是否还有领取、推进或回调
- 预期结果：
  - 取消后不再继续领取新记录
  - 活跃标记必须被清理或写入已取消终态
  - UI 应显示终止状态而非继续执行
- 关键观察点：取消后的日志静默、活跃状态是否仍存在

### TP-02-TC-005 重复启动同一会话应幂等
- 前置条件：
  - 已存在某个批量会话对象或任务入口
  - 已满足 5.4 中的执行体观察前置
- 场景/步骤：
  1. 连续点击“开始”两次或多次
  2. 观察会话数、领取数与日志
- 预期结果：
  - 只有第一次触发有效
  - 后续重复触发被忽略或合并
  - 不应出现任务倍速推进
- 关键观察点：会话实例数、重复请求次数、调度循环实例标识、任务推进节奏

### TP-02-TC-006 重复恢复同一会话应幂等
- 前置条件：
  - 批量会话处于 `paused`
  - 已满足 5.4 中的执行体观察前置
- 场景/步骤：
  1. 连续点击恢复
  2. 观察是否产生重复执行体
- 预期结果：
  - 只恢复一次
  - 不会出现多次 resume 触发多个执行体
  - 状态字段在恢复过程中不得出现重复来回切换
- 关键观察点：恢复日志、执行体数量、状态写回序列、listener 计数

### TP-02-TC-007 监听治理必须满足实现对应的可验证口径
- 前置条件：可查看 scheduler 监听相关日志、计数器或调试输出
- 场景/步骤：
  1. 启动批量会话
  2. 暂停、恢复、取消各一次
  3. 重新启动另一个会话并再次执行上述流程
  4. 记录监听注册数、解绑数或监听总数
- 预期结果：
  - 若实现为按会话注册监听：每次注册必须有一次对应解绑，第二轮会话开始前不得残留上一轮会话监听
  - 若实现为进程级单例监听：测试前后监听总数必须一致，且第二轮流程不得新增监听总数
  - 不应出现同一事件被重复订阅
- 关键观察点：监听计数、回调触发次数、重复日志

### TP-02-TC-008 启动恢复不应产生重复会话或重复执行体
- 前置条件：
  - 存在中断前的批量会话记录
  - 已满足 5.4 中的执行体观察前置
- 场景/步骤：
  1. 让批量会话执行到中间
  2. 异常退出或正常退出应用
  3. 重启后观察恢复行为
- 预期结果：
  - 重启后同一业务会话只能对应一个恢复中的会话实例和一个活跃执行体
  - 不应出现旧会话和新会话同时活跃
- 关键观察点：恢复日志、任务领取是否重复、调度循环实例标识、页面是否显示两套状态

### TP-02-TC-009 取消后再启动不应继承旧活跃状态
- 前置条件：曾执行过批量会话并取消
- 场景/步骤：
  1. 取消会话
  2. 再次对同一批任务执行 start
- 预期结果：
  - 旧活跃状态已清理
  - 新启动流程不读取旧执行体残留状态
  - 不会出现旧 listener、旧 active 标记残留导致的重复完成或重复失败计数
- 关键观察点：旧 listener、旧 active 标记、重复 completed/failed 计数

### TP-02-TC-010 混合操作风暴后最终统计必须一致且结果保持幂等
- 前置条件：
  - 批量任务包含足够多记录，且可快速触发多次操作
  - 已满足 5.4 中的执行体观察前置
- 场景/步骤：
  1. 快速触发 start / pause / resume / cancel 的交替操作，或使用自动化脚本并发发起这些请求
  2. 等待系统进入稳定状态
  3. 对比最终统计与真实记录集合
- 预期结果：
  - 不会出现同一记录重复完成
  - 不会出现总数与实际条目数不一致
  - 最终状态集合满足幂等结果，即重复操作不会产生额外记录或额外执行体残留
- 关键观察点：完成列表、失败列表、取消列表、总计数、执行体销毁情况、listener 最终计数

### TP-02-TC-011 无 pending 记录时 start 不应创建活跃执行体
- 前置条件：
  - 目标批量会话中没有 `pending` 记录，所有记录均已处于 `completed`、`failed`、`paused` 或 `canceled`
  - 已满足 5.4 中的执行体观察前置
- 场景/步骤：
  1. 对该会话点击 start
  2. 观察日志、会话状态与执行体变化
- 预期结果：
  - 不应创建新的活跃执行体
  - 不应注册新的会话级监听；若为进程级单例监听，监听总数也不得增长
  - 页面应提示无可执行记录，或保持当前状态不变
- 关键观察点：执行体数量、监听数量、页面提示、会话状态

## 10. 负向与边界场景

### 10.1 负向场景
- 在会话刚启动时立刻 pause，再 resume，不能产生两个执行体
- 在 cancel 与 resume 连续交替时，不应留下半活跃状态
- 同一会话重复点击 start，应稳定返回已在运行或被忽略，不得再创建新执行体

### 10.2 边界场景
- 会话只领取到一条记录就暂停，恢复后应继续同一会话
- 会话接近执行结束时取消，不能重新拉起残余记录
- 任务列表为空或无 `pending` 记录时 start，不应创建活跃执行体

### 10.3 恢复边界
- 重启后如果数据库中残留 active 标记，应能被识别并正确清理或恢复
- 旧版本遗留脏数据不得导致新版本重复监听或重复启动

- 同一会话只存在一个活跃执行体
- 同一条记录不会被重复领取
- 监听治理符合实现对应的可验证口径，重复 start / resume 幂等
- 重启恢复不会引入重复执行

## 12. 自动化建议

### 12.1 适合自动化的部分
- 原子领取并发单测
- 会话状态机幂等单测
- 监听注册/解绑次数断言或监听总数断言
- 恢复分支中的实例数量断言

### 12.2 建议自动化方式
- 使用服务层单元测试模拟并发抢占
- 对 listener 注册和释放增加计数断言，或对单例监听总数增加前后对比断言
- 用假调度器或 stub 驱动 start / pause / resume / cancel

### 12.3 仍需人工确认的部分
- 真实桌面环境下重复点击是否会造成额外副作用
- 重启后 UI 是否只显示一个会话
- 长时间运行是否出现重复通知、重复完成提示或卡住

### 12.4 自动化边界
- 自动化可证明“不会重复领取”，但不能替代真实退出/恢复体验验证
- 自动化结果必须和页面观测结果一起使用，不能只看内部计数

## 13. 缺陷分级与结论判定

### 13.1 缺陷分级
- S0：出现双活跃执行体、重复领取导致数据错乱、监听泄漏引发严重卡死
- S1：重复 start / resume 可观察到双执行、重复完成、恢复错误
- S2：监听治理不符合预定实现口径，但主流程仍可用
- S3：仅有轻微日志噪音，不影响实际调度

### 13.2 结论判定
满足以下条件可判定本 TP 通过：
1. 会话生命周期单一且清晰
2. 原子领取无重复
3. start / resume / cancel 幂等
4. 监听治理符合术语表中对应实现口径
5. 重启恢复不会创建额外执行体
6. 单一执行体结论已由日志标识、实例标识、listener 计数器或等价调试输出证明，而非仅凭 UI 观察

只要仍存在重复领取、重复执行、监听治理不满足口径，或无法证明执行体唯一性，本 TP 不能通过。

## 15. 阶段验收口径

TP-02 作为阶段 0 的关键调度治理项，验收时应满足：

1. 同一批量会话只存在一个活跃执行体
2. `pending` 记录领取具备原子性，不会重复领取
3. start / resume / cancel 行为幂等，不会因重复触发产生额外会话或额外执行体
4. scheduler 监听治理满足实现对应的可验证口径，重启后无残留副作用
5. 执行体唯一性已经由日志标识、调度循环实例标识、listener 计数器或等价调试输出证明，不能仅凭 UI 观察
6. 已留存足够证据，且不存在阻断级并发或生命周期缺陷

若只修复了领取逻辑但仍存在双会话、双执行体、监听残留，或无法证明执行体唯一性，则只能判定为未通过，不能进入阶段通过结论。
