# TP-02 修批量调度生命周期与原子领取

## 1. 文档目标
验证批量下载会话具备单一活跃执行体、pending 记录领取具备原子性、监听注册与解绑对称，防止重复调度和取消失效。

## 3. 风险摘要
- 重复 start 或恢复会生成多个执行体，导致重复下载和统计错乱
- pending 领取非原子时会出现同一记录被多会话同时处理
- 监听解绑不完整会造成僵尸回调、重复触发或退出后残留

## 4. 测试前置条件
- 至少准备 1 个批量下载会话、多个 pending 记录
- 可并发触发 start / pause / resume / cancel，例如：连续双击开始、从页面入口和启动恢复同时触发 start、快速 start/resume 连击
- 可重启应用并观察恢复链路与日志
- 观测位置：批量下载会话卡片、状态筛选结果、目标目录文件列表；启动应用的终端输出或应用日志文件中与批量下载/调度/启动恢复相关的日志行；通过当前项目已有的数据库查看方式检查会话与记录状态变化，若当前环境做不到则记为待补测
- 关键用例建议保留：截图、日志摘录、数据库状态对比、前后状态对比截图

## 5. 测试分层
- 配置/数据层：核对 pending 领取是否原子、同一记录归属是否唯一
- 主进程服务层：核对会话生命周期、调度器状态切换、监听注册与解绑
- IPC / preload 边界层：核对开始、暂停、恢复、取消入口是否都落到同一套会话控制
- 渲染层交互层：核对批量会话卡片状态、按钮可用性、统计展示
- 生命周期与异常场景层：核对启动恢复、退出清理、网络切换与崩溃后的残留控制

## 6. 测试范围
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/networkScheduler.ts`
- `src/main/services/database.ts`
- `src/main/services/init.ts`

## 7. 核心测试要求
- 会话活跃标记必须与真实 download loop 一致
- 同一记录不得被两个执行体重复领取
- 暂停/取消必须能命中真实 controller / session
- scheduler listener 必须存在解绑与退出清理

## 8. 详细测试用例
- `TP-02-TC-001` 同一会话重复点击开始，不会创建两个执行体
  - 触发：连续点击同一批量会话的开始按钮
  - 检查：会话活跃态、日志中执行体创建次数、是否出现并行 loop
  - 预期：最终只保留一个活跃执行体，重复点击不得产生第二个 running 实例
- `TP-02-TC-002` 同一 pending 记录在并发压力下不会被重复领取
  - 触发：并发启动多个领取入口或快速重试
  - 检查：数据库领取结果、记录归属、是否有重复写状态日志
  - 预期：单条记录只会被一个执行体领取一次，不得出现双归属或重复写入
- `TP-02-TC-003` 会话暂停后监听解除，恢复后重新注册且不重复叠加
  - 触发：对运行中会话执行 pause/resume 循环多次，再执行一次恢复后的真实操作
  - 检查：同一操作是否触发重复日志、重复状态更新，页面状态是否重复跳变
  - 预期：暂停时必须解绑，恢复后必须重新注册一次，不得出现重复回调
- `TP-02-TC-004` 会话取消后 controller、活跃态、统计口径同步清理
  - 触发：取消一个正在运行的批量会话
  - 检查：controller 释放、活跃标记、统计卡片、数据库状态
  - 预期：会话彻底停止，统计与状态同步回落
- `TP-02-TC-005` 启动恢复与手动 start 并发触发时保持幂等
  - 触发：应用启动恢复未完成会话的同时手动点开始
  - 检查：恢复日志、执行体数量、会话状态是否重复进入 running
  - 预期：最终只形成 1 个活跃执行体，同一会话不得重复进入 running，也不得重复领取同一批记录
- `TP-02-TC-006` 网络调度器状态切换不会制造僵尸会话
  - 触发：切换网络状态或限流状态，促使调度器暂停/恢复
  - 检查：scheduler 日志、会话是否仍被标记活跃、是否有残留定时器
  - 预期：状态切换后无僵尸会话或重复调度
- `TP-02-TC-007` 退出应用后恢复定时器和监听不会残留到下一轮启动
  - 触发：运行中退出应用并重新启动
  - 检查：退出清理日志、重新启动后的 listener 数量、定时器是否重建
  - 预期：旧监听和定时器被清理，新启动仅建立一套

## 9. 负向与边界场景
- 重复 start / pause / resume / cancel 快速连击
- 会话处于 running 时应用崩溃再启动
- 监听回调内部抛错
- 多会话并存且共享站点限流

## 10. 验收标准
- 同一 session 任意时刻仅有一个活跃执行体
- 同一记录不会被重复领取或重复写状态
- 监听注册、解绑、退出清理具备生命周期对称性

## 14. 缺陷分级与结论判定

### 14.1 缺陷分级
- S0：同一会话生成多个活跃执行体、同一记录被重复领取、取消 / 退出后仍持续跑旧执行体
- S1：监听注册 / 解绑不对称、恢复与手动 start 并发时幂等失效、统计与真实执行体不一致
- S2：局部日志、卡片状态或按钮反馈存在偏差，但主调度链路仍可追踪
- S3：说明文字、日志命名或记录格式仍可优化，不影响生命周期主判定

### 14.2 结论判定
满足以下条件可判定本 TP 通过：
1. 同一 session 任意时刻仅存在一个活跃执行体
2. pending 记录领取具备唯一性，不会被重复领取或重复写状态
3. 监听注册、解绑、退出清理与恢复重建已形成对称生命周期
4. 启动恢复、手动 start、pause / resume / cancel 并发触发时仍保持幂等
5. 测试记录已完整写明规则来源、实际结果、证据位置与结论

若仍存在重复执行体、重复领取、监听残留或退出后僵尸会话，则本 TP 不通过。

## 16. 阶段验收口径

TP-02 作为阶段 0 的关键批量调度治理任务，验收时应满足：

1. 批量下载会话不会因重复点击、恢复并发或网络状态切换生成多个执行体
2. 记录领取具有原子性，无法被多个执行体重复消费
3. 监听器、定时器、controller 与会话活跃态在暂停、取消、退出、重启后保持一致
4. 没有阻断级调度生命周期问题遗留

本设计文档不承载实际结果、证据路径和最终结论；这些信息必须由测试记录提供。若仅验证页面卡片变化，未验证真实执行体、领取结果或退出清理链路，则只能判定为待补测或未通过，不能进入阶段通过结论。
