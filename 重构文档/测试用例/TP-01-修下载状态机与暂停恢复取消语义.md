# TP-01 修下载状态机与暂停恢复取消语义

## 1. 文档目标

本 TP 面向下载状态机语义收敛问题，验证普通下载与批量下载在暂停、恢复、取消、失败、完成、重启恢复等场景下的状态表达是否一致、可信、可复用。

本任务的测试目标不是“下载能不能跑通”，而是证明以下三件事：

1. 用户主动行为与系统异常能够被明确区分，不会把暂停/取消误记为失败
2. 普通下载与批量下载对 `pending / running / paused / failed / completed / canceled` 的语义表达一致
3. 重启恢复后，任务不会错误自动恢复，也不会丢失真实状态

## 4. 风险摘要

1. 普通下载与批量下载当前实现路径不同，状态语义统一时容易出现一边修好另一边回退
2. 启动恢复逻辑与页面刷新逻辑可能互相覆盖，造成状态闪变或误恢复
3. 仅看数据库状态不足以判断正确性，必须同时看页面表达与实际下载行为
4. 用户主动暂停与系统异常失败的边界不清时，统计、重试与恢复行为都会失真

## 5. 测试前置条件

### 5.1 环境前置
- 应用可正常启动并进入下载相关页面
- 必须准备至少一个可稳定下载的测试资源，且支持中途暂停与恢复；没有该资源时，涉及暂停、恢复、重启恢复的用例一律记为环境阻塞
- 必须准备至少一个批量下载任务，包含多个待下载条目
- 必须准备可查看的本地日志或调试输出位置，用于核对状态写回与恢复行为

### 5.2 数据前置
- 普通下载列表中存在可重复触发的任务
- 批量下载会话中存在 `pending`、`running`、`paused`、`failed` 至少一种现成状态样本
- 若需验证重启恢复，必须保证应用退出后可再次启动并保留运行数据

### 5.3 操作前置
- 必须具备手工观察 UI 状态的条件
- 必须具备截图或录屏能力；凡涉及暂停、取消、`pauseAll()`、重启前后对比的用例，若无法留存截图或录屏，不得判通过
- 若要验证状态写回，必须能够查看日志或数据库变化；否则相关用例只能记为待补测，不得判通过

### 5.4 口径确认前置
- 在执行 TP-01 之前，必须先确认取消终态统一使用 `canceled`，历史别名仅允许映射，不允许作为独立验收口径
- 在执行 TP-01 之前，必须先确认“运行中任务在异常中断并重启后”进入哪一种唯一状态：
  - 若产品已定义“恢复候选状态”，则执行记录必须写明该状态名
  - 若产品未定义恢复候选状态，则统一按 `failed` 判定
- 若上述两项任一未确认，则涉及取消终态或重启后中断任务状态的用例只能记为待补测，TP-01 不得判通过

## 6. 状态口径表

| 状态 | 状态含义 | 允许来源 | 是否终态 | UI 表达要求 | 不得误归类为 |
|---|---|---|---|---|---|
| `pending` | 任务已入队但尚未开始执行，当前没有有效下载数据流 | 新建任务入队；恢复前待调度任务重新入队 | 否 | 显示“等待中”或“排队中”；按钮应允许开始、取消，不允许显示“继续” | `paused`、`failed` |
| `running` | 任务已被调度且正在下载，存在活动执行过程 | 用户开始；用户从 `paused` 恢复；系统按已确认规则恢复未完成运行态 | 否 | 显示进行中状态、进度条与暂停/取消操作 | `pending`、`completed` |
| `paused` | 用户主动暂停，任务保留未完成状态，且重启后不得自动恢复为运行 | 用户点击暂停；批量记录级暂停；暂停全部 | 否 | 显示“已暂停”；按钮应允许继续、取消；不得显示失败提示 | `failed`、`pending` |
| `failed` | 任务因网络、校验、文件系统或其他异常而失败，且需要用户重试或排障 | 下载异常；完整性校验失败；目标不可达；权限错误；未定义恢复候选状态时的异常中断恢复 | 是 | 显示失败状态与失败原因；允许重试，不得显示“已暂停” | `paused`、`canceled` |
| `completed` | 任务已经成功完成，结果产物可用 | 下载完成且满足完成判定条件 | 是 | 显示完成状态；不得再显示恢复或继续入口 | `running`、`failed` |
| `canceled` | 用户主动取消任务，任务终止，不再继续执行 | 用户点击取消；取消整批时对应记录被取消 | 是 | 显示已取消；不得计入失败数；不得显示继续入口 | `failed`、`paused` |

补充口径：
- 若历史实现中使用 `cancelled` 或其他拼写别名，只允许映射到 `canceled`，测试记录必须注明实际存储值与标准语义映射关系。
- 若当前实现不存在独立 `canceled` 状态，而是复用了其他状态，则 TP-01 不能判通过；最多只能记为待补测或待实现，不得把取消混入 `failed` 或 `completed`。

## 7. 六层测试口径与验证重点

### 7.1 配置与数据层
重点看：
- 任务状态是否被正确持久化
- 重启后状态是否原样恢复
- 用户主动暂停/取消是否被写成失败
- `pauseAll()` 后 `running` 与 `pending` 的状态写回是否分流正确

### 7.2 主进程服务层
重点看：
- 状态转移是否由明确操作触发
- 恢复逻辑是否尊重 `paused` 不自动恢复的硬规则
- pause / cancel / fail 是否走不同分支
- `pauseAll()` 是否只把运行中任务写成 `paused`

### 7.3 IPC / preload / shared 边界层
不适用原因：
- 本 TP 以既有下载接口和既有状态消费路径为前提，目标是校验状态语义与状态转移结果，不直接新增或修改 IPC 通道、preload 暴露面、shared 类型协议。
- 但执行时仍需确认：若页面显示状态与主进程状态不一致，应记录为跨层联动问题，并回归 TP-05 / TP-06，而不能在本 TP 内假定协议已变更。

### 7.4 渲染层交互层
重点看：
- 页面状态文案是否与真实状态一致
- 暂停、继续、取消按钮是否随状态正确切换
- 批量列表中的统计数、恢复按钮可用性是否变化正确
- `pauseAll()` 后列表项状态、汇总统计、批量恢复入口是否同步更新

### 7.5 生命周期与异常层
重点看：
- 下载进行中关闭窗口或重启应用后状态是否正确
- 崩溃或异常中断后，运行中任务只能进入已确认的恢复候选状态或 `failed`
- `paused` 在重启后不得自动恢复

### 7.6 非功能层
重点看：
- 稳定性：重复暂停、重复恢复、`pauseAll()` 不得造成状态反复跳变
- 可观测性：必须能从日志、状态写回记录或数据库中还原状态变化序列
- 统计一致性：页面统计数与任务真实状态集合必须一致
- 状态抖动：一次用户操作只允许产生与口径一致的有限状态变更，不得出现 `paused -> pending -> paused`、`paused -> failed -> paused` 等错误序列
- 可恢复性：重启后状态必须能被稳定重建，不能依赖页面首次渲染时的偶然覆盖

## 9. 详细测试用例

### TP-01-TC-001 普通下载暂停后应保持 paused 而非 failed
- 前置条件：
  - 存在一个正在进行的普通下载任务
  - 具备录屏能力，用于保留点击暂停前后状态变化
- 场景/步骤：
  1. 启动普通下载
  2. 下载过程中点击“暂停”
  3. 观察页面状态、日志与任务记录
- 预期结果：
  - 任务状态变为 `paused`
  - 不应被记为 `failed`
  - 页面应显示“已暂停”
- 关键观察点：状态字段、状态文案、按钮是否切换为“继续”

### TP-01-TC-002 普通下载恢复后应回到 running 并继续执行
- 前置条件：
  - 存在 `paused` 状态的普通下载任务
  - 具备日志查看能力，用于确认未重复创建新任务
- 场景/步骤：
  1. 点击“继续”或“恢复”
  2. 观察下载是否继续推进
- 预期结果：
  - 状态从 `paused` 变为 `running`
  - 任务继续下载，不重新创建重复任务
- 关键观察点：进度变化、日志中是否出现重复 start

### TP-01-TC-003 普通下载取消后应进入 canceled
- 前置条件：
  - 已完成 5.4 中的取消终态口径确认
  - 存在 `running` 状态的普通下载任务
  - 具备截图能力，用于保留取消后状态
- 场景/步骤：
  1. 点击“取消”
  2. 检查状态、页面提示、任务记录
- 预期结果：
  - 任务进入 `canceled`
  - 不应被写入 `failed`
  - UI 不应再显示继续下载入口
- 关键观察点：状态文本、进度条停止、后续是否还自动重试

### TP-01-TC-004 普通下载真实异常失败应保留 failed 语义
- 前置条件：可模拟网络断开或服务端返回错误
- 场景/步骤：
  1. 在下载中制造真实异常
  2. 等待任务失败落地
- 预期结果：
  - 任务状态为 `failed`
  - 页面应提示失败原因或失败态
  - 不得显示“已暂停”或“已取消”
- 关键观察点：错误消息、重试按钮、失败计数

### TP-01-TC-005 批量下载单条暂停不应污染会话状态
- 前置条件：批量下载会话中存在至少一条 `running` 记录
- 场景/步骤：
  1. 对其中一条记录执行暂停
  2. 观察列表中该记录与会话汇总状态
- 预期结果：
  - 单条记录为 `paused`
  - 会话汇总不得因此变为 `failed`
  - 其他记录不受影响
- 关键观察点：列表行状态、汇总计数、会话进度

### TP-01-TC-006 批量下载取消后应计入取消数而非失败数
- 前置条件：
  - 已完成 5.4 中的取消终态口径确认
  - 批量下载会话正在执行
- 场景/步骤：
  1. 取消单条记录或取消整批
  2. 观察状态和统计变化
- 预期结果：
  - 被取消条目进入 `canceled`
  - 取消条目必须计入取消数，不得计入失败数
  - 失败数仅统计真实异常失败的条目
- 关键观察点：完成数、失败数、取消数是否分别变化正确

### TP-01-TC-007 重启后 paused 任务不应自动恢复运行
- 前置条件：
  - 存在 `paused` 的普通或批量下载任务
  - 具备重启前后截图能力
- 场景/步骤：
  1. 暂停任务
  2. 退出应用并重新启动
  3. 观察恢复后的状态
- 预期结果：
  - `paused` 仍保持 `paused`
  - 不应在启动阶段被自动改成 `running`
  - 页面应显示手动恢复入口
- 关键观察点：启动日志、页面初始状态、是否有自动拉起行为

### TP-01-TC-008 重启后运行中中断任务与失败任务的恢复口径必须固定
- 前置条件：
  - 已完成 5.4 中的中断恢复状态口径确认
  - 存在一条运行中中断任务和一条 `failed` 任务
- 场景/步骤：
  1. 制造中断并重启应用
  2. 观察两类任务恢复后的状态
- 预期结果：
  - 原运行中但被中断的任务，进入已确认的恢复候选状态；若产品未定义该状态，则统一进入 `failed`
  - 原 `failed` 任务保持 `failed`，不应自动变成 `running`
- 关键观察点：恢复队列、状态重建、页面提示

### TP-01-TC-009 批量页面状态表达应与真实状态一致
- 前置条件：准备 `paused`、`failed`、`completed` 三种记录
- 场景/步骤：
  1. 打开批量下载页面
  2. 分别查看各状态记录的文案、图标、按钮
- 预期结果：
  - `paused`、`failed`、`completed` 三种状态必须有可区分的视觉表达
  - 各状态可用按钮必须与状态口径表一致
  - 不应出现暂停却显示失败图标的情况
- 关键观察点：状态标签、操作按钮、统计栏

### TP-01-TC-010 普通下载暂停后重复操作应幂等
- 前置条件：普通下载已处于 `paused`
- 场景/步骤：
  1. 连续点击暂停或重复触发暂停动作
  2. 再点击继续一次
- 预期结果：
  - 重复暂停不引发错误状态回写
  - 继续只恢复一次，不产生多余请求
  - 状态字段在 `paused -> running` 过程中最多发生一次有效转换，不得出现 `paused -> failed -> paused` 或 `paused -> pending -> running` 之类的错误抖动
- 关键观察点：日志中的重复调用、状态写回序列

### TP-01-TC-011 pauseAll 后 running 必须变为 paused，pending 必须保持 pending
- 前置条件：
  - 普通下载列表中同时存在至少一条 `running` 任务和至少一条 `pending` 任务
  - 具备截图或录屏能力，且可查看状态写回日志
- 场景/步骤：
  1. 记录执行前各任务状态与页面统计
  2. 触发 `pauseAll()`
  3. 观察任务状态写回、列表展示、统计数和按钮状态
- 预期结果：
  - 原本 `running` 的任务必须进入 `paused`
  - 原本已经是 `pending` 的任务必须保持 `pending`
  - 页面统计中暂停数增加、等待数保持与原 `pending` 数量一致
  - 原 `running` 任务显示恢复入口，原 `pending` 任务不得错误显示恢复入口
- 关键观察点：状态写回前后对比、统计区、每行按钮状态、日志中的 `pauseAll()` 状态序列

## 10. 负向与边界场景

### 10.1 负向场景
- 暂停后立即取消，再重启，状态不应被误写为 `failed`
- 失败后手动恢复时，必须明确走恢复逻辑，不应走“重开新任务”逻辑
- 下载目标不存在时，必须进入 `failed`，而不是 `paused` 或 `completed`

### 10.2 边界场景
- 下载刚开始、尚未产生有效进度时暂停
- 下载已接近完成时取消
- 批量任务中只有单条发生错误，整体会话应仍可区分部分失败与整体失败
- 关闭窗口后任务继续执行时，状态展示不能丢失

### 10.3 恢复边界
- 重启时若任务数据不完整，应进入可诊断失败态，或进入已定义的恢复候选状态；若两者都未定义，只能记为待补测，不得判通过
- 对于历史脏数据，需确认是否按新语义重新解释，避免二次污染

- 暂停、取消、失败三者能够稳定区分
- 重启后状态不会无条件自动恢复
- 页面表达与真实状态一致，不出现错标

## 12. 自动化建议

### 12.1 适合自动化的部分
- 状态机转移单测
- `pauseAll()` 对不同初始状态的转移单测
- 下载状态映射表单测
- 恢复逻辑的输入输出断言
- 数据持久化后的状态读取验证

### 12.2 建议自动化方式
- 对纯状态转移逻辑编写单元测试
- 对恢复分支编写服务层测试
- 对页面状态映射编写轻量快照或断言测试

### 12.3 仍需人工确认的部分
- 窗口重启后的真实可见状态
- 按钮是否符合用户预期
- 任务暂停/恢复时进度条与通知是否符合桌面交互习惯

### 12.4 自动化边界
- 自动化不应替代真实下载与重启验证
- 自动化结果只能证明逻辑正确，不能证明桌面交互闭环完全正确

## 13. 缺陷分级与结论判定

### 13.1 缺陷分级
- S0：暂停/取消导致数据丢失、任务误删、应用无法恢复
- S1：暂停/取消/失败语义混乱，影响用户判断和后续恢复
- S2：个别状态文案、图标、按钮不一致，但主链路可用
- S3：轻微文案问题，不影响状态识别

### 13.2 结论判定
满足以下条件可判定本 TP 通过：
1. 暂停、取消、失败、完成的语义均能稳定区分
2. 重启后 `paused` 不会自动恢复
3. 普通下载和批量下载状态表达一致
4. `pauseAll()` 不会把 `running` 回写为 `pending`
5. 已完成相关回归并留存证据

若任一核心状态被误判、取消未落到 `canceled`、`pauseAll()` 回写错误、或重启恢复错误，则本 TP 不通过。

## 15. 阶段验收口径

TP-01 作为阶段 0 的关键止血项，验收时应满足：

1. 用户主动暂停/取消不会污染为 `failed`
2. `paused` 不会在重启后自动恢复运行
3. 页面可以明确表达 `running / paused / failed / completed / canceled` 的差异
4. 普通下载与批量下载的状态口径统一
5. `pauseAll()` 后 `running` 正确进入 `paused`，`pending` 保持 `pending`
6. 相关证据完整，且没有阻断级缺陷未关闭

若仅有部分状态修复完成，但重启恢复、`pauseAll()` 语义或页面表达仍不一致，则只能判定为“待补测/未通过”，不能进入阶段通过结论。
