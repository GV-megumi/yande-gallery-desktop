# TP-01 修下载状态机与暂停恢复取消语义

## 1. 文档目标
验证普通下载与批量下载在 `pending / running / paused / cancelled / failed / completed` 上的状态语义一致，确保用户主动操作不再污染失败态，且重启恢复行为可解释。

## 2. 对应任务包与关联文档
- 任务包：TP-01
- 路线图阶段：阶段 0
- 主责任文档：[10-下载并发与调度链路重构.md](../10-下载并发与调度链路重构.md)
- 关联文档：[03-任务拆分总控.md](../03-任务拆分总控.md)、[00-测试规范总则.md](./00-测试规范总则.md)

## 3. 测试范围
- `src/main/services/downloadManager.ts`
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/init.ts`
- 下载页、批量下载页、应用重启后的恢复链路

## 4. 风险摘要
- 用户暂停被写成失败会污染统计、重试和恢复逻辑
- `pauseAll()` 写回 `pending` 会让 UI 无法区分等待和主动暂停
- 启动自动恢复 `paused` 会破坏用户预期

## 5. 测试前置条件
- 至少准备 3 个普通下载任务和 1 个批量下载会话
- 准备可稳定下载的测试目标，例如：可重复访问、体积较小、不会频繁超时的直链或站点样例文件
- 允许重复启动应用验证恢复链路
- 观测位置：下载页任务列表状态列、批量下载会话卡片、状态筛选结果、目标目录文件列表；启动应用的终端输出或应用日志文件中与下载/批量下载/启动恢复相关的日志行；通过当前项目已有的数据库查看方式检查对应任务/会话/下载记录的状态变化，若当前环境做不到则记为待补测
- 关键用例建议保留：截图、日志摘录、数据库状态对比、重启前后目标目录对比

## 6. 测试分层
- 配置/数据层：核对任务状态是否正确写入数据库，重启后是否保持同一语义
- 主进程服务层：核对暂停、恢复、取消、启动恢复的状态机流转
- IPC / preload 边界层：核对下载页和批量页触发的操作是否进入同一套状态语义
- 渲染层交互层：核对任务列表状态列、批量会话卡片统计和筛选展示
- 生命周期与异常场景层：核对重启、强制退出后的恢复边界

## 7. 详细测试用例

### TP-01-TC-001 普通下载单任务暂停后状态应为 paused
- 目的：验证单任务暂停不再进入 failed
- 步骤：启动单个下载 -> 在进行中点击暂停 -> 刷新列表 / 重启应用后再次查看
- 预期：任务状态为 `paused`；不计入失败统计；不会自动恢复
- 观察项：数据库状态、页面标签、日志输出是否一致
- 建议自动化：是

### TP-01-TC-002 普通下载 pauseAll 后已运行任务不应回写为 pending
- 目的：验证批量暂停语义
- 步骤：同时启动多个普通下载 -> 执行 pauseAll -> 观察列表与数据库
- 预期：原运行中任务进入 `paused`；等待中任务仍保留 `pending`
- 观察项：是否出现全部任务统一回写 `pending`
- 建议自动化：是

### TP-01-TC-003 用户取消普通下载不应被计入 failed
- 目的：验证用户主动取消的语义不落为失败
- 步骤：启动单个普通下载 -> 点击取消 -> 查看列表与详情 -> 重启后再次检查
- 预期：状态明确显示为 `cancelled`；失败统计不增加；不自动重试
- 观察项：页面状态、数据库记录、取消日志
- 建议自动化：是

### TP-01-TC-004 批量下载记录级暂停后不应污染 failed
- 目的：验证批量记录暂停的状态隔离
- 步骤：启动批量下载 -> 对单条记录执行暂停 -> 刷新会话列表
- 预期：目标记录为 `paused`；同会话其他记录状态不受影响；不会记为 `failed`
- 观察项：记录状态、会话统计、日志
- 建议自动化：是

### TP-01-TC-005 批量下载记录级取消后不应污染 failed
- 目的：验证批量记录取消的状态隔离
- 步骤：启动批量下载 -> 对单条记录执行取消 -> 继续观察其余记录完成情况
- 预期：目标记录明确进入 `cancelled`；失败计数不变；其余记录继续按预期运行
- 观察项：记录状态、统计口径、完成/取消日志
- 建议自动化：是

### TP-01-TC-006 应用重启后 paused 任务不会自动恢复
- 目的：验证暂停态的持久化与恢复边界
- 步骤：暂停一条任务 -> 退出并重启应用 -> 观察启动后的下载列表
- 预期：任务仍显示为 `paused`；不会自动进入 `running`
- 观察项：页面状态、数据库状态、启动恢复日志
- 建议自动化：是

### TP-01-TC-007 应用重启后 running/downloading 的恢复行为可解释
- 目的：验证异常退出后的恢复策略
- 步骤：让任务处于运行中时关闭应用进程，或关闭主窗口并执行退出 -> 重启 -> 查看恢复结果
- 预期：仅强制退出前处于 `running` / `downloading` 的任务可恢复；`paused`、`cancelled`、`failed` 任务不得自动恢复；恢复后的状态、数据库记录和日志三者一致，不出现假完成
- 观察项：下载页任务列表状态列、启动恢复相关日志行、数据库状态、页面是否重新进入执行
- 建议自动化：建议做集成测试

### TP-01-TC-008 页面能清晰区分 pending、paused、running、failed
- 目的：验证 UI 状态表达可辨识
- 步骤：构造不同状态任务各 1 条 -> 打开下载页与批量页 -> 对比展示
- 预期：各状态标签、颜色或文案能明确区分；不混淆等待、暂停、运行、失败
- 观察项：页面状态列、筛选结果、会话统计卡片
- 建议自动化：保留人工冒烟

## 8. 负向与边界场景
- 暂停后立即恢复
- 恢复后立刻取消
- pauseAll 与单任务 pause 混用
- 下载失败后再暂停/恢复的非法路径
- 关闭窗口、退出应用、重新打开期间的状态一致性

## 9. 验收标准
- 主动暂停/取消绝不落为 `failed`
- `pauseAll()` 不得把运行态统一回写为 `pending`
- `paused` 只能由显式恢复动作恢复
- 页面、日志、数据库三处状态一致

## 13. 缺陷分级与结论判定

### 13.1 缺陷分级
- S0：暂停 / 取消被错误写成失败、恢复链路错误拉起非运行态任务、状态污染导致主链路不可用
- S1：页面、日志、数据库三处状态不一致，或批量 / 普通下载状态语义明显割裂
- S2：局部筛选、统计、标签表达不准，但主状态流转仍可追踪
- S3：状态文案、日志命名或记录格式仍可优化，不影响状态机主判定

### 13.2 结论判定
满足以下条件可判定本 TP 通过：
1. 主动暂停 / 取消不再落为 `failed`
2. `pauseAll()` 不会把运行中任务错误回写为 `pending`
3. `paused`、`cancelled`、`failed`、`completed` 的恢复边界清晰，重启后行为与规则来源一致
4. 页面、日志、数据库三处状态已形成可回溯证据链
5. 测试记录已完整写明规则来源、实际结果、证据位置与结论

若仍存在状态污染、错误恢复或多处状态表达互相冲突，则本 TP 不通过。

## 15. 阶段验收口径

TP-01 作为阶段 0 的关键下载状态语义任务，验收时应满足：

1. 普通下载与批量下载对暂停、取消、失败、完成的语义不再互相污染
2. 启动恢复只处理允许恢复的运行态任务，不会错误拉起 `paused` 或 `cancelled`
3. 状态筛选、统计与列表展示已能区分 `pending / running / paused / failed / cancelled / completed`
4. 没有阻断级状态机语义错误遗留

本设计文档不承载实际结果、证据路径和最终结论；这些信息必须由测试记录提供。若仅完成局部 UI 对齐，但底层状态写回或恢复边界仍不可信，则只能判定为待补测或未通过，不能进入阶段通过结论。
