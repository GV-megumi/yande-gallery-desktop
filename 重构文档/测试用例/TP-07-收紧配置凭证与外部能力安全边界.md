# TP-07 收紧配置凭证与外部能力安全边界

## 1. 文档目标

本 TP 面向敏感配置越界暴露、renderer-safe DTO 缺失、`config:changed` 广播过宽、备份导出携带敏感信息，以及 `app://`、`openExternal`、webview/navigation/popup 安全边界过宽等问题，验证这些能力是否已重新收回到主进程受控边界。

本任务的测试目标不是“页面还能不能打开链接”，而是证明以下几件事：

1. renderer 不应再直接获取代理密码、Google `clientSecret`、站点凭证材料等敏感字段
2. 配置与站点对象必须以 renderer-safe DTO 形式下发，且变更广播要尽量收窄
3. 备份导出 / 导入不能默认裸带敏感信息，必须明确敏感策略
4. `app://` 只能映射受控目录，`openExternal` 只能打开允许协议和目标
5. webview、导航、新窗口与 popup 必须由主进程明确收口，而不是放任外部内容裸奔

## 3. 测试范围

### 3.1 必测范围
- renderer 侧可见的配置对象是否为脱敏后的 DTO，而非整包 config
- 站点对象、Google 配置、代理配置等敏感字段是否已从 renderer 回收
- `config:changed` 广播是否收敛到必要摘要，避免无差别广播整包配置
- 备份导出 / 导入是否明确处理敏感信息，且不会默认裸露密码类字段
- `app://` 的 root containment 是否有效
- `openExternal` 的协议白名单、目标校验与异常处理是否有效
- webview attach、navigation、popup、新窗口边界是否由主进程约束

### 3.2 不在本次范围内
- IPC 协议单一来源与通道命名统一本身，属于 TP-05
- preload / shared / handler 总体收口本身，属于 TP-06
- 启动时序、首屏优先级与退出治理的主链路，属于 TP-08
- 桌面托盘、后台常驻、通知基础，属于 TP-10

### 3.3 适用层次
- 配置与数据层
- 主进程服务层
- IPC / preload / shared 边界层
- 渲染层交互层
- 生命周期与异常层
- 非功能层

## 4. 风险摘要

1. 这是安全边界任务，失败后果通常不是“页面不好看”，而是敏感字段越界、外链误开、外部内容运行边界过宽等高风险问题
2. renderer 一旦已经依赖宽配置对象，收紧 DTO 时很容易出现局部页面兼容断裂，必须逐项验证调用面
3. `app://` 和 `openExternal` 属于系统级风险边界，测试不能只看一处路径，必须验证正常、非法、边界和恢复场景
4. webview / popup / navigation 边界如果只改主窗口设置不改页面使用方式，容易留下遗漏
5. 备份导出如果没有明确敏感策略，往往会把“可恢复”与“可泄露”混在一起，必须用证据确认实际输出

## 5. 安全边界最小判定口径

为避免不同执行者各测各的，本 TP 的安全验证至少应覆盖以下固定样本；若白名单、目录边界或敏感策略尚未在源文档、实现或测试记录中写清，则相关用例不得判通过。

| 验证对象 | 最低判定口径 |
|---|---|
| renderer-safe DTO | 必须核对配置读取返回值、renderer 可见对象结构、配置变更广播 payload 三类观察对象 |
| 备份敏感策略 | 必须分别检查导出内容与导入处理路径；若只验证导出，不得外推为“导入也通过” |
| `app://` | 必须至少覆盖合法受控路径、目录穿越样本、绝对路径样本三类路径；若项目存在等价越界样本，测试记录必须说明替代关系 |
| `openExternal` | 必须至少覆盖一类允许协议样本和一类不允许协议或异常目标样本，并说明依据来源 |
| webview / navigation / popup / 新窗口 | 四类边界至少逐项记录覆盖结果；若本次未覆盖其中某类，不得写成“外部能力边界全部通过” |

## 6. 测试前置条件

### 6.1 环境前置
- 应用可正常启动
- 可访问配置页、Google 相关页、Booru 相关页、外链入口页面
- 可查看配置返回值、日志或开发者工具，以确认 renderer 实际拿到的字段
- 若需要验证 webview / popup 行为，环境必须允许触发该类页面或其替代页面

### 6.2 数据前置
- 至少准备一组包含敏感字段的配置样本，如代理用户名 / 密码、Google `clientSecret`、站点认证材料等
- 至少准备一个可生成备份的样本
- 至少准备一个可触发外链打开、webview 导航或 popup 的样本页面
- 至少准备一个能观察 `app://` 访问结果的本地资源样本

### 6.3 操作前置
- 必须能观察配置读写、备份文件内容或导出结果；若无法观察，相关用例只能记为环境阻塞
- 必须能判断目标链接协议是否允许；若测试环境无法控制外链目标，应在记录中注明判定依据来源
- 若某安全规则尚未在源文档或实现中明确，不能用测试去反推规则，必须先补齐判断口径

## 7. 测试层次与验证重点

### 7.1 配置与数据层
重点看：
- renderer 读取到的是不是脱敏后的配置 DTO
- 敏感字段是否仍保留在 main 层或安全存储层，而不是直接下发给 renderer
- 备份导出是否按策略过滤或加密敏感字段
- `config:changed` 是否只广播必要摘要或可安全下发的数据

### 7.2 主进程服务层
重点看：
- `config:save`、备份导出、站点对象返回是否在主进程完成脱敏或收口
- `app://` 映射规则是否限制在受控目录
- `openExternal` 是否对协议、目标、异常状态有明确校验
- webview / navigation / popup 约束是否在主进程集中执行

### 7.3 IPC / preload / shared 边界层
重点看：
- renderer-safe DTO 是否通过受控 API 下发
- renderer 是否无法通过 preload 间接触到敏感字段
- 变更广播是否经过收口，而不是把整包 config 直接推给所有窗口
- 若存在站点对象或配置 DTO 的共享类型，字段是否只保留可公开部分

### 7.4 渲染层交互层
重点看：
- 页面是否还能正常消费脱敏后的配置
- 外链、webview、popup 是否仍能通过用户可理解入口使用
- 安全收紧后是否出现页面明显断裂、提示不清或入口失踪
- `app://` 访问失败时页面是否给出清晰而非误导性的反馈

### 7.5 生命周期与异常层
重点看：
- 主窗口重建、reload、重新进入页面后安全边界是否仍生效
- 导出、打开链接、webview 导航发生异常时是否有明确失败返回
- 退出或异常恢复后，敏感字段不会被缓存回 renderer 可见层

### 7.6 非功能层
重点看：
- 敏感字段是否真正从外层暴露面消失
- 安全边界是否减少外部攻击面或误操作面
- 过度收紧是否造成必要功能不可用，并需要明确说明例外理由
- 收紧后的规则是否便于长期审计

## 8. 测试设计文档与测试记录承接说明

本文档只负责定义测试设计、范围、步骤、预期结果与判定口径，不承载某次实际执行结果。

实际执行时，每条用例的以下信息必须由测试记录承接，而不是由本设计文档承载：
- 实际结果
- 证据路径 / 附件名
- 结论（通过 / 失败 / 待补测 / 环境阻塞 / N/A）
- 本次判定所引用的规则来源

如果测试记录缺少这些字段，或仅写“敏感字段看不到了”“链接大概正常”，则该用例不得判通过。对安全边界类验证，测试记录必须明确写出依据来源，例如 `../15-稳定性错误恢复与安全性修正.md`、`../12-IPC与进程边界重构.md` 或 `../03-任务拆分总控.md`。

## 9. 详细测试用例

### TP-07-TC-001 renderer 不应直接获取敏感配置与凭证字段
- 测试目的：确认敏感字段已从 renderer 收回
- 前置条件：配置中包含代理密码、Google `clientSecret`、站点认证材料等敏感字段
- 场景/步骤：
  1. 在 renderer 侧触发配置读取
  2. 观察返回对象或调试输出
  3. 检查是否仍出现敏感字段
  4. 在测试记录中写明判断依据来源
- 预期结果：
  - renderer 只能看到脱敏后的 DTO
  - 代理密码、clientSecret、站点密码材料不得直接出现
  - 测试记录必须注明引用来源
- 关键观察点：字段列表、对象结构、是否仍有敏感键名

### TP-07-TC-002 配置变更广播不应默认推送整包敏感对象
- 测试目的：验证 `config:changed` 是否已收口
- 前置条件：可触发配置保存并观察广播/订阅结果
- 场景/步骤：
  1. 修改一项普通配置
  2. 保存并观察所有窗口的变更事件
  3. 检查广播内容是否包含整包 config 或敏感字段
- 预期结果：
  - 广播内容应尽量收窄到摘要或安全 DTO
  - 不应把完整敏感对象直接广播给所有窗口
- 关键观察点：广播字段、订阅回调参数、是否有无关字段暴露

### TP-07-TC-003 备份导出不应默认裸带敏感信息
- 测试目的：确认备份策略不会直接泄露敏感字段
- 前置条件：可执行备份导出并查看导出结果
- 场景/步骤：
  1. 使用包含敏感字段的配置执行备份
  2. 查看备份文件或导出内容
  3. 检查敏感字段是否明文存在
- 预期结果：
  - 备份应按策略脱敏或加密敏感字段，或明确说明不包含这些字段
  - 不应默认明文导出代理密码、clientSecret 等
- 关键观察点：备份内容、是否明文、导出策略说明

### TP-07-TC-004 备份导入不应把敏感字段处理路径留空
- 测试目的：确认导入链路同样遵循敏感策略，而不是只在导出侧收紧
- 前置条件：可执行备份导入，且测试记录已注明本次导入策略来源
- 场景/步骤：
  1. 准备包含敏感字段样本的备份文件或等价测试样本
  2. 执行导入
  3. 观察导入过程中敏感字段是被拒绝、脱敏恢复、单独提示还是走受控恢复链路
  4. 检查 renderer 与主进程可见结果
- 预期结果：
  - 导入链路必须对敏感字段有明确处理策略
  - 不应出现“导出已脱敏，导入却默认接受明文敏感字段且无说明”的断裂
  - 若当前仅实现导出侧策略，导入侧本用例不得判通过
- 关键观察点：导入日志、敏感字段处理结果、配置恢复结果、规则来源

### TP-07-TC-005 `app://` 应只允许访问受控目录
- 测试目的：确认本地资源协议边界收紧
- 前置条件：可访问一个合法受控本地资源，以及目录穿越样本和绝对路径样本；若使用替代越界样本，测试记录必须说明替代关系
- 场景/步骤：
  1. 访问合法受控目录下的资源
  2. 尝试访问目录穿越样本
  3. 尝试访问绝对路径样本或等价越界样本
  4. 观察结果与错误信息
- 预期结果：
  - 合法目录可正常访问
  - 目录穿越与绝对路径类越界样本应被拒绝或安全失败
  - 不应映射任意本地路径
- 关键观察点：访问结果、错误提示、目录穿越痕迹、绝对路径处理结果

### TP-07-TC-006 `openExternal` 只能打开允许协议与目标
- 测试目的：验证外部链接打开白名单
- 前置条件：可触发外链入口，且已根据规则来源准备允许协议样本与不允许协议或异常目标样本
- 场景/步骤：
  1. 点击一个允许的外部链接
  2. 点击一个不允许的协议或异常目标
  3. 观察是否被打开或被拦截
  4. 在测试记录中写明本次允许/拒绝样本的依据来源
- 预期结果：
  - 允许协议可打开
  - 不允许协议或目标应被拦截并给出明确反馈
  - 不应出现无校验直接拉起系统外部程序
  - 若未写明允许/拒绝样本依据来源，则本用例不得判通过
- 关键观察点：协议类型、拦截反馈、系统行为、规则来源

### TP-07-TC-007 webview attach 应由主进程明确收口
- 测试目的：确认 webview 附着行为不再裸奔
- 前置条件：可触发 webview attach 的页面；必须在测试记录中同时准备并写明三类信息：1）允许 attach 样本，例如项目已明确允许的受控域名、页面或嵌入来源；2）非法或越界样本，例如未列入白名单的域名、协议异常样本、跨边界来源或等价未授权 attach 来源；3）规则来源，如 `../11-Electron桌面能力重构.md`、`../15-稳定性错误恢复与安全性修正.md`、实现白名单或主进程校验代码。若无法准备允许样本、非法样本或规则来源中的任一项，本用例只能记为待补测或环境阻塞，不得判通过
- 场景/步骤：
  1. 打开可触发 webview attach 的页面并记录本次允许样本
  2. 触发允许样本的 attach，观察是否按主进程规则放行
  3. 使用非法或越界样本再次触发 attach，观察是否被拦截或安全失败
  4. 核对主进程日志、白名单命中结果或等价拦截证据
  5. 若本次只能验证正常路径，必须在测试记录中明确记为待补测或环境阻塞，不能判通过
- 预期结果：
  - 非法或越界 attach 应被拦截
  - 允许 attach 应按主进程规则执行
  - 测试记录必须同时写明允许样本、非法样本与规则来源
  - 不能只测一次正常路径就判通过
- 关键观察点：允许样本、非法样本、attach 结果、主进程拦截日志、白名单命中情况、规则来源

### TP-07-TC-008 navigation 应由主进程明确收口
- 测试目的：确认页面导航不会绕过主进程边界
- 前置条件：可触发页面内跳转或外部导航行为；必须在测试记录中同时写明三类信息：1）允许样本，例如主进程规则允许跳转的受控站内路径、受控域名或等价合法导航目标；2）非法或越界样本，例如未列入白名单的导航目标、异常协议、跨边界跳转或等价未授权导航目标；3）规则来源，如 `../11-Electron桌面能力重构.md`、`../15-稳定性错误恢复与安全性修正.md`、实现白名单或主进程导航校验代码。若缺少允许样本、非法样本或规则来源中的任一项，本用例只能记为待补测或环境阻塞，不得判通过
- 场景/步骤：
  1. 打开相关页面并记录本次允许样本
  2. 触发允许的导航样本，观察是否按主进程规则执行
  3. 触发非法或越界导航样本，观察是否被拦截、重定向或安全失败
  4. 检查主进程导航日志、白名单命中结果或等价拦截证据
  5. 若本次只覆盖正常路径，或未能取得允许样本、非法样本、规则来源中的任一项，只能记录为待补测或环境阻塞，不能判通过
- 预期结果：
  - 非法导航应被拦截
  - 合法导航应按主进程规则执行
  - 测试记录必须同时写明允许样本、非法样本与规则来源
  - 仅覆盖正常路径或缺少允许样本、非法样本、规则来源中的任一项，不得判通过
- 关键观察点：允许样本、非法样本、导航去向、拦截结果、主进程日志、白名单命中情况、规则来源

### TP-07-TC-009 popup / 新窗口应由主进程明确收口
- 测试目的：确认 popup 与新窗口不会绕过边界直接弹出
- 前置条件：可触发 popup 或新窗口行为的页面；必须在测试记录中同时写明：1）允许样本，例如主进程规则允许打开的目标域名、窗口类型或受控跳转目标；2）非法或越界样本，例如未列入白名单的弹窗目标、异常协议、跨边界新窗口目标或等价未授权样本；3）规则来源，如 `../11-Electron桌面能力重构.md`、`../15-稳定性错误恢复与安全性修正.md`、实现白名单或窗口打开策略。若缺少允许样本、非法样本或规则来源中的任一项，本用例只能记为待补测或环境阻塞，不得判通过
- 场景/步骤：
  1. 打开相关页面并记录本次允许样本
  2. 触发允许的 popup 或新窗口行为，观察是否按主进程规则执行
  3. 触发非法或越界样本，观察是否被拦截、重定向或安全失败
  4. 检查白名单命中、窗口策略日志或等价主进程证据
  5. 若本次未能取得非法样本或白名单样本，只能记录为待补测或环境阻塞，不能判通过
- 预期结果：
  - 非法 popup / 新窗口应被拦截
  - 允许行为应按主进程规则执行
  - 测试记录必须同时写明允许样本、非法样本与规则来源
  - 不能只验证一次正常弹窗路径就判通过
- 关键观察点：允许样本、非法样本、窗口打开情况、拦截结果、白名单命中情况、规则来源

### TP-07-TC-010 安全收紧后页面仍应保持基本可用性
- 测试目的：确认安全边界收紧未误伤必要功能
- 前置条件：配置页、Google 页、Booru 页、外链入口均可访问；本用例必须按固定样本规则选取页面，不得泛泛写“Google 或 Booru 页面”：1）配置页固定为 Settings 中可读取非敏感配置的入口；2）外部能力页固定为实际承载 webview、navigation、popup 或外链动作的一个已知页面样本；3）若 Google 页面当前不可用，则可用一个已知 Booru 外链页面替代，但测试记录必须注明替代对象与原因
- 场景/步骤：
  1. 打开 Settings 中的固定样本入口并查看非敏感配置
  2. 打开前置条件中选定的固定外部能力页样本
  3. 使用该页面上的一个受控正常入口完成一次允许动作，例如受控外链、受控导航或受控 webview 加载
  4. 观察收紧后页面是否仍能完成主路径，并记录如有例外的限制说明
- 预期结果：
  - 必要功能可用
  - 收紧后页面不应大面积失效
  - 若某能力被限制，应有明确说明而不是静默失败
  - 测试记录必须写明本次固定样本页面及替代关系
- 关键观察点：固定样本页面、功能可用性、提示文案、是否仍能完成主路径、替代关系说明

## 10. 负向与边界场景

### 10.1 负向场景
- renderer 若仍能拿到明文密码或 clientSecret，属于严重越界
- `openExternal` 若接受任意协议或奇怪目标，属于高风险边界失败
- 备份若明文导出敏感字段，应视为阻断级问题
- webview / popup 若能绕过主进程白名单，属于安全控制失守

### 10.2 边界场景
- 某些场景下允许的协议可能只限 `http/https` 或特定白名单，必须按源文档或实现确认，不能靠猜
- 受控目录之外的资源访问若需要临时例外，必须写清原因和范围
- 导出策略若存在脱敏 / 加密 / 不包含三种方案，必须在测试记录里明确实际采用哪一种
- webview 与 navigation 在不同页面可能有不同策略，必须逐页确认而不是用一条规则覆盖全部

### 10.3 待补测前置
- 若安全白名单或导出策略尚未明确，先补齐规则再判通过
- 若无法查看备份文件内容或调试输出，相关敏感字段验证只能记为环境阻塞

## 12. 自动化建议

### 12.1 适合自动化的部分
- DTO 字段白名单测试
- 敏感字段暴露扫描
- `app://` 路径 containment 测试
- `openExternal` 协议白名单测试
- 备份导出内容的敏感字段扫描

### 12.2 建议自动化方式
- 对配置读取结果写字段断言
- 对备份输出写敏感键名扫描
- 对外链和协议校验写单元测试
- 对 `app://` 资源解析写边界测试

### 12.3 仍需人工确认的部分
- 页面对安全收紧后的用户体验是否仍清晰
- webview / popup 的真实运行行为是否被正确拦截
- 备份脱敏 / 加密策略的可理解性
- 对允许与禁止协议的交互反馈是否足够明确

### 12.4 自动化边界
- 自动化可证明“没有泄露敏感键名”“路径被拒绝”“协议被拦截”，但不能完全替代真实页面交互与系统窗口行为
- 对 webview、popup、外链唤起与备份可读性，仍需人工复核

## 13. 缺陷分级与结论判定

### 13.1 缺陷分级
- S0：敏感凭证暴露、任意路径访问、任意外链拉起、webview 越权、备份明文泄露
- S1：安全白名单不完整，或页面被明显误伤导致关键功能不可用
- S2：局部提示、边界文案、部分协议说明不清晰，但主流程可用
- S3：说明文字、命名或记录格式可继续优化，不影响安全判定

### 13.2 结论判定
满足以下条件可判定本 TP 通过：
1. renderer 不再直接获取敏感配置与凭证字段
2. 配置广播与备份策略已收紧，不默认裸带敏感内容
3. `app://` 目录 containment 与 `openExternal` 协议 / 目标校验有效
4. webview / navigation / popup 由主进程受控
5. 测试记录已写明规则来源、实际结果、证据路径和结论

若仍存在敏感字段越界、任意路径映射或外部能力裸奔，则本 TP 不通过。
