# TP-04 修页面伪能力与功能闭环

## 1. 文档目标

本 TP 面向“页面已经暴露给用户，但能力并未真正闭环或兑现”的问题，验证伪设置、假按钮、失败后无恢复路径、配置保存与运行行为不一致、以及图集删除等高频闭环是否真正完成。

本任务的验证目标不是“页面看起来可以点”，而是证明以下几件事：

1. 用户可见的设置要么真的生效，要么被明确移除/禁用并说明原因
2. 页面按钮和入口要么能走通完整闭环，要么不再以可操作形态出现
3. 配置保存、运行时行为和页面反馈必须一致，不能出现“保存成功但实际没变”的断裂
4. Google、SavedSearch、InvalidImages、FavoriteTags、Gallery 等页面的失败态、重试态、退路与删除/改名等操作闭环可解释、可回归

## 4. 风险摘要

1. 本 TP 涉及的页面很多，但问题本质并不相同：有的是伪设置，有的是调用签名不一致，有的是失败态缺失，有的是删除闭环缺失，必须按页面逐类验证，不能用一个结论覆盖全部
2. `SettingsPage`、`FavoriteTagsPage`、`GalleryPage` 这类高频管理页与配置/数据库/IPC 可能共用边界，局部修复容易带出新漂移
3. Google webview 页面存在失败后仅停止 loading 的情况，若只看视觉状态很容易误判为“加载完成”或“空数据”
4. 图集删除与改名类操作涉及不可逆语义，若确认文案不清晰，容易把“删除图集记录”误解成“删除文件”
5. 部分行为是否应“立即生效”还是“保存后需重启”需以源文档和实现为准；若当前产品规则尚未明确，必须标记为待补测前置，而不能写成“按设计即可”

## 5. 测试前置条件

### 5.1 环境前置
- 应用可正常启动，且可进入 Settings、FavoriteTags、SavedSearch、Gallery、InvalidImages、Google 相关页面
- 必须具备可观察日志、调试输出或开发者工具的条件，以便核对保存、调用和失败原因
- 必须准备至少一个可正常返回结果的 Google / webview 相关测试路径；若无可用账号或测试目标，则对应 Google 页失败/重试类用例记为环境阻塞

### 5.2 数据前置
- 至少准备一个可编辑的 FavoriteTags 图集绑定样本
- 至少准备一个 SavedSearch 样本，便于分别验证新建态和编辑态
- 至少准备一个 Gallery 样本，且该样本允许执行“删除图集记录”或“改名”相关操作的观察
- 至少准备一个 InvalidImages 场景样本，用于验证失败态与空态分离
- 至少准备一个普通下载页面中的单条记录，用于验证取消入口是否真实闭环

### 5.3 操作前置
- 必须具备截图或录屏能力；凡涉及保存后对比、失败重试、删除确认、Google 退路的用例，不保留证据不得判通过
- 若要验证“保存后是否真正生效”，必须同时能观察配置落点和运行时表现；如果只能看到表单状态，相关用例只能记为待补测
- 若某页面暂时没有可测数据源，不能用“页面为空”代替失败验证；需要标记为环境阻塞或补测前置

## 6. 测试层次与验证重点

### 6.1 配置与数据层
重点看：
- 伪设置是否真的写入了正确的配置 / 持久化位置
- 保存后配置值是否能在重启后恢复
- 页面偏好、下载默认值、路径回填是否进入正确的数据层，而不是混入整包 config
- 若某设置当前明确要求重启生效，则 UI 提示和实际行为是否一致

### 6.2 主进程服务层
重点看：
- `config:save`、页面偏好保存、下载绑定保存等主进程写入链路是否正确
- 保存后是否有必要的运行时同步，或是否明确提示需要重启
- Gallery 删除、SavedSearch 更新、FavoriteTags 回填等操作是否走到正确 service
- 错误发生时是否能返回明确失败原因，而不是只在页面里静默失败

### 6.3 IPC / preload / shared 边界层
重点看：
- SavedSearch 编辑态的调用签名是否与 preload / IPC 定义一致
- 页面调用是否走受控 API，而不是依赖隐式字段或过时参数形式
- Google 页面、Gallery 删除、FavoriteTags 回填等是否存在签名漂移
- 如果页面需要新增能力，是否已经通过正式 bridge 暴露，而不是临时绕过

### 6.4 渲染层交互层
重点看：
- 伪设置是否仍然可点击、可保存、可误导用户
- 假按钮是否仍然上屏
- 失败后是否有可见恢复入口
- 页面提示“保存成功”后，运行态是否真的变化
- 图集删除、改名、重试等关键动作是否能在页面主路径上被找到

### 6.5 生命周期与异常层
重点看：
- Google 页面失败后刷新、重新进入页面、重试时是否仍停留在正确状态
- Settings 保存后重启再打开，行为是否保持一致
- 页面重载、窗口切换、外部打开返回后，是否仍保留应保留的状态
- 若产品要求重启生效，重启后的恢复是否符合提示语义

### 6.6 非功能层
重点看：
- 高失败率或反复重试时，页面是否产生重复提示噪音
- 图集删除、Google 重试等操作是否有明显误触风险
- 大量页面状态更新时，UI 是否存在明显卡顿或状态闪烁
- 错误提示是否清晰、可追踪、可回归

## 8. 详细测试用例

### TP-04-TC-001 Settings 伪设置不应继续作为可兑现能力上屏
- 测试目的：确认 Settings / BooruSettings 中已知伪设置不会继续误导用户。本用例只负责通用伪设置扫描与初筛，不替代 `pageMode` 等专项验证；涉及专项行为的结论应以对应专项用例为准。
- 前置条件：可打开设置页，且能观察保存后的配置 / 运行效果
- 场景/步骤：
  1. 打开 Settings 或 BooruSettings 页面
  2. 逐项检查已知疑似伪设置、未接线设置、开发中入口
  3. 尝试修改并保存
  4. 重启应用或重新进入页面观察是否真的生效
- 预期结果：
  - 伪设置应被移除、禁用或明确标注为不生效
  - 若设置要求重启生效，页面必须明确提示且重启后结果一致
  - 不应出现“保存成功但运行无变化”
  - 本用例只能形成通用扫描结论，不能替代 `TP-04-TC-003` 等专项行为验证
- 关键观察点：保存按钮反馈、重启前后状态对比、配置写回位置

### TP-04-TC-002 Settings 页假按钮不应保持可点击误导
- 测试目的：确认“功能开发中”等假按钮不再作为可用入口
- 前置条件：Settings 页面可访问
- 场景/步骤：
  1. 查找页面上的“功能开发中”“TODO”“占位按钮”类入口
  2. 点击相关按钮
- 预期结果：
  - 要么按钮不存在，要么被禁用并给出明确说明
  - 不应点击后只弹无意义提示却不承接任何后续能力
- 关键观察点：按钮可见性、禁用态、提示文案是否说明真实状态

### TP-04-TC-003 BooruSettings 的 `pageMode` 保存后应有明确运行语义
- 测试目的：对 `pageMode` 做专项验证，确认其不再只是写入但不产生明确行为结果。本用例优先级高于 `TP-04-TC-001` 的通用伪设置扫描，`pageMode` 是否成立只能以本专项用例结论为准。
- 前置条件：可进入 BooruSettings / 对应 Booru 页面；执行前必须在测试记录中注明本用例的判定依据来源（需求文档、任务文档、代码实现、产品确认记录等）；若没有明确 source of truth，则本用例只能记为“待补测前置”或“环境阻塞”，不得判通过
- 场景/步骤：
  1. 修改 `pageMode`
  2. 保存设置
  3. 回到 Booru 主线页面观察列表行为、分页或展示策略是否变化
  4. 重启后再次确认
- 预期结果：
  - 若判定依据明确说明 `pageMode` 用于行为切换，则页面行为应随之变化
  - 若当前仍无真实消费链路，应明确标记为待补测前置或移除该入口
  - 本用例的专项验证结论优先于通用伪设置扫描结论
- 关键观察点：页面行为变化、配置持久化位置、是否有重启要求提示；测试记录中是否写明判定依据来源，若未写明则不得判通过

### TP-04-TC-004 `autoGenThumbnail` 应完成保存与运行链路闭环

- 场景/步骤：
  1. 打开 Settings
  2. 修改 `autoGenThumbnail`
  3. 保存并重启或触发相关扫描/导入行为
  4. 观察缩略图是否按开关行为发生变化

  - 开关值应被正确保存
  - 运行时行为应与开关状态一致
  - 不应出现 UI 显示已开启但实际不生效
- 关键观察点：配置值、缩略图生成日志、页面提示

### TP-04-TC-005 dataPath 保存后运行时路径基准应与提示一致
- 测试目的：确认 `dataPath` 这类路径项保存后不会出现 UI / 运行时不一致
- 前置条件：可修改 `dataPath` 且能观察实际数据路径；执行前必须在测试记录中注明本用例的判定依据来源（需求文档、任务文档、代码实现、产品确认记录等）；若没有明确 source of truth，则本用例只能记为“待补测前置”或“环境阻塞”，不得判通过
- 场景/步骤：
  1. 修改 `dataPath`
  2. 保存设置
  3. 按当前已确认规则判断是立即同步还是要求重启
  4. 观察数据库、缓存、下载、缩略图等路径基准是否切换
- 预期结果：
  - 若判定依据明确要求立即生效，则运行时应切换到新路径
  - 若判定依据明确要求重启生效，则页面必须明确写明，且重启后生效
  - 不允许“保存成功，但实际仍读旧路径”且无提示
- 关键观察点：路径基准、重启提示、实际写入位置；测试记录中是否写明判定依据来源，若未写明则不得判通过

### TP-04-TC-006 FavoriteTags 下载路径回填应形成闭环
- 测试目的：确认 FavoriteTags 的路径选择与图集回填链路真正闭环
- 前置条件：存在已绑定或未绑定 gallery 的 FavoriteTags 样本
- 场景/步骤：
  1. 打开 FavoriteTags 页面
  2. 检查下载路径是否由 gallery 回填或由独立选择器提供
  3. 在已绑定 gallery 的场景下修改或确认下载路径并实际保存配置，必要时继续启动一次下载
  4. 在未绑定 gallery 的场景下尝试保存配置或启动下载
  5. 观察保存结果、启动结果与页面提示
- 预期结果：
  - 已绑定场景应能正确回填并保存路径，且保存后的配置或启动下载行为与页面显示一致
  - 若继续启动下载，下载应使用已保存或已回填的有效路径
  - 未绑定场景若无独立选择器，则必须在用户可理解节点给出明确提示，阻止继续保存或启动；不得静默失败、不得只写日志不提示
- 关键观察点：路径显示、回填时机、保存是否成功、启动下载后的实际路径、无法绑定时的错误提示是否出现在用户可理解节点

### TP-04-TC-007 FavoriteTags 下载配置默认值应与已确认策略一致

  2. 检查 auto sync、concurrency、每页数量等默认值
  3. 与当前已确认产品策略比对
  4. 保存一次配置并重新打开弹窗，确认回填值与默认来源一致
- 预期结果：
  - 默认值应与已确认策略一致
  - 若某默认值尚未最终确认，应明确标记为待补测前置，不能直接写通过
  - 保存后再次打开时，回填值与默认规则或已保存值之间的关系必须可解释
- 关键观察点：默认值来源、表单回填值、保存后结果；测试记录中是否写明判定依据来源，若未写明则不得判通过

### TP-04-TC-008 SavedSearch 编辑态应与新建态保持一致的站点与更新签名
- 测试目的：确认 SavedSearch 编辑不再出现站点与更新参数漂移
- 前置条件：存在一条可编辑的 SavedSearch；执行前必须在测试记录中注明本用例的规则来源 / source of truth（如 `src/preload/index.ts` 暴露面、对应 IPC channel 定义、shared 类型、任务文档或可定位日志）；若无法明确签名来源，则本用例不得判通过
- 场景/步骤：
  1. 新建一条 SavedSearch，记录站点与查询条件
  2. 再进入编辑态修改名称、查询或站点
  3. 保存前核对本次编辑链路所依赖的 preload / IPC / shared 签名来源，或准备可映射到签名字段的日志观察点
  4. 执行保存并重新打开
  5. 核对保存请求中的参数结构、字段名与站点字段是否与规则来源一致；若只能观察到 UI “保存成功”，但无法核对跨层签名或日志参数结构，则本用例不得判通过
- 预期结果：
  - 编辑态应保留或合理展示站点信息
  - 保存时调用签名应与已注明的 preload / IPC / shared 规则来源一致
  - 不应出现保存成功但实际站点未更新的错配
  - 若 source of truth 不明确，或日志 / 参数结构无法证明跨层一致性，则只能记为待补测前置或环境阻塞，不得判通过
- 关键观察点：规则来源 / source of truth、编辑表单字段、保存请求参数结构、保存后回填、异常日志

### TP-04-TC-009 Google 页面失败后应提供可见重试或退路
- 测试目的：确认 Google Drive / Photos / Gemini 页面不会把失败伪装为完成
- 前置条件：可制造 webview 加载失败或使用无效目标
- 场景/步骤：
  1. 打开 Google 页面，先记录一次正常加载或可进入可用态的最小成功对照表现，例如页面主体内容、已登录态提示、可点击的主区域或可用导航入口
  2. 触发加载失败
  3. 观察页面是否显示错误态、重试入口或外部打开退路
  4. 对比失败终止态与步骤 1 的最小成功对照表现，确认两者不会被误判为同一种“加载完成”
- 预期结果：
  - 失败应进入显式错误态
  - 页面应提供重试、刷新或退路，而不是只停止 loading
  - 失败终止态应与最小成功对照表现可区分，不应与空白页或成功完成态混淆
- 关键观察点：最小成功对照表现、错误文案、重试按钮、是否还能重新进入可用状态

### TP-04-TC-010 InvalidImages 失败态不得伪装为空态
- 测试目的：确认无效图片页在请求失败时不会显示“没有无效图片”
- 前置条件：可模拟请求失败或断开数据源
- 场景/步骤：
  1. 打开 InvalidImages 页面
  2. 制造请求失败
  3. 观察页面状态与文案
- 预期结果：
  - 应显示显式错误态
  - 不应落入空态文案
  - 若有重试入口，应可继续恢复
- 关键观察点：空态和错态分支、错误提示、重试行为

### TP-04-TC-011 Gallery 删除 UI 应形成“只删记录不删文件”的闭环
- 测试目的：确认 Gallery 删除入口真实可见、可确认、语义清晰，且删除结果在记录层真正生效
- 前置条件：存在可删除的图集记录，并能在删除前后通过页面刷新、重新进入页面或可定位的数据层结果核对该记录是否仍存在
- 场景/步骤：
  1. 打开 Gallery 页面并找到目标图集，记录其可唯一识别的信息
  2. 尝试从主路径或右键路径触发删除
  3. 查看确认文案
  4. 执行删除后检查文件是否仍在本地
  5. 刷新当前页面、切离后重新进入页面，或通过可定位的数据层结果核对该图集记录是否已删除
  6. 确认删除后的目标记录不再在列表、详情入口或对应数据层结果中复现
- 预期结果：
  - 删除入口必须可发现
  - 确认文案必须明确仅删除图集记录 / 关联记录，不删除文件
  - 删除后不应误删本地文件
  - 删除后的图集记录应真实消失，刷新、重进页面或核对数据层后不应复现
- 关键观察点：入口位置、确认弹窗文案、文件系统保留情况、页面刷新后记录是否消失、重进后是否复现、数据层核对结果

### TP-04-TC-012 Gallery 图集改名应可闭环
- 测试目的：确认图集编辑首期至少支持改名
- 前置条件：存在可改名图集
- 场景/步骤：
  1. 打开图集详情或编辑入口
  2. 修改名称并保存
  3. 重进页面确认新名称持久化
- 预期结果：
  - 改名真实生效并持久化
  - 页面展示与数据库结果一致
- 关键观察点：编辑入口、保存反馈、重进后名称

### TP-04-TC-013 下载页单条取消入口不应仍为 TODO
- 测试目的：确认单条取消按钮不再是伪入口
- 前置条件：存在一条可操作的下载记录
- 场景/步骤：
  1. 打开 Download 页面
  2. 定位单条取消按钮
  3. 执行取消并观察状态变化
- 预期结果：
  - 取消能力应真实存在或入口被移除/禁用
  - 不应点击后只看到 TODO 或无动作
- 关键观察点：按钮文案、是否有真实动作、状态是否改变

## 9. 负向与边界场景

### 9.1 负向场景
- 保存失败后再次保存，不能把旧值误当成新值成功
- 站点编辑态缺少站点字段时，不能静默保存为错误数据
- Google 页面失败时，不能把错误当成空页面
- 删除图集时若权限不足，应明确失败原因，不能误导为已删除

### 9.2 边界场景
- `dataPath` 切换后，已有数据源尚未重建时应有明确过渡提示
- FavoriteTags 没有 gallery 绑定时，路径回填必须明确给出阻塞原因
- SavedSearch 在新建与编辑之间切换时，表单状态不能串值
- InvalidImages 在首次加载、过滤后无结果、请求失败三种场景下必须区别表达
- 图集删除二次确认重复触发时应幂等，不得重复删除或报错

### 9.3 待补测前置
- 若某一设置是否“立即生效”在现有源文档和实现里尚未明确，必须先补齐产品规则后再判通过
- 若 Google 页面当前无可用测试账号或网络环境，相关失败 / 重试 / 退路用例只能记为环境阻塞

## 11. 自动化建议

### 11.1 适合自动化的部分
- 伪设置/假按钮清单扫描
- 配置读写与默认值断言
- SavedSearch 调用签名与参数结构断言
- 页面状态映射纯函数测试
- 删除确认文案/能力存在性快照测试

### 11.2 建议自动化方式
- 为 Settings / FavoriteTags / SavedSearch 的配置逻辑编写单元测试
- 为 Google / InvalidImages 的状态机编写轻量服务层或组件层断言
- 为 Gallery 删除确认文案、是否存在真实入口编写快照或 DOM 断言测试

### 11.3 仍需人工确认的部分
- Google webview 失败后的真实视觉表现
- 保存成功后运行行为是否真正变化
- 图集删除是否足够清晰地表达“只删记录不删文件”
- 重试/退路是否足够直观

### 11.4 自动化边界
- 自动化可证明“接口和状态存在”，但不能替代真实页面交互闭环确认
- Google webview、删除确认、重启后路径变化等场景必须保留人工验证

## 12. 缺陷分级与结论判定

### 12.1 缺陷分级
- S0：伪能力导致关键数据错误、误删文件、核心页面完全不可用
- S1：设置/按钮/失败态存在明显闭环断裂，用户会被误导并影响后续操作
- S2：个别页面文案、默认值、重试入口位置不一致，但主流程可继续
- S3：轻微提示文本不清晰，不影响闭环本身

### 12.2 结论判定
满足以下条件可判定本 TP 通过：
1. 已知伪设置与假按钮不再作为可兑现能力上屏
2. Google、InvalidImages、SavedSearch、FavoriteTags、Gallery 的关键闭环已形成明确结果
3. 配置保存后运行行为或重启行为与提示一致
4. 相关回归已完成并留存证据

若仍存在可点击但无闭环的入口、失败态伪装为空态、或删除语义不清，则本 TP 不通过。
