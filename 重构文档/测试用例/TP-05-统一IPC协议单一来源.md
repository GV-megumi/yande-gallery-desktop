# TP-05 统一 IPC 协议单一来源

## 1. 文档目标

本 TP 面向 IPC 协议来源分裂、调用签名漂移、`handlers-full.ts` 造成的结构歧义，以及高频页面多入口调用链不一致的问题，验证主进程、preload、shared、renderer 四层是否已经收敛到同一套可信协议源。

本任务的测试目标不是“接口能不能用”，而是证明以下几件事：

1. IPC 通道、参数、返回值、事件名、订阅名必须存在单一可信来源，不能再由多个文件各说各话
2. `channels.ts`、`handlers.ts`、`preload/index.ts`、`shared/types.ts`、renderer 调用点之间的签名必须同源一致，已证实的漂移项必须消失
3. `handlers-full.ts` 不能继续承担“像正式实现又像 mock 归档”的歧义角色，必须在文档、文件职责和调用链上变得可解释
4. Gallery、Favorites、Character、Settings 等高频页面的 IPC 调用链在收口后仍可正常工作，且不会因协议统一而引入静默断裂
5. 新增或修改协议时，必须能在测试记录中明确引用规则来源，不允许只写“按设计”就判通过

## 4. 风险摘要

1. 本 TP 牵涉四层边界，一旦只修声明不修实际调用，就会制造“看上去统一，实际更漂移”的假收敛
2. `handlers-full.ts` 若继续保留歧义，后续维护者很容易误把它当正式协议或正式实现，导致回归遗漏
3. 高频页面对 IPC 的调用密度高，任何签名变化都会在多个入口上同步暴露，回归必须覆盖多页面而不是只测一个页面
4. 若测试记录不写明规则来源，无法区分“按文档应如此”与“代码当前如此”，这会直接降低结论可信度
5. 这类协议收口一旦遗漏旧调用，通常不会立刻崩溃，而是表现为静默失效、字段缺失或事件不触发，必须靠证据逐项确认

## 5. IPC 单一来源最小判定口径

为避免不同执行者各自抽样导致结论不可比，本 TP 的协议核对必须至少覆盖以下固定样本集；若某样本已被删除、合并或迁移，测试记录必须说明替代对象与原因，否则不得宣称“已完成单一来源验证”。

| 类别 | 最小覆盖样本 | 最低要求 |
|---|---|---|
| 查询类通道 | Gallery 列表读取或等价高频查询通道 | 必须同时核对 `channels.ts`、preload 暴露、renderer 调用点、handler 注册 |
| 保存类通道 | Settings 配置读取/保存或等价配置类通道 | 必须核对参数结构、返回结构、错误分支和旧签名是否已退场 |
| 事件/订阅类通道 | 下载、配置变更或等价高频订阅通道 | 必须核对事件名、订阅入口、取消订阅或清理路径 |
| 遗留歧义文件 | `handlers-full.ts` | 必须给出角色判定证据，证明其未再作为歧义运行来源 |

## 6. 测试前置条件

### 5.1 环境前置
- 应用可启动并进入主界面
- 可访问开发者工具、日志或调试输出，以观察 IPC 调用、错误签名和事件流
- 可切换到 Gallery、BooruFavorites、BooruCharacter、Settings 等高频页面
- 若项目启用了类型生成或协议生成产物，测试环境应能观察其是否与源文件同步；若无法查看生成产物，则相关检查只能记为待补测

### 5.2 数据前置
- 至少准备一个可触发 Gallery 典型 IPC 调用的样本
- 至少准备一个可触发 Favorites / Character 页面典型查询或动作的样本
- 至少准备一个 Settings 页面配置读取 / 保存样本
- 若存在通道列表、协议索引、生成声明或文档映射表，应准备可对照的当前版本快照

### 5.3 操作前置
- 必须能够触发页面层实际调用，不允许只看静态代码就下结论
- 若测试记录无法写明本次判断所引用的规则来源（例如 `03-任务拆分总控.md`、`12-IPC与进程边界重构.md`），该用例不得判通过
- 若某调用点已经被迁移为新协议，必须同时验证旧调用是否被消除，不能只测新入口

## 7. 测试层次与验证重点

### 6.1 配置与数据层
本 TP 不以业务配置为主，但协议定义、通道清单、共享类型与生成声明本身属于“协议数据层”。重点看：
- `channels.ts` 是否作为通道命名与分类的唯一权威来源
- `shared/types.ts` 中的协议类型是否只描述当前真实协议，而非历史残影
- 若存在协议生成产物或协议索引文件，其内容是否与源文件同步
- 测试记录必须注明本次验证引用的规则来源，避免把“文档建议”当成“已落地行为”

### 6.2 主进程服务层
重点看：
- `handlers.ts` 是否只注册与处理真实存在的通道
- 是否还存在同一能力在不同 handler 中用不同签名重复暴露
- 错误映射是否一致，尤其是参数缺失、通道不存在、返回值缺失等边界
- `handlers-full.ts` 是否仍被误用作正式实现或权威协议来源

### 6.3 IPC / preload / shared 边界层
这是本 TP 的核心。重点看：
- `channels.ts`、preload 暴露名、shared 声明、renderer 调用点是否同源一致
- 通道名、参数结构、返回结构、事件订阅名是否严格对齐
- 已证实的签名漂移是否已收敛到单一实现
- 新增通道时是否仍有裸字符串散落在 renderer 或 handler 中
- 若协议迁移存在兼容层，兼容层是否有明确退场边界而非长期并存

### 6.4 渲染层交互层
重点看：
- Gallery、Favorites、Character、Settings 等高频页面在协议统一后是否仍能完成主链路动作
- 页面是否出现“按钮可点但无响应”“接口返回但页面不更新”“旧调用静默失效”等现象
- 页面调用错误时是否能返回可理解提示，而不是只在控制台报错
- 页面中是否仍存在对旧签名、旧通道或隐式全局对象的依赖

### 6.5 生命周期与异常层
重点看：
- 重新打开页面、窗口重建、renderer reload 后协议调用是否仍稳定
- 主进程重启、热重载或异常恢复后，通道注册是否仍完整
- 若某些通道只在初始化后可用，则初始化顺序是否与调用时机一致
- 任何缺失注册、重复注册或顺序依赖问题都应记录为生命周期问题，而不是单纯 UI 问题

### 6.6 非功能层
重点看：
- 协议收敛后是否提升可维护性，而不是引入新的歧义命名
- 高频调用下是否存在明显性能退化、重复广播或重复绑定
- 权限边界是否更清晰，是否减少了 renderer 对不该触达能力的依赖
- 后续排障时是否能快速定位“唯一来源在哪里”，不能仍然靠口口相传

## 8. 测试设计文档与测试记录承接说明

本文档只负责定义测试设计、范围、步骤、预期结果与判定口径，不承载某次实际执行结果。

实际执行时，每条用例的以下信息必须由测试记录承接，而不是由本设计文档承载：
- 实际结果
- 证据路径 / 附件名
- 结论（通过 / 失败 / 待补测 / 环境阻塞）
- 本次判定所引用的规则来源

如果测试记录缺少这些字段，或仅写“通了”“没问题”“看起来一致”，则该用例不得判通过。对于需要依据文档规则确认的项目，测试记录必须写明引用来源，例如 `../12-IPC与进程边界重构.md` 或 `../03-任务拆分总控.md`，否则不能作为最终通过依据。

## 9. 详细测试用例

### TP-05-TC-001 IPC 协议源盘点应只存在一套可解释来源
- 测试目的：确认协议命名、分类和声明不再分散成多套互相竞争的来源
- 前置条件：可访问 `channels.ts`、`handlers.ts`、`preload/index.ts`、`shared/types.ts`，并能查看当前 renderer 调用点；必须按“IPC 单一来源最小判定口径”覆盖固定样本集，不能自行缩减样本
- 场景/步骤：
  1. 选取最小判定口径中的查询类、保存类、事件/订阅类固定样本各一条，并记录其实际页面入口
  2. 逐一检查主进程、preload、shared、renderer 中的协议定义与调用
  3. 记录每个固定样本的唯一来源位置
  4. 对照是否存在同一能力在多个文件重复定义
  5. 检查测试记录中是否引用了 `../12-IPC与进程边界重构.md` 作为规则来源
- 预期结果：
  - 每个固定样本都能追溯到单一可信来源
  - 不应存在“文档写 A、实现是 B、页面又调用 C”的三套口径
  - 测试记录必须注明本次判断依据来源
- 关键观察点：通道名、参数结构、返回结构、来源文件、固定样本覆盖情况、规则引用

### TP-05-TC-002 channels / preload / renderer 调用签名应严格一致
- 测试目的：验证签名漂移是否已清除
- 前置条件：必须使用“IPC 单一来源最小判定口径”中的固定样本，不得只抽测单一低频通道
- 场景/步骤：
  1. 在 `channels.ts` 读取固定样本对应的通道定义
  2. 在 `preload/index.ts` 读取对应暴露方法签名
  3. 在 renderer 页面触发同一能力
  4. 对比参数数量、字段名、返回值结构与错误分支
  5. 如存在兼容层，记录其覆盖范围与退场边界
- 预期结果：
  - 固定样本的三层签名完全一致，或存在明确且可解释的兼容层
  - 若存在兼容层，必须在测试记录中写明其退场边界
  - 不允许页面调用还能成功，但协议定义已经不同步
- 关键观察点：固定样本名称、函数参数、返回字段、事件名、空值与错误返回、兼容层说明

### TP-05-TC-003 handlers-full.ts 不应继续制造结构歧义
- 测试目的：确认 `handlers-full.ts` 的职责已经被明确处理，不再误导维护者
- 前置条件：可访问 `handlers-full.ts` 及其在代码中的引用情况
- 场景/步骤：
  1. 检查 `handlers-full.ts` 是正式注册、历史归档还是 mock 说明文件
  2. 检查启动注册链、导出链或引用链中是否仍把它当作权威来源或运行路径
  3. 若该文件保留，确认其文件头说明、命名和引用方式是否已去歧义
  4. 在测试记录中写明用于证明“未参与运行”或“已被明确归档”的证据类型，例如注册入口、导出关系、注释说明或启动日志
- 预期结果：
  - 文件角色必须唯一且明确
  - 不应再出现“看起来像正式 handler，又像历史样本”的双重解释
  - 测试记录应说明其处理结果对应哪个任务文档要求，以及依据的证据类型
- 关键观察点：文件用途、引用链、注册入口、导出关系、命名说明、是否仍参与运行

### TP-05-TC-004 高频页面 IPC 调用链应在收口后保持可用
- 测试目的：验证协议统一不会破坏真实使用入口
- 前置条件：Gallery、BooruFavorites、BooruCharacter、Settings 页面均可访问
- 场景/步骤：
  1. 在 Gallery 页面触发至少一条常用 IPC 调用
  2. 在 Favorites 页面触发查询 / 保存 / 刷新类调用
  3. 在 Character 页面触发列表加载或动作类调用
  4. 在 Settings 页面触发配置读取 / 保存类调用
- 预期结果：
  - 高频页面动作可正常完成
  - 页面不会因协议统一而出现静默失效
  - 若出现错误，错误位置可从日志或界面明确定位
- 关键观察点：页面结果、日志、错误提示、调用是否返回

### TP-05-TC-005 旧调用与漂移调用应被消除而不是并存
- 测试目的：验证已证实的协议漂移问题是否真正收敛
- 前置条件：已知至少一个历史旧调用点或旧签名样本
- 场景/步骤：
  1. 定位历史旧调用点
  2. 触发对应页面动作
  3. 检查是否仍有旧通道或旧字段被调用
  4. 检查是否存在“页面走新签名、旧逻辑仍在后台触发”的并存情况
- 预期结果：
  - 旧调用应被移除、替换或明确退场
  - 不应出现双通道并行导致的重复行为
  - 测试记录要写明旧调用是否属于已关闭遗留
- 关键观察点：旧签名是否仍被触发、日志是否出现旧通道、行为是否重复

### TP-05-TC-006 不得继续新增裸字符串通道
- 测试目的：确认新增协议没有继续把通道名硬编码散落到各层
- 前置条件：可观察最近新增或近期变更的通道实现
- 场景/步骤：
  1. 检查新增通道在 `channels.ts` 中的来源
  2. 检查 handler 与 preload 是否引用同一常量或同源映射
  3. 检查 renderer 是否没有自行拼接通道字符串
- 预期结果：
  - 新增协议必须回到唯一来源
  - 不能在 renderer 里临时硬编码通道名绕过统一入口
  - 测试记录若判通过，必须写明依据来源
- 关键观察点：硬编码字符串、引用方式、常量来源

### TP-05-TC-007 协议变更后的错误返回应可解释且可回归
- 测试目的：验证参数缺失、通道不存在、返回结构异常时的处理是否一致
- 前置条件：可通过开发者工具控制台、专用测试页、调试脚本或等价受控入口制造一条非法参数或不存在通道调用；若当前环境没有任何受控注入入口，测试记录必须记为待补测或环境阻塞，不得判通过
- 场景/步骤：
  1. 通过已注明的受控入口向选定通道传入缺失字段或错误字段
  2. 通过同一受控入口模拟调用不存在或已退场的通道
  3. 观察主进程和 renderer 的错误处理
  4. 记录本次使用的是哪一种注入方式，以及为何可代表真实错误路径
- 预期结果：
  - 错误应有清晰返回或日志
  - 不应出现无响应、无日志、无提示的静默失败
  - 错误信息要足够定位是协议问题还是页面问题
  - 若没有受控注入入口，则本用例不得直接判通过
- 关键观察点：注入方式、错误码、错误文案、日志定位、调用栈

### TP-05-TC-008 页面重开、窗口重建或 renderer reload 后协议调用仍应稳定
- 测试目的：承接本 TP 生命周期层范围，验证协议注册与消费不会因窗口生命周期变化而失序
- 前置条件：可执行页面重开、窗口重建或 renderer reload 中至少一种场景；必须覆盖“IPC 单一来源最小判定口径”中的至少一个固定样本
- 场景/步骤：
  1. 先在目标页面触发固定样本对应的 IPC 调用并确认其正常返回
  2. 执行页面重开、窗口重建或 renderer reload
  3. 再次触发同一固定样本调用
  4. 观察通道注册、订阅与返回是否仍一致
- 预期结果：
  - 生命周期变化后协议调用仍稳定
  - 不应出现缺失注册、重复注册、事件重复绑定或旧签名回流
  - 若仅覆盖一种生命周期路径，测试记录必须写明未覆盖路径及原因
- 关键观察点：固定样本名称、调用结果、注册日志、订阅数量、是否出现重复绑定

## 10. 负向与边界场景

### 9.1 负向场景
- 通道名写错或字段缺失时，页面不能假装操作成功
- 旧签名仍被局部页面引用时，必须能在回归中暴露出来，不能被新页面测试掩盖
- `handlers-full.ts` 仍被误用为运行入口时，测试结论必须判为结构风险而不是“文件存在无所谓”
- renderer 侧若出现直接拼接通道字符串的逻辑，应视为边界破坏

### 9.2 边界场景
- 同一能力同时存在查询和事件订阅形态时，必须明确区分，不可混用
- 页面重载或窗口重建后，协议注册必须仍然完整
- 若存在过渡兼容层，必须明确兼容期与退场条件
- 高并发、多页面同时调用同一通道时，不能出现串台或重复绑定

### 9.3 待补测前置
- 若当前无法确认某协议的唯一来源，必须先补齐文档或代码映射后再判通过
- 若测试环境无法查看日志或调用痕迹，相关签名核对只能记为待补测

## 11. 回归要求

### 10.1 必回归对象
- Gallery 页面
- BooruFavorites 页面
- BooruCharacter 页面
- Settings 页面
- 任何通过 preload 调用 main 能力的高频页面

### 10.2 需要联动回归
- TP-06 的 preload / shared / handler 收口
- TP-07 的 renderer-safe DTO 与敏感配置收回
- TP-08 的启动顺序与异常退出治理
- 如协议变化影响保存或恢复，再联动 TP-13 的偏好恢复链路

### 10.3 回归通过标准
- 协议来源单一且可解释
- 签名漂移消失
- 高频页面调用链可用
- `handlers-full.ts` 不再制造结构歧义
- 测试记录已引用正确规则来源

## 12. 自动化建议

### 11.1 适合自动化的部分
- 通道命名与定义一致性检查
- preload 暴露签名与 shared 类型的静态一致性检查
- renderer 调用点的字符串通道扫描
- `handlers-full.ts` 引用链扫描
- 高频页面基础调用 smoke test

### 11.2 建议自动化方式
- 通过静态分析或单元测试检查通道常量来源
- 为协议映射写快照测试，确保签名变化可见
- 为高频页面写最小 smoke test，确认核心调用无断裂

### 11.3 仍需人工确认的部分
- 协议统一后页面真实使用是否仍顺畅
- 兼容层是否已经不该继续存在
- 错误提示是否足够解释给维护者和用户

### 11.4 自动化边界
- 自动化可以证明“定义一致”和“引用一致”，但不能替代高频页面真实交互回归
- 对于 `handlers-full.ts` 角色判定、兼容层退场、规则来源引用，仍需人工核对

## 13. 缺陷分级与结论判定

### 12.1 缺陷分级
- S0：协议失配导致核心页面不可用、主进程与 renderer 断联、通道完全失效
- S1：同一能力存在多套签名或旧调用并存，造成明显回归风险
- S2：局部页面或个别事件命名不一致，但主流程可继续
- S3：命名冗长、归档说明不清晰等不影响运行的文档级问题

### 12.2 结论判定
满足以下条件可判定本 TP 通过：
1. 协议来源可追溯到单一权威来源
2. channels / preload / shared / renderer 的签名一致或存在明确且可退场的兼容层
3. `handlers-full.ts` 已不再造成结构歧义
4. 高频页面 IPC 调用链可用且无静默失效
5. 测试记录已写明规则来源、实际结果、证据路径和结论

若仍存在旧调用并存、签名漂移未清除或协议来源无法解释，则本 TP 不通过。
