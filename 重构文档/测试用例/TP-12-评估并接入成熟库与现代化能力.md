# TP-12 评估并接入成熟库与现代化能力

## 1. 文档目标

本 TP 面向“哪些高风险基础能力不应继续扩大自研面，而应在边界稳定后交给成熟基础设施接管”的问题，验证项目对成熟库与现代化能力的评估、引入顺序、替代边界与接入结果是否清晰、可执行、可回归。

本任务的测试目标不是“装上几个库、跑过构建”这么简单，而是证明以下几件事：

1. 每个候选库的引入目标、替代范围、前置条件、非目标和不引入条件都已明确
2. 成熟库接入没有绕过前置边界治理，不会把旧问题简单包装进新壳
3. 评估结论能够区分“只评估未接入”“试点接入”“正式替换”三类状态，且不能混写为通过
4. 接入后的并发、限流、虚拟化、日志、崩溃监控等能力与既有业务边界职责不冲突
5. 现代化升级后的行为、性能、可观测性和安全边界具备可验证结果，而不是只看依赖列表发生变化

## 3. 测试范围

### 3.1 必测范围
- 候选库评估结论是否明确，包括 `p-queue`、`Bottleneck`、`react-virtuoso` / `@tanstack/react-virtual`、`electron-log`、`@sentry/electron` 等
- 各专题的前置条件是否已满足，尤其是下载状态机、协议边界、退出治理、安全脱敏规则
- 已接入专题的最小闭环验证，包括：
  - 执行池 / 并发调度能力
  - 限流器与多站点速率控制能力
  - 虚拟列表对滚动性能和可见行为的影响
  - 结构化日志落盘与脱敏规则
  - 崩溃监控接入前后的异常收口与脱敏边界
- “只评估未接入”“试点接入”“正式替换”的记录与验收区分
- 成熟库接入后与既有业务逻辑、DTO 边界、Electron 原生能力之间的职责分层

### 3.2 不在本次范围内
- 下载状态机、claim 原子性、文件协议本身，属于 TP-01 / TP-02 / TP-03
- tray、notification、close / hide / quit 等 Electron 原生能力补齐本身，属于 TP-10；本 TP 只验证相关长期增强型技术基础是否准备好
- IPC / preload / shared 的边界收缩本身，属于 TP-05 / TP-06
- 页面功能闭环与交互一致性本身，属于 TP-04 / TP-09；但若虚拟化接入影响交互或状态表达，本 TP 必须联动验证

### 3.3 适用层次
- 配置与数据层
- 主进程服务层
- IPC / preload / shared 边界层
- 渲染层交互层
- 生命周期与异常层
- 非功能层

## 4. 风险摘要

1. 成熟库接入最大的假象是“依赖加上了，复杂问题就解决了”；实际上若替代边界未划清，只会把现有问题转移到更难定位的壳里
2. `p-queue` 与 `Bottleneck` 最容易发生职责重叠；若不先定义谁负责并发、谁负责速率，测试只看“能跑”会掩盖后续结构冲突
3. 虚拟列表非常容易造成功能表面可用但滚动定位、选择态、懒加载、图片布局和空错态表达回退，必须做功能与性能双线验证
4. `electron-log` 与 `@sentry/electron` 若没有先定义脱敏规则和异常边界，会把路径、URL、账号、凭证、整包配置暴露到新的观测面
5. 本 TP 强依赖 source of truth：哪些库只评估不接入、哪些先试点、哪些已正式替换，必须有明确记录；没有记录不得把“评估过”写成“已通过”

## 5. 测试前置条件

### 5.1 环境前置
- TP-01、TP-05、TP-08 至少满足本次候选库对应专题所需前置；若未满足，测试记录必须明确标注前置未完成，不得越级判通过
- 应用可正常构建、启动，并具备对应专题的可观测环境：
  - 并发 / 限流专题需要可观察请求与调度行为
  - 虚拟列表专题需要可观察滚动、渲染数量与交互结果
  - 日志 / 崩溃专题需要可观察日志文件、异常事件或上报模拟结果
- 必须具备截图或录屏能力；凡涉及性能对比、滚动表现、日志落盘、异常上报、试点接入前后对比的用例，不保留证据不得判通过

### 5.2 数据前置
- 至少准备一组下载 / 调度样本，用于验证执行池或限流器试点接入
- 至少准备一组图片密集型列表样本，用于验证虚拟列表候选接入
- 至少准备一组异常或错误样本，用于验证日志落盘与崩溃观测链路
- 若当前仅做评估未接入，则必须准备评估记录、替代边界说明与不接入条件清单；若没有这些材料，不得宣称“评估已完成”

### 5.3 操作前置
- 必须能区分“评估文档结论”“试点接入结果”“正式替换结果”三种产物，并在测试记录中分别标注
- 必须能查看日志文件、调试输出、性能指标或渲染数量；若只能看页面大致没问题，相关现代化专题只能记为待补测
- 若当前产品尚未明确某一候选库的引入时机、替代边界和不引入条件，测试记录必须先写明 source of truth 缺失，该专题不得判通过

### 5.4 口径确认前置
- 在执行 TP-12 之前，测试记录必须写明以下 source of truth：
  - 每个候选库属于“只评估未接入”“试点接入”还是“正式替换”
  - 每个候选库替代的具体对象、明确不替代的对象、前置依赖和非目标
  - `p-queue` 与 `Bottleneck` 的职责边界
  - 虚拟列表专题选用 `react-virtuoso` 还是 `@tanstack/react-virtual`，或为何暂不选型
  - `electron-log` 与 `@sentry/electron` 的脱敏和采样边界
  - 日志 / 崩溃监控专题的敏感字段清单或禁止上报字段清单，以及若当前仅有原则没有清单时应如何判定
- 若上述任一项没有明确 source of truth，则对应专题用例只能记为“待补测前置”或“环境阻塞”，不得判通过，也不得据此给出 TP-12 通过结论

## 7. 六层测试口径与验证重点

### 7.1 配置与数据层
重点看：
- 成熟库接入后是否引入新的配置项、日志目录、采样开关、限流策略配置等，且这些配置有明确落点和默认值
- 日志与监控配置是否遵守脱敏规则，不把敏感字段、整包配置或凭证写入新存储面
- 若执行池、限流器或虚拟列表依赖新配置，测试记录必须说明其来源与生效时机；未说明不得判通过

### 7.2 主进程服务层
重点看：
- `p-queue` 接入后是否只替代并发调度 / 排队子层，而没有误吞文件协议、DB 状态、恢复判定等领域逻辑
- `Bottleneck` 接入后是否只承担速率控制，不与执行池职责重叠
- `electron-log` 是否形成结构化日志落盘，并与现有生命周期、退出治理和任务链路对齐
- `@sentry/electron` 若试点接入，是否先建立异常钩子统一收口，再进行上报验证

### 7.3 IPC / preload / shared 边界层
重点看：
- 成熟库接入不应扩大 renderer 暴露面或把新库原始对象透传到页面
- 日志与监控若需要 renderer 配合，其 bridge、DTO 和事件结构必须受控
- 虚拟列表若引入新的页面数据结构，不得绕开既有 DTO / facade 设计
- 若 source of truth 未明确新库与现有边界的接口契约，则对应用例不得判通过

### 7.4 渲染层交互层
重点看：
- 虚拟列表接入后，滚动、选择态、分页、图片懒加载、详情面、空错态与既有页面闭环是否仍然成立
- 日志或崩溃监控接入不应在页面制造额外噪音或错误提示
- 若某类能力仍停留在“评估未接入”，页面上不应出现把其当已完成能力使用的误导行为
- 对“试点接入”的页面必须验证真实用户可见行为，而不能只看依赖安装成功

### 7.5 生命周期与异常层
重点看：
- 执行池或限流器接入后，启动恢复、暂停恢复、退出清理与异常中断是否仍遵守原有治理边界
- `electron-log` 与 `@sentry/electron` 接入后，异常退出、未捕获异常、崩溃和重启路径是否有明确记录或上报行为
- 虚拟列表接入后，切页、返回、滚动恢复、卸载重挂载是否稳定
- 若某专题仍缺 source of truth 或异常钩子前置未完成，则不得把其生命周期验证写成通过

### 7.6 非功能层
重点看：
- 性能：并发控制、限流、列表滚动和渲染压力是否有可观察改善或至少不退化
- 稳定性：接入后不能制造重复请求、滚动抖动、日志风暴、上报风暴或异常循环
- 可观测性：日志与监控应提高问题可定位性，而不是制造新噪音
- 安全性：脱敏规则必须成立，不能新增敏感信息泄露面
- 可维护性：评估结论、接入边界和回退策略必须可记录、可复查、可扩展

## 8. 测试设计文档与测试记录承接说明

本文档只负责定义测试设计、覆盖范围、步骤、预期结果与判定口径，不直接承接某一次实际执行结果。

实际执行时，每条用例的以下信息必须由测试记录承接，而不是由本设计文档承载：
- 实际结果
- 证据路径 / 附件名
- 结论（通过 / 失败 / 待补测 / 环境阻塞）
- 本次候选库状态（只评估 / 试点接入 / 正式替换）与 source of truth 来源

如果测试记录缺少上述字段，或仅写“库接上了”“性能更好了”“已经现代化”这类泛化描述，则该用例不得判通过。本设计文档本身也不能代替测试记录直接给出最终结论。

## 9. 详细测试用例

### TP-12-TC-001 每个候选库必须先有明确评估状态与替代边界
- 测试目的：阻止把“听说要引入”写成“已经完成”
- 前置条件：已准备候选库清单与 source of truth
- 场景/步骤：
  1. 列出本次候选库清单
  2. 标注每个候选库当前属于只评估、试点接入还是正式替换
  3. 写明替代范围、非目标和不引入条件
- 预期结果：
  - 每个候选库都有明确状态与边界
  - 若没有，则该候选库相关测试不得判通过
- 关键观察点：候选库清单、状态分类、替代边界、source of truth

### TP-12-TC-002 p-queue 试点接入不得误替代领域逻辑
- 测试目的：验证 `p-queue` 只承担执行池 / 并发调度子层职责
- 前置条件：`p-queue` 已处于试点接入或正式替换状态；已确认其替代边界的 source of truth
- 场景/步骤：
  1. 触发普通下载或批量下载的并发调度路径
  2. 观察排队、并发控制、暂停 / 恢复等行为
  3. 核对文件协议、DB 状态和恢复判定仍由既有领域逻辑负责
- 预期结果：
  - `p-queue` 只接管并发调度 / 排队子层
  - 不应把文件协议、DB 状态、恢复判定一起塞进新库适配层
  - 若 source of truth 未明确替代边界，则本用例不得判通过
- 关键观察点：并发行为、队列状态、领域逻辑边界、source of truth 记录

### TP-12-TC-003 Bottleneck 应只承担速率控制，不与执行池职责重叠
- 测试目的：验证限流器职责边界
- 前置条件：`Bottleneck` 已处于试点接入或正式替换状态；已确认与执行池职责划分的 source of truth
- 场景/步骤：
  1. 对多站点或同站点多请求场景发起请求
  2. 观察速率控制、退避或排队行为
  3. 对比并发控制仍由谁负责
- 预期结果：
  - 限流器负责速率与节流，不负责领域状态机和文件协议
  - 不应与执行池同时重复控制相同维度而导致双重节流或不可解释阻塞
  - 若职责边界未明确，测试记录不得判通过
- 关键观察点：请求节奏、退避结果、并发与速率控制分工、source of truth 记录

### TP-12-TC-004 虚拟列表候选接入必须同时验证行为与性能
- 测试目的：验证虚拟化不是只看 FPS 或只看库 demo
- 前置条件：`react-virtuoso` 或 `@tanstack/react-virtual` 已处于试点接入或正式替换状态；已确认选型依据的 source of truth
- 场景/步骤：
  1. 打开图片密集型列表页面
  2. 执行滚动、选中、懒加载、进入详情、返回、筛选或分页等操作
  3. 对比接入前后可见行为与滚动表现
- 预期结果：
  - 列表滚动、选中态、懒加载、详情回跳等行为不回退
  - 渲染数量、滚动卡顿或内存压力应有可观察改善，或至少不退化
  - 若当前仅做选型评估未接入，则不得把性能推断写成通过
- 关键观察点：滚动流畅度、渲染数量、图片显示完整性、详情回跳、source of truth 记录

### TP-12-TC-005 electron-log 接入必须先满足结构化落盘与脱敏要求
- 测试目的：验证日志现代化不是把 `console.log` 全量复制到文件
- 前置条件：`electron-log` 已试点接入或正式替换；已确认脱敏规则、日志目录、保留策略，以及敏感字段清单或禁止落盘字段清单的 source of truth
- 场景/步骤：
  1. 触发启动、关闭、任务执行、失败或恢复等关键路径
  2. 检查日志文件是否生成、结构是否可读
  3. 检查是否泄露路径、URL、账号、凭证或整包配置
  4. 对照敏感字段清单或禁止落盘字段清单核对日志字段
- 预期结果：
  - 日志应结构化落盘并支撑关键排障链路
  - 不应裸写敏感字段或整包配置对象
  - 若脱敏规则或禁止落盘字段清单未明确，则本用例不得判通过
- 关键观察点：日志文件、字段结构、敏感字段清单 / 禁止落盘字段清单、脱敏结果、source of truth 记录

### TP-12-TC-006 Sentry 接入必须建立在异常治理前置已完成基础上
- 测试目的：验证崩溃监控不是在异常边界混乱时直接上报一切
- 前置条件：`@sentry/electron` 已试点接入或正式替换；TP-08 与 `../15-稳定性错误恢复与安全性修正.md` 中异常治理前置已满足；已确认采样与脱敏边界，以及敏感字段清单或禁止上报字段清单的 source of truth
- 场景/步骤：
  1. 触发一个可控异常或未捕获错误样本
  2. 观察本地异常处理、日志、上报或模拟上报结果
  3. 检查事件中是否携带敏感字段
  4. 对照敏感字段清单或禁止上报字段清单核对事件内容
- 预期结果：
  - 异常应先进入统一治理链路，再按策略上报
  - 不应在无脱敏规则前直接把敏感数据送入监控
  - 若仅安装 SDK 但未形成可验证事件链路，则不得判通过
  - 若禁止上报字段清单未明确，则本用例不得判通过
- 关键观察点：异常钩子、事件内容、敏感字段清单 / 禁止上报字段清单、脱敏结果、source of truth 记录

### TP-12-TC-007 只评估未接入的候选库不得被写成已通过实现
- 测试目的：验证状态记录准确性
- 前置条件：至少存在一个仅完成评估、尚未接入的候选库
- 场景/步骤：
  1. 检查评估结论、实现状态和测试结论
  2. 对照依赖、代码路径和页面行为
- 预期结果：
  - 只评估未接入的候选库只能形成“评估通过 / 建议接入 / 暂不接入”之类结论
  - 不得写成“能力已完成”“实现已通过”
- 关键观察点：评估文档、依赖状态、实际代码路径、测试结论分类

### TP-12-TC-008 正式替换后必须验证旧实现边界已退场或受控保留
- 测试目的：验证新旧方案不会长期双轨失控
- 前置条件：至少一个候选库已进入正式替换阶段；已确认旧实现退场策略的 source of truth
- 场景/步骤：
  1. 检查旧实现是否仍保留
  2. 触发对应主路径操作
  3. 观察是否仍同时走旧新两套逻辑
- 预期结果：
  - 旧实现要么已退场，要么被明确保留为受控 fallback，并有明确启用条件
  - 不应新旧两套逻辑长期同时工作且无说明
  - 若退场策略未明确，则本用例不得判通过
- 关键观察点：旧实现调用痕迹、fallback 条件、source of truth 记录

### TP-12-TC-009 成熟库接入不得扩大 renderer 暴露面或引入新边界泄露
- 测试目的：验证现代化升级与边界治理不打架
- 前置条件：至少一个候选库涉及 renderer 或 bridge 层配合；可查看 preload / DTO / 页面消费结构
- 场景/步骤：
  1. 检查新库接入相关 bridge、DTO、配置返回或事件结构
  2. 在页面上触发对应能力
  3. 观察是否出现原始对象、敏感字段或新库内部对象向 renderer 透传
- 预期结果：
  - 不应因接入新库而扩大 renderer 暴露面
  - 页面仍应只消费受控 DTO 或受控命令
  - 若契约未明确，则本用例不得判通过
- 关键观察点：DTO 字段、bridge 结构、页面消费对象、source of truth 记录

### TP-12-TC-010 现代化接入后的性能或可观测性改善必须有对比证据
- 测试目的：验证“引库值得”而不是“引库完成”
- 前置条件：至少一个候选库已试点接入或正式替换；具备接入前后对比基线
- 场景/步骤：
  1. 执行接入前后的对比测试，例如并发调度、列表滚动、异常定位、日志排障
  2. 记录核心指标或观察结果
  3. 对比是否有改善、持平或退化
- 预期结果：
  - 应能给出至少一个可复核的改善点，或明确记录“当前仅结构收口、尚无性能提升”
  - 不得在没有对比证据时主观写“性能更好”“日志更清晰”
  - 测试记录至少应包含以下最低对比字段中的适用项：测试对象、候选库状态、接入前基线、接入后结果、观测时间或样本范围、关键指标或关键观察点、改善 / 持平 / 退化结论、证据路径
- 关键观察点：对比基线、测试对象、样本范围、指标或观察记录、是否有退化、结论分类

## 10. 负向与边界场景

### 10.1 负向场景
- 接入执行池后不能出现暂停 / 恢复 / 退出语义更混乱的回退
- 接入限流器后不能因为双重控制导致请求永久卡住或节流失真
- 接入虚拟列表后不能出现列表项丢失、选择态错位、滚动跳跃或空白闪烁严重问题
- 接入日志 / 崩溃监控后不能新增敏感数据泄露、日志风暴或重复上报风暴

### 10.2 边界场景
- 只在少量页面试点接入虚拟列表时，未接入页面不得被外推为通过
- 仅在某站点试点限流器时，不得外推到所有站点都已通过
- 日志只接入主进程时，不得外推为渲染层观测也已完成
- 仅有本地模拟上报时，不得外推为真实生产上报链路已通过，除非 source of truth 明确允许

### 10.3 待补测前置
- 若前置边界治理未完成，例如下载状态机、异常钩子、脱敏规则、DTO 边界仍未定义，则对应候选库专题不得判通过
- 若无接入前对比基线，则性能或可观测性改善类结论只能记为待补测，不能直接写通过

## 11. 回归要求

### 11.1 必回归对象
- 下载 / 调度主链路
- 多站点请求链路
- 图片密集型列表页
- 启动、退出、异常与恢复链路
- Settings 中涉及日志、通知、调试或高级能力入口的部分（若有）

### 11.2 需要联动回归
- TP-01 / TP-02 / TP-03 的下载稳定性与恢复边界
- TP-05 / TP-06 的边界收口与 DTO 暴露面
- TP-08 的启动、退出与异常治理
- TP-09 的虚拟列表接入后交互一致性与隐藏页行为
- TP-10 的桌面长期增强基础能力承载

### 11.3 回归通过标准
- 每个候选库的状态、边界、前置条件和非目标都有明确记录
- 已接入专题未突破既有边界治理结果
- 试点或正式接入后的行为、性能、可观测性至少有一项可复核结果
- 只评估未接入、试点接入、正式替换三类结论没有混写
- 已留存足够证据并完成必要联动回归

## 12. 自动化建议

### 12.1 适合自动化的部分
- 执行池与限流器职责边界测试
- 列表虚拟化渲染数量与滚动窗口断言
- 日志字段脱敏测试
- 异常事件内容、采样和脱敏测试
- 新旧实现切换或 fallback 路径测试
- 候选库状态与评估清单校验脚本

### 12.2 建议自动化方式
- 为 `p-queue` / `Bottleneck` 适配层编写服务层测试，断言并发与速率控制分工
- 为虚拟列表编写组件测试，断言可见项数量、滚动窗口与关键交互不回退
- 为 `electron-log` 和 `@sentry/electron` 的包装层编写脱敏与事件内容测试
- 为正式替换专题编写回归测试，验证不会同时走新旧两套无说明逻辑

### 12.3 仍需人工确认的部分
- 虚拟列表真实滚动体验、图片加载体验与详情回跳感受
- 执行池或限流器接入后的真实任务手感与排队表现
- 日志排障可读性是否真有提升
- 崩溃监控接入后的异常链路是否符合实际运维预期

### 12.4 自动化边界
- 自动化可以证明职责与结构，但不能替代真实滚动、真实排队和真实异常体验的人工观察
- 现代化库接入后若影响用户可感知行为，必须保留人工回归证据，不能只以依赖安装和单测通过作为结论

## 13. 缺陷分级与结论判定

### 13.1 缺陷分级
- S0：成熟库接入导致核心链路失效、异常治理失控、敏感数据严重泄露或新旧逻辑双轨冲突造成不可恢复问题
- S1：替代边界不清、职责重叠、虚拟列表或调度行为明显回退、日志或监控引入高风险噪音
- S2：个别专题仅形成部分试点、个别性能改善不明显或结构记录不完整，但主路径仍可用
- S3：评估文案、命名、轻微配置或记录细节问题，不影响总体现代化判断

### 13.2 结论判定
满足以下条件可判定本 TP 通过：
1. 每个候选库的状态、替代边界、前置条件、非目标和不引入条件都有明确 source of truth
2. 已试点接入或正式替换的专题都经过最小闭环验证，并未突破既有边界治理
3. `p-queue`、`Bottleneck`、虚拟列表、`electron-log`、`@sentry/electron` 等专题的职责边界可解释、可回归
4. 只评估未接入、试点接入、正式替换三类结论没有混写
5. 已留存足够证据，且测试记录已注明 source of truth 来源

若仍存在“评估状态不清”“接入边界不清”“前置未完成就越级接入”“新库扩大敏感暴露面”“无对比证据却宣称改善”等情况，则本 TP 不通过。

## 15. 阶段验收口径

TP-12 作为阶段 3 / 阶段 4 的现代化升级任务，验收时应满足：

1. 每个成熟库候选的评估结论、接入状态与替代边界都已明确，不再处于“听起来会做”的模糊状态
2. 现代化接入遵守前置依赖，不会绕过下载、边界、安全和异常治理前提
3. 已接入专题的职责边界清晰，不与既有业务逻辑和 Electron 原生能力打架
4. 已接入专题至少形成最小闭环验证，并保留可复核的对比证据
5. 所有关键结论都由测试记录写明 source of truth，未写明者不得作为通过依据

本设计文档不承载实际结果、证据路径和最终结论；这些信息必须由测试记录提供。若只有依赖变更和主观收益描述，而没有评估边界、前置条件、接入验证与对比证据，则只能判定为待补测或未通过，不能进入阶段通过结论。
