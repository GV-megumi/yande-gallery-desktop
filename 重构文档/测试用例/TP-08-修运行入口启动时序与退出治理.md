# TP-08 修运行入口启动时序与退出治理

## 1. 文档目标

本 TP 面向 `npx electron .` 默认入口可运行、主窗口展示优先级、初始化顺序、退出钩子、崩溃钩子、恢复定时器清理、数据库关闭以及 close / hide / quit 区分等问题，验证应用是否具备最小可运行、可恢复、可退出的主进程生命周期模型。

本任务的测试目标不是“程序最终能不能开起来”，而是证明以下几件事：

1. Electron 默认入口必须可运行，不能依赖额外的非默认绕路启动方式
2. 主窗口应尽早显示，非关键初始化可以延后，但不能把首屏交付卡死在后台准备项上
3. before-quit / will-quit / uncaughtException / unhandledRejection / process-gone 等退出与崩溃钩子应纳入统一治理链路
4. 恢复定时器、数据库关闭、任务冻结等收尾动作必须接入统一退出链路，不能各做各的
5. `close` 与 `quit` 在本 TP 边界内必须有明确规则且可验证；若当前产品已引入 `hide` / `close-to-tray` 语义，也必须一并给出明确规则并纳入验证

## 4. 风险摘要

1. 启动和退出是全局生命周期问题，任何小改动都可能影响整个应用是否能运行、能否展示窗口、能否正常退出
2. 主窗口显示顺序如果处理不当，可能造成“看似启动成功但首屏很慢”或“初始化依赖把界面卡死”的问题
3. 崩溃与退出治理若未统一，容易在数据库关闭、定时器清理、异常写日志之间产生重复或遗漏
4. `close` 与 `quit` 如果未明确区分，后续桌面能力（如托盘、后台常驻）会把问题放大；若当前产品已引入 `hide` / `close-to-tray` 语义，也必须一并明确边界，否则用户会误判应用是否真正退出
5. 启动类问题常常不能只靠一次手工打开判断，必须验证重启、异常、重复启动和收尾链路

## 5. 启动与退出最小判定口径

为避免“尽早显示”“合理时序”“不卡首屏”这类模糊表述，本 TP 执行时必须先在测试记录中建立可落地口径；若没有来源或阈值，只能记为待补测前置或环境阻塞，不得直接判通过。

| 验证对象 | 最低判定口径 |
|---|---|
| 默认入口启动 | 必须记录命令、终端输出、主窗口是否出现、是否可交互 |
| 主窗口优先显示 | 必须在测试记录中写明“何时算显示完成”“何时算可交互”“允许等待的前置项”，并至少给出明确时延阈值或等价可量化门槛，以及对应规则来源 |
| 非关键初始化延后 | 必须能从日志、时序记录或操作观察中区分窗口显示时刻与后台初始化时刻；若主张“未阻塞首屏”，测试记录必须给出时序证据和阈值依据 |
| 退出清理 | 必须分别记录定时器清理、数据库关闭、退出钩子触发结果；不能用一句“已退出”代替 |
| close / hide / quit 区分 | 必须在测试前写明既定语义来源；若语义未定义，不得靠执行者理解直接判通过 |

- 可正常创建或查看主窗口
- 若要验证崩溃或异常钩子，环境需要允许制造受控异常或模拟 process gone 场景；若无法制造，则相关用例记为待补测

- 若要验证 close / hide / quit 区分，需具备窗口关闭和应用退出的观察条件
- 若存在可关闭的主窗口与可重建窗口样本，应准备其对应场景

### 6.3 操作前置
- 必须能多次启动、关闭、重启应用

## 7. 测试层次与验证重点

### 7.1 配置与数据层
重点看：
- 初始化时是否需要读取配置、数据库或恢复状态
- 配置读取是否会阻塞主窗口展示
- 退出时是否需要将配置或运行状态写回持久层
- 若启动恢复依赖某些持久状态，重启后是否能从正确层恢复

### 7.2 主进程服务层
这是本 TP 的核心。重点看：
- 默认入口能否启动主进程并创建窗口
- 主窗口创建、显示、恢复和初始化任务的顺序是否合理
- before-quit / will-quit / uncaughtException / unhandledRejection / process-gone 是否进入统一治理
- 恢复定时器、数据库关闭、任务冻结是否在退出链路中执行
- 是否存在重复清理、漏清理或清理时序错误

### 7.3 IPC / preload / shared 边界层
本 TP 不以协议边界为主，但仍需核对：
- 启动时是否依赖过早的 renderer 调用
- 退出或恢复钩子是否依赖 IPC 后再执行，导致顺序竞争
- 若窗口创建依赖 preload bridge，相关 bridge 是否在正确时机可用

### 7.4 渲染层交互层
重点看：
- 主窗口是否尽早可见、可交互
- 启动过程中是否出现明显白屏、长时间无响应或卡死
- 退出时是否有清晰的用户感知，例如窗口关闭、应用退出或保持驻留的明确行为
- 若 close / hide / quit 需要区分，页面或窗口行为是否符合定义

### 7.5 生命周期与异常层
这是本 TP 的重点。重点看：
- 启动流程中关键钩子是否按顺序触发
- 异常退出、崩溃、恢复重启后，是否还有残留定时器或未关闭资源
- 单实例、重复启动、关闭后重开是否不会把旧状态错误恢复回来
- process-gone 等异常是否至少被记录、被识别、被收口

### 7.6 非功能层
重点看：
- 启动速度是否足够稳定，不因某些非关键任务而显著拖慢
- 退出是否干净，不出现卡住或长时间不结束
- 收尾资源是否完整关闭，避免泄漏或脏状态
- 生命周期治理是否让后续桌面能力更容易扩展

## 9. 详细测试用例

### TP-08-TC-001 `npx electron .` 默认入口应可正常启动
- 测试目的：确认默认运行路径可用
- 前置条件：可在项目根目录执行 `npx electron .`
- 场景/步骤：
  1. 从项目根目录运行 `npx electron .`
  2. 观察主进程是否启动
  3. 观察是否能创建主窗口
  4. 检查终端是否出现入口错误
- 预期结果：
  - 默认入口能正常启动并创建窗口
  - 不应依赖额外绕路或手工补参数才能跑起来
  - 测试记录必须写明本次判断依据来源
- 关键观察点：终端输出、窗口是否出现、是否存在入口报错

### TP-08-TC-002 主窗口应优先显示，非关键初始化可延后
- 测试目的：验证启动时序是否合理
- 前置条件：启动时存在初始化任务或恢复任务；测试记录必须先写明“显示完成”“可交互”“允许等待的前置项”分别如何判定，以及判定依据来源，否则本用例不得判通过
- 场景/步骤：
  1. 启动应用并按测试记录中的口径计时观察主窗口出现时机
  2. 观察初始化日志与窗口显示顺序
  3. 检查非关键初始化是否阻塞首屏
  4. 若窗口在显示后仍需等待才能交互，记录等待项是否属于允许的前置项
- 预期结果：
  - 主窗口应按已注明口径优先展示
  - 非关键初始化可以延后，但不能阻断已定义的首屏交付
  - 若有必须等待项，应在记录里说明原因和依据来源
  - 若未先定义判定口径，本用例不得判通过
- 关键观察点：显示完成时刻、可交互时刻、日志顺序、阻塞点、允许等待项

### TP-08-TC-003 启动恢复定时器不应阻塞窗口展示
- 测试目的：确认恢复任务与窗口显示解耦
- 前置条件：存在恢复定时器或恢复初始化任务；测试记录必须注明本次判断“阻塞”与“未阻塞”的依据来源
- 场景/步骤：
  1. 启动应用
  2. 观察恢复定时器的触发时机
  3. 检查它是否在主窗口出现后再执行，或虽提前触发但未阻塞已定义的显示/可交互口径
  4. 记录窗口显示时刻、可交互时刻与定时器触发时刻的先后关系
- 预期结果：
  - 恢复任务不应卡住主窗口首屏
  - 任务触发时机应符合已注明的初始化时序设计
  - 若无阻塞判定依据来源，本用例不得判通过
- 关键观察点：定时器触发、窗口显示、可交互时刻、日志时序、规则来源

### TP-08-TC-004 退出链路应统一清理恢复定时器
- 测试目的：确认退出时不会遗留恢复定时器
- 前置条件：启动后存在恢复定时器或延迟任务
- 场景/步骤：
  1. 启动应用并确认恢复定时器已注册
  2. 关闭或退出应用
  3. 观察定时器是否被清理
  4. 重启应用确认不会重复触发旧定时器
- 预期结果：
  - 退出时应清理恢复定时器
  - 不应在退出后继续触发旧恢复逻辑
  - 重启后不应出现重复恢复
- 关键观察点：定时器清理日志、重复触发、退出后残留行为

### TP-08-TC-005 退出链路应统一关闭数据库连接
- 测试目的：确认数据库关闭进入统一退出治理
- 前置条件：应用启动后已建立数据库连接或使用数据库服务
- 场景/步骤：
  1. 启动应用并触发数据库使用
  2. 退出应用
  3. 检查是否调用数据库关闭动作
  4. 重启后确认数据库未被脏占用
- 预期结果：
  - 退出时数据库应正常关闭
  - 不应留下锁定、脏连接或异常未关状态
- 关键观察点：关闭日志、重启是否顺畅、是否有数据库报错

### TP-08-TC-006 before-quit / will-quit 应纳入统一退出治理
- 测试目的：验证正常退出事件链是否完整
- 前置条件：可查看主进程日志
- 场景/步骤：
  1. 正常退出一次
  2. 观察 before-quit / will-quit 是否触发
  3. 检查其是否进入统一清理路径
- 预期结果：
  - before-quit / will-quit 应有统一处理路径
  - 不应出现退出了但钩子完全无迹可循的情况
- 关键观察点：钩子日志、清理日志、退出结果

### TP-08-TC-007 uncaughtException / unhandledRejection 应纳入统一异常治理
- 测试目的：验证受控异常不会脱离治理链路
- 前置条件：可制造受控异常或未处理错误；若当前环境无法安全制造，则本用例只能记为待补测或环境阻塞
- 场景/步骤：
  1. 制造一条受控异常或未处理 Promise 错误
  2. 观察 uncaughtException / unhandledRejection 是否被识别
  3. 检查是否有统一记录、收尾或失败提示
- 预期结果：
  - 异常应被识别、记录并收口
  - 不应完全无治理
  - 若无法安全制造异常，则本用例不得判通过
- 关键观察点：异常日志、收尾结果、是否进入统一异常链路

### TP-08-TC-008 process-gone 应有明确识别与处理路径
- 测试目的：验证渲染进程或相关进程异常消失时不会无记录裸奔
- 前置条件：若可能，环境允许模拟 process-gone；若无法模拟，则本用例只能记为待补测或环境阻塞
- 场景/步骤：
  1. 模拟或观察一次 process-gone 场景
  2. 记录主进程如何识别该事件
  3. 观察是否有统一处理、日志或恢复路径
- 预期结果：
  - process-gone 至少应被记录、识别并进入明确处理路径
  - 不应完全无记录、无收口
  - 若无法模拟，则本用例不得判通过
- 关键观察点：事件日志、处理结果、恢复或收尾行为

### TP-08-TC-009 close / hide / quit 行为应在本边界内区分清楚
- 测试目的：确认窗口关闭、隐藏与显式退出不是同一个语义
- 前置条件：窗口可关闭，且可观察应用是否仍在运行；测试记录必须先写明 close / hide / quit 的既定语义来源，若语义未定义，本用例不得判通过
- 场景/步骤：
  1. 点击窗口关闭按钮
  2. 观察应用是否只是隐藏、退出或保持运行
  3. 通过显式退出入口再次验证
  4. 对照记录中的既定语义判断是否一致
- 预期结果：
  - close / hide / quit 应符合已注明的既定语义
  - 不应把关闭窗口简单等同于强制退出，或反之
  - 若未先写明语义来源，本用例不得判通过
- 关键观察点：窗口是否消失、进程是否仍在、应用是否可恢复、规则来源

### TP-08-TC-010 应用已运行时再次启动应有明确处理结果
- 测试目的：承接生命周期层中“重复启动”范围，验证再次启动不会制造脏状态或无响应竞争
- 前置条件：应用已运行，且环境允许再次发起同一启动命令
- 场景/步骤：
  1. 先正常启动应用一次
  2. 在应用仍运行时再次执行启动命令
  3. 观察已有窗口、进程数量、日志与处理结果
- 预期结果：
  - 再次启动应有明确结果，例如复用已有进程、拒绝重复启动或按既定规则处理
  - 不应出现多个无控制实例争抢状态
  - 若当前项目尚未定义重复启动规则，测试记录只能记为待补测前置或环境阻塞
- 关键观察点：进程数量、窗口状态、日志、规则来源

### TP-08-TC-011 异常退出后重启不应残留旧任务或旧定时器
- 测试目的：验证退出收尾是否彻底
- 前置条件：可制造异常退出或强制关闭情形
- 场景/步骤：
  1. 启动应用并确认存在任务或恢复逻辑
  2. 制造异常退出
  3. 重启应用
  4. 观察是否出现重复恢复、旧定时器触发或脏状态
- 预期结果：
  - 重启后应从干净状态恢复
  - 不应重复执行旧任务
  - 不应把异常前的脏状态错误延续
- 关键观察点：重启后的状态、重复任务、错误日志

## 10. 负向与边界场景

### 10.1 负向场景
- 默认入口启动失败或窗口不出现，属于阻断级问题
- 退出后数据库仍被占用、恢复定时器继续运行，属于退出治理失败
- uncaughtException / unhandledRejection 完全无记录，属于异常治理失败
- close / hide / quit 被混为一谈，会直接影响后续桌面能力

### 10.2 边界场景
- 主窗口较慢初始化时，仍应能尽早显示最小可用界面
- 退出钩子与日志钩子同时触发时，不能重复清理或互相覆盖
- 重启过程中若存在恢复任务，必须避免重复注册
- 若某些退出行为暂时受平台限制，应写明平台差异和例外边界

### 10.3 待补测前置
- 若当前环境无法制造崩溃或 process-gone，相关钩子只能记为待补测
- 若无法从命令行运行默认入口，先修运行环境再判通过

## 13. 缺陷分级与结论判定

### 13.1 缺陷分级
- S0：默认入口无法运行、主窗口无法显示、退出 / 崩溃完全无治理、数据库或定时器严重脏留
- S1：启动时序明显阻塞首屏、退出链路不完整、close / hide / quit 语义混乱
- S2：个别钩子缺失、日志顺序不理想、清理动作时机可再优化但主流程可用
- S3：文案、日志格式、命名可继续优化，不影响生命周期判定

### 13.2 结论判定
满足以下条件可判定本 TP 通过：
1. `npx electron .` 默认入口可运行
2. 主窗口展示优先与“未阻塞首屏”的判断，必须按第 5 章已定义的显示完成 / 可交互口径、明确阈值、规则来源与时序证据成立
3. 退出与崩溃钩子已进入统一治理
4. 恢复定时器、数据库关闭、任务冻结已接入退出链路
5. `close` 与 `quit` 区分清楚且有证据；若当前产品已引入 `hide` / `close-to-tray`，也必须按既定语义给出证据

若仍存在默认入口启动失败、退出无治理或收尾资源泄漏，则本 TP 不通过。
