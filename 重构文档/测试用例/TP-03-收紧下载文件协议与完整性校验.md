# TP-03 收紧下载文件协议与完整性校验

## 1. 文档目标
验证普通下载与批量下载在文件落盘、临时文件命名、完整性校验、取消清理、异常恢复上的规则一致且不误删最终文件。

## 3. 风险摘要
- `.part` / 最终文件边界不清会导致半文件被误判为完成文件
- cancel 或 failed 清理过度会误删已完成的最终文件
- 重复下载与异常退出后的恢复若不幂等，会造成文件冲突或脏文件残留

## 4. 测试前置条件
- 准备可下载的中小文件，例如：可重复获取、体积适中、不会频繁超时的直链或站点样例文件
- 准备同名已存在文件场景，例如：目标目录中已存在同名最终文件，或已存在同名 `.part` 临时文件
- 准备下载中断、应用退出、取消、校验失败场景
- 观测位置：下载页任务列表状态列、状态筛选结果、目标目录文件列表；启动应用的终端输出或应用日志文件中与下载/校验/恢复相关的日志行；通过当前项目已有的数据库查看方式检查任务或下载记录状态变化，若当前环境做不到则记为待补测
- 关键用例建议保留：目标目录前后对比、截图、日志摘录、数据库状态对比

## 5. 测试分层
- 配置/数据层：核对文件路径、任务状态与恢复记录是否一致
- 主进程服务层：核对写临时文件、校验、rename、清理的落盘流程
- IPC / preload 边界层：核对普通下载与批量下载是否走同一文件协议
- 渲染层交互层：核对下载页状态列、失败提示、已完成文件可见性
- 生命周期与异常场景层：核对中断、退出、重复下载、恢复扫描后的文件边界

## 7. 详细测试用例
- `TP-03-TC-001` 下载进行中只写 `.part`，最终路径在校验通过前不可见
  - 触发：启动下载后在完成前查看目标目录
  - 检查：文件系统中是否只有临时文件，UI / 日志是否仍显示下载中
  - 预期：最终文件不可见，`.part` 作为唯一写入产物
- `TP-03-TC-002` 校验通过后才 rename 到最终路径
  - 触发：让一个下载完整完成
  - 检查：rename 时机、最终文件生成时间、日志中的校验结果
  - 预期：先校验通过，再落到最终路径
- `TP-03-TC-003` 校验失败时保留明确的失败态，不误标 completed
  - 触发：使用截断文件、错误内容、错误大小预期，或人为中断导致内容不完整的可控场景构造校验失败；若当前环境不具备则记为待补测
  - 检查：数据库状态、页面状态、失败日志、临时文件是否清理
  - 预期：状态明确为失败，不能显示完成
- `TP-03-TC-004` cancel 不会删除已完成的最终文件
  - 触发：对已完成文件执行取消或重复取消
  - 检查：文件系统、取消日志、数据库最终状态
  - 预期：最终文件保留，不被误删
- `TP-03-TC-005` failed 只清理临时产物，不清理有效最终文件
  - 触发：制造失败后观察目录清理
  - 检查：目录中最终文件是否存在、`.part` 是否被清掉
  - 预期：只清临时文件，保留有效文件
- `TP-03-TC-006` 重复下载同一目标时 DB 唯一约束/幂等逻辑成立
  - 触发：对同一目标重复发起下载
  - 检查：数据库唯一记录、是否出现重复任务、文件命名结果
  - 预期：不会产生重复有效记录或冲突写入
- `TP-03-TC-007` 异常退出后恢复逻辑不会把半文件误判为完成文件
  - 触发：下载中强制退出应用后重启
  - 检查：恢复扫描、半文件标记、页面状态和日志
  - 预期：半文件仍被视为未完成，需继续或重新下载，不得直接当作完成文件
- `TP-03-TC-008` 批量下载与普通下载的文件协议表现一致
  - 触发：分别走普通下载与批量下载完成同类文件
  - 检查：临时文件命名、rename、校验、清理规则是否一致
  - 预期：两条链路的文件协议行为保持一致

## 8. 边界与异常场景
- rename 前进程退出
- 下载过程中磁盘空间不足
- 已存在目标文件但大小错误
- `.part` 文件残留后再次发起下载

## 9. 验收标准
- 成功前不污染最终文件路径
- 异常和取消不误删完整文件
- 恢复逻辑不依赖脆弱的“文件存在即完成”假设

## 10. 交叉回归要求
- 回归 TP-01 的下载状态语义与取消/失败口径，确认文件协议调整不会把取消、失败、完成三类状态重新混淆
- 回归 TP-02 的批量领取、恢复与会话生命周期，确认批量链路在 `.part`、rename、恢复扫描后仍保持单一会话语义
- 回归下载相关配置项，至少确认下载目录、目标路径或同类配置在重启后仍生效，且文件协议变更不会把文件写到错误位置
- 回归应用启动入口与重启恢复入口，确认启动后的恢复扫描不会把半文件误判为完成文件，也不会跳过待恢复文件
- 回归下载页、批量下载页等阶段 0 高频页面中的文件存在性提示、失败提示与恢复入口

## 13. 缺陷分级与结论判定

### 13.1 缺陷分级
- S0：半文件被误判为完成文件、取消 / 失败清理误删最终文件、恢复逻辑破坏真实文件边界
- S1：校验、rename、清理顺序不一致，或普通下载与批量下载文件协议明显割裂
- S2：局部提示、失败表达或同名文件处理说明不够清晰，但主落盘链路仍可追踪
- S3：日志命名、证据格式或说明文字仍可优化，不影响文件协议主判定

### 13.2 结论判定
满足以下条件可判定本 TP 通过：
1. 下载完成前不会污染最终文件路径
2. 校验、rename、临时文件清理与失败处理遵循同一文件协议
3. cancel / failed / 异常退出不会误删有效最终文件，也不会把 `.part` 误判为完成文件
4. 普通下载与批量下载在文件协议上的行为保持一致
5. 测试记录已完整写明规则来源、实际结果、证据位置与结论

若仍存在误删最终文件、半文件误判、协议不一致或恢复扫描错误，则本 TP 不通过。

## 15. 阶段验收口径

TP-03 作为阶段 0 的关键文件协议治理任务，验收时应满足：

1. 普通下载与批量下载对 `.part`、校验、rename、清理、恢复的行为已统一
2. 成功、失败、取消、恢复不会互相污染文件系统结果
3. 恢复扫描不再依赖“文件存在即完成”的脆弱假设
4. 没有阻断级文件协议或完整性问题遗留

本设计文档不承载实际结果、证据路径和最终结论；这些信息必须由测试记录提供。若只验证最终文件是否出现，而未验证 `.part`、校验、清理与恢复边界，则只能判定为待补测或未通过，不能进入阶段通过结论。
