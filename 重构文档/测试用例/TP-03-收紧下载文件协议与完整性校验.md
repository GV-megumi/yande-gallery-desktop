# TP-03 收紧下载文件协议与完整性校验

## 1. 文档目标

本 TP 面向下载文件协议收口问题，验证普通下载与批量下载是否已采用“.part 临时文件 + 成功后 rename”的落盘模式，验证完成态是否经过完整性校验，验证取消、失败、异常退出后的清理边界是否清晰，验证数据库约束和幂等写入语义是否能支撑重复下载与重启恢复。

本任务的测试目标不是单纯确认“文件有落盘”，而是确认：

1. 文件只有在校验通过后才进入最终文件名
2. 取消与失败只清理临时产物，不误删最终文件
3. 重复下载、重复启动、异常退出恢复时不会产生脏文件或覆盖错误
4. 队列表与文件协议具备幂等与约束能力

## 4. 风险摘要

1. 文件协议改造最容易牵连取消逻辑，若边界不清，可能误删真实完成文件
2. 完整性校验如果放置不当，会导致成功率下降或重复校验造成性能问题
3. 数据库唯一约束和应用层幂等如果不同步，可能出现“库里重复、盘上冲突”的两难情况
4. 异常退出恢复往往只在中断后重启时显现，手工验证必须覆盖

## 5. 测试前置条件

### 5.1 环境前置
- 应用可启动并可执行普通下载、批量下载
- 测试环境必须具备可观察磁盘文件变化的能力
- 必须准备一个可控大小的测试文件，便于观察下载中间态
- 必须准备可用于完整性校验的已知资源，或具备可重复模拟校验失败的测试条件；若两者都不具备，完整性校验相关用例只能记为环境阻塞，不得判通过

### 5.2 数据前置
- 准备至少一个可成功下载的资源
- 准备至少一个可模拟失败、取消、中断的资源或测试条件
- 准备一个重复下载同一资源的场景
- 准备一个目标路径已存在文件的场景，用于验证重复下载与冲突处理

### 5.3 操作前置
- 必须可人工查看临时文件目录与最终文件目录变化
- 必须具备录屏或截图工具
- 必须具备查看日志、错误消息、数据库写入结果的能力
- 若无法查看数据库记录，则“无重复有效记录”只能通过应用层去重证据辅助判定，不能直接判完全通过

## 6. 文件协议与完整性校验最小口径表

| 对象 | 最低口径 | 通过判定 | 不通过判定 |
|---|---|---|---|
| 临时文件阶段 | 下载进行中只能写入临时文件，不得把未完成内容写入最终文件名 | 下载过程中仅出现 `.part` 或已明确约定的唯一临时后缀文件 | 下载过程中直接生成最终文件名且内容未完成 |
| 最终文件阶段 | 只有完成判定成立后，最终文件名才允许出现 | 最终文件在完成后出现，且内容可读、状态与 UI 一致 | 最终文件在未完成时已暴露，或内容损坏却仍显示完成 |
| rename 时机 | rename 必须发生在完成下载并通过完整性校验之后 | 日志或可观察事件能证明“先校验，后 rename” | rename 发生在校验前，或根本无 rename 过程 |
| 完整性校验最低口径 | 至少满足以下之一：文件大小与预期一致；内容哈希或校验值匹配；上游协议明确返回可验证长度且本地结果一致 | 执行记录能明确写出本次使用的校验依据，并有日志或结果支撑 | 校验依据不存在，或只有“文件看起来能打开”这类主观判断 |
| 失败后清理边界 | `failed` 只能清理临时文件，不得删除已完成最终文件 | 失败后最终文件保留完整，临时文件按实现被清理或保留，但行为有证据可追踪 | 失败后误删最终文件，或未区分临时文件与最终文件 |
| 恢复时半文件识别 | 重启恢复时必须以临时后缀、未完成状态或校验未通过结果识别半文件，不能把半文件当最终文件 | 重启后半文件被识别为未完成，并进入恢复或清理路径 | 半文件被误识别为最终文件并标记完成 |

补充口径：
- 若当前实现使用的临时后缀不是 `.part`，测试记录必须明确记录实际后缀名称，并证明其只用于未完成文件。
- 若当前实现尚未定义完整性校验依据，则 TP-03 不得判通过；完整性校验类用例只能记为待补测。
- 若当前实现对失败后临时文件采取“保留待诊断”，则必须有日志和文件命名规则支撑；如果没有明确规则，只能记为待补测，不能判通过。
- 每次执行记录必须注明本次采用的校验依据类型：长度 / hash / 上游可验证长度 / 其他；若未注明，则该次执行结果不能用于横向比较，也不能支撑 TP-03 通过结论。

## 7. 观察口径

执行 TP-03 时，至少必须观察并记录以下对象：

1. 临时文件目录
   - 观察临时文件是否生成、是否增长、是否在成功后消失
2. 最终文件目录
   - 观察最终文件是否只在完成后出现，是否被 cancel 或 failed 误删
3. 数据库记录
   - 观察是否出现重复有效记录，或同一资源被重复插入多条有效队列项
4. 关键日志事件
   - 必须关注 rename、完整性校验、失败清理、取消清理、恢复识别等关键事件

如果某次执行没有覆盖上述四类观察对象中的关键项，则该次执行不能支持 TP-03 通过结论。

## 8. 六层测试口径与验证重点

### 8.1 配置与数据层
重点看：
- 队列表是否具备数据库唯一约束，或至少具备可验证的应用层幂等写入能力
- 重复任务是否不会在数据库中产生重复有效记录
- 恢复后是否能区分临时文件与最终文件

### 8.2 主进程服务层
重点看：
- 文件是否先写 `.part` 或已明确定义的唯一临时扩展名
- 完成后是否经过校验再 rename
- 取消与失败是否只清理临时文件
- rename 前后 cancel 是否走不同边界处理

### 8.3 IPC / preload / shared 边界层
不适用原因：
- 本 TP 的核心是主进程下载文件协议、磁盘落盘、数据库约束与恢复清理策略，文件协议本身不直接以 IPC / preload / shared 作为主要验收对象。
- 若实际执行中发现页面上报的下载结果与磁盘实际状态不一致，应记录为跨层问题并联动 TP-05 / TP-06，但不把该层作为本 TP 的主要改造范围。

### 8.4 渲染层交互层
重点看：
- 下载完成、失败、取消时 UI 提示是否与实际磁盘状态一致
- 重复下载或已存在文件时是否给出明确反馈
- rename 前后 cancel 的页面提示是否与文件状态一致

### 8.5 生命周期与异常层
重点看：
- 应用异常退出后临时文件是否可识别、可恢复或可安全清理
- 重启后是否出现重复完成、重复覆盖、脏文件残留
- 半文件在恢复路径中是否被误判为完成文件

### 8.6 非功能层
重点看：
- 数据安全：最终文件不得被误删、误覆盖、误标记完成
- 可恢复性：异常退出后必须能识别半文件并进入恢复或清理路径
- 可观测性：rename、校验、失败清理、取消清理必须能从日志中观察
- 幂等性：重复下载、重复提交、重复恢复不得制造重复有效记录或磁盘脏文件
- 稳定性：同一文件在多次触发 cancel / retry / restart 后仍保持边界一致

## 9. 测试设计文档与测试记录承接说明

本文档负责定义测试设计、覆盖范围、用例步骤、预期结果与判定口径，不直接承接某一次执行结果。

实际执行时，每条用例必须由对应测试记录补充以下最低字段：
- 实际结果
- 证据路径或附件名
- 结论：通过 / 失败 / 待补测 / 环境阻塞

如果测试记录缺少上述字段，或只有“已验证”“看起来正常”这类主观描述，则该用例不得判通过。本文档本身也不能替代测试记录直接给出通过结论。

## 10. 详细测试用例

### TP-03-TC-001 普通下载应先生成临时文件而非直接写最终文件
- 前置条件：
  - 有一个可成功下载的普通任务
  - 可查看临时文件目录与最终文件目录
- 场景/步骤：
  1. 开始下载
  2. 在下载进行中查看目录
- 预期结果：
  - 下载进行中只能出现临时文件
  - 不应直接把未完成内容写成最终文件名
- 关键观察点：文件后缀、文件大小、目录中文件列表

### TP-03-TC-002 下载完成后应在校验通过后 rename 到最终文件名
- 前置条件：
  - 存在一个完成型下载任务
  - 可查看 rename 与校验日志
- 场景/步骤：
  1. 完成一次下载
  2. 查看最终目录和日志
- 预期结果：
  - 下载完成前不应暴露最终文件名的未完成内容
  - 校验通过后才 rename 成最终文件
  - 最终文件可正常打开，且满足本次记录中声明的校验依据
- 关键观察点：rename 时点、校验日志、最终文件完整性

### TP-03-TC-003 完整性校验失败时不应进入完成态
- 前置条件：必须可模拟损坏下载内容或校验失败
- 场景/步骤：
  1. 触发一次校验失败
  2. 观察状态和磁盘残留
- 预期结果：
  - 任务应进入 `failed`
  - 最终文件不应被错误认定为可用
  - 若实现定义失败后清理临时文件，则必须删除临时文件；若实现定义失败后保留临时文件待诊断，则必须保留临时文件且日志中明确记录保留原因
- 关键观察点：失败原因、残留文件类型、是否生成最终文件
- 判定补充：若当前实现未定义失败后保留还是删除临时文件，则该用例只能记为待补测，不得判通过

### TP-03-TC-004 cancel 时序边界必须覆盖 rename 前与 rename 后两个子场景
- 前置条件：存在接近完成的下载任务，且可观察 rename 日志与文件目录变化
- 场景/步骤：
  - 子场景 A：rename 前 cancel
    1. 在下载尚未完成、仍处于临时文件阶段时触发 cancel
    2. 观察临时文件、最终文件、日志
  - 子场景 B：rename 后 cancel
    1. 在日志显示 rename 已完成后触发 cancel，或对刚完成任务立即触发 cancel
    2. 观察最终文件、日志和页面提示
- 预期结果：
  - 子场景 A：cancel 只影响临时文件或未完成部分，不得生成或误删最终文件
  - 子场景 B：已完成最终文件不得被删除，页面不得把已完成文件当作可清理临时文件处理
- 关键观察点：rename 前后日志、目录差异、最终文件是否保留

### TP-03-TC-005 failed 清理边界应仅覆盖临时文件
- 前置条件：存在一个失败下载任务
- 场景/步骤：
  1. 触发失败
  2. 观察清理行为
- 预期结果：
  - 清理操作不得删除已存在的最终文件
  - 临时文件的去留必须符合当前实现的明确规则，并能在日志中找到对应证据
- 关键观察点：最终文件是否完整、临时文件是否保留、日志是否记录清理动作

### TP-03-TC-006 重复下载同一资源时应满足业务幂等且磁盘无脏文件
- 前置条件：
  - 同一资源可重复发起下载
  - 可查看最终目录、临时目录和应用层去重日志
- 场景/步骤：
  1. 完成一次下载
  2. 再次发起同一资源下载
  3. 对比两次操作后的磁盘与页面结果
- 预期结果：
  - 不应生成重复脏文件
  - 最终目录中不得出现重复完成文件或异常半文件残留
  - 页面反馈必须与实际处理结果一致
- 关键观察点：最终目录文件数、临时文件残留、页面提示、应用层去重日志

### TP-03-TC-007 已存在最终文件时再次下载不得静默覆盖
- 前置条件：目标目录已有同名最终文件
- 场景/步骤：
  1. 再次执行下载
  2. 观察系统行为
- 预期结果：
  - 系统必须执行以下其一，且产品已有明确实现：跳过并提示、复用现有文件并提示、要求用户确认覆盖
  - 不得静默覆盖现有最终文件
  - 不得把最终文件误当成临时文件删除
- 关键观察点：冲突提示、覆盖确认、文件内容变化
- 判定补充：若当前产品未明确同名文件处理规则，则该用例只能记为待补测，不得判通过

### TP-03-TC-008 异常退出后重启应能识别临时文件并避免污染完成态
- 前置条件：正在下载的任务已生成临时文件
- 场景/步骤：
  1. 在下载中异常退出或强制关闭应用
  2. 重启后检查目录和任务状态
- 预期结果：
  - 临时文件被正确识别
  - 临时文件不应被错误当成最终文件
  - 恢复流程只能走恢复未完成任务或清理半文件路径，不得直接标记完成
- 关键观察点：临时文件残留、恢复日志、任务状态

### TP-03-TC-009 批量下载中单条失败不应破坏其他条目的文件协议
- 前置条件：批量任务中包含多个文件
- 场景/步骤：
  1. 让其中一条下载失败
  2. 检查其他条目的文件状态
- 预期结果：
  - 失败条目只影响自身临时产物
  - 其他条目仍按协议完成校验和 rename
- 关键观察点：批量目录中各条记录的文件状态

### TP-03-TC-010 数据库唯一约束或并发重复提交不得产生重复有效队列项
- 前置条件：可重复并发提交相同下载条目，且可查看数据库记录或数据库错误返回
- 场景/步骤：
  1. 对同一资源连续或并发发起多次下载请求
  2. 查看数据库记录变化与去重证据
- 预期结果：
  - 不应插入重复有效队列项
  - 数据库唯一约束是优先验证项；如果存在，必须优先验证数据库层拒绝或合并重复写入
  - 若当前没有数据库唯一约束而仅依赖应用层幂等，则必须用日志、记录数对比或请求去重证据证明重复请求未生成第二条有效记录
- 关键观察点：记录唯一键、插入结果、错误返回、数据库记录数、去重日志

## 11. 负向与边界场景

### 11.1 负向场景
- 下载失败后强制退出应用，再重启，最终文件不能被误认为已完成
- 取消刚好发生在 rename 前后边界，不能误删已完成文件
- 校验失败后不应继续进入“完成”提示

### 11.2 边界场景
- 文件接近完成时取消，必须精确区分临时文件与最终文件
- 下载目录中已有同名文件且内容不同，不能直接覆盖
- 临时文件损坏但最终文件存在时，清理行为不能影响最终文件

### 11.3 恢复边界
- 历史旧版本遗留的最终文件不能被新版本临时文件逻辑误处理
- 断点恢复时必须重新校验，不可只依据文件大小判断完成

## 12. 回归要求

### 12.1 必回归对象
- 普通下载落盘路径
- 批量下载落盘路径
- 取消与失败后的清理逻辑
- 重启恢复后的目录状态

### 12.2 需要联动回归
- TP-01 的取消/失败语义是否仍正确
- TP-02 的会话生命周期变化是否导致清理重复触发
- 与下载完成通知或页面提示相关的交互链路

### 12.3 回归通过标准
- 任何完成态文件都必须先经过临时文件与校验流程
- cancel / failed 不得误删最终文件
- 异常退出后不会产生脏最终文件或重复有效记录

## 13. 自动化建议

### 13.1 适合自动化的部分
- 文件协议状态机单测
- 完整性校验逻辑单测
- 临时文件清理与 rename 顺序测试
- 数据库唯一约束或幂等写入单测

### 13.2 建议自动化方式
- 用临时目录模拟下载目标路径
- 对文件写入与 rename 过程使用可控 stub
- 对数据库插入用事务或内存测试库模拟重复提交

### 13.3 仍需人工确认的部分
- 真实磁盘上 `.part` 文件的出现与消失
- rename 是否真发生在校验后
- 取消时最终文件是否被误删
- 重启后文件是否仍可正常打开

### 13.4 自动化边界
- 自动化可以证明协议顺序与约束正确，但不能完全替代真实退出恢复验证
- 自动化与人工验证应共同确认“目录状态 + 任务状态 + 用户反馈”三者一致

## 14. 缺陷分级与结论判定

### 14.1 缺陷分级
- S0：误删最终文件、写坏最终文件、校验失败仍进入完成态
- S1：临时文件与最终文件边界混乱，导致重复覆盖或恢复错误
- S2：个别残留文件或重复提示，但主流程仍可恢复
- S3：纯清理策略细节或提示文案问题

### 14.2 结论判定
满足以下条件可判定本 TP 通过：
1. 下载先经临时文件协议，再校验，再 rename
2. cancel / failed 不会误删最终文件
3. 重复下载和异常退出恢复具备幂等与边界控制
4. 队列表满足“不出现重复有效记录”目标，且已验证数据库唯一约束或应用层幂等证据
5. 已完成必要回归并保留证据

若仍存在直接写最终文件、误删完成文件、校验缺失、或无法证明“不出现重复有效记录”，则本 TP 不通过。

## 16. 阶段验收口径

TP-03 作为阶段 0 的文件协议止血项，验收时应满足：

1. 文件必须先进入临时态，校验通过后再 rename
2. 完成态文件不应被 cancel / failed 误删
3. 重复下载和异常退出恢复时不会生成脏最终文件
4. 队列表满足“不出现重复有效记录”的验收目标；数据库唯一约束为优先验证项，若当前仅依赖应用层幂等，也必须提供可观察证据
5. 每次执行记录都已注明校验依据类型，便于横向复核
6. 已留存足够证据，且不存在阻断级数据破坏风险

只要仍存在直接写最终文件、误删完成文件、校验缺失、无法证明去重有效，或执行记录未注明校验依据任一问题，就不能判定 TP-03 通过，也不能进入阶段通过结论。
