# TP-06 收缩 preload / shared / handler 边界

## 1. 文档目标

本 TP 面向 preload 过胖、shared 类型失去权威、handler 职责越界以及 child-window 能力默认继承的问题，验证主进程与渲染进程之间的能力边界是否已经明显收缩，并恢复为可解释、可维护、可分级的结构。

本任务的测试目标不是“API 还能不能调”，而是证明以下几件事：

1. preload 暴露面必须收缩到 renderer-safe 范围，不能继续做 God API
2. shared 只能承载可信且同步维护的共享类型，不再冒充过时协议总表
3. handler 角色必须回到边界校验、权限控制、转换与错误映射，不再承担过多业务编排
4. child-window 与主窗口的能力必须分层，不能默认继承同等高权限 bridge
5. 测试设计文档与测试记录的职责必须分清，实际结果、证据路径和结论必须由执行记录承接

## 3. 测试范围

## 4. 风险摘要

1. preload、shared、handler 三层边界一旦一起收缩，极容易把“收口”误做成“功能裁掉”，必须区分真实必要能力与历史残影
2. child-window 能力分层若处理不当，可能导致子窗口失去必要功能，或者仍然继承主窗口高权限 bridge
3. `handlers-full.ts` 若仍处于歧义状态，会让测试和维护都误判“哪个文件才是正式职责边界”
4. 若 renderer 仍依赖过宽对象或隐式全局，则边界收缩后可能暴露一批潜在兼容问题，必须通过执行记录逐条确认
5. 任何“按设计应该收缩”的说法都不能替代实际验证，测试记录必须保留来源和结果

## 5. 边界收缩最小判定口径

为避免“明显收缩”“更安全”这类主观判断，本 TP 执行时必须至少按下表建立统一口径；若缺少基线、白名单或对象清单，则相关用例只能记为待补测前置或环境阻塞，不得直接判通过。

| 验证对象 | 最低判定口径 |
|---|---|
| preload 暴露面 | 必须对照允许暴露的最小能力集、现有白名单或执行记录中列出的基线对象清单；若没有基线，不得宣称“已收缩” |
| shared 共享类型 | 必须核对类型是否仍被真实调用消费，且不再承担历史协议总表职责 |
| handler 职责 | 必须能指出该 handler 承担的是参数校验、权限控制、转换、错误映射中的哪些边界职责；若还承担业务编排，必须明确列出 |
| child-window 能力集 | 必须给出至少一种主窗口样本和一种子窗口样本的能力对照表；若项目存在多个子窗口实现，应在记录中写明本次覆盖对象 |
| renderer 可见对象 | 必须能列出一个已允许字段样本和一个明确不应暴露字段样本，证明页面消费的是受控 DTO 而非整包对象 |

## 7. 测试层次与验证重点

### 6.1 配置与数据层
本 TP 的配置层验证重点不是业务配置本身，而是“共享类型与 DTO 是否只承载必要字段”。重点看：
- `shared/types.ts` 中是否还有把历史协议总表当权威的写法
- DTO 是否与实际消费对象一致，且没有把内部状态、敏感字段或实现细节暴露给 renderer
- child-window 如果需要独立 DTO，其字段是否明显少于主窗口高权限对象

### 6.2 主进程服务层
重点看：
- handler 是否只做边界校验、参数转换、权限控制、错误映射
- 是否还有业务编排被长期堆在 handler 中
- `handlers-full.ts` 是否已被明确归档、去歧义或限制用途
- 主窗口与 child-window 是否被主进程明确区分能力集

### 6.3 IPC / preload / shared 边界层
这是本 TP 的核心。重点看：
- preload 暴露能力是否显著收缩，不再把内部实现一股脑交给 renderer
- shared 是否只保存可信、同步维护的共享类型
- renderer 是否通过受控 API 和 DTO 调用主进程，而不是直接读内部对象
- child-window 是否不再默认继承主窗口完整高权限 bridge

### 6.4 渲染层交互层
重点看：
- 页面是否还能完成原本应有的主链路动作
- renderer 侧是否因边界收缩而出现“对象字段消失”“方法不见了”“调用路径改了但页面未跟进”的问题
- 页面使用 DTO 后是否更清楚地只消费 view-state 与 commands
- 子窗口是否仍能完成其限定能力，而不是既太少又太多

### 6.5 生命周期与异常层
重点看：
- 窗口重建、reload、重新打开后 preload 暴露面是否一致
- child-window 初始化是否仍能在主窗口之外正确挂载受控能力
- 若 handler / preload 依赖初始化顺序，则异常恢复后是否仍一致

### 6.6 非功能层
重点看：
- preload 变瘦后是否更容易维护和审计
- shared 变得更可信后，后续调试是否更快定位来源
- child-window 能力分层是否更有安全感而不是更混乱
- 边界收缩是否减少“有能力但不该暴露”的风险

## 9. 详细测试用例

### TP-06-TC-001 preload 暴露面应收缩为 renderer-safe 最小能力集
- 测试目的：确认 preload 不再继续作为无边界扩张的 God API
- 前置条件：可查看 preload 暴露对象或日志，并能访问主窗口页面；测试记录必须给出允许暴露的最小能力集基线、白名单或等价规则来源，否则本用例不得判通过
- 场景/步骤：
  1. 打开开发者工具或等效调试输出
  2. 读取 `window.electronAPI` 或当前项目实际暴露对象的结构
  3. 将当前暴露对象与测试记录中列出的最小能力集基线逐项对照
  4. 记录超出白名单的对象、内部实现字段或无边界大对象
- 预期结果：
  - 暴露能力应收缩到已注明的最小可用面
  - 不应看到超出基线的大对象、内部实现或原始内部实体
  - 若没有基线或白名单，本用例不得判通过
- 关键观察点：基线对象清单、暴露对象字段数量、能力分类、是否存在内部实现外泄

### TP-06-TC-002 shared/types.ts 不应继续冒充过时协议总表
- 测试目的：确认 shared 类型恢复为可信共享层
- 前置条件：可访问 `shared/types.ts` 与相关引用位置
- 场景/步骤：
  1. 检查 shared 中是否仍保留明显历史协议总表或过时定义
  2. 检查引用方是否还把过时定义当作权威来源
  3. 对比类型与实际调用签名是否一致
- 预期结果：
  - shared 只保留可信且同步维护的共享类型
  - 不应继续把历史残影当权威协议
  - 测试记录需注明判断来源
- 关键观察点：类型命名、引用链、是否仍与实际实现脱节

### TP-06-TC-003 handler 应主要承担边界职责而非业务编排
- 测试目的：确认 handler 角色已回归边界层
- 前置条件：可查看 `handlers.ts` 中高频通道处理逻辑；测试记录必须说明本次选取的查询类、保存类、事件类 handler 样本及其对应规则来源
- 场景/步骤：
  1. 选取一条查询类、一条保存类、一条事件类 handler
  2. 逐项标记其承担的是参数校验、权限控制、转换、错误映射中的哪些边界职责
  3. 记录是否仍包含业务组合、页面策略、重试/轮询编排或深层流程编排
  4. 若存在业务编排，记录其是否已在任务文档中被允许保留，若无明确依据则视为未收口
- 预期结果：
  - handler 应主要承担边界职责
  - 重业务编排应下沉到 service / facade / hooks
  - 若无法明确列出边界职责与越界部分，本用例不得判通过
- 关键观察点：边界职责清单、越界编排清单、跨层调用深度、是否存在页面流程上浮

### TP-06-TC-004 handlers-full.ts 的角色必须明确且不再歧义
- 测试目的：确认 `handlers-full.ts` 已被明确处理，不会再误导维护者
- 前置条件：可查看 `handlers-full.ts` 内容与引用链
- 场景/步骤：
  1. 检查文件是否仍作为运行实现、归档样例或兼容过渡存在
  2. 检查其说明是否足够明确
  3. 检查代码引用是否已去歧义
- 预期结果：
  - 文件角色必须唯一明确
  - 不应再出现正式实现、历史样例、mock 说明三种解释并存
  - 测试记录需引用相应任务文档说明
- 关键观察点：命名、注释、引用链、是否仍参与运行

### TP-06-TC-005 renderer 应仅消费 renderer-safe DTO 与受控命令
- 测试目的：确认页面不再直接触达内部实现对象
- 前置条件：可查看页面调用或调试输出；测试记录必须列出一个允许暴露字段样本和一个明确不应暴露字段样本，否则本用例不得判通过
- 场景/步骤：
  1. 在 renderer 页面读取配置、站点或对象类数据
  2. 观察页面拿到的是 DTO 还是整包对象
  3. 主动检查记录中列出的“不应暴露字段样本”是否仍可被读取
  4. 触发一个典型命令，检查是否通过受控 API 调用 main
- 预期结果：
  - 页面只消费安全 DTO、命令和事件
  - 不应直接拿到内部对象或过宽桥接能力
  - 记录中列出的“不应暴露字段样本”必须不可见，否则不得判通过
- 关键观察点：允许字段样本、不应暴露字段样本、对象结构、API 入口、字段暴露量

### TP-06-TC-006 child-window 能力应与主窗口分层
- 测试目的：确认子窗口不会默认继承主窗口完整高权限能力
- 前置条件：存在 child-window 或嵌入式子页面场景；测试记录必须写明本次覆盖的是哪一种子窗口实现；若项目存在多个子窗口实现而本次只覆盖其中一种，不得把结论外推为“所有子窗口均通过”
- 场景/步骤：
  1. 记录本次使用的主窗口样本和子窗口样本
  2. 比较主窗口与该子窗口可用 API
  3. 检查子窗口是否仅拥有该场景所需能力
  4. 触发一个子窗口限定动作，确认其在受控范围内工作
- 预期结果：
  - 子窗口能力应明显少于或不同于主窗口
  - 不应默认共享完整高权限 bridge
  - 若保留额外能力，必须能说明理由并在记录中引用来源
  - 若只覆盖一个子窗口实现，结论只能对该实现成立
- 关键观察点：子窗口实现类型、能力对照表、权限差异、是否存在越权能力

### TP-06-TC-007 过渡兼容层不得长期与新边界并存而无说明
- 测试目的：确认边界收缩过程中是否存在无期限兼容层
- 前置条件：存在旧接口、旧类型或过渡桥接
- 场景/步骤：
  1. 定位兼容层
  2. 观察其是否只是短期迁移手段
  3. 检查是否有明确退场边界或替换计划
- 预期结果：
  - 兼容层要么退场，要么有明确迁移说明
  - 不应长期同时存在新旧双边界而无解释
- 关键观察点：兼容层说明、调用方数量、是否仍被新代码依赖

## 10. 负向与边界场景

### 9.1 负向场景
- preload 收缩后某页面直接访问原始对象，应被视为越界依赖
- child-window 仍可拿到主窗口才应有的权限时，属于边界失败
- `handlers-full.ts` 被误用作正式协议参考时，必须在记录中标记结构风险
- shared 中仍留有过时协议总表但引用方未清理，应视为未完成

### 9.2 边界场景
- 同一页面在主窗口和子窗口都可运行时，必须验证两者能力差异
- 若某些能力必须在 preload 中保留，应明确为“renderer-safe 且必要”的最小集合
- 兼容层若存在，应在记录中区分“临时迁移”与“长期保留”
- 窗口重建或 reload 后暴露面不能失序

### 9.3 待补测前置
- 若无法查看 preload 暴露对象，相关 shrink 验证只能记为待补测
- 若 child-window 场景当前不可用，需先补齐测试入口再判通过

- 对 shared 中的协议类型写静态校验
- 对 child-window 能力列表写白名单测试
- 对 handler 复杂度和职责分布做轻量静态检查

### 11.3 仍需人工确认的部分
- 暴露面收缩后页面是否仍顺手可用
- 子窗口能力差异是否符合产品心智
- `handlers-full.ts` 是否已不再制造歧义
- renderer 是否真正停止了对过宽对象的依赖

### 11.4 自动化边界
- 自动化可验证结构收敛，但不能替代真实窗口交互与权限感知
- 对过渡兼容层、文件角色歧义和边界解释，仍需人工核对

## 13. 缺陷分级与结论判定

### 12.1 缺陷分级
- S0：preload 仍然是无边界高权限桥、子窗口越权、shared 失去可信性导致核心页面不可用
- S1：handler 仍承担大量业务编排，或能力分层不清导致明显回归风险
- S2：局部 DTO 字段或个别页面调用方式不够收敛，但主流程可继续
- S3：命名、注释、归档说明可再整理，但不影响边界判定

### 12.2 结论判定
满足以下条件可判定本 TP 通过：
1. preload 暴露面明显收缩且只保留 renderer-safe 能力
2. shared 恢复可信共享类型职责，不再冒充过时协议总表
3. handler 角色回到边界职责，不再大量编排业务流程
4. child-window 能力与主窗口分层清晰
5. 测试记录、执行记录和规则来源引用完整

若仍存在过宽暴露、职责越界或 child-window 越权，则本 TP 不通过。
