# TP-10 建立桌面基础能力：后台常驻、托盘、通知基础

## 1. 文档目标

本 TP 面向 Electron 桌面应用基础运行形态补齐问题，验证应用是否已经从“窗口关闭即结束的大型页面容器”收敛为“可承载后台任务的桌面工具”。

本任务的测试目标不是“托盘图标能不能显示”或“通知弹没弹出来”这么简单，而是证明以下几件事：

1. `close / hide / quit` 三种语义已经明确区分，且不会互相串义
2. 主窗口生命周期与长任务生命周期已解耦，关闭窗口不再默认中断后台任务
3. tray 能承接应用常驻、主窗口恢复 / 重建、状态观察与显式退出入口
4. 通知能力要么形成可复核的触发-展示-设置控制链路，要么不再作为可兑现能力暴露给用户
5. 单实例、启动恢复、托盘驻留、退出治理之间的边界清晰，不会制造“应用还活着但找不到窗口”之类的新问题

## 3. 测试范围

### 3.1 必测范围
- 关闭主窗口、隐藏主窗口、显式退出应用三种路径的行为差异
- tray 图标存在性、菜单入口、点击恢复、显式退出入口、窗口已销毁时的重建能力
- 长任务进行中关闭窗口、隐藏到托盘、从托盘恢复后的连续性
- 单实例场景下重复启动时的唤醒已有窗口 / 恢复主窗口行为
- 通知设置、通知触发时机、通知点击后的恢复或回跳行为
- 主窗口关闭、托盘驻留、退出治理、启动恢复之间的时序联动

### 3.2 不在本次范围内
- 下载状态机语义本身，属于 TP-01
- 批量调度单一执行体与 claim 原子性，属于 TP-02
- 运行入口默认可启动、崩溃钩子与统一退出治理主体，属于 TP-08；但本 TP 必须验证桌面形态是否正确消费这些基础能力
- 交互一致性中的导航、Modal、空错态统一规则，属于 TP-09
- `electron-log`、`@sentry/electron` 等长期增强型观测能力接入本身，属于 TP-12

### 3.3 适用层次
- 配置与数据层
- 主进程服务层
- IPC / preload / shared 边界层
- 渲染层交互层
- 生命周期与异常层
- 非功能层

## 4. 风险摘要

1. 若 `close / hide / quit` 口径未先固定，托盘与通知会把原本的退出边界问题放大为更多不可解释状态
2. 若下载状态或长任务语义仍不可信，后台常驻会掩盖问题而不是解决问题，因此测试必须同时观察任务真实状态与桌面行为
3. tray 与单实例恢复若未联动好，容易出现“进程存在但窗口丢失”“再次启动无反应”“托盘恢复和 second-instance 抢同一窗口状态”等问题
4. 通知若只有页面设置但没有系统行为兑现，会再次制造 TP-04 已要求清理的伪能力
5. 不同平台、不同系统通知权限与焦点规则差异较大，测试记录必须写明本次执行环境；不能把某一平台现象直接外推为通用通过

## 5. 测试前置条件

### 5.1 环境前置
- 应用可正常启动，并已具备 TP-08 所要求的最小可运行启动链路
- 当前环境必须支持系统托盘；若系统禁用托盘或运行环境不提供托盘承载面，tray 相关用例只能记为环境阻塞，不得判通过
- 当前环境必须支持系统通知；若通知被系统全局关闭或权限被拒绝，通知触发类用例只能记为环境阻塞，不得判通过
- 必须能观察应用进程、窗口状态、托盘图标与通知弹出结果

### 5.2 数据前置
- 至少准备一个可持续运行数十秒以上的长任务样本，用于验证关闭窗口后任务不中断
- 至少准备一组固定通知样本，并在测试记录中逐条绑定事件类别、触发入口与观察目标：成功通知样本、失败通知样本、需人工处理通知样本、可点击通知回跳样本；若产品当前只定义了其中部分事件类别，必须写明 source of truth，未覆盖类别只能记为待补测、环境阻塞或 N/A，不得由相邻类别外推通过
- 至少准备一个可重复触发 second-instance 的启动入口

### 5.3 操作前置
- 必须具备截图或录屏能力；凡涉及关闭窗口、托盘恢复、通知弹出、second-instance 唤醒的用例，不保留证据不得判通过
- 必须具备查看日志、调试输出或进程状态的能力；若只能看到 UI，无法核对“应用仍在后台运行”或“显式退出已真正结束进程”，相关用例只能记为待补测
- 必须能区分“关闭窗口”“最小化”“隐藏到托盘”“退出应用”这几类用户动作；若当前产品尚未明确其中某一动作的唯一语义，测试记录必须先写明 source of truth，未写明不得判通过

### 5.4 口径确认前置
- 在执行 TP-10 之前，必须先确认以下 source of truth：
  - 主窗口右上角关闭按钮触发的是 `hide`、`close-to-tray`、`ask-user` 还是 `quit`
  - tray 左键 / 双击 / 菜单动作中，哪个是恢复主窗口的主入口
  - tray 菜单的最小动作集合是什么，至少应明确：恢复主窗口、显式退出，以及产品认可的最小状态查看动作清单；若存在“打开下载队列”“查看同步状态”“查看最近通知”等状态查看动作，必须逐项写明正式命名与语义
  - 通知是否受设置页开关控制，若受控制，默认值与生效时机是什么
  - second-instance 到来时，若当前无可见主窗口，应唤醒已有窗口还是重建主窗口
  - 当前产品允许进入“主窗口已销毁但后台实例仍存活”的方式是什么，哪些路径是正式支持路径，哪些只是测试注入路径
  - 通知由哪一层统一触发（主进程服务、统一通知适配层或等价受控层），以及哪些事件类别必须通知、哪些事件类别不得通知
- 若上述任一项没有明确 source of truth，则对应测试用例只能记为“待补测前置”或“环境阻塞”，不得判通过，也不得据此给出 TP-10 通过结论

## 6. 六层测试口径与验证重点

### 6.1 配置与数据层
重点看：
- 通知开关、后台常驻相关设置、关闭行为设置是否有明确持久化落点
- 关闭窗口到托盘后，任务运行状态是否继续保持且可恢复观察
- 若桌面形态需要保存“上次隐藏到托盘”或“是否显示提示”的轻量偏好，落点是否与既有配置边界一致
- 执行记录必须注明本次验证依赖的配置来源与实际生效时机；若只看到 UI 勾选变化而未核对持久化与运行行为，不得判通过

### 6.2 主进程服务层
重点看：
- 主窗口关闭、隐藏、恢复、销毁、重建是否由统一主进程模型驱动
- tray 初始化、销毁、菜单项与窗口恢复逻辑是否存在重复注册、重复创建、空引用操作
- 长任务是否真正与窗口关闭解耦，而不是“窗口一关任务也跟着停但页面没提示”
- 通知触发是否在主进程受控层统一发出，且不会重复触发或在错误时机提前触发
- second-instance 是否走单一路径唤醒已有窗口或重建窗口
- 若存在“主窗口已销毁但后台实例仍存活”的桌面形态，测试记录必须先写明该状态如何进入、是否是正式支持路径，以及恢复后是否沿用原主窗口状态还是按定义重建；未写明不得判通过

### 6.3 IPC / preload / shared 边界层
重点看：
- 设置页若提供后台常驻或通知开关，其页面调用必须走受控 API，而不是隐式依赖渲染层本地状态
- renderer 若需要请求“显示主窗口”“退出应用”“打开通知相关设置页”等能力，必须通过正式 bridge 暴露
- 若当前实现尚无相关 bridge，而页面却宣称能力已存在，则应记录为伪能力或跨层缺口，不能判通过
- 若 source of truth 未明确通知回跳行为、托盘菜单指令返回值或相关 DTO 字段，则测试记录必须先写明依据；未写明不得判通过

### 6.4 渲染层交互层
重点看：
- Settings 页中的通知 / 后台运行相关入口是否真实可用，不能只是“看起来可配”
- 关闭窗口后的用户反馈是否可理解，例如是否有“已最小化到托盘”提示、是否有显式退出入口
- 从托盘恢复后，窗口焦点、可见性、主要页面状态是否符合预期
- 若通知被点击，窗口恢复或页面跳转是否与产品定义一致；若无明确定义，测试记录必须注明 source of truth，未注明不得判通过

### 6.5 生命周期与异常层
重点看：
- 关闭窗口后应用是否继续存活、任务是否继续、tray 是否仍可恢复
- 显式退出后是否真正结束进程、清理 tray、停止后台任务承载入口
- second-instance 到来时，是否稳定唤醒或重建主窗口，而不是无窗口响应
- 主窗口已销毁、仅剩后台进程时，托盘与 second-instance 是否都能恢复窗口
- 异常退出后重启，如果产品规定要恢复窗口 / 恢复任务 / 不恢复隐藏态，测试记录必须写明 source of truth；未写明不得判通过

### 6.6 非功能层
重点看：
- 可解释性：用于区分“窗口关了但应用没退”和“真正退出”的提示文案、菜单入口或状态反馈是否已按 source of truth 明确存在且可达
- 稳定性：重复关闭、重复恢复、重复 second-instance 不得制造多份窗口、多份 tray 或无响应状态
- 资源治理：窗口隐藏后 CPU / 日志 / 轮询 / 任务执行不能出现 source of truth 未定义的额外增长；若需判定增长异常，测试记录必须写明基线、观察窗口与判定字段
- 可观测性：必须能从日志、进程状态、tray 状态和通知记录中还原桌面生命周期序列
- 兼容性：测试记录必须注明系统环境、通知权限状态、tray 环境差异，不能忽略平台差异

## 7. 测试设计文档与测试记录承接说明

本文档只负责定义测试设计、覆盖范围、步骤、预期结果与判定口径，不直接承接某一次实际执行结果。

实际执行时，每条用例的以下信息必须由测试记录承接，而不是由本设计文档承载：
- 实际结果
- 证据路径 / 附件名
- 结论（通过 / 失败 / 待补测 / 环境阻塞 / N/A）
- 本次执行环境与 source of truth 来源

如果测试记录缺少上述字段，或仅写“托盘正常”“通知没问题”“窗口能回来”这类主观描述，则该用例不得判通过。本设计文档本身也不能代替测试记录直接给出最终结论。

## 8. 详细测试用例

### TP-10-TC-001 关闭主窗口不应默认等价于退出应用
- 测试目的：验证 `close` 与 `quit` 已分离
- 前置条件：已确认主窗口关闭按钮的 source of truth；存在至少一个可运行中的长任务；可观察进程与 tray 状态
- 场景/步骤：
  1. 启动应用并开始长任务
  2. 点击主窗口关闭按钮
  3. 观察窗口、进程、tray、任务状态
- 预期结果：
  - 若 source of truth 定义关闭按钮为隐藏或 close-to-tray，则窗口关闭后应用进程仍存在
  - tray 应保留可恢复入口
  - 长任务不应被无提示中断
  - 若 source of truth 明确关闭即退出，则不得同时宣称支持后台常驻
- 关键观察点：窗口可见性、进程是否仍在、tray 图标、长任务状态、测试记录中的 source of truth

### TP-10-TC-002 显式退出入口应真正结束应用并清理桌面承载面
- 测试目的：验证 `quit` 语义独立且可信
- 前置条件：应用已处于可托盘常驻状态；存在显式退出入口，例如 tray 菜单、应用菜单或确认弹窗
- 场景/步骤：
  1. 让应用进入后台常驻或保持前台运行
  2. 触发显式退出入口
  3. 观察窗口、tray、进程和任务承载状态
- 预期结果：
  - 应用进程真正退出
  - tray 图标被清理
  - 不再残留可恢复但不可见的后台实例
  - 若产品规定退出前需要确认或冻结任务，测试记录必须注明 source of truth 并据此判定
- 关键观察点：进程结束、tray 消失、任务是否按规则停止、是否存在退出确认

### TP-10-TC-003 tray 应形成可复测的恢复主窗口入口
- 测试目的：验证 tray 不是只显示图标而无恢复闭环
- 前置条件：系统支持 tray；应用已隐藏到托盘；已确认主恢复入口的 source of truth，并写明本次验证采用的固定恢复动作与观察窗口
- 场景/步骤：
  1. 将应用隐藏到托盘
  2. 按 source of truth 触发 tray 主恢复动作，例如左键、双击或菜单项
  3. 在预先约定的观察窗口内记录窗口恢复结果
- 预期结果：
  - 主窗口应恢复为可见、可聚焦状态，或按 source of truth 恢复到最近一次可见状态
  - 不应出现点击 tray 无响应、只恢复到任务栏但不聚焦、恢复出空白窗口等问题
  - 若当前产品未明确左键 / 双击规则，则用例只能记为待补测前置，不能判通过
- 关键观察点：固定 tray 动作、观察窗口、窗口 show / focus、恢复后的页面状态、source of truth 记录

### TP-10-TC-004 主窗口已销毁时 tray 或 second-instance 应能重建窗口
- 测试目的：验证“只有后台进程存活”时的窗口恢复能力
- 前置条件：应用可处于无主窗口但仍在后台的状态；可触发 tray 恢复或 second-instance；测试记录必须先写明该状态的进入方式、是否属于正式支持路径，以及对应 source of truth；若 source of truth 明确该形态不是正式支持的桌面主路径，则本用例只能作为补充验证记录为 N/A、待补测或补充观察项，不纳入 TP-10 主路径通过门槛；若只能通过临时调试注入进入该状态，也必须如实记录，未记录不得判通过
- 场景/步骤：
  1. 按已记录的正式支持路径或受控测试注入路径，让主窗口被销毁或不可见，仅保留后台实例
  2. 通过 tray 或 second-instance 触发恢复
  3. 观察是否创建或重建主窗口
- 预期结果：
  - 可恢复出新的主窗口实例，且不会重复创建多个主窗口
  - 重建后主上下文与主要页面状态符合产品定义
  - 若 source of truth 未定义“销毁后恢复”与“隐藏后恢复”的差异，用例不得判通过
  - 若该状态本身并非正式支持形态，测试记录不得把结果外推为日常桌面主路径已通过，且不得据此阻断其他已定义主路径结论
- 关键观察点：状态进入方式、窗口实例数、重建日志、恢复后的初始页面、source of truth 记录

### TP-10-TC-005 长任务进行中关闭窗口后应继续运行
- 测试目的：验证长任务与窗口生命周期解耦
- 前置条件：TP-01、TP-02、TP-08 的基础链路已可运行；存在一个长任务样本；可观察任务真实进度
- 场景/步骤：
  1. 启动长任务
  2. 在任务运行中关闭或隐藏主窗口
  3. 等待一段时间
  4. 从 tray 恢复主窗口后检查任务进度
- 预期结果：
  - 长任务在窗口不可见期间继续推进，或按 source of truth 维持后台承载状态
  - 恢复窗口后可看到连续进度，不应从头丢失或静默停止
  - 若产品规定某类任务在后台必须暂停，则测试记录必须注明 source of truth，未注明不得判通过
- 关键观察点：任务进度连续性、后台期间日志、恢复后状态、source of truth 记录

### TP-10-TC-006 second-instance 到来时应唤醒已有窗口或重建主窗口
- 测试目的：验证单实例从“防重复启动”升级为“可恢复桌面入口”
- 前置条件：应用已在运行；已确认 second-instance 的 source of truth；可再次执行启动命令
- 场景/步骤：
  1. 让应用处于前台、隐藏到托盘、无主窗口三种状态中的至少一种
  2. 再次执行应用启动入口
  3. 观察已有实例响应
- 预期结果：
  - 不应产生第二个完整应用实例
  - 已有实例应按 source of truth 唤醒已有窗口或重建主窗口
  - 不应出现进程仍单例但窗口没有恢复、用户无反馈的状态
- 关键观察点：实例数、窗口恢复结果、日志中的 second-instance 处理路径、source of truth 记录

### TP-10-TC-007 通知设置必须真实控制通知行为
- 测试目的：验证通知相关 UI 不是伪设置
- 前置条件：系统支持通知；Settings 中存在通知相关设置；已确认默认值与生效时机的 source of truth；测试记录必须绑定一条固定通知样本并写明其事件类别、触发入口与为何可代表本用例，未绑定固定样本不得判通过
- 场景/步骤：
  1. 打开通知设置并记录当前值
  2. 对本用例绑定的固定通知样本，分别设置为开启与关闭
  3. 触发同一事件类别的通知样本
  4. 观察通知是否实际按设置生效
- 预期结果：
  - 开启时通知应能按定义触发
  - 关闭时同一固定样本不应再触发通知
  - 不允许“设置页可切换但系统行为无变化”
  - 未覆盖的其他通知类别不得由本用例外推通过，只能按各自样本另行记录
  - 若产品定义为需要重启后生效，测试记录必须注明 source of truth 且重启后复测；未注明不得判通过
- 关键观察点：固定通知样本、设置项持久化、通知是否出现、重启前后差异、source of truth 记录

### TP-10-TC-008 成功 / 失败 / 需人工处理通知应与任务真实结果一致
- 测试目的：验证通知闭环不是“只要有结束事件都弹成功”
- 前置条件：已准备成功、失败、需人工处理三类固定通知样本，且测试记录必须逐条绑定事件类别、触发入口、是否属于必须通知 / 不应通知 / 未定义类别；已确认每类结果是否应通知、由哪一层触发通知、哪些事件不得通知的 source of truth。若本次只覆盖其中部分事件类别，未覆盖类别只能记为待补测、环境阻塞或 N/A，不得由已覆盖类别外推通过
- 场景/步骤：
  1. 分别触发本次已绑定的成功、失败、需人工处理固定样本中的已定义类别
  2. 观察各类别通知内容、触发时机与任务真实状态
  3. 核对通知是否由已定义的受控层统一触发，而不是页面零散触发
  4. 记录未覆盖类别与未覆盖原因，不得留空
- 预期结果：
  - 通知内容必须与任务真实结果一致
  - 不得出现任务失败却弹成功通知，或中间态过早弹最终成功通知
  - 不得把未定义为“必须通知”的事件类别写成通过，也不得把明确“不应通知”的事件错误弹出通知
  - 对未被产品定义或本次未覆盖的类别不得擅自判通过，测试记录必须写明 source of truth 与覆盖状态
- 关键观察点：固定通知样本、通知触发层、通知文案、触发时机、任务最终状态、source of truth 记录

### TP-10-TC-009 点击通知后的恢复行为必须可解释
- 测试目的：验证通知不仅能弹，还要有受控后续行为
- 前置条件：系统支持点击通知回调；已确认点击通知后的 source of truth，例如恢复主窗口、跳到下载页、仅聚焦应用；测试记录必须绑定一条固定可点击通知样本并写明其事件类别、触发入口与回跳预期，未绑定固定样本不得判通过
- 场景/步骤：
  1. 触发本用例绑定的固定可点击通知样本
  2. 点击通知
  3. 观察窗口和页面行为
- 预期结果：
  - 应按 source of truth 恢复窗口、聚焦应用或跳到指定上下文
  - 不应点击后无反应，也不应跳到与通知无关页面
  - 本用例结论仅对已绑定的固定可点击通知样本成立，不得外推为所有通知类别点击行为都已通过
  - 若当前平台不支持点击通知回调，测试记录必须标注环境限制并记为环境阻塞，不得默认通过
- 关键观察点：固定可点击通知样本、通知点击结果、窗口焦点、页面跳转、平台限制、source of truth 记录

### TP-10-TC-010 托盘菜单应提供最小可理解动作集合
- 测试目的：验证 tray 菜单不是只有图标没有能力
- 前置条件：tray 已创建；产品已定义最小动作集合并写明正式动作清单，至少包含恢复主窗口、显式退出，以及产品认可的最小状态查看动作；若无正式动作清单，本用例不得判通过
- 场景/步骤：
  1. 打开 tray 菜单
  2. 对照 source of truth 核对菜单是否包含恢复主窗口、显式退出，以及已明确列出的最小状态查看动作清单
  3. 分别触发这些动作，并记录每一项动作的入口文案、触发结果与目标状态
- 预期结果：
  - 菜单中至少存在恢复与退出两类核心动作
  - 每一项最小状态查看动作都能与 source of truth 中的正式命名和语义一一对应，不得由测试执行人临时解释
  - 动作文案和行为一致，不制造歧义
  - 若产品尚未定义最小动作集合，或只写“可查看状态”但未枚举动作清单，测试记录必须注明 source of truth 缺失，该用例不得判通过
- 关键观察点：菜单项集合、正式动作清单、逐项触发结果、文案可解释性、source of truth 记录

### TP-10-TC-011 关闭到托盘的重复操作必须幂等
- 测试目的：验证重复 close / hide 不会产生多份 tray 或多份窗口状态污染
- 前置条件：应用支持关闭到托盘；可观察 tray 数量与窗口状态；测试记录必须写明本次重复操作的固定次数与观察窗口，若未写明不得判通过
- 场景/步骤：
  1. 按测试记录中预先声明的固定次数重复关闭窗口或触发隐藏动作
  2. 按同样固定次数通过 tray 重复执行恢复再隐藏
  3. 在统一观察窗口内观察实例与托盘状态
- 预期结果：
  - tray 不应重复创建多份
  - 窗口状态不应出现“已隐藏但仍可见一层残影”或“多次恢复产生多个窗口”
  - 重复操作只允许产生幂等结果
- 关键观察点：固定重复次数、统一观察窗口、tray 数量、窗口实例数、日志中的重复初始化 / 重复销毁

### TP-10-TC-012 退出后再次启动应恢复到明确初始桌面形态
- 测试目的：验证退出、重启与桌面形态恢复边界
- 前置条件：已确认冷启动、显式退出后重启的 source of truth；测试记录必须明确本用例只验证“桌面形态初始化边界”，不替代 TP-08 的统一退出治理结论，也不替代 TP-01 / TP-02 的任务恢复语义结论
- 场景/步骤：
  1. 让应用进入后台常驻状态
  2. 执行显式退出
  3. 再次启动应用
  4. 观察主窗口、tray、通知设置状态，以及是否错误继承上次隐藏态
- 预期结果：
  - 冷启动结果应符合 source of truth，不能继承上一次不该保留的“隐藏态”脏状态
  - tray 应按启动规则创建或延迟创建
  - 若本次观察到任务恢复行为，也只能记录其是否与桌面形态初始化冲突；任务恢复本身是否正确，仍须回到 TP-08 / TP-01 / TP-02 的既定口径判定
- 关键观察点：冷启动窗口状态、tray 初始化时机、是否误继承隐藏态、与任务恢复是否冲突、source of truth 记录

## 9. 负向与边界场景

### 9.1 负向场景
- 系统通知权限被拒绝时，设置页不能假装通知已正常启用
- tray 初始化失败时，不能继续把关闭窗口当成 hide-to-tray 成功；必须有可见退路或记录为失败
- second-instance 到来时若主窗口引用失效，不能静默吞掉恢复动作
- 长任务失败后不应继续保留误导性的“后台运行正常”提示

### 9.2 边界场景
- 主窗口最小化后再关闭，与直接关闭的行为边界必须可解释
- 应用启动后尚未完全初始化完成就关闭窗口，不能造成半退出半常驻状态
- 任务接近完成时关闭窗口，通知与恢复行为不能串义
- 仅剩 tray、无主窗口时再执行显式退出，必须仍然可完全退出

### 9.3 恢复边界
- 异常退出后重启时，如果 source of truth 未定义是否恢复到托盘态或前台态，测试记录必须先写明依据；未写明不得判通过
- 若旧版本无 tray / 常驻数据而新版本引入相关设置，兼容迁移行为必须可解释；否则只能记为待补测

## 11. 自动化建议

### 11.1 适合自动化的部分
- 主进程 close / hide / quit 状态机分支断言
- tray 初始化与销毁幂等测试
- second-instance 处理分支测试
- 通知设置开关与事件映射断言
- 窗口已销毁时的恢复 / 重建逻辑测试

### 11.2 建议自动化方式
- 为主进程生命周期控制器编写单元测试或集成测试
- 对 tray 菜单动作、窗口 show / hide / focus 调用编写可控 stub 断言
- 对通知事件映射编写服务层测试，验证不同任务结果触发的通知类型与条件
- 对 second-instance 逻辑编写集成断言，验证不会生成多实例且能恢复窗口

### 11.3 仍需人工确认的部分
- 真正的系统 tray 是否可见、可交互、不会消失
- 不同关闭路径下用户是否能理解应用仍在后台运行
- 系统通知真实弹出、点击和焦点恢复体验
- 长任务在窗口不可见期间的真实连续性

### 11.4 自动化边界
- 自动化可以证明分支和调用顺序，但不能替代系统级 tray / 通知 / 焦点行为的人工观察
- 桌面形态相关结论必须保留人工验证证据，尤其是托盘图标、通知弹出和 second-instance 唤醒体验

## 12. 缺陷分级与结论判定

### 12.1 缺陷分级
- S0：关闭窗口即误退出导致任务中断、显式退出失效、second-instance 处理失控产生多实例、找不到窗口且无法恢复
- S1：tray / 通知 / 窗口恢复链路明显错误，用户会被误导或后台运行不可信
- S2：个别通知类型、菜单项、恢复焦点或提示文案不一致，但主闭环仍可走通
- S3：轻微文案、图标、提示时机问题，不影响桌面基础能力成立

### 12.2 结论判定
满足以下条件可判定本 TP 通过：
1. `close / hide / quit` 语义明确且可验证
2. tray 已形成恢复主窗口、显式退出及已声明状态查看动作的可复核动作闭环
3. 长任务不会因关闭窗口而无提示中断
4. 通知设置与系统通知行为一致，未兑现能力不再伪装为已生效
5. second-instance 能唤醒已有窗口或重建主窗口
6. 已留存可复核证据，且测试记录已注明执行环境与 source of truth 来源

若仍存在关闭窗口等价误退出、tray 不可恢复、通知设置伪生效、second-instance 不能恢复窗口、或关键口径没有 source of truth 支撑，则本 TP 不通过。

## 14. 阶段验收口径

TP-10 作为阶段 2 的桌面形态基础任务，验收时应满足：

1. 应用已具备“窗口可关闭但应用不必退出”的最小桌面运行模型
2. tray 能承接常驻、恢复与显式退出入口
3. 长任务与窗口生命周期已经解耦，后台承载基础成立
4. 通知能力已形成可复核的触发-展示-设置控制链路，或未兑现部分已从 UI 中明确收口
5. second-instance、窗口恢复 / 重建与退出治理之间没有明显冲突
6. 所有关键结论都由测试记录写明 source of truth，未写明者不得作为通过依据

本设计文档不承载实际结果、证据路径和最终结论；这些信息必须由测试记录提供。若仅有 tray 或仅有通知单点成立，但 `close / hide / quit`、长任务后台承载或 second-instance 恢复仍不稳定，则只能判定为待补测或未通过，不能进入阶段通过结论。
