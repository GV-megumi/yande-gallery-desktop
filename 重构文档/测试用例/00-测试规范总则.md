# 00-测试规范总则

## 1. 文档目的

本文件用于统一 `重构文档/测试用例/` 下全部 TP 文档的测试口径，避免不同任务包出现：

- 术语说明：TP 文档指单个测试包文档；任务包指对应整改任务；阶段包指同一阶段内的多个 TP 集合；高风险包指需要优先回归的依赖/邻近 TP 或风险点集合。

- 用例粒度不一致
- 通过标准不一致
- 证据要求不一致
- 只跑单点功能、不做交叉回归
- 只看 UI 表象、不验证主进程/IPC/配置/生命周期链路

## 2. 测试原则

1. **测试对象是“整改需求”，不是单点页面截图**
   - 必须同时覆盖页面行为、主进程服务、IPC/preload、配置/数据库、启动/关闭/恢复链路。
2. **测试通过不等于仅跑通自动化**
   - 任何 TP 若涉及桌面行为、窗口生命周期、隐藏页副作用、通知、托盘、webview、安全边界，必须有人机验证或明确说明无法验证的原因。
3. **必须保留证据**
   - 至少保留一种最小证据（如命令输出、日志、页面行为描述、异常现象、截图路径或复现步骤）；若形成完整测试记录，再补齐第 7 节所列字段。
4. **必须做负向和边界验证**
   - 不能只验证黄金路径。
5. **必须做交叉回归**
   - 每个 TP 修复后都要回归其依赖链和相邻高风险包。

## 3. 统一文档结构

每个 TP 文档原则上必须包含以下结构；如为保持精简而合并小节，也不得遗漏对应信息：

1. 文档目标
2. 对应任务包与关联文档
3. 测试范围
4. 风险摘要
5. 测试前置条件
6. 测试分层
7. 详细测试用例
8. 负向与边界场景
9. 验收标准
10. 交叉回归要求
11. 自动化建议
12. 不在本包主责任范围的说明

## 4. 用例编号规范

- 编号格式：`[TP编号]-TC-[三位序号]`
- 例如：
  - `TP-01-TC-001`
  - `TP-01-TC-002`
  - `TP-10-TC-014`
- 同一文档编号必须连续，不允许重复编号。

## 5. 测试分层定义

### 5.1 配置/数据层
- 配置读写、数据库状态、唯一约束、恢复数据、迁移结果。

### 5.2 主进程服务层
- 调度器、生命周期、文件系统、状态机、异常治理、通知、托盘等主进程职责。

### 5.3 IPC / preload / shared 边界层
- 通道名、参数签名、返回 DTO、安全收敛、事件广播。

### 5.4 渲染层交互层
- 页面入口、按钮、弹窗、空错态、列表、反馈消息、路径选择、过滤器、排序与分页。

### 5.5 生命周期与异常场景层
- 启动、关闭、最小化、隐藏、恢复、重复启动、崩溃、退出清理。

### 5.6 非功能层
- 性能、稳定性、并发、安全性、一致性、可解释性（例如：结果异常时能说明触发条件、日志位置和失败原因，便于复现与定位）。

## 6. 缺陷级别与结论状态统一定义

### 6.1 缺陷级别

- **S0（阻断）**：直接导致功能不可用、状态污染、安全失守、数据误删误写。
- **S1（严重）**：主链路可走但结果不可信、伪能力、明显错误恢复缺失。
- **S2（中等）**：交互割裂、状态表达错误、回归风险高但存在替代路径。
- **S3（一般）**：文案、提示、低频边角问题，不影响本次阶段验收主结论。

说明：后续 TP 文档统一使用 `S0` ～ `S3` 作为缺陷分级写法；若个别说明中仍出现“阻断 / 严重 / 中等 / 一般”，应按本节映射理解，不再视为另一套分级体系。

### 6.2 结论状态

- **通过**：前置条件满足，执行步骤完成，证据充分，且结果符合用例预期。
- **失败**：前置条件满足且已实际执行，但结果不符合预期，或发现缺陷需要按对应级别记录。
- **待补测**：当前缺少样本、基线、规则来源、可观测条件或必要前置，导致无法形成有效结论；不得按通过处理。
- **环境阻塞**：受当前设备、系统权限、运行环境、外部依赖或平台能力限制，无法完成验证；不得按通过处理。
- **N/A**：该用例对当前状态、专题或实现形态不适用，例如仅完成选型评估、尚无试点代码；测试记录必须写明不适用原因，不得用 `N/A` 掩盖本应执行的验证。

## 7. 证据留存要求

每条关键用例至少记录：

- 关键用例：直接覆盖主链路、回归高风险点、跨层边界或失败代价较高的场景；简单重复验证不算关键用例。

- 执行时间
- 代码版本 / 分支 / 提交号
- 前置数据
- 执行步骤
- 实际结果
- 预期结果
- 是否通过
- 证据位置（日志、截图、录屏、命令输出）

## 8. 回归规则

1. 阶段 0 包至少回归：下载、配置、启动入口、关键高频页面。
2. 阶段 1 包至少回归：IPC、preload、Settings、Gallery、FavoriteTags、SavedSearch。
3. 阶段 2 包至少回归：App 主导航、Modal、空错态、隐藏页行为、桌面入口。
4. 阶段 3/4 包至少回归：性能、结构稳定性、成熟库接入前后行为一致性。
5. 当一个 TP 跨多个阶段或主题时，按其覆盖范围取并集回归；若与高风险包重叠，优先补足重叠链路并单独标注。

## 9. 自动化建议规则

- **必须自动化优先**：状态机、唯一约束、DTO 签名、纯函数转换、文件协议。
- **建议自动化**：高频页面查询条件、配置保存链路、错误态切换。
- **保留人工为主**：托盘、系统通知、窗口关闭/恢复、webview、真实桌面行为。
- 无法自动化的场景需在 TP 中注明原因、人工验证方式和替代证据，不可空缺。

## 10. 禁止的验收方式

建议执行顺序：前置条件 -> 分层覆盖 -> 负向/边界 -> 交叉回归 -> 证据留存 -> 验收。

以下方式单独存在时，不能视为“测试通过”：

- 只跑 `npm run build`
- 只跑 `npm run test`
- 只开页面看一眼
- 只看代码没报错
- 只验证一条黄金路径
- 没有任何日志/行为/步骤证据就宣布通过
