# 00-测试规范总则

## 1. 文档目的

本文件用于统一整套重构测试文档（以下简称 TP 文档）的测试口径，明确**测试范围、证据标准、验收条件、回归规则和缺陷分级**，避免不同任务包各写各的、测法不一致、结论不可复用。

它的核心目标不是“多写一些测试建议”，而是把以下几件事统一起来：

1. 测什么：每个 TP 文档必须覆盖哪些层次、哪些边界、哪些反例
2. 怎么测：人工验证、自动化补测和回归验证各自承担什么职责
3. 怎么算通过：完成条件、证据要求、缺陷判定和回归门槛是什么
4. 怎么复用：一个 TP 的验证结果如何支撑阶段验收和跨任务回归

本总则适用于 `重构文档/测试用例/` 目录下所有 `TP-xx` 测试文档。任何 TP 文档都可以在不违背本总则的前提下补充更细的专项规则，但**不能低于本总则要求**。

## 1.1 术语边界

为避免后文混用，本文中的几个核心术语作如下区分：

- **TP 文档**：围绕某个任务包长期维护的测试设计文档，用来定义测试范围、层次、用例、回归要求和验收口径
- **测试记录**：某一次实际执行测试后的结果记录，用来记录本次执行了哪些用例、结果如何、证据在哪里、有哪些阻塞
- **阶段验收结论**：跨多个 TP 的汇总结论，用来判断某一阶段是否达到放行条件，以及有哪些阻断项和遗留风险

后文若提到“文档结构”“固定章节”“用例清单”，默认指 TP 文档；若提到“实际结果”“附件路径”“本次未执行原因”，默认指测试记录；若提到“是否允许进入下一阶段”，默认指阶段验收结论。

---

## 2. 适用范围

本总则覆盖整套重构测试体系中的以下内容：

- 任务包级验证
- 阶段验收
- 跨任务包回归
- 手工验收与自动化补测协同
- 缺陷分级与结论记录
- 证据留存与复核

本总则不替代以下内容：

- 具体实现方案
- 任务拆分与改造路线图
- 代码级重构细节
- 单个页面或服务的临时排障笔记

如果测试结论与实现观察、文档描述出现冲突，应优先回到代码行为、运行结果和留存证据重新核对，不允许只靠主观印象下结论。

---

## 3. 与上层文档的关系

测试文档体系与上层重构文档的关系如下：

- `00-重构文档总览与使用说明.md`：说明整套重构文档怎么组织、怎么使用
- `03-任务拆分总控.md`：说明每个 TP 任务包改什么、范围多大、验收关注点是什么
- 本文件：说明每个 TP 文档应该怎么测、怎么判定、怎么留证据

因此，TP 文档不是独立于重构路线图存在的。测试应当围绕任务包目标、风险点、验收标准和建议回归项展开，而不是另起炉灶设计一套与任务无关的测试清单。

## 4. 每个 TP 文档必须包含的固定结构

本章分为**必须项**和**建议项**。必须项是最低要求，缺失即视为文档不合格；建议项用于提升可读性和复用性，但不能替代必须项。

### 4.1 必须项：必备章节顺序

每份 TP 文档都必须按以下结构编写。允许根据任务包特性增补小节，但以下内容不得缺失，也不得缺少可执行性。

1. 文档目的
2. 测试目标与覆盖范围
3. 依赖与前置条件
4. 测试层次与验证重点
5. 测试用例清单
6. 负向 / 边界 / 异常验证
7. 回归范围
8. 自动化建议
9. 缺陷分级与结论判定
10. 证据留存要求
11. 阶段验收口径

如果某个 TP 文档确实不适用某一项，也必须明确写明“不适用原因”，不能直接删掉不提。

### 4.2 必须项：不适用判定规则

只有当任务包**客观上不涉及该层能力或该类验证对象**时，才可以把某一节标记为“不适用”。例如：纯 renderer 交互修正任务，若完全不涉及主进程服务和配置持久化，可以把对应层次标记为“不适用”，但仍需说明为什么不涉及。

以下情况**不允许**标记为“不适用”：

- 时间不足
- 环境未备齐
- 尚未执行
- 本轮暂时没测
- 证据还没收集完

以上情况必须标记为：

- 待补测
- 环境阻塞
- 需要后续补证

不能用“不适用”掩盖未执行项。

### 4.3 必须项：每份 TP 文档必须写清的字段

每份 TP 文档至少要交代以下信息：

- TP 编号和任务名称
- 对应的主阶段
- 任务包目标
- 涉及模块或页面
- 前置依赖
- 本次测试重点
- 必测的正常路径
- 必测的反例和边界条件
- 需要回归的相邻任务包或共享边界
- 验收通过的条件
- 需要保留的证据
- 是否适合自动化，适合哪些部分

### 4.4 必须项：不允许缺失的测试结论字段

每个 TP 文档最终都必须能回答下面四个问题：

1. 这次是否达到任务包验收条件
2. 如果没有达到，阻塞点是什么
3. 哪些问题必须修，哪些可以作为后续任务
4. 哪些相邻功能已经回归确认过

如果无法回答，说明测试文档结构不完整。

### 4.5 建议项：展示顺序与补充字段

在满足前述必须项的前提下，TP 文档建议进一步补充以下内容，以提高复测和复核效率：

- 风险摘要
- 依赖链说明
- 关键术语说明
- 关联缺陷编号
- 证据索引表
- 变更影响对象列表

这些内容属于建议项，不作为最低合格门槛，但对于跨任务包回归和阶段复核有明显帮助。

## 5. 测试原则

### 5.1 不能只做构建或单元测试

重构测试不是“只要能 build / test 通过就算过”。

凡是涉及以下内容的任务，都必须补充手工验证：

- Electron 桌面生命周期
- 窗口创建、隐藏、恢复、退出
- 托盘、通知、开机自启、单实例
- webview / 外链 / 新窗口 / 导航安全边界
- 主进程与渲染进程交互
- IPC / preload / shared 边界变更
- 页面交互闭环、空错态、弹窗、隐藏页副作用
- 配置保存、持久化、恢复、迁移
- 下载、恢复、取消、文件落盘、恢复清理

也就是说，构建和测试命令只是基础门槛，不是最终验收本身。

### 5.2 必须覆盖人工验证

只要任务涉及真实用户会感知的运行路径，就必须给出手工验证步骤，并且至少覆盖：

- 正常路径
- 负向路径
- 边界条件
- 恢复或重试路径
- 与相邻功能的交叉影响

尤其是以下场景，不能只靠自动化断言代替人工观察：

- 桌面窗口展示和恢复
- 页面切换和副作用
- webview 加载和导航行为
- 托盘交互和系统级动作
- 通知弹出与关闭
- 异常后的可恢复性
- 用户提示是否与实际行为一致

### 5.3 必须保留证据

测试结论必须有证据支撑。凡是影响验收的验证项，不能只写“已验证”“看起来正常”“未发现问题”而不留痕迹。

证据至少应包括：

- 关键步骤记录
- 运行结果截图或录屏
- 终端输出或日志片段
- 关键状态变化说明
- 缺陷复现链路

如果结论涉及“某个问题已修复”，还应保留修复前后的对比证据或明确的复测说明。

### 5.4 必须测负向和边界

只测 happy path 不足以支撑重构验收。每个 TP 文档至少要考虑：

- 输入为空或非法
- 目标数据不存在
- 重复触发同一操作
- 中途取消或中断
- 依赖服务不可用
- 配置缺失或过期
- 权限不足或路径异常
- 页面关闭、重启、恢复之后状态是否正确

### 5.5 必须做跨回归

重构任务通常不是局部改动。一个任务包可能影响：

- 邻近任务包
- 共享服务
- 进程边界
- 启动 / 退出 / 恢复链路
- 页面状态与配置持久化

因此，TP 文档中的“回归范围”不能只写本页、本功能，而必须明确哪些相邻任务包或共享边界要一起回归。

### 5.6 必须区分“测试通过”和“风险接受”

如果某个问题暂时无法验证、无法修复或不在本次范围内，TP 文档必须明确标记为：

- 已确认通过
- 已确认失败
- 已知风险，暂不处理
- 需要补测
- 需要后续任务承接

不能把“没测到”“暂时没时间测”“看上去还行”写成通过。

## 6. 测试层次定义

每个 TP 文档都必须先按下面六个层次检查是否涉及对应能力。只要任务涉及该层，就必须写入测试范围并给出验证方法；只有任务包客观上不涉及该层能力时，才可以标记为“不适用”。

### 6.1 配置与数据层

关注点包括：

- 配置读取、写入、默认值与迁移
- 数据持久化与恢复
- 数据结构兼容性
- 路径、缓存、数据库、临时文件、导出文件
- 配置与运行数据是否正确落在预期位置

典型验证项：

- 新旧配置是否能兼容
- 修改配置后是否真的生效
- 重启后是否保持
- 异常中断后数据是否损坏
- 清理动作是否误删最终数据

### 6.2 主进程服务层

关注点包括：

- 下载服务
- 调度服务
- 初始化与退出治理
- 缓存、数据库、文件系统、托盘、通知
- 与系统能力相关的主进程逻辑

典型验证项：

- 服务是否按预期启动和停止
- 失败后是否可恢复
- 重启后是否能续接状态
- 是否存在重复调度、重复监听、重复执行
- 是否有资源泄漏或退出卡住

### 6.3 IPC / preload / shared 边界层

关注点包括：

- IPC 通道定义是否唯一
- preload 暴露面是否收缩到安全范围
- shared 类型是否与实际协议一致
- renderer 是否只通过受控 API 调用主进程能力

典型验证项：

- 通道名称、参数、返回值是否一致
- 协议修改后旧调用是否已收敛
- renderer 是否还能直接触达不该暴露的敏感字段
- 类型声明与真实行为是否一致

### 6.4 渲染层交互层

关注点包括：

- 页面切换
- 按钮、表单、弹窗、菜单
- 空态、错态、加载态
- 列表、筛选、分页、详情、预览
- 用户能直接感知的交互闭环

典型验证项：

- 用户操作后页面是否有明确反馈
- 功能按钮是否真有效
- 失败后是否有恢复路径
- 状态切换是否符合预期
- 隐藏或切页后是否还有不该发生的副作用

### 6.5 生命周期与异常层

关注点包括：

- 启动顺序
- 退出顺序
- 崩溃与异常恢复
- 重启恢复
- 单实例、后台常驻、窗口重建
- process gone、未捕获异常、退出钩子等

典型验证项：

- 程序启动慢或快时是否都稳定
- 主窗口还未完全初始化时是否会出错
- 异常退出后是否留下脏状态
- 恢复时是否重复执行旧任务

### 6.6 非功能层

关注点包括：

- 性能
- 稳定性
- 安全性
- 可恢复性
- 可观测性
- 兼容性

典型验证项：

- 大量数据下是否明显卡顿
- 长时间运行是否出现泄漏或失效
- 外部链接、webview、新窗口是否安全受控
- 错误是否能被记录和追踪
- 相同操作在不同平台或不同状态下是否一致

## 7. 用例编号规范

### 7.1 编号格式

所有测试用例统一使用如下格式：

`TP-01-TC-001`

含义如下：

- `TP-01`：所属任务包编号
- `TC`：Test Case
- `001`：同一 TP 内按顺序递增的三位编号

### 7.2 编号规则

#### 必须项

1. 一个 TP 文档内的用例编号必须唯一
2. 编号必须连续，除非文档明确说明用例已删除或合并
3. 若同一用例拆成多个子场景，可用 `TP-01-TC-001-1`、`TP-01-TC-001-2` 作为子场景编号，但主编号仍需保留

#### 建议项

1. 编号按测试执行顺序排列，而不是按页面名称随意排序
2. 同一层次的用例尽量相邻排列，便于批量执行和复测

### 7.3 编号与结论关联

每个用例应能在结果记录中被单独引用，便于：

- 复测时定位
- 阶段验收时追踪
- 缺陷回归时确认覆盖范围
- 后续任务复用已有测试链路

## 8. 测试用例的基本写法

### 8.1 必须项：最小用例字段

每个用例至少必须包含以下信息：

- 用例编号
- 前置条件
- 操作步骤
- 预期结果
- 实际结果
- 证据路径或附件名
- 结论（通过 / 失败 / 待补测 / 环境阻塞）

如果 TP 文档只给测试设计，不直接承载某次执行结果，也必须明确实际结果和证据字段将由对应测试记录承接，不能省略这一层。

### 8.2 建议项：补充字段

在满足最小字段后，建议继续补充：

- 用例标题
- 测试目的
- 是否属于必须项
- 是否影响阶段验收
- 对应测试层次
- 关联缺陷编号

### 8.3 最小测试记录模板

可按下面格式记录某次实际执行结果：

```md
- 用例编号：TP-01-TC-001
- 前置条件：已有一个进行中的下载任务
- 步骤：点击“暂停”，关闭窗口，重新打开应用
- 预期结果：任务保持 paused，不会在重启后自动恢复
- 实际结果：重启后任务仍为 paused，未自动恢复
- 证据路径/附件名：TP-01-TC-001-暂停前后录屏.mp4
- 结论：通过
```

### 8.4 写法要求

用例描述必须写成可执行、可复测、可判断的表达，不要只写“验证功能是否正常”这种空话。

### 8.5 好的用例描述

- 打开应用后，点击托盘图标，验证主窗口是否恢复并保持原状态
- 在下载进行中关闭窗口，验证任务是否继续运行且没有误退出
- 修改配置后重启应用，验证新配置是否生效且数据未丢失

### 8.6 不合格的用例描述

- 测一下这个页面
- 看看功能有没有问题
- 走一遍流程
- 确认正常即可

## 9. 缺陷分级指导

TP 文档在记录问题时，应尽量使用统一的严重程度描述，便于阶段验收和跨任务排序。

### 9.1 严重级别建议

#### S0：阻断级

满足任一条件即可视为阻断级：

- 应用无法启动或主窗口无法正常显示
- 核心功能完全不可用
- 进程退出、崩溃、卡死、无法恢复
- 数据丢失、误删、不可逆破坏
- 安全边界失效，可能泄露敏感数据或执行危险外部行为
- 任务包的核心验收目标无法完成

#### S1：高风险级

- 主要功能可用，但关键路径错误
- 用户操作会进入错误状态且恢复困难
- 重要配置、下载、持久化、IPC 行为明显不一致
- 大概率影响阶段验收结果

#### S2：中风险级

- 局部功能异常，但有替代路径或临时绕过方式
- 页面交互、提示、状态展示不一致
- 某些边界条件失败，但主流程仍可用
- 需要修复，但不一定阻断当前阶段

#### S3：低风险级

- 文案、提示、样式、交互细节问题
- 不影响主流程和数据正确性
- 可纳入后续优化，不应伪装成已通过

### 9.2 分级原则

分级时优先看以下因素：

1. 是否影响数据安全
2. 是否影响启动、退出、恢复和主流程
3. 是否影响任务包验收目标
4. 是否影响其他共享边界
5. 是否只有局部体验问题

### 9.3 分级记录要求

每个缺陷至少要写清：

- 复现步骤
- 影响范围
- 严重程度
- 是否阻塞验收
- 是否已回归修复

## 10. 证据留存要求

### 10.1 必须项：最低证据要求

凡是涉及结论、验收或缺陷判定的测试项，都必须保留至少一种证据。最低可接受证据类型包括：

- 页面截图
- 录屏
- 终端输出
- 日志片段
- 配置文件变更前后对比
- 数据库或文件状态变化说明
- 异常堆栈或错误消息

### 10.2 证据必须能支撑结论

证据不是装饰，而是用来证明结论成立。证据应当能回答：

- 执行了什么
- 看到什么结果
- 为什么认为通过或失败
- 如果有异常，异常发生在哪一步

### 10.3 建议项：证据命名样式

建议在 TP 文档或测试记录中按下面方式组织证据：

- `TP-01-TC-001-启动前.png`
- `TP-01-TC-001-恢复后.png`
- `TP-01-TC-002-日志片段.txt`
- `TP-05-TC-003-通道错误录屏.mp4`

### 10.4 证据保留时长与可追踪性

测试文档本身应保留到该阶段及其后续回归周期结束。若项目对审计、复核或问题追踪有更高要求，应同时保留原始日志和原始截图，不要只保留二次整理后的摘要。

### 10.5 无法留证时的处理

如果某项测试因为环境限制无法留证，必须在结果里明确说明原因，并标注：

- 是否影响结论可信度
- 是否需要补测
- 是否需要换环境复核

不能因为无法留证就直接默认通过。

## 11. 回归规则

### 11.1 按阶段回归

阶段验收时，回归不只看当前 TP，还要回看该阶段的共享前置和相邻任务包。

建议原则如下：

- 阶段 0：重点回归下载主链路、状态机、文件协议、启动退出基础
- 阶段 1：重点回归 IPC、preload、共享边界、页面闭环、安全边界
- 阶段 2：重点回归导航、弹窗、空错态、桌面常驻相关体验
- 阶段 3：重点回归结构优化后对已有流程的影响、性能和稳定性

### 11.2 按任务性质回归

不同性质的任务，回归重点不同：

- **状态机 / 调度类**：回归暂停、恢复、取消、重启续接、并发、重复触发
- **协议 / 边界类**：回归所有调用方、签名一致性、错误调用兜底、兼容性
- **页面闭环类**：回归相关页面的按钮、弹窗、保存、重试、空错态
- **生命周期类**：回归启动、退出、重启、崩溃恢复、单实例、后台常驻
- **安全类**：回归受控协议、受控窗口、敏感信息泄露、外链与 webview
- **性能类**：回归大列表、长时间运行、重复操作、资源占用

### 11.3 按变更面回归

如果某次改动涉及共享边界，应默认回归更广：

- 改 main 服务，要看 renderer 是否仍然可用
- 改 IPC，要看所有调用方是否都能连通
- 改 preload，要看全局 API 是否兼容
- 改配置，要看重启恢复和页面初始化
- 改 webview / 外链，要看安全拦截与系统行为

### 11.4 必须项：回归范围的最小原则

回归范围不能只写“相关页面”。每个 TP 文档至少必须写出：

- 本次改动直接影响的对象
- 至少一个相邻依赖对象
- 至少一个共享边界对象
- 至少一个重启或恢复场景

## 12. 自动化测试指导

### 12.1 自动化分级

TP 文档中的自动化建议分为三类：

#### 必须自动化

适合在每次变更后高频重复执行、且结果明确稳定的场景，例如：

- 纯逻辑函数
- 数据转换
- 协议校验
- 配置读写的稳定规则
- 状态机转移
- 关键边界判定

#### 建议自动化

适合做成回归脚本，但仍需要人工复核的场景，例如：

- 主进程服务的基础启动检查
- IPC 通道连通性
- 下载流程的部分链路
- 数据恢复、重启恢复
- 页面交互中的稳定子集

#### 手工优先

更依赖真实运行环境、系统交互或视觉判断的场景，例如：

- Electron 窗口生命周期
- 托盘与通知
- webview、外链、新窗口
- 复杂 UI 交互和多步骤状态切换
- 异常恢复与用户体验确认

### 12.2 自动化边界

自动化不是为了替代所有人工测试，而是为了减少重复劳动。若自动化会掩盖真实桌面行为、系统边界或手工可见问题，则应以人工为主。

### 12.3 自动化编写原则

如果 TP 文档建议加入自动化，应明确说明：

- 自动化覆盖什么
- 不覆盖什么
- 依赖什么测试环境
- 运行失败时如何判定是用例问题还是产品问题
- 是否可作为回归门禁

### 12.4 自动化与人工的关系

建议遵循以下顺序：

1. 先确认人工步骤完整
2. 再判断哪些步骤适合脚本化
3. 最后把稳定的子步骤转成自动化补测

不要反过来：先写脚本，再用脚本结果代替人工验收。

## 13. 阶段验收规则

### 13.1 必须项：TP 验收通过的最低条件

一个 TP 文档只有在满足以下条件时，才可以写“通过”或“完成”：

- 核心目标已验证
- 必测正常路径已覆盖
- 必测反例已覆盖
- 与该 TP 强相关的回归项已完成
- 已留存足够证据
- 已记录未解决风险或后续任务

### 13.2 必须项：阶段验收的判定方式

阶段验收不能只看单个 TP 是否完成，而应综合：

- 当前阶段所有 TP 的完成情况
- 是否存在阻断级缺陷
- 是否存在共享边界未回归
- 是否存在仅凭自动化无法覆盖的桌面风险
- 是否存在已知风险但未明确记录

### 13.3 必须项：未通过时的输出要求

若未通过，TP 文档或测试记录必须输出：

- 未通过原因
- 影响范围
- 复测条件
- 建议修复顺序
- 是否阻塞下一个阶段

### 13.4 最小结论示例

#### 通过示例

- TP 结论：TP-01 已通过。核心状态流转、暂停后重启保持、相邻恢复链路回归均已完成，证据见 `TP-01-TC-001` 至 `TP-01-TC-006` 附件。
- 阶段结论：阶段 0 当前已满足放行条件，未发现阻断级缺陷，下载主链路相关 TP 已完成最低回归。

#### 未通过示例

- TP 结论：TP-05 未通过。IPC 通道签名已部分收敛，但 Favorites 页面仍存在旧调用，阻塞本 TP 验收，需补修后复测。
- 阶段结论：阶段 1 当前不通过。TP-05 存在阻断共享边界的一致性问题，不能进入下一阶段。

## 14. 禁止的验收模式

以下做法一律不视为合格验收：

1. **只 build / 只 test**
   - 没有人工验证桌面行为、页面闭环和边界条件，就不能算完成

2. **只看一眼 UI**
   - 没有完整步骤、没有反例、没有留证据，不能算测试

3. **只测 happy path**
   - 只验证成功场景，不验证失败、取消、重试、恢复，不足以支持重构验收

4. **没有证据就下结论**
   - 仅凭“我看过了”“应该没问题”不能作为结论

5. **把未测当通过**
   - 没验证的项必须标记为未测、待补测或风险接受，不能暗中归类为通过

6. **只测本任务，不测共享边界**
   - 影响 IPC、preload、配置、状态机、生命周期的变更，必须补测相邻边界

7. **自动化结果替代全部人工验收**
   - 特别是桌面生命周期、webview、安全、通知、托盘、弹窗等场景，不允许只靠脚本结论

8. **没有缺陷分级**
   - 出现问题却不区分阻断级、高风险和低风险，会破坏阶段验收判断

9. **没有回归说明**
   - 只写本次结果，不写回归了哪些相邻功能，会导致结果不可复用

10. **结论与证据不一致**
    - 证据显示有问题，却仍写通过；或证据不足却写已完成，均不允许

## 15. TP 文档编写模板建议

为保证各 TP 文档风格一致，建议采用下列模板组织内容：

### 15.1 基本信息

- TP 编号
- 任务名称
- 主阶段
- 主责任文档
- 相关模块

### 15.2 测试目标

- 本任务要证明什么
- 最关键的验收点是什么

### 15.3 测试范围

- 必测范围
- 不在本次范围内的内容
- 与其他 TP 的交叉点

### 15.4 测试层次

- 配置与数据层
- 主进程服务层
- IPC / preload / shared 边界层
- 渲染层交互层
- 生命周期与异常层
- 非功能层

### 15.5 用例清单

- TP-xx-TC-001
- TP-xx-TC-002
- TP-xx-TC-003

### 15.6 负向与边界用例

- 非法输入
- 重复操作
- 中断 / 取消
- 恢复 / 重启
- 依赖失败

### 15.7 自动化建议

- 必须自动化项
- 建议自动化项
- 手工优先项

### 15.8 缺陷与结论

- 发现的问题
- 严重程度
- 是否阻断验收
- 是否完成回归
- 最终结论

### 15.9 证据

- 截图
- 录屏
- 日志
- 关键输出

## 16. 执行要求小结

所有 TP 文档在实际执行时，都应遵守以下底线：

- 先明确范围，再开始测试
- 先做正常路径，再补负向和边界
- 先做人工验证，再决定是否补自动化
- 先留证据，再下结论
- 先回归共享边界，再宣布阶段通过
- 先记录风险，再谈后续排期

如果某个任务包的测试文档无法做到以上要求，说明不是“测试写得简略”，而是测试口径本身还没有准备好。
