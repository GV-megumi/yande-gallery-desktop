# TP-09 统一导航弹窗空错态与隐藏页副作用

## 1. 文档目标

本 TP 面向全局交互一致性问题，验证导航、弹窗关闭模型、空态 / 错态 / 加载态、隐藏页副作用、反馈噪音以及高频管理动作的信息面是否形成统一规则。

本任务的测试目标不是“页面能不能切换”，而是证明以下几件事：

1. 一级菜单、二级菜单与右侧内容之间的联动模型稳定可预期
2. 同类 Modal 的关闭规则一致，特殊场景例外规则明确可验证
3. 错误不能再伪装成空态，加载、失败、无结果三种状态必须互斥且可区分
4. 隐藏页不应继续执行用户不可见副作用，同时 Hub 类页面切换状态不能丢失
5. 反馈提示和高频管理动作的信息面要一致，不能制造噪音或让用户猜入口

## 3. 测试范围

### 3.1 必测范围
- App 全局一级菜单 / 二级菜单 / 右侧内容联动

- Google、InvalidImages、SavedSearch、BlacklistedTags 等页面的加载、失败、空态区分
- DownloadHub / TagManagement / FavoriteTags 等 keep-alive / 隐藏页副作用治理
- 反馈消息层级、重复提示与过早成功提示治理
- 图标按钮 tooltip、Hub 默认子页、单独窗口打开边界、图集信息面一致性等一致性要求；其中高频动作入口的信息面可发现性与位置一致性由 `TP-09-TC-010` 承接，若涉及动作真实闭环则转入 TP-04，不再单独展开“可检索选择器”专项用例

- 下载状态机本身的状态语义，属于 TP-01
- IPC / preload / shared 边界与协议收口，属于 TP-05 / TP-06
- 配置持久化边界分层，属于 TP-13

### 3.3 适用层次
- 渲染层交互层
- 生命周期与异常层
- 主进程服务层中的可见副作用控制部分
- 非功能层中的噪音、可解释性与一致性部分

## 4. 风险摘要

1. 本 TP 改动范围看似集中在 renderer，但实际会牵连页面容器、挂载策略、隐藏页副作用和部分主进程窗口行为，容易出现“统一了一处，另一个页面坏了”
2. 空态、错态、加载态的区分如果不按固定规则统一，很容易继续把失败伪装成无数据
3. Modal 关闭模型一旦未按场景区分，用户会在不同页面反复猜测为什么同类弹窗行为不一样
4. 隐藏页副作用治理必须尊重 Hub 页面“切换不丢状态”的既有承诺，不能用简单卸载把状态一起丢掉
5. 反馈提示的治理需要区分阶段性提示和最终结果提示，否则会出现重复成功、过早成功或失败后仍报成功的噪音

## 5. 测试前置条件

### 5.1 环境前置
- 应用可正常启动，且可进入 App 主导航、Google、InvalidImages、FavoriteTags、DownloadHub、TagManagement、Gallery、PostDetails 等页面
- 必须具备截图或录屏能力；凡涉及导航变化、Modal 关闭、隐藏页切换、反馈消息的用例，不保留证据不得判通过
- 必须能查看页面状态或日志，以便区分副作用是否仍在执行

### 5.2 数据前置
- 至少准备一个有二级菜单或默认子页的 section
- 执行前必须建立弹窗样本清单，至少分为三类：普通表单弹窗、危险确认弹窗、沉浸式/查看型例外弹窗
- 每类至少选取 2 个真实页面样本；若某类不足 2 个，必须在测试记录中说明样本唯一性和无法扩充原因
- 若未建立弹窗样本清单，不得宣称“关闭模型已统一”
- 至少准备一个 Google 页失败样本、一个 InvalidImages 失败或无结果样本、一个 SavedSearch 或 BlacklistedTags 的无结果样本
- 至少准备一个 DownloadHub / TagManagement / FavoriteTags 的可切页样本，用于观察隐藏页副作用和状态保留

### 5.3 操作前置
- 必须能连续点击、切换页面、关闭弹窗并观察状态变化
- 若某页面当前没有可实际触发的失败态或无结果态样本，则相关用例应标记为环境阻塞，不得直接判通过
- 若某页面的关闭模型依赖特殊规则而产品规则尚未明确，必须先补足验证口径；不能写“按设计”代替验证规则

## 6. 测试层次与验证重点

### 6.1 配置与数据层
重点看：
- 哪些页面状态属于跨会话或应持久化状态，哪些只是临时 UI 状态
- 切页后是否保留应保留的工作上下文
- 隐藏页治理是否误伤需要保留的数据

### 6.2 主进程服务层
重点看：
- 导航、窗口、webview / popup 相关约束是否被主进程收口后正确执行
- 隐藏页不应继续产生不必要的主进程副作用，例如重复恢复、重复监听或重复轮询
- 若某些动作依赖主进程返回错误或状态变化，页面是否能正确接收并表达

### 6.3 IPC / preload / shared 边界层
本 TP 不以协议重构为主，但以下情况需要核对：
- 页面切换后依赖的状态是否仍通过受控 API 获取
- 管理动作、信息面数据、页面偏好等是否有跨层一致性问题
- 若导航或弹窗行为依赖 bridge 返回值异常，则应记录为跨层问题而不是纯 UI 问题

### 6.4 渲染层交互层
重点看：
- 一级菜单只切左侧二级菜单，二级菜单和右侧内容同步
- 同类 Modal 关闭模型是否一致
- 错误是否被空态吞掉
- 反馈提示是否重复、过早或前后矛盾
- 高频管理动作是否在主要信息面可见，不只藏在右键里

### 6.5 生命周期与异常层
重点看：
- 切页、隐藏、重进、外部打开返回后状态是否保持
- Google/webview 失败后是否还能重新进入可用状态
- 隐藏页是否在不可见状态下仍持续执行恢复、轮询、监听等动作

### 6.6 非功能层
重点看：
- 反馈噪音是否过多
- 状态切换是否引发明显闪烁或误导
- 大列表或频繁切页下，是否仍可快速判断当前页面规则
- 可发现性、可解释性、可恢复性是否足够稳定

## 8. 详细测试用例

### TP-09-TC-001 一级菜单切换后左侧二级菜单与右侧内容必须同步
- 测试目的：验证导航联动模型是否稳定
- 前置条件：App 主界面可正常显示，且存在至少两个一级菜单 section
- 场景/步骤：
  1. 点击一级菜单 A
  2. 观察左侧二级菜单、右侧主内容与标题
  3. 再点击一级菜单 B
  4. 再次观察三者联动
- 预期结果：
  - 一级菜单仅切换左侧二级菜单与对应右侧内容上下文
  - 左侧二级菜单默认项与右侧主内容保持一致
  - 不应出现左栏与右侧内容不属于同一 section 的情况
- 关键观察点：菜单高亮、默认子页、页面标题、右侧内容联动

### TP-09-TC-002 二级菜单默认子页切换后应保持上下文一致
- 测试目的：验证同一 section 内默认页和二级菜单切换规则
- 前置条件：存在多个二级菜单项的 section
- 场景/步骤：
  1. 进入某 section
  2. 切换二级菜单项
  3. 观察右侧内容、标题与导航栈
- 预期结果：
  - 二级菜单与右侧内容同步变化
  - 页面标题应至少能稳定标识当前一级 section 与当前二级子页，不得只显示一个模糊总标题而无法判断当前落点
  - 默认子页规则清晰且稳定
- 关键观察点：当前选中二级菜单、标题口径是否能区分 section / 子页、内容区

### TP-09-TC-003 普通表单弹窗应统一支持一致的关闭模型
- 测试目的：验证同类表单弹窗关闭行为是否统一
- 前置条件：已建立弹窗样本清单，且普通表单弹窗类别至少包含 2 个真实页面样本；若不足 2 个，测试记录必须说明样本唯一性；若未建立样本清单，不得据此宣称关闭模型已统一
- 场景/步骤：
  1. 打开普通表单弹窗
  2. 分别尝试点击遮罩、按 Esc、点击 X、点击取消
  3. 对同类弹窗重复相同操作
- 预期结果：
  - 同类弹窗的关闭行为一致
  - 若产品决定禁用系统 X，则 mask / keyboard / footer 规则必须同步且可解释
  - 不应出现同为表单弹窗却一个能 Esc 一个不能的漂移
- 关键观察点：closable、maskClosable、keyboard、footer 按钮行为

### TP-09-TC-004 沉浸式例外弹窗与危险确认弹窗的关闭规则必须明确且可验证
- 测试目的：验证特殊场景的关闭例外是否仅限于明确对象，同时覆盖危险确认弹窗的实际关闭规则
- 前置条件：已建立弹窗样本清单，且危险确认弹窗类别与沉浸式/查看型例外弹窗类别都已纳入清单；其中沉浸式/查看型例外弹窗类别至少包含 2 个真实页面样本，若不足 2 个，测试记录必须说明样本唯一性；若未建立样本清单，不得据此宣称关闭模型已统一
- 场景/步骤：
  1. 分别打开一个危险确认弹窗样本和一个沉浸式/查看型例外弹窗样本
  2. 对危险确认弹窗尝试系统 X、Esc、遮罩关闭与显式取消 / 关闭入口，记录哪些方式允许关闭、哪些方式被禁止
  3. 对沉浸式/查看型例外弹窗尝试系统 X、Esc、遮罩关闭
  4. 对比普通表单弹窗、危险确认弹窗与沉浸式例外弹窗三类规则
- 预期结果：
  - 危险确认弹窗的关闭规则必须能被实际测试覆盖并明确记录，不能只在前置条件里提到
  - 特殊弹窗可以有例外规则，但必须明确体现
  - 例外不应扩散到普通表单弹窗
  - 危险确认弹窗若禁止某类关闭方式，必须仍保留清晰可见的显式退出入口
- 关键观察点：危险确认弹窗与沉浸式例外弹窗是否允许系统关闭、Esc、遮罩关闭，是否有明确退出入口，规则是否与普通表单弹窗区分清楚

### TP-09-TC-005 Google 页面失败不得伪装为空态
- 测试目的：验证 Google Drive / Photos / Gemini 的失败态表达
- 前置条件：可制造 webview 加载失败
- 场景/步骤：
  1. 打开 Google 页面
  2. 触发加载失败或断网
  3. 观察 loading、空白、错误文案、重试入口
- 预期结果：
  - 应显示显式错误态
  - 不应仅仅把 loading 关掉然后留空白
  - 不应把失败当作空数据
- 关键观察点：错误状态、重试按钮、外部打开退路

### TP-09-TC-006 InvalidImages、SavedSearch、BlacklistedTags 的空态与错态必须区分
- 测试目的：验证筛选无结果和加载失败的表达区分
- 前置条件：可准备无结果与失败两类样本
- 场景/步骤：
  1. 在 InvalidImages 页面触发失败和无结果两类场景
  2. 在 SavedSearch 或 BlacklistedTags 上触发无结果与异常加载
  3. 对比页面表现
- 预期结果：
  - 无结果应显示空态
  - 失败应显示错态
  - 加载中与完成态不能混淆
- 关键观察点：空态文案、错误文案、加载指示器是否互斥

### TP-09-TC-007 隐藏页不应继续执行用户不可见副作用
- 测试目的：验证 keep-alive 或 display:none 页面的副作用治理
- 前置条件：存在一个带轮询、恢复或监听的隐藏页样本，例如 DownloadHub、TagManagement 或 FavoriteTags；若业务明确要求保留某类后台行为，测试记录中必须注明该例外的依据来源，否则不得把持续活动视为合理例外
- 场景/步骤：
  1. 进入目标页并开始其副作用行为
  2. 记录页面可见时的轮询请求、IPC 监听 / 恢复调度、日志活动记录、页面状态写入行为
  3. 切换到其他 tab 或页面让其隐藏
  4. 继续观察隐藏后轮询请求是否继续发送、IPC 监听或恢复调度是否继续执行、日志是否继续产生活动记录、页面隐藏后是否仍自动写状态
  5. 再切回页面，确认状态保留是否符合预期
- 预期结果：
  - 页面隐藏后，轮询请求不应继续发送，除非测试记录已注明经确认允许保留的后台例外及其依据
  - 页面隐藏后，IPC 监听或恢复调度不应继续执行，除非测试记录已注明经确认允许保留的后台例外及其依据
  - 页面隐藏后，日志不应持续产生活动记录，除非测试记录已注明经确认允许保留的后台例外及其依据
  - 页面隐藏后，不应继续自动写状态
  - 若某页面必须 keep-alive，则副作用语义应显式绑定到可见性或统一上提控制
  - Hub 页面切换状态不应丢失
- 关键观察点：轮询请求、IPC 监听与恢复调度、日志活动记录、自动状态写入、页面状态保留、例外依据是否已在测试记录中注明

### TP-09-TC-008 Hub 类页面切换后状态不得丢失
- 测试目的：验证隐藏页治理不能破坏 Hub 切换保态承诺
- 前置条件：可切换多个 Hub 子页
- 场景/步骤：
  1. 在 Hub 页 A 设置筛选 / 进度 / 选项
  2. 切到 Hub 页 B
  3. 再切回 A
- 预期结果：
  - A 页应保持原状态
  - 不应因为治理隐藏副作用而重置用户上下文
- 关键观察点：筛选状态、进度、选项、滚动位置

### TP-09-TC-009 反馈提示不应重复、过早或前后矛盾
- 测试目的：验证消息噪音治理
- 前置条件：存在会产生阶段性提示的操作，例如批量下载或任务创建
- 场景/步骤：
  1. 触发一个会产生多阶段反馈的操作
  2. 观察是否重复弹“成功”或先成功后失败
  3. 观察最终汇总提示是否与中间提示冲突
- 预期结果：
  - 不应出现同链路重复成功提示
  - 结果提示应表达阶段、结果和下一步，不制造噪音
  - 最终结果应与实际执行一致
- 关键观察点：Toast 顺序、消息层级、阶段反馈文本

### TP-09-TC-010 高频管理动作应在主要信息面具有一致可见性
- 测试目的：仅验证图集、标签、下载等高频动作入口的信息面可发现性与位置一致性，不验证这些动作本身是否真正形成功能闭环；动作真实可用性与结果正确性归 TP-04
- 前置条件：存在 Gallery、FavoriteTags 或相关管理页；最小检查集至少覆盖 Gallery 删除/改名、FavoriteTags 下载配置/同步、下载页取消/重试；若某页面没有对应动作，必须在测试记录中说明替代动作及原因
- 场景/步骤：
  1. 分别进入 Gallery、FavoriteTags、下载页或其替代页面
  2. 按最小检查集逐项检查 Gallery 删除/改名、FavoriteTags 下载配置/同步、下载页取消/重试是否只藏在右键菜单或隐蔽入口中
  3. 对比主界面、详情面、右键菜单是否都提供稳定可发现入口，并记录入口位置是否与同类页面保持一致
  4. 若点击入口进入后续弹窗或流程，只确认入口位置和信息面可发现性，不把后续动作成功与否计入本用例结论
- 预期结果：
  - 高频动作至少在主要信息面或显式操作区具备稳定入口
  - 不应只依赖右键让用户猜操作
  - 本用例只就入口可发现性与位置一致性给结论，不外推动作功能闭环已通过；相关功能闭环应转入 TP-04
  - 若使用替代动作，测试记录必须说明替代原因，且不得把替代动作的通过外推为原动作已通过
- 关键观察点：最小检查集覆盖情况、主 CTA、右键菜单、详情面入口一致性、入口位置是否稳定、替代动作及原因

### TP-09-TC-011 纯图标按钮应统一具备 tooltip
- 测试目的：验证纯图标按钮解释文本是否统一补齐
- 前置条件：页面中存在纯图标按钮
- 场景/步骤：
  1. 扫描常见页面中的纯图标按钮
  2. 悬停查看是否有 tooltip
  3. 对比 tooltip 文案与按钮真实动作是否一致
  4. 检查多个不同按钮是否错误共用无法区分的相同文案
- 预期结果：
  - 所有纯图标按钮均应有可解释 tooltip
  - tooltip 文案必须与实际动作一致
  - tooltip 不能是“操作”“功能”“点击执行”等通用占位词
  - 多个不同按钮不能共用无法区分的相同文案
  - 不应出现无文字说明的动作按钮
- 关键观察点：tooltip 文案、是否唯一且准确、是否与真实动作一致、是否存在通用占位词

### TP-09-TC-012 单独窗口打开能力不应扩散到一级菜单
- 测试目的：验证“单独窗口打开”边界
- 前置条件：存在支持单独窗口打开的页面或菜单项
- 场景/步骤：
  1. 尝试对不同层级菜单执行单独窗口打开
  2. 对比一级菜单和二级菜单的允许范围
- 预期结果：
  - 单独窗口打开只作用于左侧二级菜单，不向一级菜单扩散
  - 边界必须清楚
- 关键观察点：可触发层级、窗口打开结果、菜单文案

### TP-09-TC-013 图集信息面应保持一致，不得一个地方有另一个地方缺
- 测试目的：验证详情面和悬浮信息面的字段一致性
- 前置条件：可打开 Gallery 的图集信息面与悬浮信息面
- 场景/步骤：
  1. 查看选中图集的 Popover 信息
  2. 打开独立详情或信息 Modal
  3. 对比字段
- 预期结果：
  - 关键信息面字段应保持一致
  - 不应出现一个地方展示来源，另一个地方缺失的情况
- 关键观察点：字段差异、来源标签、元数据一致性

## 9. 负向与边界场景

### 9.1 负向场景
- 菜单切换过程中快速连点，不能导致内容区与标题不同步
- Modal 连续打开 / 关闭不能引发状态残留
- Google 页面失败后反复重试不能停留在假空态
- 隐藏页切换后继续执行恢复任务，必须被视为问题而不是正常现象

### 9.2 边界场景
- 首次进入某 section 与返回该 section 时，默认子页必须一致
- 无结果与失败在同一页面连续切换时，状态不能串台
- Hub 页面在隐藏状态下若必须保留局部状态，其副作用必须按统一规则暂停或受控
- 纯图标按钮在窄窗口下仍应可识别其意义
- 大列表或频繁消息下，提示层级不得互相覆盖

### 9.3 待补测前置
- 若某特殊弹窗是否允许系统 X 的规则尚未明确，需先补齐产品规则或交互规范后再验收
- 若未建立弹窗样本清单，或样本数不足且未说明唯一性，不得宣称关闭模型已统一
- 若 Google 页面失败场景无法稳定复现，相关用例只能记为环境阻塞

## 11. 自动化建议

### 11.1 适合自动化的部分
- 导航状态模型断言
- Modal 关闭行为矩阵断言
- 空态 / 错态 / 加载态映射断言
- Tooltip 存在性断言
- 纯展示信息面字段一致性快照

### 11.2 建议自动化方式
- 为导航与页面状态写组件测试或状态机测试
- 为 Modal 关闭矩阵写参数化测试
- 为空错态和 tooltip 写 DOM 断言
- 为信息面一致性写快照对比测试

### 11.3 仍需人工确认的部分
- 实际导航层级感受是否符合用户心智
- 隐藏页副作用是否真的停下
- Google webview 失败后的真实反馈体验
- 消息噪音是否在真实操作链路中足够克制

### 11.4 自动化边界
- 自动化可验证规则存在，但不能替代桌面实际感受
- Hidden / keep-alive / webview 类问题必须保留人工复核

## 12. 缺陷分级与结论判定

### 12.1 缺陷分级
- S0：导航错乱导致页面不可用、隐藏页继续执行导致严重资源 / 状态问题、错误被空态吞掉影响判断
- S1：Modal 关闭规则混乱、反馈噪音严重、信息面不一致影响高频操作
- S2：局部页面文案、tooltip、信息面字段轻微不一致
- S3：视觉细节或文案微调，不影响规则理解

### 12.2 结论判定
满足以下条件可判定本 TP 通过：
1. 一级菜单、二级菜单与右侧内容联动规则一致
2. 同类 Modal 关闭模型一致，特殊例外清晰
3. 错误不再伪装为空态，加载 / 空 / 错三态互斥可区分
4. 隐藏页不再持续执行不可见副作用，Hub 状态仍可保留
5. 反馈噪音与高频动作信息面问题已被收口

若导航、空错态或隐藏页副作用仍存在明显错乱，则本 TP 不通过。
