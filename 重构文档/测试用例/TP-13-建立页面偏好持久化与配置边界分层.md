# TP-13 建立页面偏好持久化与配置边界分层

## 1. 文档目标

本 TP 面向页面偏好持久化和配置边界收口问题，验证页面级排序 / 筛选 / 分页 / 视图偏好是否落在正确的数据层，是否能跨会话恢复，以及 `config`、数据库、`localStorage` 的边界是否清晰。

本任务的测试目标不是“页面重新打开还能记住状态”这么简单，而是证明以下几件事：

1. 高频管理页的工作上下文能够跨会话恢复，且恢复来源明确
2. renderer 不再直接读写整包 config，而是消费收窄后的 DTO 或独立偏好接口
3. 页面偏好、产品数据与轻量本地偏好分层明确，不再互相混装
4. 高风险状态迁移不会把 UI 行为继续塞回配置容器

## 3. 测试范围

### 3.1 必测范围
- FavoriteTags、Gallery、BlacklistedTags 等高频管理页的排序、筛选、分页、每页数量、选中视角等状态恢复
- config / database / localStorage 三者边界是否符合文档约定
- renderer-safe DTO 是否能满足页面消费，而不暴露整包 config
- 页面偏好是否在重启后恢复，并且恢复来源清晰可追踪
- 已写入 config 的 UI 行为状态是否被迁移、替换或收口
- 页面偏好在多窗口 / 多页面场景下是否保持一致或符合定义

### 3.2 不在本次范围内
- 页面的功能闭环本身，例如删除、改名、失败重试，属于 TP-04
- 全局导航、空错态、Modal 关闭规则，属于 TP-09
- IPC 单一来源与 preload / shared 收缩本身，属于 TP-05 / TP-06；但本 TP 需要关注其对偏好接口的支撑是否到位
- 下载状态机、文件协议、桌面生命周期主问题，属于 TP-01 / TP-02 / TP-03 / TP-08 / TP-10

### 3.3 适用层次
- 配置与数据层
- 主进程服务层
- IPC / preload / shared 边界层
- 渲染层交互层
- 生命周期与异常层
- 非功能层

## 4. 风险摘要

1. 页面偏好一旦没有统一存储层，很容易继续被各页随手写进 config 或 localStorage，导致跨窗口不一致和后续迁移困难
2. `config` 当前承担应用级配置、敏感配置和部分 UI 行为状态的混合职责，若不严格分层，任何迁移都可能把旧问题换个容器重复一次
3. FavoriteTags、Gallery、BlacklistedTags 都是高频管理页，如果恢复规则不明确，用户会误以为状态丢失或恢复错误
4. 若 renderer 仍直接消费整包 config，后续 DTO 收口会和页面实现互相打架
5. 某些状态是否应入 DB、入 config 或仅入 localStorage，必须按现有文档与实现明确验证；如果当前规则尚未落定，不能用“按设计”代替判断

## 5. 测试前置条件

### 5.1 环境前置
- 应用可正常启动，且可进入 FavoriteTags、Gallery、BlacklistedTags 页面
- 必须能重启应用，并观察重启前后状态变化
- 必须能查看日志、配置文件或数据库记录，以便确认偏好落点
- 必须具备截图或录屏能力；凡涉及重启恢复、状态切换、配置边界对比的用例，不保留证据不得判通过

### 5.2 数据前置
- 至少准备一个具有多个筛选项、排序项、分页或每页数量设置的 FavoriteTags 样本
- 至少准备一个具有多个可排序 / 可筛选对象的 Gallery 样本
- 至少准备一个 BlacklistedTags 样本，用于验证页面偏好恢复
- 若需验证多窗口一致性，必须能同时打开多个窗口或至少通过切页观察同一偏好源

### 5.3 操作前置
- 必须具备修改页面偏好并保存、关闭、重启、重新打开的操作条件
- 若当前没有可用于核对存储层的日志或文件权限，相关用例只能标记为环境阻塞或待补测，不得判通过
- 若某偏好当前尚未明确归属层级，必须先补齐产品规则；不能用测试去倒推规则

## 6. 测试层次与验证重点

### 6.1 配置与数据层
重点看：
- 哪些状态属于应用配置，哪些属于业务数据，哪些只是轻量本地偏好
- 页面状态是否写入了正确层级，而不是继续混进整包 config
- 重启后恢复是否来自正确持久层
- 配置 DTO 是否只包含 renderer 安全字段，不把敏感或无关字段暴露出去

### 6.2 主进程服务层
重点看：
- 页面偏好保存是否经由主进程统一收口
- 数据库中是否存在页面偏好落点，且能按页面 / 维度查回
- config 修改后广播是否只携带必要摘要或 DTO，而不是整包对象
- localStorage 是否仅用于轻量、无敏感、renderer 独享偏好

### 6.3 IPC / preload / shared 边界层
重点看：
- 页面偏好 API 是否通过正式 bridge 暴露
- renderer 是否拿到的是 DTO / 偏好接口，而不是整包 config 对象
- 保存与读取签名是否与 shared 定义一致
- 若页面状态恢复需要新接口，是否已通过边界层定义而不是临时绕过

### 6.4 渲染层交互层
重点看：
- 页面切换后，排序 / 筛选 / 分页 / 每页数量是否按规则恢复
- 恢复后是否与当前过滤结果一致
- 页面状态是否能清晰区分“最后一次工作上下文”与“当前临时视图”
- 若页面状态应该重置，也必须能明确看出重置发生，而不是悄悄丢失

### 6.5 生命周期与异常层
重点看：
- 重启后页面偏好是否正确恢复
- 关闭窗口、重新打开、切换页面后状态是否按定义一致
- 若某偏好依赖主进程或数据库恢复，异常中断后是否能继续恢复

### 6.6 非功能层
重点看：
- 偏好恢复是否稳定，不出现闪回或抖动
- 多页共享偏好时是否没有明显冲突
- 数据分层是否让后续迁移更可解释，而不是更混乱
- 敏感字段是否仍有越界暴露风险

## 8. 详细测试用例

### TP-13-TC-001 FavoriteTags 排序与筛选偏好应跨会话恢复
- 测试目的：验证高频管理页偏好恢复是否稳定
- 前置条件：FavoriteTags 页面可正常打开，且有足够数据用于排序 / 筛选；测试记录必须注明本用例关于“应恢复 / 应重置 / 应保持”的规则来源；若规则未明确，只能记为待补测前置或环境阻塞，不得判通过
- 场景/步骤：
  1. 在 FavoriteTags 中修改筛选条件、排序方式、页码、每页数量
  2. 切换到其他页面再返回
  3. 关闭应用并重启
  4. 再次打开 FavoriteTags
- 预期结果：
  - 排序、筛选、分页、每页数量按规则恢复
  - 恢复结果与最后一次工作上下文一致
  - 若某项不应持久化，则必须明确重置而不是随机回落
- 关键观察点：恢复来源、初始值、重启前后状态、测试记录中是否写明规则来源；若未写明则不得判通过

### TP-13-TC-002 Gallery 偏好恢复应与高频管理上下文一致
- 测试目的：验证 Gallery 页面工作上下文能跨会话恢复
- 前置条件：Gallery 页面存在可排序、可筛选、可分页样本；测试记录必须注明本用例关于“应恢复 / 应重置 / 应保持”的规则来源；若规则未明确，只能记为待补测前置或环境阻塞，不得判通过
- 场景/步骤：
  1. 修改 Gallery 的搜索、排序、页码或每页数量
  2. 切到其他页面
  3. 重启应用后返回 Gallery
- 预期结果：
  - 之前的工作上下文应恢复
  - 不应每次回到默认视图
  - 恢复来源应与页面偏好存储层一致
- 关键观察点：页面状态、恢复时机、是否发生回落、测试记录中是否写明规则来源；若未写明则不得判通过

### TP-13-TC-003 BlacklistedTags 偏好不应只停留在页面内存
- 测试目的：验证 BlacklistedTags 的排序 / 筛选 / 分页偏好是否跨会话持久化
- 前置条件：BlacklistedTags 页面可访问且有数据；测试记录必须注明本用例关于“应恢复 / 应重置 / 应保持”的规则来源；若规则未明确，只能记为待补测前置或环境阻塞，不得判通过
- 场景/步骤：
  1. 修改 BlacklistedTags 的筛选与分页
  2. 切页并重启应用
  3. 重新打开 BlacklistedTags
- 预期结果：
  - 页面偏好应保持或按定义恢复
  - 不应因为页面重开而完全回到默认值
- 关键观察点：本地状态、恢复后的页码、每页数量、测试记录中是否写明规则来源；若未写明则不得判通过

### TP-13-TC-004 页面偏好不应继续写入整包 config
- 测试目的：验证 UI 行为状态是否已从 config 容器中分层出去；本用例侧重存储落点与广播边界，不替代 renderer 暴露面 / DTO 收窄验证
- 前置条件：可查看 config 读取 / 写入日志、数据库或调试输出
- 场景/步骤：
  1. 修改一个页面偏好项，例如排序或页码
  2. 触发保存
  3. 检查 preload 暴露的配置读取接口返回值、renderer 运行时可见对象结构、config 变更广播 payload
  4. 检查写入落点和广播内容
- 预期结果：
  - 页面偏好不应再混装进整包 config
  - 若需要存储，应进入专门偏好层或数据库
  - 广播内容不应携带无关或敏感字段
  - 若只看到页面表现而未核对 preload 返回值、renderer 可见对象结构和广播 payload，则不得判通过
- 关键观察点：preload 配置读取接口返回值、renderer 运行时可见对象结构、config 变更广播 payload、config 写回内容、DTO 是否收窄、数据库记录

### TP-13-TC-005 renderer 应只消费 renderer-safe DTO，而非整包配置对象
- 测试目的：验证前端侧暴露面是否收窄；本用例侧重 renderer 可见暴露面、DTO 字段收窄与受控消费方式，不替代存储落点与广播边界验证
- 前置条件：可查看 preload 暴露面或调试对象
- 场景/步骤：
  1. 打开开发者工具或日志
  2. 检查 preload 暴露的配置读取接口返回值
  3. 检查 renderer 运行时可见的配置对象结构
  4. 检查 config 变更广播 payload
  5. 确认页面消费的是 DTO 或受控接口
- 预期结果：
  - renderer 不应直接拿到整包 config
  - 敏感字段和内部实现细节不应直接暴露给页面
  - 若只看页面表现而未核对 preload 返回值、renderer 可见对象结构和广播 payload，则不得判通过
- 关键观察点：preload 配置读取接口返回值、renderer 运行时可见对象结构、config 变更广播 payload、API 入口

### TP-13-TC-006 localStorage 仅应承载轻量 renderer 独享偏好
- 测试目的：确认本地存储边界不会膨胀成产品上下文
- 前置条件：可查看 localStorage 内容或对应读写逻辑；执行前必须在测试记录中注明本次选取的 localStorage 偏好项及其依据，且该偏好项必须明确归属于 renderer 轻量本地偏好，例如主题、界面展示偏好或其他只影响本地显示的轻量设置
- 场景/步骤：
  1. 修改一个已在测试记录中注明依据的轻量本地偏好项
  2. 观察 localStorage 是否写入
  3. 对比页面级工作上下文是否仍落在更合适的层
- 预期结果：
  - 轻量偏好可进入 localStorage
  - 产品级工作上下文不应混入 localStorage
  - 若未在测试记录中注明所选 localStorage 偏好项及其依据，则本用例不得判通过
- 关键观察点：所选偏好项及依据、localStorage 键名、数据大小、是否包含敏感或业务数据

### TP-13-TC-007 页面偏好保存后重启应恢复到最后一次工作上下文
- 测试目的：作为跨页面抽样总验证或代表页主路径复核，确认“保存 -> 关闭 -> 重启 -> 恢复”主链路成立；不替代 `TP-13-TC-001` ~ `TP-13-TC-003` 对各页面细分偏好项的逐项验证
- 前置条件：至少一个页面可持久化偏好，并可重启；测试记录必须注明本用例关于“应恢复 / 应重置 / 应保持”的规则来源；若规则未明确，只能记为待补测前置或环境阻塞，不得判通过
- 场景/步骤：
  1. 在目标页面调整偏好并保存
  2. 关闭应用
  3. 重启并重新打开页面
- 预期结果：
  - 状态恢复到保存前最后一次工作上下文
  - 恢复后无需用户重复设置
- 关键观察点：恢复值、重启前后对比、是否需要重载、测试记录中是否写明规则来源；若未写明则不得判通过

### TP-13-TC-008 多窗口或多页面访问同一偏好时应符合统一规则
- 测试目的：验证跨窗口一致性或多页面共享同一偏好源的一致性边界
- 前置条件：测试记录必须注明本用例关于“应同步 / 应延迟同步 / 应保持独立”的规则来源；若规则未明确，只能记为待补测前置或环境阻塞，不得判通过；若应用支持稳定多窗口入口，则可按多窗口路径执行；若当前无稳定多窗口入口，则本用例只能验证“多页面共享同一偏好源”，不得外推为多窗口一致性通过
- 场景/步骤：
  1. 按前置条件确定本次执行路径：多窗口路径或多页面共享偏好源路径
  2. 在窗口 A 或页面 A 修改页面偏好
  3. 在窗口 B 或页面 B 观察同一偏好
  4. 再次保存并检查一致性
- 预期结果：
  - 若按多窗口路径执行，偏好源应在多窗口间表现为一致或符合明确规则
  - 若按多页面路径执行，只能证明多页面共享同一偏好源的一致性，不得外推为多窗口一致性通过
  - 不应出现一个窗口或页面已变更、另一个窗口或页面长期不更新且无说明
- 关键观察点：执行路径类型、广播 / 订阅时机、是否出现陈旧状态、测试记录中是否写明规则来源；若未写明则不得判通过

### TP-13-TC-009 config 只应暴露收窄后的 DTO，不应泄露敏感字段
- 测试目的：确认 config 边界分层对 renderer 的安全效果
- 前置条件：可查看配置返回对象或调试输出
- 场景/步骤：
  1. 在 renderer 触发配置读取
  2. 检查 preload 暴露的配置读取接口返回值
  3. 检查 renderer 运行时可见对象结构
  4. 检查 config 变更广播 payload
  5. 核对对象中是否出现敏感字段或无关内部字段
- 预期结果：
  - 只返回 renderer-safe 字段
  - 不应出现代理密码、clientSecret 等敏感字段
  - 页面不应拿到整包配置对象
  - 若只看页面表现而未核对 preload 返回值、renderer 可见对象结构和广播 payload，则不得判通过
- 关键观察点：preload 配置读取接口返回值、renderer 运行时可见对象结构、config 变更广播 payload、DTO 字段列表、敏感字段是否被隐藏

### TP-13-TC-010 已写入 config 的 UI 状态应完成迁移或明确退场
- 测试目的：验证历史混装状态是否被清理
- 前置条件：历史配置中可能存在 UI 行为状态字段
- 场景/步骤：
  1. 使用旧配置或导入历史数据
  2. 打开应用并观察页面行为
  3. 检查旧字段是否被迁移、忽略或说明
- 预期结果：
  - 历史 UI 状态要么被迁移到新偏好层，要么被明确弃用
  - 不应继续以混装字段驱动新行为
- 关键观察点：迁移后值、兼容路径、是否产生重复来源

## 9. 负向与边界场景

### 9.1 负向场景
- 页面偏好保存失败时不能假装已保存成功
- DB 不可用时，不能把产品级上下文悄悄降级到错误的本地存储层
- 多窗口同时修改同一偏好时，不能出现明显脏读但无提示
- 旧 config 中存在过时 UI 字段时，不能继续让其覆盖新偏好层

### 9.2 边界场景
- 首次安装无历史偏好时，应落入明确默认值，而不是随机值
- 页面只改了一项偏好时，其他维度不应被连带重置
- 切页但不重启时，局部临时视图和持久化偏好不能混淆
- localStorage 清空后，轻量偏好可以回落默认，但页面级工作上下文应按规则独立恢复

### 9.3 待补测前置
- 若当前产品尚未明确某一偏好应落在 config、DB 还是 localStorage，必须先补齐边界规则后再测
- 若无法查看存储落点或调试输出，相关分层验证只能记为环境阻塞

## 11. 自动化建议

### 11.1 适合自动化的部分
- 偏好保存 / 读取的纯逻辑测试
- DTO 字段收窄断言
- localStorage 轻量偏好读写测试
- 页面偏好恢复状态快照测试
- 历史字段迁移规则测试

### 11.2 建议自动化方式
- 为页面偏好模型编写单元测试或服务层测试
- 为 DTO 转换与广播内容编写断言测试
- 为页面初始状态和重启恢复编写组件级测试

### 11.3 仍需人工确认的部分
- 重启后页面是否真的符合工作习惯
- 多窗口一致性是否在真实交互中自然
- 历史配置迁移提示是否足够可理解
- `localStorage` 与 DB 的边界是否不会让用户困惑

### 11.4 自动化边界
- 自动化可证明存储和 DTO 规则，但不能替代真实重启和多窗口观察
- 配置分层一旦涉及历史兼容，必须保留人工回归

## 12. 缺陷分级与结论判定

### 12.1 缺陷分级
- S0：敏感字段越界、页面偏好落错层导致数据污染、重启后严重丢失工作上下文
- S1：高频管理页偏好无法恢复，或 config / DB / localStorage 分层明显混乱
- S2：个别页面状态恢复不完整、默认值不一致、广播摘要过宽但主流程可用
- S3：轻微字段命名或文案问题，不影响分层与恢复

### 12.2 结论判定
满足以下条件可判定本 TP 通过：
1. FavoriteTags、Gallery、BlacklistedTags 等高频管理页的偏好能按规则恢复
2. renderer 不再直接消费整包 config，而是使用收窄后的 DTO 或受控偏好接口
3. config、DB、localStorage 的边界分层明确且可回归
4. 历史混装状态已迁移、收口或明确退场
5. 已留存可复核证据，且测试记录已注明 source of truth 来源

若仍存在偏好落层混乱、重启恢复丢失或敏感字段越界，则本 TP 不通过。

## 14. 阶段验收口径

TP-13 作为阶段 1 / 阶段 2 的关键边界任务，验收时应满足：

1. 高频管理页偏好能跨会话恢复，且来源清晰
2. config、数据库、localStorage 分层明确，不再混装 UI 行为状态
3. renderer 侧暴露对象已收窄为安全 DTO 或受控接口
4. 历史 UI 状态混装问题已收口，不再扩散
5. 没有阻断级分层或恢复问题未关闭

本设计文档不承载实际结果、证据路径和最终结论；这些信息必须由测试记录提供。若只有单页恢复完成，但边界层仍混乱，或恢复来源无法解释，则只能判定为待补测或未通过，不能进入阶段通过结论。
