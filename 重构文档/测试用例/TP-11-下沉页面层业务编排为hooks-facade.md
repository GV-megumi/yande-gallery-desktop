# TP-11 下沉页面层业务编排为 hooks / facade

## 1. 文档目标

本 TP 面向 renderer 页面层业务编排过重、重复流程四散和边界职责失衡问题，验证页面是否已经从“自己拼接下载、收藏、喜欢、轮询、防抖、刷新和错误恢复”收敛为“消费稳定 view state 与 commands”的结构。

本任务的测试目标不是“页面提取了几个 hook”或“文件看起来更整洁”，而是证明以下几件事：

1. 多页面重复业务流程已经被识别、归并并通过 hooks / facade 统一承接
2. 页面层职责已收缩，且这种收缩必须能通过固定样本、统一触发动作、统一观察窗口与迁移前基线证明；页面不再直接拼装重业务编排、跨层状态拼接和重复错误处理
3. hooks / facade 的输入、输出、错误、加载、刷新、重试、取消等语义已形成可复用且可验证的统一口径
4. 边界收口没有被新中间层重新打散，facade 不是把旧问题“包装一下继续存在”
5. 迁移后的页面行为与迁移前业务闭环保持一致，不会因为抽层导致功能回退或交互漂移

## 3. 测试范围

### 3.1 必测范围
- Booru 主线多个页面中重复业务流程的归并情况，至少覆盖收藏、喜欢、下载、搜索 / 刷新、轮询或回填中的若干组合
- hooks / facade 对外暴露的 view state、commands、加载态、错态、刷新策略、重试策略的一致性
- 页面迁移后是否仍通过受控 API / DTO 消费数据，而不是重新把主进程细节暴露到页面
- 页面层是否已不再重复实现防抖、请求竞态控制、轮询订阅、结果回填等共性编排
- 一组试点页面迁移后的行为一致性与回归结果

### 3.2 不在本次范围内
- IPC 协议单一来源本身，属于 TP-05
- preload / shared / handler 边界收缩本身，属于 TP-06；但本 TP 必须确认 hooks / facade 未反向突破这些边界
- 导航、Modal、空错态统一规范本身，属于 TP-09；但本 TP 必须验证抽层后页面状态表达没有退回不一致
- 下载状态机、文件协议、桌面形态基础能力，属于 TP-01 / TP-02 / TP-03 / TP-10
- 现代化库接入选型本身，属于 TP-12

### 3.3 适用层次
- 配置与数据层
- 主进程服务层
- IPC / preload / shared 边界层
- 渲染层交互层
- 生命周期与异常层
- 非功能层

## 4. 风险摘要

1. hooks / facade 容易被误做成“把页面代码搬到另一个文件”，若输入输出契约没有形成统一且可验证的口径，只会增加中间层而不减少复杂度
2. 若 TP-05、TP-06 尚未基本收口，facade 很容易再次直接依赖漂移协议或过宽配置对象，形成新的假抽象
3. 试点页面如果只验证 happy path，不验证错误、重试、竞态和刷新，容易把结构问题伪装成重构成功
4. 迁移后最常见的回退不是页面直接报错，而是收藏状态回填慢一步、下载入口重复触发、轮询未清理、刷新与筛选不同步等“可用但不可信”的问题
5. 本 TP 强依赖 source of truth：哪些流程必须统一，哪些页面允许保留页面特化逻辑，必须有明确依据；没有依据不得用“看起来差不多”判通过

## 5. 测试前置条件

### 5.1 环境前置
- TP-05、TP-06、TP-09 至少已完成本次试点页面所需的基础前置；若未完成，测试记录必须注明残留前置风险，不得直接宣称结构收敛已通过
- 应用可正常启动，并可进入至少两到三个已纳入 hooks / facade 试点的页面
- 必须具备查看日志、开发者工具、网络请求、组件状态或调试输出的条件，以便核对页面是否仍在重复执行同类业务编排

### 5.2 数据前置
- 至少准备一组存在重复业务操作的 Booru 页面样本，例如 Favorites、Artist、Popular、Pools 或其他主线页面
- 至少准备一组固定对比样本，并在测试记录中写明样本 ID、查询词 / 筛选条件、初始状态与适用页面，用于跨页比较收藏、喜欢、下载、刷新、筛选、重试、失败回填等行为；若跨页对比使用了不同样本、不同查询词或不同初始状态，则相关用例只能记为待补测或补充观察，不得判通过
- 至少准备一个失败样本和一个需要重试或刷新恢复的样本

### 5.3 操作前置
- 必须具备截图或录屏能力；凡涉及迁移前后行为对比、重复请求抑制、错误重试、轮询清理的用例，不保留证据不得判通过
- 必须能观察页面层是否仍重复创建请求、订阅或轮询；若只能从 UI 外观判断，相关结构性用例只能记为待补测
- 若当前产品尚未明确哪些流程必须统一抽到 facade、哪些流程允许页面特化，则测试记录必须先写明 source of truth；未写明不得判通过

### 5.4 口径确认前置
- 在执行 TP-11 之前，测试记录必须写明以下 source of truth：
  - 本次试点页面清单
  - 本次要求归并的重复业务流程清单，例如收藏、喜欢、下载、刷新、轮询、防抖、状态回填等
  - facade / hook 对页面承诺的最小输入输出契约，例如 `viewState`、`commands`、错误与刷新语义
  - 哪些页面特有逻辑允许留在页面层，哪些必须下沉
  - 本次迁移前基线：至少包括迁移前涉及的页面清单、主要重复流程样本、关键交互闭环和已知差异说明；若无基线，不得把“迁移后更好”或“重复明显减少”写成通过
  - 跨页对比统一口径：至少写明本次固定对比样本、统一触发动作、统一观察窗口与统一判定字段；若无法统一，则相关跨页对比用例不得判通过
- 若上述任一项没有明确 source of truth，则对应用例只能记为“待补测前置”或“环境阻塞”，不得判通过，也不得据此给出 TP-11 通过结论

## 6. hooks / facade 下沉最小判定口径

| 验证对象 | 最低判定口径 |
|---|---|
| 必须下沉的共性流程 | 至少写明哪些重复业务编排必须收口到 facade / hook，例如收藏、喜欢、下载、刷新、轮询、防抖、状态回填；未写明不得以“明显减少”判通过 |
| 允许留在页面层的逻辑 | 必须列出页面特化逻辑清单及保留理由；不能只写“页面特殊” |
| view state + commands 契约 | 必须先定义页面最小消费结构、命令语义、错误 / 刷新 / 重试语义；未定义不得判通过 |
| 迁移前基线 | 必须记录迁移前的页面清单、关键业务闭环和重复流程样本；无基线不得主张“重复流程已基于 5.4 固定样本、统一触发动作、统一观察窗口与迁移前基线证明得到收敛” |
| 结构性证据 | 对“已下沉”“已统一”“已减少重复”这类结论，至少要有页面消费结构、请求 / 日志 / 状态对象或对比录屏中的一种硬证据；只看代码拆分表象不得判通过 |

## 7. 六层测试口径与验证重点

### 7.1 配置与数据层
重点看：
- facade / hook 是否消费了收窄后的 DTO、页面偏好与查询条件，而不是重新拼接整包 config 或脏数据对象
- 页面迁移后，搜索条件、分页、排序、收藏状态、下载状态等数据来源是否仍然清晰
- 若某个页面需要保留局部本地状态，测试记录必须说明该状态为什么不进入 facade 契约；没有依据不得判通过

### 7.2 主进程服务层
重点看：
- 页面抽层后是否仍通过既定主进程服务与 IPC 链路拿数据，而不是绕过既有 service 设计
- 同一业务动作在多个页面触发时，主进程侧是否不再出现多套不同调用序列
- handler 是否已从下载、收藏、喜欢、轮询、防抖、刷新恢复等重业务编排中退出，只保留边界校验、转换、错误映射等职责
- 若抽层后主进程日志显示不同页面仍在走不同签名或不同恢复路径，应记录为结构未收敛，而不是把差异当成合理特化
- 若页面变薄但 handler 同时变厚，测试记录必须判为“职责未真正下沉”，不得视为通过

### 7.3 IPC / preload / shared 边界层
重点看：
- hooks / facade 是否只消费受控 bridge、shared DTO 或正式定义的 commands，不再直接依赖裸字符串、过时签名或整包对象
- 页面迁移后是否减少了页面直接碰 IPC 细节的情况
- facade 层不能反向成为新的 God layer：若其对外暴露原始 IPC 细节、错误码杂糅、裸响应对象，则不得判通过
- 若 source of truth 未定义 facade 的最小输入输出契约，用例不得判通过

### 7.4 渲染层交互层
重点看：
- 页面是否只消费 `view state + commands`，页面代码更聚焦展示与交互
- 收藏、喜欢、下载、刷新、筛选、重试等操作在不同试点页面上的交互结果是否保持一致
- 同一类错误、加载、刷新状态在多个试点页面上是否由统一结构提供，而不是每页各自兜底
- 页面迁移后不应出现按钮还能点但状态回填延迟、重复 toast、重复请求或局部状态错乱

### 7.5 生命周期与异常层
重点看：
- 页面切换、重复进入、筛选条件变化、组件卸载重挂载后，facade / hook 的订阅、轮询、防抖和竞态清理是否仍符合预先定义的观察窗口与判定字段
- 同一页面重复进入时，不应累积多份监听、轮询或请求序列
- 页面隐藏或切换后，抽层逻辑应与 TP-09 的可见性规则一致，不得继续残留不可见副作用
- 失败后重试、刷新后恢复、重新搜索等异常恢复路径是否由统一结构承接

### 7.6 非功能层
重点看：
- 可维护性：试点页面重复业务编排是否已基于 5.4 固定样本、统一触发动作、统一观察窗口与迁移前基线证明得到收敛，而不是仅把体量平移
- 稳定性：相同操作在不同页面上不再表现出彼此矛盾的竞态结果、重复请求或状态回填差异；若仍有差异，测试记录必须写明 source of truth
- 可观测性：日志、调试输出、状态对象命名与结构必须足够清晰，便于复测和定位抽层问题
- 性能：抽层后不能因为重复计算、无意义 rerender 或重复订阅导致更差表现
- 可扩展性：新增同类页面时，测试记录必须能说明其是否能复用现有 facade / hook 契约；若无法说明，则说明结构仍未形成稳定复用边界

## 8. 测试设计文档与测试记录承接说明

本文档只负责定义测试设计、覆盖范围、步骤、预期结果与判定口径，不直接承接某一次实际执行结果。

实际执行时，每条用例的以下信息必须由测试记录承接，而不是由本设计文档承载：
- 实际结果
- 证据路径 / 附件名
- 结论（通过 / 失败 / 待补测 / 环境阻塞 / N/A）
- 本次试点页面清单与 source of truth 来源

如果测试记录缺少上述字段，或仅写“已抽成 hook”“页面更干净了”“逻辑都复用了”这类结构性空话，则该用例不得判通过。本设计文档本身也不能代替测试记录直接给出最终结论。

## 9. 详细测试用例

### TP-11-TC-001 试点页面清单与归并流程清单必须先被明确
- 测试目的：阻止“看起来抽了几个 hook 就算完成”的模糊验收
- 前置条件：准备本次试点页面与目标流程清单；具备 source of truth
- 场景/步骤：
  1. 记录本次纳入试点的页面
  2. 记录本次要求归并的重复流程，例如收藏、喜欢、下载、刷新、轮询、防抖、状态回填
  3. 对照实现与测试记录核对
- 预期结果：
  - 页面清单、流程清单、契约清单必须明确
  - 若未明确，不得宣称“页面层业务编排已下沉”
- 关键观察点：source of truth、试点页面列表、归并流程列表

### TP-11-TC-002 多页面相同业务动作应通过统一 commands 语义承接
- 测试目的：验证 facade / hook 不是“同名不同义”的表面统一
- 前置条件：至少两个试点页面存在相同业务动作，例如收藏或下载；已确认 commands 契约的 source of truth；测试记录必须为页面 A 与页面 B 绑定同一组固定对比样本、同一触发动作和同一观察窗口，若样本、动作或观察窗口不一致，则本用例不得判通过
- 场景/步骤：
  1. 在页面 A 对固定对比样本触发同一业务动作
  2. 在页面 B 对同一固定对比样本触发同一业务动作
  3. 以同一观察窗口对比 view state 变化、反馈时机、错误处理和日志序列
- 预期结果：
  - 相同业务动作应表现出统一语义
  - 不应页面 A 成功后立即回填、页面 B 仍需手工刷新才能回填
  - 若页面确有特化差异，测试记录必须写明 source of truth，未写明不得判通过
  - 若跨页使用了不同样本、不同动作或不同观察窗口，本用例结论无效，不得判通过
- 关键观察点：固定对比样本、统一触发动作、统一观察窗口、commands 命名、状态回填、错误反馈、日志序列、source of truth 记录

### TP-11-TC-003 页面应主要消费 view state，而非重新拼接跨层原始对象
- 测试目的：验证页面职责已收缩到展示与交互
- 前置条件：可查看页面调试输出、props / state 或 hook 返回结构；测试记录必须至少保留一种结构性证据，例如页面消费结构截图、hook 返回对象样本、调试输出或日志片段
- 场景/步骤：
  1. 打开试点页面
  2. 检查页面消费的数据结构
  3. 对比是否仍直接拼接 IPC 原始返回、整包 config、复杂领域对象
- 预期结果：
  - 页面主要消费 facade / hook 提供的 view state 与 commands
  - 不应在页面里继续大量拼接请求状态、错误映射、竞态标记和回填逻辑
  - 若只从文件拆分表象判断，未核对实际消费结构，则不得判通过
- 关键观察点：view state 结构、页面本地状态数量、原始对象暴露情况、结构性证据

### TP-11-TC-004 失败、重试、刷新语义应由统一结构承接
- 测试目的：验证抽层不只覆盖 happy path
- 前置条件：可触发失败和重试；已确认错误 / 刷新 / 重试契约的 source of truth；测试记录必须为页面 A 与页面 B 绑定同一类失败样本、同一重试或刷新动作以及同一观察窗口，若不一致则本用例不得判通过
- 场景/步骤：
  1. 在试点页面 A 对固定失败样本制造失败
  2. 触发已约定的重试或刷新动作
  3. 在试点页面 B 对同类固定失败样本重复相同流程
  4. 以同一观察窗口对比错误状态、重试入口、恢复结果
- 预期结果：
  - 失败、重试、刷新语义应在试点页面之间保持一致或符合明确定义的差异
  - 不应一个页面吞错、一个页面重复 toast、一个页面重试后不清 error
  - 若存在差异，测试记录必须写明 source of truth，未写明不得判通过
  - 若跨页使用了不同失败样本、不同恢复动作或不同观察窗口，本用例结论无效，不得判通过
- 关键观察点：固定失败样本、统一恢复动作、统一观察窗口、error 状态、retry / refresh 命令、恢复时机、source of truth 记录

### TP-11-TC-005 防抖、请求竞态与重复触发不应在各页重新实现
- 测试目的：验证共性编排是否真正归并
- 前置条件：可快速连续触发搜索、筛选、刷新或收藏动作；可观察网络请求或日志；测试记录必须写明固定对比样本、统一连续触发动作、统一观察窗口与竞态裁决规则来源，若任一项不一致则本用例不得判通过
- 场景/步骤：
  1. 在试点页面 A 对固定对比样本执行统一的连续触发动作，例如连续输入同一组查询词或连续切换同一组筛选值
  2. 观察请求数量、结果回填顺序和最终状态
  3. 在试点页面 B 对同一固定对比样本重复相同行为
  4. 以同一观察窗口对比竞态结果
- 预期结果：
  - 不应出现每页各自不同的竞态结果，例如旧请求覆盖新请求、重复触发多份无效请求
  - 共性防抖或竞态控制应在试点页面之间表现一致
  - 若当前产品未定义最终竞态裁决规则，测试记录必须先写明 source of truth，未写明不得判通过
  - 若跨页使用了不同样本、不同连续触发动作或不同观察窗口，本用例结论无效，不得判通过
- 关键观察点：固定对比样本、统一连续触发动作、统一观察窗口、请求数量、响应回填顺序、最终筛选结果、source of truth 记录

### TP-11-TC-006 轮询、订阅或自动刷新逻辑必须在页面卸载或隐藏后正确清理
- 测试目的：验证 hooks / facade 不会制造新的隐藏副作用
- 前置条件：至少一个试点页面存在轮询、订阅或自动刷新；可观察日志、定时器或请求
- 场景/步骤：
  1. 进入试点页面并触发自动刷新或订阅
  2. 切页、卸载组件或隐藏页面
  3. 继续观察请求、日志或订阅活动
  4. 再次返回页面
- 预期结果：
  - 页面不可见或已卸载后，共性副作用应按规则停止或受控上提
  - 再次进入页面时不应叠加多份订阅或轮询
  - 若某页面明确允许后台轮询，测试记录必须注明 source of truth，未注明不得判通过
- 关键观察点：轮询次数、订阅数量、日志活动、返回后的副作用数量、source of truth 记录

### TP-11-TC-007 抽层后不应重新突破 IPC / preload / shared 边界，且不得把重编排上浮到 handler
- 测试目的：验证 facade 没有变成新的边界绕过层，也没有通过“页面变薄、handler 变厚”制造假收敛
- 前置条件：可查看 facade / hook 依赖对象、bridge 暴露、shared 类型和对应 handler；测试记录必须至少提供一个允许输入 / 输出样本、一个不应透传样本，以及一个 handler 职责样本，否则不得判通过
- 场景/步骤：
  1. 检查 facade / hook 输入输出
  2. 检查其是否直接依赖裸 IPC 字符串、过时签名、整包 config 或敏感字段
  3. 抽取与该主路径对应的一条 handler，核对其是否仍承担下载、收藏、喜欢、轮询、防抖、刷新恢复等重业务编排
  4. 在试点页面实际触发一次主路径操作
- 预期结果：
  - facade / hook 只依赖正式 bridge 与受控 DTO
  - 不应把原始 IPC 响应直接无加工透传给页面
  - handler 不应借抽层之名继续承接本应下沉到 facade / hook 的重业务编排
  - 若当前 facade 契约或 handler 边界未定义清楚，则用例不得判通过
- 关键观察点：bridge 调用入口、shared DTO、原始对象透传情况、handler 职责样本、允许样本 / 禁止样本、source of truth 记录

### TP-11-TC-008 页面特化逻辑必须有明确保留理由，不得拿“特殊情况”逃避归并
- 测试目的：验证“允许特化”不是无限口袋
- 前置条件：至少存在一个保留页面特化逻辑的试点页面；已确认保留依据的 source of truth
- 场景/步骤：
  1. 找出页面中仍保留的业务编排逻辑
  2. 对照 source of truth 检查其是否属于明确允许的特化
  3. 验证该特化不会破坏统一 commands / view state 契约
- 预期结果：
  - 保留在页面层的逻辑必须有明确理由
  - 特化不得破坏统一抽象层输出语义
  - 若没有明确依据，只能视为结构未收敛，不得判通过
- 关键观察点：特化逻辑清单、保留依据、与统一契约的兼容性

### TP-11-TC-009 迁移后的试点页面应保持原有关键业务闭环不回退
- 测试目的：验证结构优化没有换来功能回退
- 前置条件：至少两个试点页面可执行收藏、下载、刷新、重试或回填主流程；具备迁移前后的对比口径；测试记录必须先挂接 5.4 中定义的迁移前基线，并为迁移前后绑定同一组固定对比样本、统一触发动作和统一观察窗口，否则不得判通过
- 场景/步骤：
  1. 在试点页面对固定对比样本执行关键业务闭环
  2. 以同一观察窗口对比迁移前基线或既有验收要求
  3. 检查是否出现功能存在但状态回填、按钮反馈、失败恢复退化
- 预期结果：
  - 关键闭环不应因抽层而回退
  - 若存在差异，测试记录必须区分“设计允许变化”与“回归缺陷”，并写明 source of truth，未写明不得判通过
  - 若迁移前后使用了不同样本、不同动作或不同观察窗口，则本用例结论无效，不得判通过
- 关键观察点：固定对比样本、统一触发动作、统一观察窗口、按钮反馈、状态回填、错误恢复、迁移前基线、与既有 TP-04 / TP-09 要求的一致性

### TP-11-TC-010 新增同类页面应能复用现有 facade / hook 契约
- 测试目的：验证抽层结果具备扩展价值，而不是一次性搬迁
- 前置条件：至少有一个未纳入试点但业务相近的页面可作为对照；已确认本次复用目标的 source of truth；测试记录必须至少保留一份复用差异清单或契约适配清单，否则不得判通过
- 场景/步骤：
  1. 选择一个业务相近页面作为复用对照
  2. 对比其与试点 facade / hook 的契约适配情况
  3. 记录哪些能力可直接复用、哪些是明确允许的差异
- 预期结果：
  - 应能说明现有抽象对同类页面的复用边界
  - 若完全无法说明，只能说明抽象仍未形成稳定复用边界，不能判 TP-11 通过
  - 本用例不要求立即完成新页面接入，但必须能形成可复用判断
- 关键观察点：契约可复用性、差异点清单、复用适配清单、source of truth 记录

## 10. 负向与边界场景

### 10.1 负向场景
- facade / hook 内部失败后，页面不能同时收到多份冲突状态，例如既成功又失败
- 页面快速切换导致组件卸载时，不应发生 setState after unmount、重复订阅或重复 toast
- 页面特化逻辑不能绕回直接碰 IPC 或整包 config
- 相同命令在不同页面上不能出现“名字一样、行为不同且无说明”的情况

### 10.2 边界场景
- 首次进入页面与多次重复进入页面，view state 初始值与恢复逻辑必须一致或符合定义
- 搜索条件快速变化、分页切换、收藏状态连续点击时，最终结果必须符合 source of truth 规定的竞态裁决规则与统一判定字段
- 页面隐藏但 keep-alive 存在时，副作用是否暂停或上提必须可解释
- 同一 facade 被多个页面实例同时消费时，不得相互污染局部状态，除非 source of truth 明确要求共享

### 10.3 待补测前置
- 若当前未明确试点页面清单、目标流程清单和契约清单，则 TP-11 相关用例不得判通过
- 若无法观察请求、日志、订阅或状态对象，仅能看到表层 UI，则结构性验证只能记为待补测或环境阻塞

## 11. 回归要求

### 11.1 必回归对象
- 所有已纳入试点的 Booru 主线页面
- 与试点页面共享命令语义的下载、收藏、喜欢、刷新入口
- 页面切换、重复进入、隐藏 / 显示链路
- 相关 facade / hook 的服务层或状态层测试对象

### 11.2 需要联动回归
- TP-05 的协议来源收口
- TP-06 的 preload / shared / handler 边界收缩
- TP-09 的空错态、隐藏页副作用与交互一致性
- TP-04 的页面功能闭环与按钮真实能力
- TP-13 的页面偏好与查询条件持久化边界

### 11.3 回归通过标准
- 试点页面重复业务编排已基于 5.4 固定样本、统一触发动作、统一观察窗口和迁移前基线证明得到收敛
- 页面主要消费 view state 与 commands
- 共性错误、刷新、重试、竞态和副作用治理在试点页面间表现一致
- facade / hook 未重新突破进程边界和 DTO 边界
- 关键业务闭环不因抽层而回退

## 12. 自动化建议

### 12.1 适合自动化的部分
- facade / hook 输入输出契约测试
- commands 语义与状态转移断言
- 错误、刷新、重试状态机测试
- 请求竞态、防抖和重复触发测试
- 轮询 / 订阅清理测试
- 页面消费 DTO 而非原始对象的结构断言

### 12.2 建议自动化方式
- 为 facade / hook 编写单元测试和服务层测试，断言 `viewState` 与 `commands` 语义
- 使用组件测试验证试点页面只消费 facade / hook 返回值，不再手写同类编排
- 使用假的网络层或 bridge 层模拟失败、重试、竞态与重复进入场景
- 为隐藏 / 卸载后的清理行为编写计时器与订阅断言

### 12.3 仍需人工确认的部分
- 多页面在真实交互下的状态回填时机、展示字段与反馈顺序是否与统一判定字段一致
- 抽层后页面反馈、刷新、重试与按钮触发后的可见反馈是否与 source of truth 规定的反馈时机和结果一致
- 隐藏 / 切页后的副作用是否真的停下，且返回页面后的恢复路径与观察窗口一致
- 试点页面之间的交互一致性是否已在固定样本、统一触发动作、统一观察窗口和统一判定字段下得到验证

### 12.4 自动化边界
- 自动化可以证明契约和状态机，但不能替代真实页面间行为一致性的人工观察
- 页面抽层若影响可见交互和反馈，必须保留人工回归证据，不能只看 hook 单测通过

## 13. 缺陷分级与结论判定

### 13.1 缺陷分级
- S0：抽层后关键页面主流程失效、跨页面状态严重错乱、边界突破导致敏感或原始对象越界暴露
- S1：重复编排未真正收敛、统一命令语义失真、错误 / 重试 / 竞态在不同页面表现明显不一致
- S2：个别试点页面仍保留较多特化逻辑、某些副作用清理不完整，但主流程可继续
- S3：命名、日志、轻微结构或提示细节问题，不影响整体抽层目标

### 13.2 结论判定
满足以下条件可判定本 TP 通过：
1. 本次试点页面清单、归并流程清单和 facade / hook 契约都有明确 source of truth
2. 多页面重复业务编排已基于 5.4 固定样本、统一触发动作、统一观察窗口和迁移前基线证明得到收敛，页面主要消费 `view state + commands`
3. 错误、刷新、重试、竞态和副作用治理在试点页面之间保持一致或有明确差异依据
4. facade / hook 没有重新突破 TP-05 / TP-06 已收紧的边界
5. 抽层后关键业务闭环未回退，并已留存可复核证据

若仍存在“只是拆文件未收敛语义”、跨页面命令语义不一致、facade 重新暴露原始边界对象、或没有 source of truth 支撑试点范围与契约，则本 TP 不通过。

## 15. 阶段验收口径

TP-11 作为阶段 3 的结构优化任务，验收时应满足：

1. 页面层重复业务编排已在试点范围内得到真实归并，而非仅代码搬家
2. 页面主要消费 `view state + commands`，且页面职责收缩已基于 5.4 固定样本、统一触发动作、统一观察窗口和迁移前基线得到证明
3. hooks / facade 契约已形成可复用且可验证的统一语义，错误、刷新、重试、竞态和副作用治理在试点页面间保持一致或具备明确差异依据
4. hooks / facade 未重新打破 IPC / preload / shared / DTO 边界
5. 所有关键结论都由测试记录写明 source of truth，未写明者不得作为通过依据

本设计文档不承载实际结果、证据路径和最终结论；这些信息必须由测试记录提供。若只是少量页面试探性提取了 hook，但试点范围、统一语义、边界契约和回归证据都不完整，则只能判定为待补测或未通过，不能进入阶段通过结论。
