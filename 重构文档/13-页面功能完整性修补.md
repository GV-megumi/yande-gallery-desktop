# 13-页面功能完整性修补

## 1. 本文档范围

本文件负责处理“页面已经暴露给用户，但能力并未真正闭环或兑现”的问题。

主责任内容包括：
- 伪设置
- 假按钮 / TODO 入口
- 页面闭环缺失
- 错误后无恢复路径
- 配置保存与运行行为不一致
- 页面状态结果与用户输入脱节

不展开的内容：
- 导航模型、弹窗规则、空错态统一规范（见 [14](./14-UI交互一致性修正.md)）
- 下载状态机底层实现（见 [10](./10-下载并发与调度链路重构.md)）
- IPC / preload / handler 架构边界（见 [12](./12-IPC与进程边界重构.md)）

## 2. 本专项核心问题概览

本专项的关键词不是“界面不好看”，而是“页面承诺没有兑现”。

这里需要明确区分两层内容：
- **需求层**：当前产品已经承诺了哪些页面能力与行为。
- **整改原则层**：为了提升可信度和可解释性，本轮文档额外采用哪些质量约束来治理这些页面。

当前多个页面都存在：
- 用户看得见的设置不会真正生效
- 用户可以点击的按钮不会真正执行
- 页面失败后没有恢复入口
- 页面当前看到的条件与实际结果并不一致
- 页面提示“保存成功”后，运行行为却没有同步

这些问题一旦积累，会直接破坏用户对产品的信任；但“上屏即承诺”“可见能力要么生效要么撤下”等表述，应理解为本轮整改原则，而不是反向冒充需求来源。

## 3. 详细问题拆分

### 3.1 子问题簇 A：伪设置治理

#### 主要问题
- P1-17 `pageMode` 不是完全未接线，而是已进入配置读写和页面本地状态，但未见明确驱动分页 / 无限滚动行为
- P1-18 `autoGenThumbnail` 开关未保存
- P1-22 `dataPath` 问题当前更偏配置路径系统；现有 `SettingsPage` 未直接暴露该项，但后续若恢复入口仍需解决保存后运行时路径基准不刷新
- P1-14 通知开关上屏但能力未兑现（与桌面能力交叉）

#### 需要解决什么
- 按本轮整改原则，页面上暴露的设置应真正生效，或临时撤下
- 配置保存后的提示必须与实际运行行为一致
- 页面不能继续保留“点了也没变化”的伪设置

#### 当前主要落点文件
- `src/renderer/pages/SettingsPage.tsx`
- `src/renderer/pages/BooruSettingsPage.tsx`
- `src/renderer/pages/BooruPage.tsx`
- `src/renderer/pages/BooruFavoritesPage.tsx`
- `src/renderer/pages/BooruArtistPage.tsx`
- `src/renderer/pages/BooruTagSearchPage.tsx`
- `src/main/services/config.ts`
- `src/shared/types.ts`

#### 代码复核补充
- `BooruSettingsPage` 确实会写入 `pageMode`，`BooruPage` / `BooruFavoritesPage` / `BooruArtistPage` / `BooruTagSearchPage` 也会把它读入 `appearanceConfig`；本轮把这些文件列为主要落点，主因是它们都属于同一 appearance 配置的直接消费点，但当前代码里尚未形成明确的渲染分支或分页策略切换抓手
- `SettingsPage` 的 `loadConfig` / `handleSave` 当前只处理 `downloads.path` 与 `thumbnails.maxWidth/maxHeight/quality`，未见 `autoGenThumbnail` 对应读写链路
- `dataPath` 当前主要落点仍在 `src/main/services/config.ts` 的路径系统，而不是 `SettingsPage` 直接输入项

#### 推荐实施顺序
1. 先识别所有伪设置
2. 短期无法实现的，先移除或禁用
3. 能快速闭环的，补配置读写和真实消费链路
4. `dataPath` 这类运行时强耦合项，必须把 UI 提示与实际行为对齐

#### 验收要点
- 页面设置改动后，行为真实变化或明确提示需重启
- 伪设置不再继续暴露给用户

### 3.2 子问题簇 B：假按钮 / 假模块治理

#### 主要问题
- P1-19 单条下载取消按钮只有 TODO
- 设置页“功能开发中”按钮已经上屏
- 帖子详情页 `RelatedPostsSection` 已挂载但内部仍是 TODO

#### 需要解决什么
- 页面上所有可点击入口都必须形成真实闭环，或在短期内被隐藏 / 禁用 / 明确说明

#### 当前主要落点文件
- `src/renderer/pages/BooruDownloadPage.tsx`
- `src/renderer/pages/SettingsPage.tsx`
- `src/renderer/pages/BooruPostDetailsPage.tsx`
- `src/renderer/components/BooruPostDetails/RelatedPostsSection.tsx`
- `src/renderer/components/BooruPostDetails/Toolbar.tsx`

#### 代码复核补充
- `BooruDownloadPage` 中单条取消下载的 `Popconfirm` 仍直接写着 `/* TODO: 实现取消 */`
- `SettingsPage` 中至少还有 `reindexDb` 和更新设置入口只弹 `功能开发中`
- `RelatedPostsSection` 已被挂载，但 `useEffect` 中仍只打印日志并返回空列表
- 帖子详情工具栏里的幻灯片入口当前也只是 `message.info(t('details.slideshowDev'))`

#### 推荐实施顺序
1. 先清最误导的假按钮
2. 再清挂载但不工作的模块
3. 最后补真实能力或重新设计入口

#### 验收要点
- 用户能点到的主入口不再是空实现
- 页面上不再挂着明知不会工作的模块

### 3.3 子问题簇 C：页面闭环缺失

#### 主要问题
- P1-10 Google webview 页面没有失败态/重试/退路
- P2-03 FavoriteTags 下载路径选择闭环缺失，但真实缺口应收窄为“未绑定图库时没有独立路径选择器，下载路径只能随 gallery 回填”
- SavedSearch 当前更明确的问题不是“站点编辑不生效”，而是编辑态调用 `updateSavedSearch` 时参数签名与 preload / IPC 定义不一致
- Gallery recent 搜索入口与结果源断裂
- 无效图片页失败态伪装成空态

#### 需要解决什么
- 按本轮整改原则，页面在成功路径之外还应提供可解释的失败路径和恢复路径
- 配置与运行结果之间不能出现“输入了但系统没兑现”的断裂

#### 当前主要落点文件
- `src/renderer/pages/FavoriteTagsPage.tsx`
- `src/renderer/pages/BooruSavedSearchesPage.tsx`
- `src/renderer/pages/GalleryPage.tsx`
- `src/renderer/pages/InvalidImagesPage.tsx`
- `src/renderer/pages/GoogleDrivePage.tsx`
- `src/renderer/pages/GooglePhotosPage.tsx`
- `src/renderer/pages/GeminiPage.tsx`
- `src/preload/index.ts`
- `src/main/ipc/handlers.ts`
- `src/main/services/booruService.ts`

#### 代码复核补充
- Google Drive / Photos / Gemini 页面都监听了 `did-fail-load`，但只会把 loading 关掉，没有把失败落成错误状态、更没有重试和退路
- `FavoriteTagsPage` 已具备 `upsertFavoriteTagDownloadBinding`、启动下载、历史查看、失败重试、图集绑定修复等链路；当前最明显的闭环缺口是 `downloadPath` 输入框只读且依赖 `galleryId` 回填
- `BooruSavedSearchesPage` 新建态可以设置站点，但编辑态不再展示站点选择；同时页面调用 `window.electronAPI.booru.updateSavedSearch(editingSearch.id, formName.trim(), formQuery.trim())`，而 preload / IPC 实际定义是 `(id, updates)`
- `InvalidImagesPage` 请求失败时只 toast，不记录错误状态；随后 `!loading && images.length === 0` 会直接渲染“没有无效图片”

#### 推荐实施顺序
1. 先修最明显的闭环缺口：Google、InvalidImages、FavoriteTags
2. 再修 SavedSearch、Gallery 等容易误导用户的页面行为
3. 最后补帖子详情、次级模块的闭环与边界反馈

#### 验收要点
- 页面失败时不再只剩日志或空白
- 页面结果与用户输入条件一致
- 路径、站点、搜索等配置项真正参与行为

### 3.4 子问题簇 D：结果表达与用户输入脱节

#### 主要问题
- Booru 主线当前搜索词与“已选标签”不一致
- 高级过滤器变更后结果不自动刷新
- 黑名单过滤后可能出现看似渲染坏掉的空白区
- 下载任务“创建成功”提示过早，后续仍可能失败

#### 需要解决什么
- 页面应该让用户看得懂“现在系统究竟按什么条件运行、当前结果代表什么”

#### 当前主要落点文件
- `src/renderer/pages/BooruPage.tsx`
- `src/renderer/components/BooruPageToolbar.tsx`
- `src/renderer/pages/BooruBulkDownloadPage.tsx`
- `src/renderer/components/BulkDownloadSessionCard.tsx`

#### 推荐实施顺序
1. 先让查询条件与结果绑定一致
2. 再减少过早成功提示和误导性状态表达
3. 最后补更细的阶段化进度与反馈

#### 验收要点
- 当前展示条件与实际查询条件一致
- 用户不会再看到前后矛盾的成功/失败提示
- 干运行、扫描、开始下载等阶段有更可解释的反馈

### 3.5 子问题簇 E：高频管理页的持久化与操作闭环

#### 主要问题
- `FavoriteTagsPage`、`GalleryPage`、`BlacklistedTagsPage` 的排序 / 筛选 / 页码 / 每页数量状态都只存在于页面本地 `useState`，重新进入页面后会回到默认值
- 图库删除能力后端已存在，但 `GalleryPage` 右键菜单当前只有“编辑”，用户无法在 UI 中完成删除图集闭环
- FavoriteTags 下载配置的默认策略仍偏保守：`autoSyncGalleryAfterDownload` 默认 `false`、`concurrency` 默认 `3`，与“下载后立即同步图集、提高批量吞吐”的常见路径不一致

#### 需要解决什么
- 高频管理页应能记住用户上一次工作上下文，而不是每次都从默认排序重新开始
- 已存在后端能力的管理动作，应在 UI 上形成明确、可发现、可确认的闭环
- 下载配置默认值应对齐当前已确认的主路径预期，而不是长期停留在历史保守值
- 图库页应明确区分“刷新”与“同步文件夹”，同步按钮命名和图标语义需稳定保留
- 图集编辑能力首期至少承接“改名”这一已确认需求

#### 当前主要落点文件
- `src/renderer/pages/FavoriteTagsPage.tsx`
- `src/renderer/pages/GalleryPage.tsx`
- `src/renderer/pages/BlacklistedTagsPage.tsx`
- `src/preload/index.ts`
- `src/main/ipc/handlers.ts`
- `src/main/services/galleryService.ts`

#### 代码复核补充
- `FavoriteTagsPage` 的 `filterSiteId`、`sortKey`、`sortOrder`、`keyword`、`page`、`pageSize` 都是纯前端局部 state；`GalleryPage` 的 `gallerySearchQuery`、`gallerySortKey`、`gallerySortOrder` 也仅保存在页面 state；`BlacklistedTagsPage` 同样只在本地保存 `filterSiteId / keyword / page / pageSize`
- `GalleryPage` 右键菜单当前只提供 `edit`，但 preload 已暴露 `gallery.deleteGallery`，主进程 `gallery:delete-gallery` handler 与 `galleryService.deleteGallery()` 也都已存在
- FavoriteTags 下载配置弹窗在打开时仍把 `autoSyncGalleryAfterDownload` 默认回落到 `false`、`concurrency` 默认回落到 `3`；表单 `initialValues` 也仍是 `concurrency: 3`

#### 推荐实施顺序
1. 先明确哪些页面状态属于“跨会话产品级偏好”，不要继续停留在纯本地 state
2. 再补图集删除 UI 闭环，并把“仅删除图集和关联记录、不删除文件”写进确认文案
3. 最后统一 FavoriteTags 默认策略与页面回填规则

#### 验收要点
- 重新进入 FavoriteTags / Gallery / BlacklistedTags 后，排序、筛选、分页状态能按规则恢复
- 图集删除在 UI 中可见、可确认、可解释，且不会误导用户为“删除文件”
- FavoriteTags 下载配置默认值与当前已确认产品策略一致，或至少给出明确可见的默认值来源说明
- 图库页“刷新 / 同步文件夹”语义清晰，图集编辑至少支持改名
- 下载相关表单 / 列表每页默认值与已确认需求一致

## 5. 推荐实施顺序

### 第一阶段：清理伪能力
- 假按钮
- 伪设置
- 已挂载但不工作的模块

### 第二阶段：补页面闭环
- 失败态 / 重试 / 外部打开
- 配置保存与运行行为对齐
- 路径、站点、搜索条件等真正闭环

### 第三阶段：修结果与输入一致性
- 查询条件可见性
- 结果刷新触发
- 阶段反馈表达

## 6. 前置依赖与并行关系

### 前置依赖
- 若某页面问题依赖新 IPC 能力或协议修正，应同步看 [12](./12-IPC与进程边界重构.md)
- 若某页面问题涉及下载状态语义，应同步看 [10](./10-下载并发与调度链路重构.md)

### 可并行关系
- Settings、FavoriteTags、Google、InvalidImages、SavedSearch 可以分组并行
- 但如果多个页面共享同一配置 / 同一 IPC 接口，不应各自独立重写边界逻辑

## 7. 风险点

1. 容易只补 UI 提示，不补真实行为链路
2. 容易把暂时做不到的能力继续保留在页面上，导致后续再度失真
3. 多页面共享同一主进程接口时，局部修补可能引入新的不一致

## 8. 验收要点

至少应确认：
- 页面暴露给用户的设置和按钮不再是伪能力
- 关键页面在失败时存在明确恢复入口
- 页面结果和用户输入条件不再脱节
- 保存成功提示与真实运行行为一致
