# 11-Electron桌面能力重构

## 1. 本文档范围

本文件负责把 `审查报告.md` 中与 Electron 桌面应用形态相关的问题，重组为一条可执行的桌面能力补齐路线。

主责任内容包括：
- 后台常驻
- 托盘
- 关闭行为
- 长任务运行与窗口生命周期关系
- 系统通知
- 启动恢复
- 单实例能力完善
- 开机自启
- 崩溃治理的桌面承载面

不展开的内容：
- 下载状态机和调度细节（见 [10](./10-下载并发与调度链路重构.md)）
- IPC / preload / shared 边界（见 [12](./12-IPC与进程边界重构.md)）
- 安全协议、敏感配置、异常治理细节（见 [15](./15-稳定性错误恢复与安全性修正.md)）
- 成熟库对日志和监控的引入策略（见 [16](./16-成熟库替代与现代化升级建议.md)）

## 2. 本专项核心问题概览

报告在第 7 章中的判断非常明确：当前应用更像“功能很多的 Electron 窗口应用”，而不是“成熟的后台图库/下载管理工具”。

这条结论不是视觉问题，而是运行形态问题。当前缺口主要分三组。

### 2.1 运行形态不桌面化

- 非 macOS 下关闭所有窗口后直接退出
- 没有托盘
- 没有 close / hide / quit 区分
- 没有“关闭窗口但任务继续”这一运行模型

### 2.2 长任务承载能力不成立

- 后台常驻缺失
- 通知开关已出现但系统通知未落地
- 启动恢复有基础但语义不严谨
- 单实例只做了基础防重复启动，没有窗口恢复与唤醒策略

### 2.3 应用级治理能力不足

- 缺少开机自启
- 缺少崩溃治理
- 退出链路没有形成统一策略

## 3. 详细问题拆分

### 3.1 子问题簇 A：建立 close / hide / quit 的单一桌面模型

#### 核心问题
- P1-13 无托盘、无关闭不退出、后台常驻链路不成立
- 第 7.1、7.2、7.3 明确指出当前关闭窗口就会退出应用，且没有托盘与隐藏逻辑

#### 需要解决什么
- 把窗口关闭与应用退出拆开
- 为“用户只是关掉窗口”和“用户明确退出应用”定义不同行为
- 给后台下载、恢复、通知提供稳定承载点

#### 当前主要落点文件
- `src/main/index.ts`
- `src/main/window.ts`

#### 推荐实施顺序
1. 先定义 close / hide / quit 行为模型
2. 再引入 tray
3. 再接入关闭确认、最小化到托盘等交互策略

#### 验收要点
- 关闭主窗口不再默认等价于杀死应用
- 应用存在明确的显式退出入口
- 托盘可作为恢复应用与查看状态的入口

---

### 3.2 子问题簇 B：让长任务真正可以被桌面应用承载

#### 核心问题
- 第 7.4 指出主进程层面部分支持长任务，但被窗口退出策略破坏
- P1-14 指出通知开关有模型无兑现
- 第 7.6 指出启动恢复存在基础能力但不严谨

#### 需要解决什么
- 让长任务的生命周期不被窗口生命周期直接打断
- 让任务完成、失败、需人工处理等状态具备最小反馈闭环
- 让应用启动时的恢复模型与用户主动暂停/关闭行为不冲突

#### 当前主要落点文件
- `src/main/index.ts`
- `src/main/window.ts`
- `src/main/services/init.ts`
- `src/renderer/components/BulkDownloadTaskForm.tsx`
- `src/shared/types.ts`

#### 推荐实施顺序
1. 先以主进程生命周期治理为基础，明确长任务与窗口关闭解耦
2. 再补最小通知能力
3. 再细化任务完成 / 失败 / 人工处理的通知策略

#### 验收要点
- 关闭窗口后长任务不会被不必要中断
- 通知设置要么真实生效，要么从 UI 暂时撤下
- 启动恢复不再与 paused 语义冲突

---

### 3.3 子问题簇 C：补齐单实例、启动恢复与开机自启

#### 核心问题
- 第 7.7 指出单实例只有基础能力
- 第 7.8 指出开机自启缺失
- 第 7.6 指出启动恢复模型不严谨

#### 需要解决什么
- 单实例不只防止重复启动，还要会恢复已存在窗口或重建主窗口
- 若应用长期承担后台任务，就需要开机自启的设计预留
- 启动恢复要与任务状态、托盘状态和窗口状态对齐

#### 当前主要落点文件
- `src/main/index.ts`
- `src/main/window.ts`
- `src/main/services/init.ts`

#### 推荐实施顺序
1. 先补单实例恢复能力
2. 再整理启动恢复
3. 最后再引入开机自启

#### 验收要点
- 重复启动时能唤醒已有窗口或恢复主窗口
- 启动恢复不再制造隐藏状态错乱
- 开机自启作为长期能力，有明确前置条件和非目标说明

---

### 3.4 子问题簇 D：让桌面应用具备最小崩溃治理承载面

#### 核心问题
- 第 7.9 和 P1-20 都指出主进程缺少 quit / crash / unhandled exception 统一兜底

#### 需要解决什么
- 给桌面运行形态建立统一退出与崩溃承载面
- 让数据库关闭、下载冻结、恢复定时器清理进入同一治理链路
- 为后续日志与崩溃监控接入创造条件

#### 当前主要落点文件
- `src/main/index.ts`
- `src/main/services/database.ts`
- `src/main/services/init.ts`

#### 推荐实施顺序
1. 明确 before-quit / will-quit / process-gone / unhandled exception 的处理策略
2. 再决定哪些信息写本地日志、哪些进入监控

#### 验收要点
- 异常退出不再处于完全裸奔状态
- 应用退出时关键资源有统一收口

## 4. 影响模块 / 页面 / 链路

### 主进程层
- `src/main/index.ts`
- `src/main/window.ts`
- `src/main/services/init.ts`
- `src/main/services/database.ts`

### 渲染层相关入口
- `src/renderer/pages/SettingsPage.tsx`
- `src/renderer/components/BulkDownloadTaskForm.tsx`

### 关键链路
- 启动 -> 创建主窗口 -> 初始化 -> 恢复任务
- 关闭窗口 -> 应用隐藏 / 托盘驻留 / 显式退出
- 长任务进行中 -> 任务完成 / 失败 -> 系统通知
- 重复启动 -> 唤醒已有窗口 / 恢复主窗口

## 5. 推荐实施顺序

### 第一阶段：定义应用级生命周期模型
- close / hide / quit 区分
- 退出治理与恢复定时器收口
- 单实例恢复能力补全

### 第二阶段：补后台常驻基础设施
- tray
- 关闭到托盘
- 主窗口恢复/重建

### 第三阶段：补最小桌面反馈闭环
- 系统通知基础能力
- 下载完成 / 失败 / 需人工处理通知

### 第四阶段：长期桌面增强
- 开机自启
- 崩溃恢复策略
- 更细的后台运行策略

## 6. 前置依赖与并行关系

### 前置依赖
- 若要做后台常驻、托盘、通知，必须先让下载状态语义足够可信（见 [10](./10-下载并发与调度链路重构.md)）
- 若要做退出与崩溃治理，需要同步看安全与稳定性边界（见 [15](./15-稳定性错误恢复与安全性修正.md)）

### 可并行关系
- tray 基础设施 与 通知基础链路可以并行设计
- 单实例恢复能力 与 启动恢复整理可以并行推进
- 但不要在退出治理未明确前，先做复杂后台驻留细节

## 7. 风险点

1. 若下载状态机仍不稳，后台能力会放大状态错误
2. 若退出治理未成型，关闭到托盘可能让资源泄漏与恢复竞争更难排查
3. 若通知先于能力落地暴露给用户，会继续制造伪设置问题
4. 若单实例与主窗口恢复设计不统一，会产生“应用没退出但也找不到窗口”的体验问题

## 8. 验收要点

至少应确认：
- 关闭窗口不再默认结束任务
- 应用存在可理解的后台驻留入口和显式退出入口
- 通知策略要么真实成立，要么不暴露给用户
- 重复启动、恢复主窗口、退出资源清理具备一致策略
