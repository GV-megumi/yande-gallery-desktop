# 15-稳定性错误恢复与安全性修正

## 1. 本文档范围

本文件负责处理那些跨越运行入口、协议边界、配置边界、窗口/webview 边界和异常退出治理的问题。

主责任内容包括：
- `app://` 协议边界
- `openExternal` 协议 / 目标校验
- Electron 安全面设置与 webview 附着、导航、新窗口约束
- 敏感配置、站点凭证与 renderer-safe DTO
- 启动入口、异常退出、崩溃治理与统一兜底
- 与文件完整性相关的错误恢复边界

不展开的内容：
- 下载状态机和会话调度（见 [10](./10-下载并发与调度链路重构.md)）
- 托盘、通知、后台常驻等桌面形态（见 [11](./11-Electron桌面能力重构.md)）
- IPC 单一来源与 shared / preload 重构（见 [12](./12-IPC与进程边界重构.md)）
- 成熟库选型与长期现代化升级（见 [16](./16-成熟库替代与现代化升级建议.md)）

## 2. 本专项核心问题概览

报告在第 4、7、9、10.5 章里已经确认，这一组问题不是零散的安全点，而是系统边界整体过松：

- 本地资源协议边界过宽
- renderer 权限与敏感数据暴露过宽
- Electron 窗口 / webview 安全面设置过宽
- 运行入口和异常退出治理不完整
- 文件完整性和恢复策略在异常边界下不可靠

## 3. 详细问题拆分

### 3.1 子问题簇 A：收紧本地资源与外部链接边界

#### 主要问题
- P0-03 `app://` 可映射任意本地路径
- P1-03 `system:open-external` 无协议 / 目标校验

#### 需要解决什么
- `app://` 只允许访问受控目录
- `openExternal` 只允许安全协议和可接受目标
- 渲染层与远端数据不能随意驱动本机打开任意协议或读取任意路径

#### 当前主要落点文件
- `src/main/index.ts`
- `src/renderer/utils/url.ts`
- `src/main/ipc/handlers.ts`
- `src/renderer/components/BooruImageCard.tsx`
- `src/renderer/pages/BooruArtistPage.tsx`

#### 推荐实施顺序
1. 先限制 `app://` 的 root containment
2. 再限制 `openExternal` 协议与目标
3. 再梳理 renderer 中所有外部链接消费点

#### 验收要点
- `app://` 不再映射任意本地路径
- `openExternal` 只打开允许的协议/目标

---

### 3.2 子问题簇 B：收紧窗口 / webview / navigation 安全面

#### 主要问题
- P1-04 Electron 安全面过宽
- 第 9.2.1 指出当前项目与 Electron 官方安全清单存在明显差距

#### 需要解决什么
- 对 webview、新窗口、导航、附着行为建立主进程白名单治理
- 审视 `webSecurity: false`、`webviewTag: true`、`allowpopups="true"` 的必要性
- 避免继续让外部内容在过宽边界下运行

#### 当前主要落点文件
- `src/main/window.ts`
- `src/main/index.ts`
- `src/renderer/pages/GoogleDrivePage.tsx`
- `src/renderer/pages/GooglePhotosPage.tsx`
- `src/renderer/pages/GeminiPage.tsx`

#### 推荐实施顺序
1. 先补主进程 `setWindowOpenHandler` / `will-navigate` / `will-attach-webview` 约束
2. 再梳理页面对 webview 的真实需求
3. 最后决定哪些设置能关、哪些必须保留并加白名单

#### 验收要点
- webview / navigation / popup 不再裸奔
- 主窗口和子窗口安全边界更清晰

---

### 3.3 子问题簇 C：收回敏感配置与凭证暴露面

#### 主要问题
- P0-07 敏感配置与站点凭证被完整下发到 renderer
- 保存配置时还会广播整包配置
- 备份文件会包含敏感字段

#### 需要解决什么
- 定义 renderer-safe 配置与站点 DTO
- 不再把代理密码、Google `clientSecret`、站点认证材料直接交给 renderer
- `config:changed` 事件只广播摘要
- 备份策略支持脱敏或加密

#### 当前主要落点文件
- `src/main/ipc/handlers.ts`
- `src/preload/index.ts`
- `src/shared/types.ts`
- `src/main/services/config.ts`
- `src/main/services/backupService.ts`
- 依赖相关字段的 renderer 页面

#### 代码复核补充
- `config:get` 当前直接返回 `getConfig()` 全量对象，`config:save` 成功后又把 `newConfig` 原样广播给所有窗口
- `preload` 直接把 `config.get/save/onConfigChanged` 暴露给 renderer；已有页面（如 `BooruPage`）会订阅整包配置变更
- `config.ts` 中明确包含 `network.proxy.username/password`、`google.clientSecret` 等敏感字段
- 备份导出 `createAppBackupData()` 会直接携带 `getConfig()` 结果，导入时也会整包 `saveConfig()`

#### 推荐实施顺序
1. 先盘点当前敏感字段流向
2. 再设计 renderer-safe DTO
3. 再收紧 `config:get`、`config:save`、站点返回对象、广播事件和备份链路

#### 验收要点
- renderer 无法直接读到敏感配置与站点密码材料
- 配置变更广播不再携带完整敏感对象
- 备份导出不会默认裸带敏感信息

### 3.4 子问题簇 D：修运行入口、异常退出与崩溃治理

#### 主要问题
- P0-05 `npx electron .` 默认入口启动失败
- P1-20 缺少 quit / crash / unhandled exception 统一兜底
- 第 7.9 明确指出崩溃治理缺失

#### 需要解决什么
- 建立最小可运行入口矩阵
- 把 before-quit / will-quit / process-gone / unhandled exception 等纳入统一治理链路
- 让数据库关闭、任务冻结、恢复定时器清理进入统一收口

#### 当前主要落点文件
- `src/main/index.ts`
- `src/main/window.ts`
- `src/main/services/init.ts`
- `src/main/services/database.ts`
- `build/main/window.js`（用于定位当前入口问题现象）

#### 代码复核补充
- 主入口当前已有单实例锁、`window-all-closed` 退出和启动初始化链路，说明并非“完全无生命周期代码”
- `init.ts` 已通过 `resumeDownloadsTimer` 延迟恢复普通下载和批量下载，但该定时器没有接入统一退出清理
- `database.ts` 已提供 `closeDatabase()`，但主入口未见在 `before-quit` / `will-quit` 等时机调用
- 当前仍未见 `process-gone`、`uncaughtException`、`unhandledRejection`、`before-quit`、`will-quit` 等统一收口

#### 推荐实施顺序
1. 先修运行入口问题
2. 再补退出与崩溃治理钩子
3. 再决定日志与监控挂载策略

#### 验收要点
- Electron 默认入口可用
- 异常退出、崩溃、退出流程不再处于完全无治理状态

---

### 3.5 子问题簇 E：收紧异常场景下的文件恢复边界

#### 主要问题
- P1-08 / P1-09 / P1-24 都说明文件完整性与恢复在异常边界下不可靠

#### 需要解决什么
- 在异常退出、崩溃、取消、恢复场景下，明确“哪些文件可删、哪些文件不可删、哪些文件需校验”
- 让错误恢复语义不再依赖脆弱的最终文件路径存在性

#### 当前主要落点文件
- `src/main/services/downloadManager.ts`
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/init.ts`

#### 代码复核补充
- 现状并非完全没有恢复和清理：普通下载失败时会直接删除 `targetPath`；批量下载在失败、取消、恢复路径里也已有多处文件清理逻辑
- 但普通下载恢复语义本质上是“把 `downloading` 重置回 `pending` 后整单重下”，不是断点续传
- 批量下载恢复会把 `downloading` 记录重置为 `pending`，若统计上已无待下载项则直接把会话标成完成；这说明系统已有部分恢复判定，但规则仍分散在不同服务里
- 后续文档应把“已有部分清理 / 校验 / 恢复”与“仍缺统一规则”同时写清，避免误写成完全空白状态

#### 边界说明
- 本节只处理异常退出 / 崩溃 / 取消 / 恢复场景下的文件存在性、清理边界与误删误判问题，不重写下载调度与状态流转语义；后者仍以 [10](./10-下载并发与调度链路重构.md) 为主责任文档

#### 推荐实施顺序
1. 与下载文件协议同步定义异常态清理规则
2. 明确退出、重启、恢复时对临时文件和最终文件的处理策略

#### 验收要点
- 异常场景下不误删完整文件
- 恢复逻辑不再把半文件误判成已完成文件

### 3.6 子问题簇 F：建立 config / 数据库 / localStorage 的持久化边界

#### 主要问题
- 当前项目已经同时使用 `config.yaml`、SQLite 和 renderer `localStorage` 持久化状态，但缺少明确分层规则
- `config:get` / `config:save` 直接以整包配置作为 renderer 读写对象，导致应用级配置、敏感字段和页面 UI 偏好被混装在同一个容器里
- `App.tsx` 已把 `ui.menuOrder`、`ui.pinnedItems` 写入配置，而 `AppConfig` 类型只正式声明了 `menuOrder`，说明配置容器已经开始承接页面行为状态并出现类型漂移
- 收藏标签、图库、黑名单等管理页的排序 / 筛选 / 分页状态目前只存在于页面内存里；一旦要做跨会话恢复，就必须先回答“这类状态该落在哪一层”

#### 需要解决什么
- 给三类持久化载体建立明确边界：什么属于应用配置，什么属于产品数据，什么只是 renderer 局部偏好
- 防止 config 继续膨胀成“既放凭证、又放页面视图状态、还直接广播给 renderer”的混合大包
- 为后续页面排序/筛选持久化建立统一落点，而不是每个页面各自随手写到 config 或 localStorage

#### 推荐边界
1. `config.yaml`
   - 只放应用级、环境级、跨进程基础配置
   - 例如：`dataPath`、数据库/缓存/日志路径、代理配置、Google OAuth 基础配置、Booru 全局 appearance 默认值、下载默认模板等
2. 数据库
   - 放业务实体、关系、运行历史，以及“跨会话且属于产品工作上下文”的页面级偏好
   - 例如：FavoriteTags / Gallery / BlacklistedTags 的排序、筛选、分页、选中视角等需要跨窗口/跨会话一致的状态；至于菜单固定项等是否最终入库，应在后续实施中结合边界与兼容策略确定
3. renderer `localStorage`
   - 只放轻量、无敏感性、renderer 独享的体验偏好
   - 例如：主题、语言；这类状态即使不同窗口不完全一致，也不会破坏产品数据语义

#### 代码复核补充
- `config:get` 当前直接返回 `getConfig()` 全量对象，`config:save` 成功后又把 `newConfig` 原样广播给所有窗口
- `config.ts` 的 `AppConfig` 同时承载路径、代理密码、Google `clientSecret`、Booru appearance 等；而 `App.tsx` 又额外把 `ui.pinnedItems` 写回配置，形成“类型未声明但运行时已在使用”的漂移
- renderer 当前真正适合放在 `localStorage` 的例子只有轻量偏好：`useTheme.ts` 保存主题模式，`locales/index.ts` 保存语言
- FavoriteTags 下载绑定这类业务配置已经在数据库 `booru_favorite_tag_download_bindings` 中落地，说明“与领域对象强相关且需要跨会话一致”的数据，本项目本就更适合放 DB 而不是 config

#### 推荐实施顺序
1. 先冻结“哪些页面状态允许继续写进 config”，避免边改边扩散
2. 再定义 renderer-safe `AppConfigDTO` 与数据库化的 `page_preferences` / 视图状态模型
3. 最后逐页把高频管理页排序/筛选持久化迁移到统一偏好存储层，并收紧 `config:changed` 广播内容

#### 验收要点
- 敏感配置、应用基础配置、页面工作上下文三者不再混装
- config 对 renderer 只暴露脱敏且收窄后的 DTO
- 页面级持久化状态有统一落点，不再各页随意选择 config 或 localStorage

## 5. 推荐实施顺序

### 第一阶段：先修高风险边界
- `app://`
- `openExternal`
- 敏感配置与站点凭证暴露
- webview / navigation / popup 主进程约束

### 第二阶段：再修运行与异常治理
- Electron 入口
- quit / crash / unhandled exception
- 恢复定时器、数据库关闭、任务冻结统一纳管

### 第三阶段：收口异常文件恢复边界
- 结合下载文件协议定义清理与恢复规则

## 6. 前置依赖与并行关系

### 前置依赖
- 若某项安全边界变更影响 bridge 暴露面，应同步看 [12](./12-IPC与进程边界重构.md)
- 若某项恢复边界变更影响下载链路，应同步看 [10](./10-下载并发与调度链路重构.md)

### 可并行关系
- `app://`、`openExternal`、敏感配置 DTO 可以并行子线推进
- 运行入口修复与退出治理可以协同
- 但不要在日志/监控尚未定义脱敏策略前先全量接入观测系统

## 7. 风险点

1. 安全边界一旦收紧，可能暴露历史依赖宽松行为的兼容问题
2. renderer 端可能直接依赖当前暴露的敏感字段，收紧后需要同步清理消费方式
3. 运行入口与异常治理涉及全局生命周期，变更范围大

## 8. 验收要点

至少应确认：
- 本地资源与外部链接边界收紧
- webview / popup / navigation 不再裸奔
- renderer 不再直接拿到敏感配置和凭证材料
- 入口可运行，异常与退出有统一治理链路
- 异常恢复下的文件策略更可解释
