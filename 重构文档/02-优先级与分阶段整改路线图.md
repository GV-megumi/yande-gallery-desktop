# 02-优先级与分阶段整改路线图

## 1. 文档目的

本文件用“实施阶段 / 依赖关系 / 并行关系”重组 `审查报告.md` 中的问题，不按原报告章节顺序，也不直接按 P0/P1/P2 平铺。

本文件主要回答：

- 哪些问题必须先修
- 哪些问题共享前置改造
- 哪些工作可以并行推进
- 每个阶段的完成标志和回归重点是什么

## 2. 使用方式

- 需要排期：先看本文件
- 需要分发任务：先看本文件，再看 [03-任务拆分总控.md](./03-任务拆分总控.md)
- 需要做阶段验收：先看本文件的阶段完成标志，再对照专项文档验收点

## 3. 路线图设计原则

1. 主组织轴按实施阶段与依赖关系排列
2. 原始 P0/P1/P2 继续保留，但只作为风险标签
3. 共享前置改造优先，不做彼此打架的并行开发
4. 先修运行语义、安全边界、明显伪能力，再做交互统一和现代化增强
5. 每个阶段都必须有可检查的完成条件，而不是只列一串问题名

## 4. 阶段划分总览

| 阶段 | 目标 | 核心主题 | 主要对应文档 |
|---|---|---|---|
| 阶段 0 | 立即止血 | 运行阻断、安全硬伤、状态机失真、假入口 | 10 / 12 / 13 / 15 |
| 阶段 1 | 收敛核心链路 | 进程边界、页面伪能力、主窗口初始化与恢复 | 01 / 10 / 12 / 13 / 15 |
| 阶段 2 | 建立一致性 | 导航、弹窗、空错态、隐藏页副作用、桌面运行形态基础 | 11 / 14 / 15 |
| 阶段 3 | 结构优化与现代化升级 | facade/hooks、执行池、限流、虚拟化、preload 瘦身 | 12 / 14 / 16 |
| 阶段 4 | 长期桌面增强 | 托盘、通知、开机自启、日志、崩溃监控 | 11 / 15 / 16 |

## 5. 阶段 0：立即止血

### 5.1 阶段目标
先阻断那些会直接导致：
- 下载状态污染
- 重复调度
- 安全面失守
- 运行入口不可用
- 用户操作明显失真

### 5.2 必须先做的内容

#### A. 下载状态机与调度硬伤
- 修批量下载会话生命周期 bug
- 修批量记录领取原子性
- 修普通下载 `pauseAll()` 语义
- 修普通下载单任务 pause 污染 `failed`
- 修 paused 会话自动恢复

#### B. 安全和边界硬伤
- 收紧 `app://` 任意路径映射
- 修 `openExternal` 协议/目标校验
- 先把敏感配置与站点凭证从 renderer 暴露面收回
- 给 webview / new window / navigation 建立主进程收口规则

#### C. 明显伪能力和假入口
- 删除或禁用下载页单条取消 TODO 入口
- 清理 `pageMode` 伪设置
- 清理 `autoGenThumbnail` 伪设置
- 清理或归档 `handlers-full.ts`

#### D. 运行入口问题
- 建立最小可运行启动矩阵
- 解决 `npx electron .` 默认入口启动失败

### 5.3 为什么这一阶段必须先做

如果不先解决这一层问题：
- 后续任何托盘、通知、后台运行能力都只是在错误状态机上叠加功能
- 后续任何页面交互统一都可能建立在伪能力上
- 后续任何 hooks/facade、执行池替代都无法在不稳定边界上可靠落地

### 5.4 可并行关系

可以并行：
- 下载状态机修复 与 页面伪能力清理
- 运行入口修复 与 `handlers-full.ts` 清理
- `app://` / `openExternal` / 敏感配置边界收紧（同属安全边界，可在一个专题包内并行子项）

不应并行：
- 在协议单一来源未收敛前，多个任务同时新增 IPC 接口
- 在下载状态机未统一前，先做托盘/通知/后台常驻

### 5.5 阶段完成标志

达到以下条件后，才算阶段 0 完成：

- 下载暂停/取消/恢复不再污染失败语义
- 批量会话不会重复领取同一记录
- `app://` 不再映射任意本地路径
- renderer 不再直接获取敏感配置和站点凭证
- `npx electron .` 能以受控方式启动
- 页面上不再保留明显假按钮或伪设置入口

### 5.6 回归重点

- 普通下载 / 批量下载的 pause / resume / cancel / restore
- 配置保存与页面消费链路
- Electron 启动入口、主窗口创建、子窗口能力
- FavoriteTags、Download、Settings 等高频页面入口

## 6. 阶段 1：收敛核心链路

### 6.1 阶段目标
在止血后，把系统中最容易持续制造新问题的边界收口。

### 6.2 核心内容

#### A. IPC 与进程边界收口
- 统一 IPC 协议单一来源
- preload API 与声明同源维护
- shared 类型纠偏
- handler 职责瘦身
- 页面层业务编排逐步下沉为 hooks / facade

#### B. 页面功能闭环修复
- `dataPath` 保存后的运行时同步或明确重启要求
- FavoriteTags 路径配置闭环
- SavedSearch 站点编辑与运行闭环
- Google 集成页失败态 / 重试 / 外部打开退路
- InvalidImages 失败态与空态分离
- 图库删除入口闭环与“仅删图集记录、不删文件”确认模型

#### C. 页面偏好持久化与配置边界收口
- FavoriteTags / Gallery / BlacklistedTags 的排序、筛选、分页状态持久化
- 明确 config / DB / localStorage 三者边界
- 收紧 renderer 可见配置 DTO，避免继续把页面行为状态写进整包 config

#### D. 启动与恢复链路收口
- 主窗口优先显示，再延迟非关键初始化
- 启动恢复、退出清理、下载恢复定时器对齐

### 6.3 依赖关系

- 阶段 1 默认依赖阶段 0 完成
- 页面功能闭环可与 IPC 收口部分并行，但不要在接口还漂移时同时大改多页消费方式

### 6.4 可并行关系

可以并行：
- 页面功能闭环组
- IPC 单一来源组
- preload 瘦身准备工作
- 启动与恢复优化

不应并行：
- 在 shared / preload / handler 结构未定义清楚前，就全面把页面抽到 hooks / facade

### 6.5 阶段完成标志

- IPC 通道与 preload 暴露面有单一来源
- shared 不再继续冒充过时协议总表
- 关键伪能力已经变成真实闭环或被移除
- 启动不再阻塞主窗口首屏显示到不可接受程度
- 恢复、退出和初始化的时序不再互相竞争

### 6.6 回归重点

- Gallery、SavedSearch、FavoriteTags、Google、Settings
- 页面间 IPC 调用
- 启动、关闭、恢复链路

## 7. 阶段 2：建立一致性

### 7.1 阶段目标
把“功能已经基本能用，但用户仍难以建立稳定预期”的问题统一掉。

### 7.2 核心内容

#### A. 全局导航一致性
- 保留“一级菜单只切左侧二级菜单”的既定规则
- 建立单一导航状态模型，收敛二级菜单、默认子页与右侧内容的同步关系
- 固定页面、嵌入页、导航栈统一在同一主上下文模型上

#### B. Modal 关闭模型一致性
- 为普通表单弹窗建立统一规则
- 特殊沉浸式页面明确例外规则
- 显式统一 `closable` / `maskClosable` / `keyboard`

#### C. 空态 / 错态 / 加载态一致性
- 错误不能再伪装成空态
- 筛选无命中与完全无数据明确区分
- Google / InvalidImages / SavedSearch 等页面统一恢复入口

#### D. 容器可见性与副作用模型一致性
- 隐藏页不再继续恢复、轮询、监听
- keep-alive 与副作用显式解绑或上提到更高层统一控制
- 治理隐藏页副作用时，不得破坏 Hub 页面“切换不丢状态”的既有承诺

#### E. 桌面形态基础能力
- close / hide / quit 区分
- 为托盘与后台常驻建立最小基础设施

### 7.3 依赖关系

- 阶段 2 默认依赖阶段 1
- 但若阶段 0 已先清掉明显伪能力，可先启动导航与 Modal 规范设计，不一定要等全部功能修补完毕

### 7.4 可并行关系

可以并行：
- 导航模型
- Modal 规范
- 空错态规范
- 隐藏页副作用治理

不应并行：
- 在未确定最终桌面关闭策略前，同时做托盘与通知的全量交互细节

### 7.5 阶段完成标志

- 一级菜单规则、二级菜单默认项与右侧内容联动关系清晰且一致
- 同类弹窗关闭规则一致
- 错态不再被空态吞掉
- 不可见页面不再继续执行用户不可感知副作用，且 Hub 页面切换仍不丢状态
- 后台常驻链路开始具备可承载能力的基础结构

### 7.6 回归重点

- App 主导航
- 所有高频 Modal
- DownloadHub / TagManagement / FavoriteTags
- Google / InvalidImages / Gallery / PostDetails

## 8. 阶段 3：结构优化与现代化升级

### 8.1 阶段目标
在边界清晰后，把最容易反复出结构性问题的基础设施逐步交给成熟方案。

### 8.2 核心内容

- 下载执行池接入 `p-queue`
- API 限流接入 `Bottleneck`
- 列表虚拟化方案落地
- 页面层业务编排下沉 hooks / facade
- preload 进一步瘦身
- 构建分块与 vendor 大包优化

### 8.3 依赖关系

- 默认依赖前两阶段完成
- 如果边界还没收口，不应先大规模接入新库，否则只会把旧问题包进新壳里

### 8.4 可并行关系

可以并行：
- 虚拟列表探索与构建分块优化
- hooks/facade 下沉与 preload 瘦身
- 执行池与限流器的方案评估

不应并行：
- 在下载状态机未稳定前直接切执行池
- 在页面容器行为未稳定前直接做大规模虚拟列表重构

### 8.5 阶段完成标志

- 队列与限流不再完全依赖手写 while-loop
- 图片密集型列表开始具备真正虚拟化
- 页面层重复业务编排明显下降
- preload 暴露面较阶段 1 明显收缩

### 8.6 回归重点

- 下载相关性能与稳定性
- Gallery / Booru 列表滚动表现
- 页面 hooks / facade 与原页面行为一致性

## 9. 阶段 4：长期桌面增强

### 9.1 阶段目标
在边界稳定后，把项目补到“更像成熟桌面应用”的层级。

### 9.2 核心内容

- `electron-log`
- `@sentry/electron`
- 托盘、关闭不退出、后台常驻细化
- 通知策略
- 开机自启
- 崩溃恢复与优雅退出

### 9.3 依赖关系

- 依赖前述阶段已经完成
- 特别依赖下载状态机、退出治理、桌面基础能力已稳定

### 9.4 阶段完成标志

- 日志和异常具备结构化落盘与上报能力
- 长任务场景下托盘、通知、后台运行形成闭环
- 异常退出后的可观测性和恢复策略明显增强

## 10. 原始级别与阶段的关系

### P0 主要落在
- 阶段 0：下载会话生命周期、原子 claim、`app://`、默认入口启动失败、敏感配置越界
- 阶段 1：协议单一来源的第一轮收口

### P1 主要落在
- 阶段 0：pause 语义、paused 自动恢复、假按钮、伪设置、`openExternal`
- 阶段 1：页面闭环、`dataPath`、隐藏页副作用、配置同步
- 阶段 2：导航、弹窗、桌面基础能力、错误态规范

### P2 主要落在
- 阶段 1：`handlers-full.ts` 清理
- 阶段 3：构建分块、成熟库引入、虚拟化
- 阶段 4：长期桌面增强项

## 11. 阶段验收建议

- 不要只验代码通过或测试通过
- 每个阶段结束时至少做：
  1. 任务包验收
  2. 相关页面回归
  3. 边界核查
  4. 与 `04-覆盖矩阵与原报告映射.md` 对照，确认没有遗漏
  5. 将复核结论写入 `05-复核记录与整合结论.md`
