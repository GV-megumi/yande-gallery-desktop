# 16-成熟库替代与现代化升级建议

## 1. 本文档范围

本文件负责处理“哪些能力不应继续扩大自研面，而应逐步交给成熟基础设施接管”的问题。

主责任内容包括：
- Electron 原生安全 / 生命周期能力优先补齐的原则
- `p-queue`、`Bottleneck` 等执行池 / 限流器候选
- `react-virtuoso` / `@tanstack/react-virtual` 等虚拟列表候选
- `electron-log` / `@sentry/electron` 等日志与崩溃监控候选
- 现代化引入顺序、收益、成本、风险

不展开的内容：
- 先修哪些现有硬伤（见 [02](./02-优先级与分阶段整改路线图.md)）
- 进程边界、preload、shared 重构细节（见 [12](./12-IPC与进程边界重构.md)）
- 下载状态机与桌面形态细节（见 [10](./10-下载并发与调度链路重构.md)、[11](./11-Electron桌面能力重构.md)）

## 2. 本专项核心判断

报告第 9 章的结论不是“把自研全部推倒重来”，而是：

- 继续让生命周期、安全边界、托盘、通知等 Electron 原生能力缺位，是不合理的
- 继续让下载执行池、限流、虚拟列表、日志、崩溃监控全部靠当前手写或缺失方案支撑，也已经进入高风险区
- 但成熟库引入必须建立在边界已经基本收口的前提上，否则只是把旧问题包进新壳

## 3. 替代建议原则

1. **不是全盘换掉自研**
   - 领域状态、数据库结构、业务约束仍应保留在项目内部掌控

2. **先补边界，再引库**
   - 协议边界、状态机、安全边界没收口前，不应急着大规模接入新库

3. **优先替换结构性高风险领域**
   - 执行池
   - 限流器
   - 虚拟列表
   - 日志
   - 崩溃监控

4. **Electron 原生能力优先用原生补齐**
   - tray
   - notification
   - app lifecycle
   - window / webContents 安全控制

## 4. 详细替代专题

### 4.1 执行池：`p-queue`

#### 适用问题
- 当前下载 / 批量下载仍以重型自研调度为主
- 报告已确认会话生命周期、claim、pause/resume 语义、监听泄漏等问题

#### 推荐用途
- 作为下载执行池的运行时收口层
- 让内存执行队列 API 更可控

#### 适合引入的前提
- 先完成下载状态机与调度边界收口（见 [10](./10-下载并发与调度链路重构.md)）

#### 代码复核补充
- 当前实现并不只是“简单 while-loop”，而是已经在 `src/main/services/downloadManager.ts`、`src/main/services/bulkDownloadService.ts`、`src/main/services/networkScheduler.ts` 中混合了启动恢复、失败清理、批量会话统计、浏览态网络调度等一套重型自研运行时
- 因此当前并不存在一个边界非常整齐、可被整块替换的单一“执行池层”
- `p-queue` 更适合优先替换普通下载与批量下载里可抽离的并发调度 / 排队子层，而不是替换文件命名、DB 状态、恢复判定等领域逻辑

#### 收益
- 比继续扩大自研调度器更稳
- 更容易建立 pause / resume / priority 等控制面

#### 风险
- 若 DB 状态与内存运行态边界没梳清，会把旧问题封装后继续存在

### 4.2 限流器：`Bottleneck`

#### 适用问题
- Booru 多站点差异化限流
- 429、动态退避、per-site 策略

#### 推荐用途
- 作为主进程服务层的限流基础设施
- 与下载执行池分层：状态持久化留在 DB，限流与节流交给成熟 limiter

#### 适合引入的前提
- 先明确不同站点请求的边界与控制面

#### 收益
- 比单纯 while-loop 和 sleep 更适合多站点策略

#### 风险
- 如果接入点不清晰，容易和现有调度逻辑互相重叠

### 4.3 虚拟列表：`react-virtuoso` / `@tanstack/react-virtual`

#### 适用问题
- 图片密集型列表目前没有真正虚拟化
- 构建产物大、列表渲染压力大、滚动性能受限

#### 候选差异
- `react-virtuoso`：适合较快获得 variable-size / grid 场景收益
- `@tanstack/react-virtual`：适合需要长期深度定制、保留复杂布局控制的场景

#### 适合引入的前提
- 先明确 Gallery / Booru 列表最终布局策略
- 先解决“当前查询条件与结果脱节”等页面语义问题，避免在不稳定行为上套虚拟化

#### 收益
- 滚动性能、内存、DOM 压力显著改善

#### 风险
- 当前 CSS columns / 瀑布流实现与虚拟化不天然兼容

### 4.4 结构化日志：`electron-log`

#### 适用问题
- 当前主要依赖 `console.log` / `console.error`
- 异常定位、退出治理、生产排障能力不足

#### 推荐用途
- 统一主进程 / 渲染进程落盘日志
- 为退出治理、恢复链路、长任务排障提供基础支撑

#### 适合引入的前提
- 先定义脱敏规则，避免路径、账号、URL、凭证进日志

#### 收益
- 接入成本低，Electron 亲和度高，能快速替换散落日志

#### 风险
- 若不做脱敏规则，日志本身会变成新的敏感泄露面

### 4.5 崩溃监控：`@sentry/electron`

#### 适用问题
- 主进程 quit / crash / unhandled exception 治理不足
- 缺少真正能看生产问题的能力

#### 推荐用途
- 接住主进程、渲染进程、预加载层异常与 crash 事件

#### 适合引入的前提
- 先补基础异常治理钩子（见 [15](./15-稳定性错误恢复与安全性修正.md)）
- 先确定脱敏和采样策略

#### 代码复核补充
- 现有主入口已经有单实例、窗口关闭退出、启动恢复等基础生命周期代码，但还没有统一异常收口，因此 Sentry 前置条件不是“从零开始接管生命周期”，而是先把现有钩子补齐并统一出口
- 若仍保留当前整包配置下发 / 广播和备份裸导出，直接接入崩溃监控会放大敏感数据暴露风险

#### 收益
- 能真正看见生产异常而不是只靠本地复现

#### 风险
- 若在边界和脱敏未定义前接入，可能扩大敏感数据暴露面

## 5. 哪些能力应先用 Electron 原生补齐

成熟库不是所有问题的答案。以下能力应优先使用 Electron 原生能力：

- tray
- notification
- app lifecycle
- `setWindowOpenHandler`
- `will-navigate`
- `will-attach-webview`
- 单实例与窗口恢复相关基础能力

原因是：
- 这些能力本身就是 Electron 的运行时边界，不应先交给第三方库包起来再理解
- 先把原生能力补齐，后续是否需要外围封装再决定

## 6. 推荐引入顺序

### 第一批：边界收口后优先评估
1. `electron-log`
2. `p-queue`
3. `Bottleneck`

### 第二批：页面行为稳定后推进
4. `react-virtuoso` 或 `@tanstack/react-virtual`

### 第三批：基础异常治理到位后推进
5. `@sentry/electron`

### 第四批：与桌面能力长期增强一起推进
6. 托盘、通知、开机自启等长期桌面增强策略细化

## 7. 影响模块 / 当前主要落点文件

### 下载 / 调度 / 限流
- `src/main/services/downloadManager.ts`
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/networkScheduler.ts`

### 图片密集型列表
- `src/renderer/App.tsx`
- `src/renderer/pages/GalleryPage.tsx`
- `src/renderer/pages/BooruPage.tsx`
- 其他图片密集型页面

### 日志 / 崩溃 / 生命周期
- `src/main/index.ts`
- `src/main/window.ts`
- `src/main/services/init.ts`
- `src/main/services/database.ts`

## 8. 风险点

1. 在边界没收口前引库，只会把旧问题包进去
2. 虚拟列表与当前布局冲突风险较高，不能只看库能力宣传页
3. 日志和崩溃监控若不做脱敏，会制造新的安全问题
4. 执行池与限流器若同时引入，必须先定义谁管并发、谁管速率
5. 不能把 `p-queue` / `Bottleneck` 误当成对下载领域逻辑的整体替代；文件协议、DB 状态、恢复判定仍需项目内部掌控

## 9. 验收要点

至少应确认：
- 每个候选库的引入目标、替代范围、前置条件都明确
- 不把库引入当成替代边界设计的捷径
- 现代化升级顺序与阶段路线图一致，不与前置硬伤修复冲突
