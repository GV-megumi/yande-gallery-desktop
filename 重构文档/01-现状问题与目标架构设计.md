# 01-现状问题与目标架构设计

## 1. 文档目的

本文件不负责逐条复述 `审查报告.md`，而是把报告中已经确认的关键判断，上升为一套可以指导后续全部专项改造的目标结构。

它主要回答四个问题：

1. 当前项目到底失稳在哪里
2. 为什么不能再继续按现状堆功能
3. 目标架构应该长成什么样
4. 各专项文档之间的边界和演进顺序应该怎样定义

## 2. 当前现状的总判断

基于 `审查报告.md` 第 2 章、第 8 章、第 10 章、第 12 章，可以把当前问题概括为七个问题域：

1. 下载并发与调度链路失稳
2. Electron 桌面能力缺位
3. IPC / preload / shared / renderer 边界失控
4. 页面功能完整性不足，出现大量伪能力
5. UI 交互一致性没有形成统一规则
6. 稳定性 / 错误恢复 / 安全边界偏松且存在硬伤
7. 自研面已经超出轻量实现的安全区，开始需要成熟库接管基础设施

这些问题不是彼此独立的。真正的结构性风险在于：

- 页面层为了补功能，开始承担越来越多业务编排
- preload 为了方便页面消费，暴露面越来越大
- main handler 为了快速对接页面，把职责一路上浮
- 下载与恢复链路为了先跑起来，状态机、调度、文件协议、页面副作用逐渐缠在一起
- 桌面能力、安全边界、崩溃治理又没有把这些问题兜住

这意味着项目已经从“功能很多但不够完善”，进入了“边界一旦不收口，再继续加功能只会让问题指数级扩散”的阶段。

## 3. 最重要的决策基线

后续所有重构都应建立在以下八条判断之上：

1. 当前最大风险不在少几个按钮，而在下载与恢复链路的一致性设计不稳。
2. 当前应用还没有成为合格的桌面下载/图库工具，因为后台常驻闭环没有建立起来。
3. preload / IPC / shared 边界已经开始失控，继续堆功能只会放大维护成本。
4. 页面上已经出现“看起来能配、实际上不生效”的伪能力，这是可信度问题，不只是小 bug。
5. 测试全绿不能证明产品合格，因为桌面生命周期、运行入口、后台体验与交互一致性没有被真实覆盖。
6. 导航、弹窗、空错态、消息反馈缺少统一规则，说明交互规范并未建立。
7. 项目已经到了该借力成熟库的时候，再继续扩大自研基础设施，风险会更高。
8. 最值得立即投入的不是继续加功能，而是先还掉核心技术债。

## 4. 当前架构症状与根因

### 4.1 下载链路症状

当前下载体系的问题不是单点 bug，而是多个层级同时失真：

- 批量下载会话生命周期绑定错层
- 领取记录非原子
- 普通下载与批量下载的 pause / cancel / failed 语义互相冲突
- 文件写入协议不完整，缺少 `.part + rename` 与严格完成校验
- 渲染层隐藏页面仍在执行恢复、轮询、监听等副作用

根因是：
- 状态机没有被定义成单一可信模型
- 调度器、数据库状态、运行时内存态和页面副作用边界不清
- 下载功能既缺主进程级结构治理，也缺页面层消费约束

### 4.2 进程边界症状

当前 `main / preload / shared / renderer` 的问题集中表现为：

- preload 过于臃肿，已经成为 God API
- IPC 通道来源分裂为 `channels.ts`、preload 本地常量、handlers 裸字符串三套来源
- `shared` 中的部分协议定义已不能代表真实接口
- 页面直接调用不存在的 API 或依赖不一致签名
- 配置、站点凭证、子窗口高权限 bridge 穿透到了 renderer

根因是：
- 没有把“最小暴露、单一来源、职责隔离”当成硬边界
- 为了短期便利，把协议定义、类型定义、bridge 暴露、页面消费分散维护
- 页面层承担了太多本该由 facade / service / hooks 层吸收的复杂度

### 4.3 页面层症状

当前页面问题可分成三类：

1. **功能兑现问题**
   - 设置项不生效
   - 按钮只有 TODO
   - 页面闭环缺失
   - 错误态/重试入口缺失

2. **交互规则问题**
   - 一级菜单只切左栏不切右侧内容
   - Modal 关闭模型不统一
   - 空态 / 错态 / 加载态混淆
   - 隐藏页面仍在运行副作用
   - 消息反馈噪音过大

3. **页面工作上下文问题**
   - FavoriteTags / Gallery / BlacklistedTags 的排序、筛选、分页状态跨会话丢失
   - 高选项量选择器仍停留在不可检索下拉
   - 同一对象在不同信息面展示字段不一致

根因是：
- 页面暴露能力后缺少统一的整改原则与兑现约束，导致用户看到的能力与实际行为经常脱节
- 页面容器与导航状态缺少单一模型
- 缺少统一的交互约束与共享页面行为层
- 更底层地说，项目没有先定义“页面偏好应该持久化到哪里、以什么 DTO 暴露”，所以高频管理页一直停留在临时态

### 4.4 桌面能力与安全边界症状

当前 Electron 层的问题并不是“以后再补就行”的增强项，而是已经影响核心产品形态：

- 无托盘、无关闭不退出、无后台常驻闭环
- 无系统通知、无开机自启、无崩溃治理
- `app://` 协议过宽
- `openExternal` 无协议/目标校验
- webview、navigation、新窗口能力没有在主进程收口

根因是：
- 当前主进程只做了最基础的 BrowserWindow 管理
- 没有建立明确的应用级生命周期治理策略
- 没有把外部内容、窗口、安全、恢复视作统一边界问题

## 5. 目标架构

### 5.1 总体目标

目标不是一次性重写，而是把系统演进到下面这个状态：

- main 负责系统边界、外部 I/O、调度、持久化、桌面生命周期和安全治理
- preload 只做受控 bridge，不再承载业务编排
- shared 只承载稳定、可信、同源维护的共享类型与协议结构
- renderer 页面只负责展示、用户交互、消费 view-model 与 command，不再直接拼装复杂业务流程
- 下载体系拥有清晰的状态机、调度器边界、文件协议、恢复模型
- 桌面应用具备合格的后台运行、通知、退出治理、安全边界与错误恢复能力

### 5.2 目标分层

#### 主进程层（main）
主进程应成为以下能力的唯一责任层：

- 下载执行与调度
- 数据库状态与文件系统协议
- 配置、凭证、站点认证材料
- 窗口生命周期、托盘、通知、退出治理
- `app://`、`openExternal`、webview、新窗口等安全边界
- IPC handler 的参数校验、权限控制、错误映射

不应继续把大量业务流程临时塞进 handler 或 preload。

#### bridge 层（preload）
preload 目标应收缩为：

- 暴露最小必要的 query / command / event 接口
- 所有 API 由单一协议源生成或同源维护
- 对 renderer 返回 renderer-safe DTO，而不是原始内部对象
- 子窗口与嵌入页使用差异化能力集，而不是复用主窗口全部权限

preload 不应继续：
- 自己维护另一套协议常量
- 聚合大量业务逻辑
- 暴露敏感配置或凭证字段

#### 共享层（shared）
shared 的目标是：

- 只保存可信、同步维护的共享类型
- 明确区分：协议结构、领域 DTO、页面消费 DTO、事件 payload
- 不再让过时类型继续冒充“完整协议总表”

#### 持久化分层（config / database / localStorage）
目标状态下，三类持久化载体必须各司其职：

1. **config / AppConfigDTO**
   - 只承载应用级、环境级、跨进程基础配置
   - 例如：`dataPath`、数据库/缓存/日志路径、代理基础配置、Google OAuth 基础配置、Booru 全局外观默认值、下载默认模板
   - renderer 只能拿到脱敏且收窄后的 `renderer-safe AppConfigDTO`

2. **database**
   - 承载业务实体、关系、运行历史，以及“跨会话且属于产品工作上下文”的页面偏好
   - 例如：FavoriteTags / Gallery / BlacklistedTags 的排序、筛选、分页、选中视角等需要跨窗口/跨会话一致的状态；至于某些导航或固定项是否最终落 DB，仍应在后续实施时结合约束再定

3. **renderer localStorage**
   - 只承载轻量、无敏感性、renderer 独享的体验偏好
   - 例如：主题、语言这类即使不同窗口不完全一致，也不会破坏产品数据语义的状态

如果不先建立这一层分工，后续任何“页面状态持久化”都会继续把 config 写成混合大包，或让各页各自挑一个落点野生生长。

#### 页面层（renderer）
renderer 页面层的目标是：

- 专注于页面状态、组件组合和用户交互
- 通过 hooks / facade / 稳定 data-access 层消费能力
- 不再重复写下载、收藏、服务端喜欢、轮询、防抖、刷新等业务编排
- 不再承载不可见副作用
- 页面工作上下文通过统一 page preferences 层持久化，而不是各页自己决定写 config / DB / localStorage
- 导航、空态、错态、弹窗、反馈遵循统一规则

### 5.3 下载链路的目标结构

下载体系应演进为四个清晰分层：

1. **状态模型层**
   - 统一定义普通下载和批量下载的状态语义
   - 明确 `pending / running / paused / cancelled / failed / completed`
   - 用户主动暂停/取消不再污染失败语义

2. **调度与执行层**
   - 活跃会话与真实执行循环绑定
   - 领取记录具备原子性
   - 会话执行池单实例化
   - 恢复、重试、限流与调度器监听具备生命周期对称性

3. **持久化与文件协议层**
   - 队列表具备数据库约束
   - 文件写入采用 `.part + rename`
   - 完成后执行显式完整性校验
   - 取消/失败时只清理明确属于临时态的文件

4. **页面消费层**
   - 页面只消费稳定状态，不驱动隐式恢复
   - 隐藏页不再继续自动恢复或轮询
   - 下载页、批量页、收藏标签下载页共享清晰的状态表达和交互闭环

### 5.4 桌面运行形态的目标结构

目标桌面形态应具备：

- close / hide / quit 明确区分
- 托盘为后台常驻提供入口
- 下载进行中支持“关闭窗口但继续运行”
- 系统通知承接长任务完成、失败和需人工处理事件
- 启动恢复与退出清理有一致策略
- 单实例能力不只防重复启动，还要处理唤醒已存在窗口和主窗口重建
- 开机自启、崩溃治理、异常日志在后续阶段逐步补齐

### 5.5 安全与稳定性边界的目标结构

系统安全边界应演进为：

- `app://` 只映射受控目录
- `openExternal` 只允许安全协议和受控目标
- webview / navigation / popup / sub-window 统一由主进程白名单治理
- renderer 永远只能接收脱敏后的配置与站点对象
- 子窗口能力不再默认继承主窗口高权限 bridge
- 主进程补齐 quit / crash / unhandled exception 统一治理链路
- 日志与监控分阶段补齐，但前提是先定义脱敏与边界规则

## 6. 专项文档边界

### 6.1 10-下载并发与调度链路重构
负责回答：
- 下载任务怎么正确运行
- 下载状态机、调度、恢复、文件协议怎么收敛

### 6.2 11-Electron桌面能力重构
负责回答：
- 应用怎么以一个合格桌面应用的形态承载这些长任务

### 6.3 12-IPC与进程边界重构
负责回答：
- 谁该暴露什么、协议从哪里来、页面为什么不该再直接承载这些复杂度

### 6.4 13-页面功能完整性修补
负责回答：
- 页面已经暴露出来的能力，怎样真正闭环并兑现承诺

### 6.5 14-UI交互一致性修正
负责回答：
- 不同页面和组件之间，如何形成稳定、一致、可预期的交互规则

### 6.6 15-稳定性错误恢复与安全性修正
负责回答：
- 上述能力应该在什么安全与稳定边界内运行

### 6.7 16-成熟库替代与现代化升级建议
负责回答：
- 哪些基础设施不应继续扩大自研面，应由成熟库接管

## 7. 总体演进路径

### 阶段 A：先止血
优先修：
- 批量会话生命周期
- 原子 claim
- 普通下载 pause 语义
- paused 会话自动恢复
- `app://` / `openExternal` / 敏感配置暴露
- 假按钮、伪设置、明显失真的入口

目标：先让运行语义、安全边界、用户可见的最明显失真停止恶化。

### 阶段 B：再收口
优先修：
- IPC 单一来源
- preload 瘦身
- shared / handler / facade 边界收敛
- 页面隐藏副作用、导航、弹窗、空错态统一
- 主窗口先显示，再延迟非关键初始化

目标：让系统从“能跑但混乱”进入“边界清晰、可持续维护”。

### 阶段 C：后增强
优先做：
- 执行池、限流器、虚拟列表
- 结构化日志与崩溃监控
- 托盘、通知、开机自启等长期桌面增强能力

目标：在边界清晰之后，再逐步把高风险自研基础设施交给成熟方案。

## 8. 非目标

本轮文档体系和后续实施不应做以下事情：

- 不把全部历史问题都重写为“新产品设计”
- 不以大规模重写替代边界收口
- 不把成熟库引入当成短期修复所有问题的万能方案
- 不在没有统一状态机和边界设计前，先做大规模 UI 重做
- 不因为测试通过，就跳过桌面生命周期、安全边界和运行语义问题

## 9. 使用建议

- 凡是跨两个以上专项的任务，先读本文件再开工
- 凡是准备新建任务包或做阶段排序，必须用本文件作为边界基线
- 凡是独立复核者发现专项文档互相抢边界，应以本文件为准做整合修正
