# 10-下载并发与调度链路重构

## 1. 本文档范围

本文件负责重组 `审查报告.md` 中与下载执行、并发控制、恢复语义、文件写入协议和页面副作用边界直接相关的问题。

主责任内容包括：
- 普通下载与批量下载状态机
- 会话生命周期与记录领取
- 调度器监听与恢复链路
- 下载队列表约束
- 文件写入、完整性校验、取消清理策略
- 下载相关页面的副作用边界与交互闭环依赖

不展开的内容：
- 托盘、通知、关闭不退出等桌面运行形态（见 [11](./11-Electron桌面能力重构.md)）
- IPC 单一来源、preload 边界（见 [12](./12-IPC与进程边界重构.md)）
- 安全协议、敏感配置、异常治理（见 [15](./15-稳定性错误恢复与安全性修正.md)）
- 成熟库选型细节（见 [16](./16-成熟库替代与现代化升级建议.md)）

## 2. 核心问题概览

报告对下载链路的判断可以收敛成四个问题簇。

### 2.1 状态机失真

当前最危险的问题不是“缺少一个下载功能”，而是状态机语义本身不可信：

- 批量下载中，用户主动暂停/取消会污染为 `failed`
- 普通下载中，`pauseAll()` 会把任务写回 `pending`
- 单任务 pause 又会在通用失败分支中被覆盖成 `failed`
- 启动恢复还会把 `paused` 会话自动恢复为 `running`

这意味着：
- 用户操作与系统异常没有被区分
- 恢复逻辑、失败统计、重试逻辑都建立在脏状态上
- 页面层很难用状态做可信表达

### 2.2 调度器边界失稳

批量下载链路中，调度器与会话生命周期存在设计级问题：

- 会话活跃标记过早清除
- 记录领取非原子
- 同一会话可能被重复 start / 重复恢复
- `networkScheduler` 只有注册没有解绑
- 启动阶段的恢复定时器没有接入退出清理

这意味着：
- “现在到底有没有活跃执行体”这个问题没有可靠答案
- 幂等性和并发安全性都无法保证

### 2.3 持久化与文件协议不完整

下载链路目前缺乏严格的文件协议：

- 普通下载和批量下载直接写最终文件路径
- 普通下载完成后缺少显式二次完整性校验
- 普通下载队列表缺少数据库层唯一约束
- cancel 会在特定时序下误删完整文件

这意味着：
- 磁盘上的“文件存在”不等于“文件有效”
- 崩溃恢复、重复下载、取消清理会互相打架

### 2.4 页面副作用边界和交互收口不足

下载中心及相关页面当前仍有两个明显问题：

- 页面上的取消 / 配置入口并不总是真闭环
- 隐藏页继续恢复、轮询、监听，破坏交互可解释性

这类问题虽然外显在 renderer，但主因仍是下载链路与页面容器边界不清，因此在本文件中作为执行边界说明保留。

## 3. 详细问题拆分

### 3.1 子问题簇 A：统一下载状态机

#### 主要问题
- P0-06 用户主动暂停/取消污染为 `failed`
- P1-01 `pauseAll()` 把任务改回 `pending`
- P1-02 启动时把 `paused` 会话恢复为 `running`
- P1-21 `pauseDownload()` 会把单任务暂停污染为 `failed`

#### 需要解决什么
- 定义普通下载与批量下载都认可的统一状态集合
- 明确用户主动操作、系统异常、自动恢复三类语义的边界
- 让页面不再需要猜测“这个 failed 到底是异常还是用户暂停造成的”

#### 推荐改造顺序
1. 先定义统一状态模型
2. 再修普通下载 pause 语义
3. 再修批量下载记录级 pause/cancel 语义
4. 最后对齐启动恢复与 UI 状态表达

#### 当前主要落点文件
- `src/main/services/downloadManager.ts`
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/init.ts`
- `src/renderer/pages/BooruDownloadPage.tsx`
- `src/renderer/pages/BooruBulkDownloadPage.tsx`

#### 验收要点
- 主动暂停/取消后不再写 `failed`
- `paused` 只能手动恢复，不能启动自动恢复
- 普通下载和批量下载对 `pending / paused / running / failed / completed` 的语义一致

---

### 3.2 子问题簇 B：修批量会话生命周期与原子领取

#### 主要问题
- P0-01 会话活跃标记过早清除
- P0-02 记录领取非原子
- P1-07 监听注册后不解绑

#### 需要解决什么
- 让活跃会话与真实下载循环绑定
- 让领取 pending 记录具备原子性
- 让调度器监听有对称的注销机制

#### 为什么必须一起处理
如果只修生命周期不修 claim：
- 仍可能并发领取同一记录

如果只修 claim 不修生命周期：
- 仍可能重复 start / pause / cancel 失效

因此这三项必须视为同一条“调度器可信性”问题链处理。

#### 推荐改造顺序
1. 把会话活跃态绑定到真实 `downloadLoop`
2. 建立原子 claim
3. 会话执行池单实例化
4. 为 scheduler listener 提供 unsubscribe
5. 启动恢复与退出清理对齐

#### 当前主要落点文件
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/networkScheduler.ts`
- `src/main/services/init.ts`
- `src/main/services/database.ts`

#### 验收要点
- 同一 session 不会重复被启动成多个执行体
- 同一 pending 记录不会被两个调度循环同时领取
- 会话结束/暂停/取消后监听显式解绑

---

### 3.3 子问题簇 C：补齐数据库约束与文件落盘协议

#### 主要问题
- P1-06 下载队列表缺少数据库层唯一约束
- P1-08 直接写最终文件路径，没有 `.part + rename`
- P1-09 cancel 在特定时序下可能误删完整文件
- P1-24 普通下载完成后缺少显式完整性校验

#### 需要解决什么
- 让数据库层具备幂等/唯一保护
- 让文件写入过程与完成态物理隔离
- 让失败清理只作用于明确的临时产物
- 让完成态经过可解释的二次确认

#### 推荐改造顺序
1. 增加唯一约束或幂等写入语义
2. 改为临时文件写入
3. 成功后显式校验再 rename
4. 调整 cancel / failed 清理策略
5. 设计残留临时文件恢复与清理策略

#### 当前主要落点文件
- `src/main/services/database.ts`
- `src/main/services/booruService.ts`
- `src/main/services/downloadManager.ts`
- `src/main/services/bulkDownloadService.ts`

#### 验收要点
- 下载成功前不会直接污染最终文件路径
- 普通下载补齐完成后校验
- cancel 不再对 `pending` 对应文件一刀切删除
- DB 层不再完全依赖应用层去重

---

### 3.4 子问题簇 D：收紧下载相关页面副作用与交互闭环

#### 主要问题
- P1-19 单条下载取消按钮只有 TODO
- P1-23 隐藏页仍执行恢复、轮询和监听副作用
- P2-03 FavoriteTags 下载路径闭环缺失（与下载配置相关）

#### 需要解决什么
- 页面暴露出的下载能力应真实可用
- 不可见页面不应继续驱动主流程副作用
- 与下载相关的配置入口应真正闭环
- 治理隐藏页副作用时，不得破坏 Hub 页面“切换不丢状态”的既有承诺

#### 推荐改造顺序
1. 先删除或实现假按钮 / 假入口
2. 把页面副作用与 `activeTab` / 可见性绑定
3. 再补 FavoriteTags 路径配置闭环

#### 当前主要落点文件
- `src/renderer/pages/BooruDownloadHubPage.tsx`
- `src/renderer/pages/BooruDownloadPage.tsx`
- `src/renderer/pages/BooruBulkDownloadPage.tsx`
- `src/renderer/pages/BooruTagManagementPage.tsx`
- `src/renderer/pages/FavoriteTagsPage.tsx`

#### 验收要点
- 页面上不再保留点击无效的取消入口
- 隐藏页不再自动恢复任务或轮询会话
- FavoriteTags 下载路径可由用户明确配置或明确显示回退来源

## 4. 影响模块 / 页面 / 链路

### 主进程服务层
- `src/main/services/downloadManager.ts`
- `src/main/services/bulkDownloadService.ts`
- `src/main/services/networkScheduler.ts`
- `src/main/services/init.ts`
- `src/main/services/database.ts`
- `src/main/services/booruService.ts`

### 渲染层主要页面
- `src/renderer/pages/BooruDownloadPage.tsx`
- `src/renderer/pages/BooruBulkDownloadPage.tsx`
- `src/renderer/pages/BooruDownloadHubPage.tsx`
- `src/renderer/pages/FavoriteTagsPage.tsx`
- `src/renderer/pages/BooruTagManagementPage.tsx`

### 关键链路
- 启动 -> 初始化 -> 自动恢复 -> 下载调度
- 下载进行中 -> 暂停 / 取消 -> 状态更新 -> 页面刷新
- 下载完成 -> 文件校验 -> 状态完成 -> 页面反馈
- 页面切换 -> tab 隐藏 / 显示 -> 副作用运行 / 停止

## 5. 推荐实施顺序

### 第一阶段：状态机止血
- 统一普通下载和批量下载的 pause / cancel / failed 语义
- 修 paused 自动恢复

### 第二阶段：调度器收口
- 修会话活跃标记
- 修原子 claim
- 修监听解绑

### 第三阶段：持久化与文件协议
- 补 DB 约束
- `.part + rename`
- 完整性校验
- 清理策略收口

### 第四阶段：页面副作用与交互收口
- 清假按钮
- 清隐藏页副作用
- 补路径闭环

## 6. 前置依赖与并行关系

### 前置依赖
- 若涉及后台常驻、托盘、通知，应先完成本文件前两阶段（相关内容见 [11](./11-Electron桌面能力重构.md)）
- 若涉及新的下载接口暴露，应先确认协议边界不继续扩散（相关内容见 [12](./12-IPC与进程边界重构.md)）
- 若涉及取消、删除、异常退出后的文件与状态处理，应同步参考 [15](./15-稳定性错误恢复与安全性修正.md)

### 可并行关系
- 状态机修复 与 页面伪能力清理可并行
- DB 约束 与 文件协议可协同，但最好不要在状态机完全不稳时同时重写所有下载路径
- 页面副作用治理可与 UI 规范设计协同（见 [14](./14-UI交互一致性修正.md)）

## 7. 风险点

1. 普通下载与批量下载当前并非同一套实现，统一语义时容易出现“修一边坏另一边”
2. 文件协议改造会影响恢复策略、取消清理、已存在文件判断
3. 调度与恢复定时器治理容易牵扯应用退出链路
4. 页面隐藏副作用治理若做得太粗暴，可能误伤 keep-alive 需求

## 8. 验收要点

至少应覆盖：

- 下载状态语义可信：暂停不再等于失败，待恢复与真正待执行区分清楚
- 会话幂等可信：重复开始 / 重复恢复不会产生多个执行体
- 文件完整性可信：完成态经过明确校验，失败态不误删完整文件
- 页面副作用可信：用户不可见页面不再继续偷偷执行关键副作用
