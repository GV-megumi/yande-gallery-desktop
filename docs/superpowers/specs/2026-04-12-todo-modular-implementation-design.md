# TODO 模块化总实现方案设计

> 目标：将 [TODO.md](TODO.md) 中列出的全部需求整理成一份统一方案，按模块拆解，明确导航、图库、下载、标签管理、通用交互与更新检查的最终落地方式。

## 1. 方案范围

本方案覆盖 [TODO.md](TODO.md) 中全部 11 类需求，并将其收敛为以下 7 个实施模块：

1. 下载链路与任务去重
2. 图库 / 图集管理
3. 标签管理页重构
4. 下载页重构
5. 导航与窗口交互
6. 通用 UI 一致性
7. 检查更新修复

这份设计不是逐项罗列补丁，而是给出一套统一的功能组织方式，避免多个页面重复实现、规则冲突或后续返工。

## 2. 已确认的产品决策

以下结论已经由用户明确确认，后续实现计划与代码设计必须遵守：

### 2.1 导航结构

- Booru 左侧导航仍然只保留 **一级菜单 + 二级菜单**。
- 不在左侧继续展开所谓“三级菜单”。
- `收藏标签 + 黑名单` 合并为一个二级菜单：**标签管理**。
- `下载管理 + 批量下载` 合并为一个二级菜单：**下载**。
- 所谓“三级”改为 **右侧页面内部的页内切换**，表现方式参考 [SettingsPage.tsx](src/renderer/pages/SettingsPage.tsx#L188-L235)。
- `标签管理` 页面内部切换：`喜欢 / 黑名单`。
- `下载` 页面内部切换：`下载 / 批量下载`。
- `标签管理` 默认子页：**喜欢**。
- `下载` 默认子页：**下载**。
- **需求 10** 的“单独窗口打开”只作用于 **左侧二级菜单**。
- **需求 11** 的“点击一级菜单只切左边的二级菜单，不直接切右边页面”只作用于 **一级菜单**。

### 2.2 图库 / 图集交互

- 新增一个明确命名的 **“同步文件夹”按钮**。
- 同步按钮应带同步图标。
- 现有刷新按钮保留，但语义应明确区分于同步。
- 图集列表右键菜单新增：**编辑**。
- 图集编辑页 / 编辑弹窗至少支持：**修改图集名称**。

### 2.3 通用交互规范

- 去掉通用界面中的右上角 `X` 关闭入口。
- 保留明确的 `关闭` / `返回` 操作。
- **所有纯图标按钮统一增加 tooltip**。

### 2.4 下载去重规则

- 批量下载统一按 **下载路径 + 标签合集** 联合判重。
- 收藏标签页下载按钮不单独定义另一套去重规则，应复用批量下载任务层的统一判重逻辑。
- 如果重复点击导致同一任务已存在，可以直接提示“任务已存在”或“任务进行中”。

### 2.5 默认值

- 下载相关表单 / 列表里的 **每页数量默认值统一改为 200**。

## 3. 现状核对与关键发现

### 3.1 图库页刷新按钮当前行为

已核对 [GalleryPage.tsx](src/renderer/pages/GalleryPage.tsx#L829-L837) 与 [GalleryPage.tsx](src/renderer/pages/GalleryPage.tsx#L243-L298)：

- 当前图集详情页右上角的刷新按钮调用 `loadGalleryImages(selectedGallery.id)`。
- `loadGalleryImages` 的行为是：
  - 读取图集信息
  - 再按图集 `folderPath` 调用 `window.electronAPI.gallery.getImagesByFolder`
  - 重新显示该图集当前可见图片列表
- 它本质是 **刷新当前图集图片展示**，不是“同步文件夹结构”。

因此本方案明确区分：

- **刷新** = 重取当前图集图片列表
- **同步文件夹** = 重新扫描该图集目录状态并更新图集内容 / 元数据

### 3.2 图集扫描 / 同步相关能力已经存在一部分，可优先复用

主进程已存在“扫描子文件夹并创建图集”的 IPC 与服务：
- [handlers.ts:588-595](src/main/ipc/handlers.ts#L588-L595)
- [galleryService.ts:347-478](src/main/services/galleryService.ts#L347-L478)

同时，图片目录扫描并导入数据库的底层能力也已存在：
- [imageService.ts:520-594](src/main/services/imageService.ts#L520-L594)

这说明图库体系已经具备“扫描目录 → 发现图片 → 导入数据库 / 更新图集”的后端基础能力，只是当前前端图集页没有把它作为“同步文件夹”的明确动作暴露出来。

## 4. 模块 A：下载链路与任务去重

### 4.1 要解决的问题

对应 TODO：
- 收藏标签下载按钮防重复点击
- 有批量下载任务时提示“进行中”
- 创建成功要提示
- 批量下载本身防同任务重复启动
- 已保存任务按规则去重
- 默认每页数量改为 200

问题的根因不是前端按钮，而是：

- 多个入口可能直接触发“创建下载任务”
- 任务创建层缺少统一唯一键
- UI 层只能被动重复发请求

### 4.2 设计方案

采用 **任务层统一判重 + 前端轻量防重复反馈**。

#### 任务唯一键

统一定义：

- `下载路径`
- `标签合集`

这两个字段组合后，作为批量下载任务唯一标识。

其中“标签合集”不能直接使用原始字符串，应做标准化：
- 去掉多余空格
- 统一分隔符
- 同义空项剔除
- 排序或稳定序列化

保证语义相同的任务不会因为输入顺序不同而绕过去重。

#### 统一任务入口

以下入口都必须走同一任务服务：
- 收藏标签页点击下载
- 批量下载页手动创建
- 已保存任务重新触发
- 未来其他下载发起入口

#### UI 返回结果规范

任务服务应至少返回以下结果类型之一：
- 新任务创建成功
- 任务已存在
- 任务正在执行中
- 任务已存在且已完成（可视业务决定是否允许复用）

前端据此做统一提示：
- 新建成功 → `任务创建成功`
- 已存在 → `任务已存在`
- 正在执行 → `任务进行中`

#### 收藏标签页按钮行为

[FavoriteTagsPage.tsx](src/renderer/pages/FavoriteTagsPage.tsx#L93-L123) 所在页面不应自己实现独立去重逻辑，只做：
- 点击时短暂 loading / disabled
- 调用统一入口
- 根据返回结果提示状态

这样即使用户连续点击三次，也只会：
- 第一次创建任务
- 后两次收到“已存在/进行中”的统一响应

#### 已保存任务去重

“已保存任务”应按同样规则校验。
如果目标下载路径与标签合集一致，则不再重复下发任务，而是：
- 关联现有任务
- 或直接提示任务存在

#### 默认值统一

下载相关配置中的 `每页数量` 默认值统一为 **200**，避免：
- 收藏标签页一套默认值
- 批量下载页另一套默认值
- 保存配置再恢复后产生漂移

### 4.3 推荐实现边界

本轮方案中，推荐做到：
- 服务层判重
- 收藏标签页提示统一
- 批量下载页创建逻辑接入统一入口
- 已保存任务去重
- 默认值统一

不建议只做“按钮防抖”而不治理服务层。

## 5. 模块 B：图库 / 图集管理

### 5.1 图集改名

对应 TODO：
- 图库-图集支持修改名称，放在右键
- 编辑页支持改图集名字

#### 设计

在图集列表卡片增加右键菜单，至少包含：

编辑入口打开后，采用轻量编辑弹窗或独立编辑面板，首期至少支持：
- 修改图集名称

建议兼容未来扩展字段，但不把本轮需求扩大为“图集完整设置中心”。

### 5.2 图集同步文件夹

#### 刷新按钮职责

保留现有刷新按钮，职责定义为：
- 重新加载当前图集图片列表
- 不负责发现新图集
- 不负责重新扫描目录结构

按钮应补 tooltip，例如：
- `刷新当前图集`

#### 同步文件夹按钮职责

新增明确按钮：
- 名称：`同步文件夹`
- 图标：同步图标
- tooltip：简要说明同步作用

职责定义为：
- 扫描当前图集对应目录
- 将新增图片同步到当前图集展示 / 统计中
- 更新最后扫描时间等图集元数据

#### 复用策略（本轮必须先确认）

实现“同步文件夹”前，应优先复用现有后端能力，而不是重写扫描逻辑：
1. 评估 `scanAndImportFolder` 是否可直接用于“当前图集目录图片补扫”
2. 评估 `updateGalleryStats` 是否可直接用于同步后刷新图片数量 / 最后扫描时间
3. 如现有 `scanSubfoldersAndCreateGalleries` 仅适用于“批量发现子图集”，则为当前图集同步补一个更轻量的主进程封装，而不是直接复用整个“扫描子文件夹建图集”入口

换句话说，本轮的“同步文件夹”更像是：
- **复用现有扫描导入逻辑 + 新增一个当前图集级别的显式 UI 入口**

而不是：
- 全新发明一套图库同步机制

对于“是否一并发现子目录图集”，本轮建议拆开：
- **当前图集详情页的同步文件夹按钮**：只聚焦当前图集目录同步
- 现有“扫描子文件夹并创建图集”能力继续保留为更高层级动作，不混入当前图集按钮语义

这样用户认知最清晰：
- 刷新 = 重载显示
- 同步 = 扫描该图集目录变化
- 扫描子文件夹 = 新建图集管理动作

并且工程上也最稳，因为它直接建立在已有后端扫描能力之上。

### 5.3 图集编辑入口与编辑页范围

对应需求确认如下：
- 图集列表右键新增 `编辑`
- 编辑页 / 编辑弹窗首期必须支持：`图集名称修改`

本轮设计推荐做法：
- 图集卡片右键菜单打开“编辑图集”入口
- 编辑界面至少包含图集名称字段
- 若实现成本低，可顺带显示只读路径、递归扫描、监视目录等已有字段，但不把它们列为本轮必须修改项

这样既满足用户要求，又避免把改名需求无限扩大成完整设置中心。

### 5.4 图集自动更新

用户描述“文件夹里有新图片，图集不会主动更新”。

本轮方案建议：
1. **必做**：给出显式手动同步能力
2. **可选增强**：检查现有 `isWatching` 字段是否真的有监控链路；若有则补足联动，若无则不要在方案中承诺“自动同步已完成”

### 5.5 图集详情页信息显示

图集详情页现有信息区（见 [GalleryPage.tsx:840-899](src/renderer/pages/GalleryPage.tsx#L840-L899)）已包含：
- 名称

## 8. 模块 E：导航与窗口交互

### 8.1 一级菜单点击行为

当前 [App.tsx](src/renderer/App.tsx#L177-L205) 使用 `selectedKey / selectedSubKey / selectedBooruSubKey / selectedGoogleSubKey` 管理导航状态。

需求 11 要求：
- 点击一级菜单时，只更新左侧二级菜单集合
- 不立即强制切换右侧页面内容

#### 方案

一级菜单点击后：
- 更新当前 section（gallery / booru / google）
- 保留右侧当前选中页，直到用户主动点二级菜单

这能避免：
- 用户只是想看某一级菜单下有哪些子项，却被强制跳页
- 合并型二级菜单重构后出现误切换

### 8.2 二级菜单右键“单独窗口打开”

需求 10 仅作用于二级菜单。

#### 现状

[SubWindowApp.tsx](src/renderer/SubWindowApp.tsx#L14-L29) 目前只支持：
- tag-search
- artist
- character

#### 方案

扩展子窗口体系，支持“通用二级菜单页面路由”。

二级菜单右键增加：
- `单独窗口打开`

子窗口路由应包含：
- 一级 section
- 二级 key
- 若该二级页有默认子页，也携带默认子页状态

例如：
- `booru:tag-management?tab=favorite`
- `booru:download?tab=downloads`
- `gallery:galleries`

这样可以复用现有子窗口框架，而不是新增特例逻辑。

### 8.3 固定菜单与缓存兼容

[App.tsx:202-301](src/renderer/App.tsx#L202-L301) 已存在 pinned item 与 mounted cache 机制。合并导航后，需要同步更新：
- 固定菜单 metadata
- 激活页面标题
- pinned 缓存 key 规则

否则合并后的 `标签管理` / `下载` 在固定到底部时可能与旧 key 冲突。

#### 迁移策略（本轮需明确执行）

二级菜单合并后，旧数据不能只停留在“有冲突风险”描述，必须给出迁移方案。建议采用：

1. **配置层映射迁移**
   - 旧 pinned key `favorite-tags`、`blacklisted-tags` → 新 key `tag-management`
   - 旧 pinned key `downloads`、`bulk-download` → 新 key `download`

2. **默认子页映射**
   - 若旧固定项来自 `favorite-tags`，迁移后默认激活 `tag-management?tab=favorite`
   - 若旧固定项来自 `blacklisted-tags`，迁移后默认激活 `tag-management?tab=blacklist`
   - 若旧固定项来自 `downloads`，迁移后默认激活 `download?tab=downloads`
   - 若旧固定项来自 `bulk-download`，迁移后默认激活 `download?tab=bulk`

3. **mounted cache 直接失效重建**
   - 已挂载缓存的旧 pinId 不做复杂兼容，进入新版本后直接按新 key 重新挂载
   - 这样可以避免旧缓存内容和新页面结构不一致导致 UI 残留

4. **标题映射同步更新**
   - 页面标题与 pin 标签统一使用合并后的菜单名称
   - 具体子页标题由右侧页内导航状态决定

5. **旧路由最小兼容**
   - 若代码中仍存在直接跳转到旧二级页 key 的逻辑，不应该直接报错，而应在进入时映射到新页面与默认 tab

本轮推荐原则是：
- **持久化配置做映射迁移**
- **运行时缓存直接清旧建新**

这样实现成本最低，且用户已有固定菜单不会无故丢失。

### 8.4 设置页风格复用边界

`标签管理` / `下载` 的页内子导航只复用设置页的交互风格，不要求把页面完全改造成设置页代码结构。复用的重点应是：
- 页面内部子导航形式
- 分组 / 面板视觉层次
- 当前子页高亮与切换逻辑

不强制要求直接抽象成与设置页同一套组件，除非实现时发现足够可复用。

### 9.5 一致性规范落地

本方案将把以下规范写入正式文档：

## 10. 模块 G：检查更新报错

### 10.1 现状

前端设置页已存在更新检查入口：
- [SettingsPage.tsx:213-235](src/renderer/pages/SettingsPage.tsx#L213-L235)

preload 已暴露：
- [preload/index.ts:543-551](src/preload/index.ts#L543-L551)

这说明问题更可能在：
- IPC 通道实现
- 更新服务
- 返回结构与前端预期不一致
- 异常未被友好处理

### 10.2 方案

按整条链路排查并固化返回结构：
1. 前端点击能稳定调用到 IPC
2. preload 暴露方法与主进程接口一致
3. 更新服务返回统一结构：
   - currentVersion
   - latestVersion
   - hasUpdate
   - releaseUrl
   - releaseName
   - publishedAt
   - error
   - checkedAt
4. 失败时前端不只显示技术性异常字符串，而是给出可理解提示

### 10.3 验收目标

点击“检查更新”后，用户至少能得到：
- 成功：当前版本 / 最新版本 / 是否可更新 / 发布地址
- 失败：明确失败原因或最少是稳定的失败提示

## 12. 实施约束

为避免范围失控，本次设计明确以下边界：

- 不把 `标签管理` / `下载` 合并扩展成左侧多层菜单系统重构
- 不把图集编辑页默认扩展成全量配置中心，首期只要求改名，其他字段视实现成本顺带整合
- 不把“图集自动更新”默认承诺为全自动监听完成，而是先保证明确可用的手动同步能力
- 不以按钮防抖代替服务层任务去重
- 不因为页面合并而删除原有功能能力

## 13. 方案结论

本次 TODO 的正确落地方式不是逐项打补丁，而是建立以下统一结构：

- **Booru 左侧二级菜单收口**：标签管理 / 下载
- **右侧页内子导航承载原“三级”内容**
- **批量下载任务服务层统一去重**
- **图库页刷新 / 同步语义彻底拆分**
- **图集右键编辑入口显式化**
- **所有纯图标按钮统一 tooltip**
- **关闭交互统一去掉 X**
- **更新检查走标准化返回结构**

这样既能覆盖 TODO 中所有条目，又能确保未来维护时规则统一、页面行为一致、入口清晰。
