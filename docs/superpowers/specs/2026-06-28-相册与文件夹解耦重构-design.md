# 相册与文件夹解耦重构 — 需求设计说明

- 日期：2026-06-28
- 范围：本地图库（相册 / Gallery）的数据模型、扫描行为、设置页与「相册信息」对话框、跨机器迁移
- 架构选型：**方案 B（彻底解耦，现在就引入 `gallery_folders` + `gallery_images` 关联表，原生支持一相册对多文件夹）**

---

## 1. 背景与问题

### 1.1 现状模型（重构前）

- `galleries` 表里 `folderPath` 为 `TEXT NOT NULL UNIQUE`，**相册本身就等于一个文件夹**；`isWatching / recursive / extensions` 也直接挂在相册行上。
- **图片归属是隐式的**：`images.filepath`（全局唯一）落在某相册 `folderPath` 前缀下，就算属于该相册。没有任何 `gallery↔image` 外键，全靠 `LIKE 'folderPath%'` 前缀匹配。
- 添加一个父文件夹时，`scanSubfoldersAndCreateGalleries` **递归遍历所有层级**，对每个有图片的子目录建一个 `recursive=false` 的相册。
- booru 收藏标签批量下载会自动建相册（`booru_favorite_tag_download_bindings.galleryId` 关联），与用户手动添加的相册**混在同一张表、同一个列表里**，没有任何区分标记。

### 1.2 两个待修复的问题

1. **bug #1：程序自建相册污染设置页。** 设置页「图库文件夹」把所有相册（含 booru 自动建的）逐行列出，杂乱。
2. **bug #2：动态扫描过深。** 例如 `M:\booru` 自身没有图片、子文件夹全是图片，期望是"扫描该文件夹和一级子文件夹"，但当前实现递归到所有层级，**任何深层子目录只要有图片就被收编成相册并展示**。

### 1.3 用户提出的方向与更深层动机

- 设置页**去掉相册列表**，只保留「指定文件夹进行扫描」这一个动作。
- 把相册管理收进**「相册信息」对话框**，并赋予它交互能力：自动扫描开关、立即扫描、更改文件夹路径、解绑。
- 核心理念：**「相册是相册、文件夹是文件夹」**——相册是长期实体，文件夹是它能获得/失去的关联；**一个相册可以对应多个文件夹**。
- 「解绑」是解除"相册↔文件夹"的关联，**不是**把文件夹拉黑（黑名单 = `gallery_ignored_folders`，已存在，是另一套）。
- **更深层动机（核心驱动）**：把 DB**连同文件一起**搬到另一台机器时，能**保留相册结构**、只把文件夹根路径重指到新位置，而不是"删档重建"。旧模型做不到（相册就是不可改的 `folderPath`），这正是采用方案 B 的根本原因。

## 2. 目标与非目标

### 2.1 目标

- 把"相册"与"文件夹"拆成两个独立概念，引入显式的 **文件夹绑定**（`gallery_folders`，1 相册 : N 文件夹）与显式的 **图片成员**（`gallery_images`）。
- 修复扫描深度：只在**一级子文件夹**层面建相册。
- 设置页只留「扫描文件夹」动作；相册信息对话框升级为**多文件夹管理面板**。
- 扫描遇到**同名相册**时提示用户选择"合并到现有相册"或"新建独立相册"。
- 把现有数据**无损迁移**到新模型（图片归属一张不丢）。
- **DB 跨机器可迁移（核心目标）**：文件随库一起搬到新机器、目录结构不变、仅根路径/盘符变化时，相册、图片、文件夹绑定、booru 关联可**整体无损重定位**到新路径前缀——**不重扫、不丢**图片身份/标签/封面/下载关联；结构变化或重新获取文件的场景，则用逐个相册的「更改路径 + 重扫」。

### 2.2 非目标（本期明确不做）

- **不做**真正的文件系统实时监视器；"自动扫描"只是**进入相册时扫一次**。
- **不改**「最近图片 / 所有图片」两个全局 Tab 的语义（它们仍直接读 `images` 全表）。
- **不改**标签系统、缩略图生成管线本身。
- **不放开** `gallery_folders.folderPath` 的全局唯一约束（即仍保持"一个文件夹只属于一个相册"，避免图片被两个相册重复收编；如未来确有需要再单独立项）。
- 跨机器重定位**不做自动探测/猜测新根路径**，由用户**显式给出**前缀映射。

## 3. 术语

| 术语 | 含义 |
| --- | --- |
| 相册 Gallery | 纯逻辑实体（`galleries` 行），有名字、封面、统计，但**不直接持有文件夹路径**。 |
| 文件夹绑定 Folder binding | `gallery_folders` 行，把一个磁盘文件夹关联到一个相册，并带该文件夹的 `recursive / extensions`。 |
| 图片成员 Membership | `gallery_images` 行，表示某 `images` 行属于某相册。 |
| 孤儿图片 Orphan image | 不再属于**任何**相册（无任何 `gallery_images`）的 `images` 行，应被回收。 |
| 解绑 Unbind | 删除一条文件夹绑定及其带来的图片成员（并回收孤儿），**保留相册记录**，不进黑名单。 |
| 黑名单 / 忽略列表 | `gallery_ignored_folders`，已存在；删除整个相册时使用，避免再次扫描父目录自动重建。与"解绑"无关。 |
| 根目录重定位 Relocate | 跨机器迁移时，按 `旧前缀→新前缀` 无损改写库内所有路径字符串，使其指向新机器上的文件。 |

## 4. 目标数据模型

> `images` 表保持不变（全局、`filepath UNIQUE`、由扫描去重创建）。

### 4.1 `galleries`（重建：去掉文件夹相关列，新增 `autoScan`）

```sql
galleries (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  name          TEXT    NOT NULL,
  coverImageId  INTEGER,                 -- FK images(id) ON DELETE SET NULL
  imageCount    INTEGER NOT NULL DEFAULT 0,   -- 缓存：= COUNT(gallery_images WHERE galleryId)
  lastScannedAt TEXT,
  autoScan      INTEGER NOT NULL DEFAULT 1,    -- 原 isWatching 改名：进入相册时自动扫一次
  createdAt     TEXT    NOT NULL,
  updatedAt     TEXT    NOT NULL
)
-- 移除：folderPath / isWatching / recursive / extensions
```

### 4.2 `gallery_folders`（新表：相册 ↔ 文件夹，1:N）

```sql
gallery_folders (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  galleryId  INTEGER NOT NULL,           -- FK galleries(id) ON DELETE CASCADE
  folderPath TEXT    NOT NULL UNIQUE,    -- 全局唯一：一个文件夹只属于一个相册
  recursive  INTEGER NOT NULL DEFAULT 1, -- 每个文件夹独立的递归设置
  extensions TEXT,                       -- JSON 数组；为空时用默认扩展名
  createdAt  TEXT    NOT NULL,
  updatedAt  TEXT    NOT NULL,
  FOREIGN KEY (galleryId) REFERENCES galleries(id) ON DELETE CASCADE
)
CREATE INDEX idx_gallery_folders_galleryId ON gallery_folders(galleryId);
```

### 4.3 `gallery_images`（新表：相册 ↔ 图片成员）

```sql
gallery_images (
  galleryId INTEGER NOT NULL,            -- FK galleries(id) ON DELETE CASCADE
  imageId   INTEGER NOT NULL,            -- FK images(id)   ON DELETE CASCADE
  addedAt   TEXT    NOT NULL,
  PRIMARY KEY (galleryId, imageId)
)
CREATE INDEX idx_gallery_images_imageId ON gallery_images(imageId);  -- 反查：某图片属于哪些相册
```

## 5. 成员关系语义与操作原语

> 这是 B 的核心。所有"相册看到哪些图片""删哪些图片"都改为读写 `gallery_images`，不再用 `filepath` 前缀。

### 5.1 不变量

- 图片属于相册 G ⟺ 存在 `gallery_images(G, imageId)`。
- 某相册的 `imageCount` = `COUNT(gallery_images WHERE galleryId = G)`。
- 某相册详情列表 = `images JOIN gallery_images WHERE galleryId = G`。
- 由于 `gallery_folders.folderPath` 全局唯一，扫描产生的每张图片在**正常情况下只归属一个相册**；`gallery_images` 仍允许"同图多集"，为未来与嵌套绑定边界留出空间。

### 5.2 操作原语（行为定义，非代码）

| 原语 | 效果 |
| --- | --- |
| `scanFolderIntoGallery(galleryId, folderPath, recursive, extensions)` | 遍历文件夹（按 recursive）→ 每个图片文件：保证 `images` 行存在（filepath 去重）+ 保证 `gallery_images(galleryId,imageId)` 存在 → 刷新 `imageCount/lastScannedAt` → 刷新 app:// 登记表。 |
| `createGalleryFromFolder(folderPath, recursive=true)` | 新建相册（name = 文件夹名，重名去重）+ 插一条 `gallery_folders` + `scanFolderIntoGallery`。 |
| `bindFolder(galleryId, folderPath, recursive=true)` | 给已有相册**加一条**文件夹绑定 + 扫描入成员。用于「+ 添加文件夹」与「合并到现有相册」。 |
| `unbindFolder(galleryId, folderPath)` | 删该 `gallery_folders` 行 → 删它带来的 `gallery_images` 成员 → **孤儿回收**（见 5.3）→ 刷新登记表。**相册记录保留**；**不**写黑名单。 |
| `changeFolderPath(galleryId, oldPath, newPath)` | = `unbindFolder(old)` + `bindFolder(new)`，相册与其它文件夹绑定不受影响。**适用于换文件来源/结构变化**；会重扫，代价见 §5.4。 |
| `deleteGallery(id)`（沿用现有、但走新表） | 删相册 → 级联删其所有 `gallery_folders` 与 `gallery_images` → 孤儿回收 → **把每个被删文件夹写入黑名单**（避免扫父目录自动重建）。这是唯一会拉黑的操作，入口在图库页右键菜单。 |
| `scanParentFolder(rootPath)` | 设置页「扫描文件夹」：只遍历**一级子文件夹**，对每个执行幂等的"建/合并/跳过"（见 §6）。 |
| `relocateRoot(mappings[])` | 跨机器迁移：按一组 `旧前缀→新前缀` 在一个事务内**无损**改写库内所有路径字符串（见 §11）。**不删图片、不重扫**，保留全部成员/标签/封面/booru 关联；改写后刷新登记表。 |

### 5.3 孤儿回收（解绑 / 改路径 / 删相册时）

删除成员后，对受影响的每个 `imageId`：若它**不再属于任何相册**（无任何 `gallery_images`），则回收——删 `images` 行（`image_tags` 随 FK CASCADE）、删缩略图、复位对应 `booru_posts`（`downloaded=0, localPath=NULL, localImageId=NULL`）。该回收语义与重构前 `deleteGallery` 对图片的清理一致；磁盘原文件始终保留。
> 1:1 常态下，被解绑文件夹的图片只属这一个相册 → 等价于"从库里删掉这些图片"，符合用户对"删除相册图片信息"的预期；多文件夹/共享时不会误删别的相册仍在用的图。

### 5.4 「更改路径」与「重定位」的取舍（重要）

- **`changeFolderPath`（逐个相册，对话框内）** = 解绑旧 + 绑新 + 重扫：解绑会回收旧图片行，于是**封面重置、绑定在旧图片身份上的本地标签不保留**（新扫描按新 filepath 建新 `images` 行）。适用于"换了文件来源 / 目录结构变了"。
- **`relocateRoot`（跨机器整体，设置页内）** = 纯路径字符串前缀重写：**无损**，图片身份/标签/封面/booru 关联全部保留，且**不重扫**。**搬库（文件一起搬、结构不变）首选此路径。**

## 6. 扫描行为（修复 bug #2 + 同名碰撞合并 G）

### 6.1 深度修复

设置页「扫描文件夹」选定父目录 `R` 后，**只遍历 `R` 的一级子文件夹**（不再递归到更深层建相册）。对每个有图片的一级子文件夹 `F`，创建的相册/绑定为 **`recursive = true`**，因此 `F` 内部更深层的图片会被纳入**该相册**（不丢图），但深层目录本身**不再**单独成集。若 `R` 自身有直接图片，`R` 也作为一个相册（同样 recursive=true）。

### 6.2 幂等 + 同名碰撞处理（G）

遍历每个一级子文件夹 `F`（已在黑名单或无图片的跳过）后：

1. **`F.path` 已存在于 `gallery_folders`** → 静默跳过（已导入，保证幂等）。
2. **存在同名相册（名字 = `F` 文件夹名，但路径不同）** → 记为一次**碰撞**，交由用户选择：
   - **合并**：`bindFolder(existingGallery, F.path)`——把 `F` 加入该现有相册的文件夹列表。
   - **新建**：`createGalleryFromFolder(F.path)`（名字按现有规则去重，如 `name (2)`）。
3. **以上都不是** → `createGalleryFromFolder(F.path)`。

**批量碰撞 UX（本期设计，可在评审推翻）**：一次扫描中所有碰撞**收集后用一个汇总弹窗**呈现，逐项选择"合并/新建"，并提供"全部合并 / 全部新建"快捷应用，避免逐个弹窗。扫描结束给出汇总提示（新增 N 个相册、合并 M 个文件夹、跳过 K 个）。

## 11. 数据迁移、回填与跨机器重定位

### 11.1 一次性 Schema 迁移与回填（**最高风险**）

放在 `database.ts` 兼容迁移流程内，幂等：

1. 建 `gallery_folders`、`gallery_images` 两张新表。
2. **`galleries` 重建**：新表结构（§4.1），从旧表复制 `id/name/coverImageId/imageCount/lastScannedAt/createdAt/updatedAt`，`autoScan ← 旧 isWatching`；保留 `coverImageId` 等 FK 与索引、保住 `id`（其它表按 `galleryId` 引用）。
3. 对每个旧相册插一条 `gallery_folders`：`folderPath/recursive/extensions ← 旧列`。
4. **回填 `gallery_images`**（关键）：对每个旧相册，按**旧的 recursive 感知前缀逻辑**（即现 `deleteGallery` 用的匹配：recursive=1 → `filepath LIKE 'F{sep}%' OR filepath = 'F'`；recursive=0 → `LIKE 'F{sep}%' AND NOT LIKE 'F{sep}%{sep}%'`）选出图片，逐一插入成员行。这一步等价复制现有"图片→相册"归属，**一张不能丢**。
5. 迁移需**幂等**：以"新表是否已存在/已回填"为判据，重复启动不重复插入、不破坏数据。

> 嵌套相册边界：若历史上同时存在 `M:\booru` 与 `M:\booru\x` 两个相册，回填后深层图片会同时进入两个相册的成员（与旧前缀匹配下两个 `getImagesByFolder` 都能查到它一致）。需要"某图片归属单一相册"的反查（如 `invalid_images.galleryId`、`deleteImage` 找相册）按**最深 folderPath 优先**选择，沿用现有 `ORDER BY LENGTH(folderPath) DESC` 行为。

### 11.2 跨机器重定位适用场景

DB 与**文件一起**搬到新机器、目录**结构不变、仅根路径/盘符变化**（如 `N:\hk\yande_download\*` → `D:\art\*`，`M:\booru\*` → `E:\booru\*`）。此时应**整体无损重定位**，不重扫、不丢任何 per-image 数据。结构变化或重新获取文件的场景，仍走 §8 对话框里逐个相册的「更改路径 + 重扫」。

### 11.3 重定位原语 `relocateRoot(mappings[])`

输入：一组或多组 `{ oldPrefix, newPrefix }`（支持多根一次处理）。在**单个事务**内，对下列字段做**边界感知**的前缀改写（仅当 `path === oldPrefix` 或 `path.startsWith(oldPrefix + sep)` 才替换；经 `normalizePath` 归一；win32 大小写不敏感）：

| 表.字段 | 作用 |
| --- | --- |
| `gallery_folders.folderPath` | 相册的文件夹绑定 |
| `images.filepath` | 图片实际路径（保留图片身份/标签/封面） |
| `booru_posts.localPath` | booru 下载追踪 |
| `booru_favorite_tag_download_bindings.downloadPath` | booru 收藏下载目标目录 |
| `gallery_ignored_folders.folderPath` | 黑名单路径保持有效 |

**无损**：不删除任何 `images`/成员行、不重扫，`gallery_images`、`image_tags`、`coverImageId`、booru 关联全部原样保留。改写后刷新 `galleryRootRegistry`（app:// 白名单）。

### 11.4 重定位预检与安全

- **Dry-run 预览 + 确认**：应用前给出各表将改写的行数；并检测改写后是否与既有路径产生 `UNIQUE` 冲突（`images.filepath` / `gallery_folders.folderPath`，即 newPrefix 与目标机已存在路径重叠）——有冲突则**整体中止并报告**，不做部分改写。
- **事务 + 幂等**：全部改写在一个事务内提交；对已应用过的映射重复运行为空操作（不再有匹配 oldPrefix 的行）。多根映射应一次性提交，避免中间态。
- **缩略图**：缩略图是按源 `filepath` 派生的缓存；前缀改写后旧缩略图可能失配 → **惰性重建**即可，无数据损失。

### 11.5 重定位 UI 入口

设置页新增 **「重定位根目录」** 维护入口（与「已忽略文件夹」同区）：填写一组 `旧前缀 → 新前缀` → 预览影响行数与冲突 → 确认应用。

### 11.6 丢失文件夹检测

进入图库页 / 打开相册信息时，对相关 `gallery_folders.folderPath` 做存在性检查；不存在则标记 **「文件夹丢失」**：相册卡片加角标、对话框对应文件夹行显示提示，并引导去「重定位根目录」或「更改路径」。**绝不**因路径缺失自动删除任何图片或成员（防离线盘 / 未挂载盘误删）。

## 13. 边界与风险

- **回填错误 → 图片消失**：§11.1 步骤 4 是全局最高风险，必须有针对性测试（含 recursive=0/1、嵌套、空相册等）。
- **booru 自建相册**被解绑/改路径/删除 → 复位下载追踪，需二次确认与提示（§9）。
- **解绑 ≠ 拉黑**：解绑后再扫描父目录会**重新收编**该子文件夹（因为它不在黑名单）——这是预期行为，文案需让用户理解二者区别。
- **`galleries` 重建**是表重建迁移，需保住 `coverImageId` 等 FK 与索引、保证幂等。
- **全局唯一 folderPath**：「合并/添加文件夹」绑定一个已被别的相册占用的文件夹时要拒绝并提示。
- **重定位前缀碰撞**：newPrefix 与目标机既有路径重叠会触发 `UNIQUE` 冲突 → 由 §11.4 预检拦截、整体中止；多根映射一次性给全，避免分批改写产生中间态。
- **重定位边界匹配**：必须按路径边界匹配（`N:\art` 不能误伤 `N:\artists`）。
