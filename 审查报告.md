## 2. 总体结论

这个项目不是“基本可用但还需优化”，而是**功能面很大，但完成度不均，桌面化能力明显不达标，且边界设计已经出现失稳信号**。

直接结论如下：

- **功能残缺**：多个页面和设置项已经暴露给用户，但能力并未真正闭环。
- **体验粗糙**：导航模型混乱，关闭行为不统一，失败态和反馈严重不足。
- **架构债明显**：preload 成了 God API，IPC 协议来源不唯一，shared / preload / main 已发生类型漂移。
- **下载链路薄弱**：普通下载与批量下载都能跑，但恢复、去重、取消、状态一致性存在明显漏洞。
- **并发不足且调度设计不稳**：批量下载会话生命周期管理存在设计级 bug，重复恢复和重复领取任务的风险是实打实的。
- **Electron 桌面能力仍未完成阶段验收**：tray、关闭不退出、后台常驻、通知、single-instance 恢复、基础崩溃治理的代码主链路已补齐并有自动化支撑，但真实桌面环境的人机验证证据仍缺；开机自启仍未实现。
- **安全边界偏松且有硬伤**：`app://` 自定义协议当前可映射任意本地路径，`system:open-external` 也没有做协议/目标校验。
- **敏感信息最小暴露原则失守**：代理密码、Google `clientSecret`、站点 `passwordHash/apiKey` 会越过 main/preload 边界直接抵达 renderer，且保存配置时还会广播到全部窗口。
- **现代化程度不足**：大量自研逻辑已经超出“轻量实现”的安全区，开始显露出应由成熟库接管的信号。
- **测试通过不能证明产品合格**：当前测试更像“单元与局部行为通过”，不足以覆盖桌面应用体验与运行时一致性。

一句话总结：

> 这是一个已经堆出大量功能面的 Electron 应用，但核心桌面能力、下载体系稳定性、协议边界和交互一致性都还没有达到一个成熟桌面产品应有的标准；桌面主链路虽已补齐，真实桌面验收仍未完成。它不是不能用，而是**离“合格的桌面图库/下载器”还有明显距离**。

### P0-01 批量下载会话活动标记过早清除
- 严重级别：P0
- 模块/页面：批量下载服务
- 现象：会话启动后，`downloadLoop()` 在后台继续运行，但函数 `finally` 立即执行，把 `activeDownloadSessions` 删除。
- 证据：
  - [src/main/services/bulkDownloadService.ts:1117-1123](src/main/services/bulkDownloadService.ts#L1117-L1123)
  - [src/main/services/bulkDownloadService.ts:1232-1239](src/main/services/bulkDownloadService.ts#L1232-L1239)
- 为什么这是问题：
  - 防重启标记提前失效；
  - 暂停/取消拿不到 controller；
  - 恢复逻辑会误以为会话未运行。
- 复现方式/触发条件：
  - 启动某个批量下载会话；
  - 再次触发恢复/开始；
  - 或尝试暂停/取消。
- 根因分析：
  - 会话生命周期绑定错了层级，绑定在 `startDownloadingSession()` 退出而不是 `downloadLoop()` 完结。
- 用户影响：重复下载、取消无效、状态抖动。
- 技术影响：状态一致性崩坏，后续修复越来越难。
- 修复建议：
  - 把活跃标记生命周期绑定到下载循环真实结束；
  - 监听器和 controller 在循环退出后再清理；
  - 为会话引入单一状态机。
- 是否建议用成熟库替代：建议部分替代
- 候选库：`p-queue`（执行池），保留 DB 状态层
- 收益：减少自研调度 bug
- 成本/风险：中；需要重构会话调度核心

### P0-02 批量下载记录领取非原子，重复调度会重复领取同一记录
- 严重级别：P0
- 模块/页面：批量下载服务
- 现象：调度器先查 `pending`，后异步把记录改成 `downloading`，中间没有原子 claim。
- 证据：
  - [src/main/services/bulkDownloadService.ts:1193-1223](src/main/services/bulkDownloadService.ts#L1193-L1223)
  - [src/main/services/bulkDownloadService.ts:1254-1259](src/main/services/bulkDownloadService.ts#L1254-L1259)
- 为什么这是问题：一旦出现重复调度器，同一记录会被两个下载流程同时领取。
- 复现方式/触发条件：
  - 会话重复恢复；
  - 或重复调用 start；
  - 记录仍是 pending。
- 根因分析：数据库层没有“抢占任务”语义，完全依赖单调度器假设。
- 用户影响：重复下载、文件覆盖、统计失真。
- 技术影响：状态污染，失败重试与补偿逻辑全都不可靠。
- 修复建议：
  - 增加原子 claim；
  - 用事务或条件更新抢占记录；
  - 会话内执行池必须单实例化。
- 是否建议用成熟库替代：部分替代
- 候选库：执行池用 `p-queue`，限流层用 `Bottleneck`

### P0-03 `app://` 协议可映射任意本地路径
- 严重级别：P0
- 模块/页面：主进程协议处理、本地文件访问
- 现象：自定义 `app://` 协议当前直接把 URL 映射为本地文件系统路径，没有受控根目录限制。
- 证据：
  - [src/main/index.ts:12-20](src/main/index.ts#L12-L20)
  - [src/main/index.ts:32-46](src/main/index.ts#L32-L46)
  - [src/renderer/utils/url.ts:8-30](src/renderer/utils/url.ts#L8-L30)
- 为什么这是问题：这不是单纯“边界偏松”，而是把本地文件系统暴露成渲染层可消费的协议入口。
- 用户影响：一旦渲染层出现危险 URL 装配、XSS 或未来引入不可信内容，本地文件读取面会被放大。
- 技术影响：后续所有图片、本地预览、web 内容混用都会叠加风险。
- 修复建议：
  - 协议层只允许映射受控目录（如缓存、下载、图库根目录）；
  - 对路径做 canonicalize 和 root containment 校验；
  - 非白名单路径直接拒绝，不要继续 `callback({ path })`。

### P0-06 用户主动暂停/取消会把批量下载记录污染为 failed
- 严重级别：P0
- 模块/页面：批量下载服务
- 现象：暂停/取消通过 `AbortController.abort()` 中止请求，但记录级错误处理把包含 `aborted` 的情况视为可重试下载错误；超过重试后，记录最终仍会被写成 `failed`。
- 证据：
  - [src/main/services/bulkDownloadService.ts:1638-1667](src/main/services/bulkDownloadService.ts#L1638-L1667)
  - [src/main/services/bulkDownloadService.ts:1695-1701](src/main/services/bulkDownloadService.ts#L1695-L1701)
- 为什么这是问题：用户主动操作与真实网络失败被混在一起，恢复语义、失败统计和重试逻辑都会失真。
- 复现方式/触发条件：
  - 启动批量下载；
  - 在有记录进行中时执行暂停或取消；
  - 观察记录状态和失败计数。
- 根因分析：中断语义没有单独建模为 `paused` / `cancelled`，而是复用了统一失败路径。
- 用户影响：用户主动暂停后仍会看到失败记录，难以判断到底是自己暂停还是任务异常。
- 技术影响：状态机失真，后续恢复、补偿和统计都不可靠。
- 修复建议：
  - 为记录级状态补齐 `paused` / `cancelled` 语义；
  - 中断异常不应走通用失败分支；
  - 会话级暂停/取消与记录级最终状态必须对齐。
- 是否建议用成熟库替代：否
- 候选方向：保留 DB 状态层，自行修正状态机即可
- 收益：恢复语义清晰，统计可用
- 成本/风险：中；要重梳理会话与记录状态流转

- 为什么这是问题：这不是普通的“preload 太胖”，而是**敏感信息最小暴露原则已经失守**。代理密码、Google `clientSecret`、站点认证材料不应进入所有 renderer，更不该在保存配置时被广播，或随备份文件整包导出。

- 候选方向：这是边界设计问题，应先在现有架构内收紧
- 收益：显著降低敏感信息暴露面
- 成本/风险：中；需要梳理当前前端对凭证字段的依赖

### P1-09 cancel 会删除 pending 记录对应文件，在特定时序下可能误删完整文件

  - 长期看应把“临时文件状态”和“任务状态”分离。
- 是否建议用成熟库替代：否
- 候选方向：修正取消清理策略即可
- 收益：降低边界时序下的误删风险
- 成本/风险：低到中

### 6.2 串行化在哪里
- 启动流程串行化，窗口被迫等待初始化完成。
- 批量 dry-run 顺序扫页、重试、sleep，启动一个会话本身就可能长时间阻塞 IPC。

### 6.3 调度不合理点
- 会话活动标记提前清掉；
- 记录领取非原子；
- paused 会话被自动恢复；
- 普通下载 pauseAll 把任务写回 pending；
- 普通下载单任务 pause 会继续落入 failed；
- `networkScheduler` 只有注册没有注销，存在监听器积累风险；
- 启动阶段 2 秒后自动恢复的定时器没有接入退出清理，关闭应用时仍可能与退出过程竞争。
  - 证据：[src/main/services/init.ts:60-105](src/main/services/init.ts#L60-L105)、[src/main/index.ts:97-101](src/main/index.ts#L97-L101)

### 6.4 后台能力当前缺口
- tray / close-to-tray / 后台常驻 / 下载完成通知的代码主链路已补齐，且已有自动化支撑；
- 但真实桌面环境的人机验证证据仍未补齐，当前只能判定为阶段验收待补测；
- 开机自启仍缺失；
- 崩溃后的运行中补偿策略仍未形成完整产品级方案。

### 6.5 最值得优先优化的点
1. 修复批量会话生命周期与原子 claim
2. 修复普通下载与批量下载的“暂停/恢复”语义
3. 把 dry-run 改成后台异步阶段
4. 让窗口先显示，再延迟做非关键初始化
5. 引入成熟执行池/限流器

## 9. 联网对标与替代方案建议

### 9.1 先进实践摘要
- 下载器/媒体管理器类 Electron 应用，2026 年已把“托盘 + 后台运行 + 关闭不退出 + 会话级通知”当作基础能力。
- 图片密集型列表，懒加载已不够，通常会做**严格虚拟化**。
- 队列/限流通常不再完全手写 while-loop，而是引入成熟调度器，再把业务状态持久化在 DB。
- 生产问题定位依赖结构化日志与崩溃监控，而不是 console。

### 9.2 联网对标结论（带来源）

#### 9.2.1 Electron 安全与生命周期
- 官方安全清单明确建议：不要关闭 `webSecurity`、限制 navigation、限制新窗口、谨慎使用 `<webview>`、不要滥用 `allowpopups`、在主进程统一收口权限和窗口行为。
- 当前项目差距：
  - 主窗口启用了 `webSecurity: false` 与 `webviewTag: true`：[src/main/window.ts:62-68](src/main/window.ts#L62-L68)
  - Google 集成页直接使用 `<webview allowpopups="true">`：[src/renderer/pages/GoogleDrivePage.tsx:44-48](src/renderer/pages/GoogleDrivePage.tsx#L44-L48)
  - 主进程未见 `setWindowOpenHandler` / `will-navigate` / `will-attach-webview` 约束
- 是否值得引入：值得，且应优先做
- 收益：直接收紧当前最明显的安全暴露面，也为后续后台常驻和外部页面集成打基础
- 改造成本：中
- 风险点：收紧后可能暴露历史依赖 `webSecurity: false` 的兼容问题
- 来源：
  - Electron Security Tutorial: https://www.electronjs.org/docs/latest/tutorial/security
  - Electron webContents API: https://www.electronjs.org/docs/latest/api/web-contents
  - Electron Tray Tutorial: https://www.electronjs.org/docs/latest/tutorial/tray
  - Electron app API: https://www.electronjs.org/docs/latest/api/app
  - Electron Notifications: https://www.electronjs.org/docs/latest/tutorial/notifications

#### 9.2.2 React 图片密集型列表 / 虚拟化
- 先进方案：图片密集型应用不再只靠 lazy-load，而是把渲染范围限制在可视区及 overscan；对 variable-size / grid / masonry 场景优先使用成熟虚拟化方案。
- 当前项目差距：当前图库和 Booru 列表仍以整批缩略图 + 普通渲染为主，没有真正虚拟化。
- 更适合当前项目的候选：
  - `react-virtuoso`：更适合快速获得 variable-size / grid 场景收益
  - `@tanstack/react-virtual`：更适合保留现有复杂布局并做长期深度定制
- 是否值得引入：值得，高优先级
- 收益：滚动性能、内存占用、DOM 压力、可视区内网络请求都会明显改善
- 成本：中到偏高
- 风险点：当前 CSS columns / 瀑布流实现与虚拟化不天然兼容，可能需要改布局策略
- 来源：
  - TanStack Virtual: https://tanstack.com/virtual
  - TanStack Virtual Introduction: https://tanstack.com/virtual/latest/docs/introduction
  - React Virtuoso: https://virtuoso.dev/
  - react-window: https://github.com/bvaughn/react-window
  - virtua: https://github.com/inokawa/virtua

#### 9.2.3 队列 / 限流 / 后台任务
- 先进方案：把“持久化任务状态”和“运行时并发/限流”分层处理，而不是把所有语义都塞进手写 while-loop。
- 当前项目差距：当前下载/批量下载以自研状态机 + axios stream + 简单并发控制为主，已出现状态一致性和调度器生命周期问题。
- 候选方案：
  - `Bottleneck`：更适合多站点限流、动态退避、per-site 策略
  - `p-queue`：更适合轻量 Promise 执行池收口
  - `BullMQ`：能力最强，但对当前桌面单机应用过重，不建议现阶段直接上
- 是否值得引入：值得，优先从 `Bottleneck` 或 `p-queue` 开始
- 收益：能显著减少自研调度 Bug，并为多站点下载限流提供成熟基础设施
- 成本：中
- 风险点：需要重新梳理 DB 状态与内存运行态的边界
- 来源：
  - Bottleneck: https://github.com/SGrondin/bottleneck
  - p-queue: https://github.com/sindresorhus/p-queue
  - BullMQ Overview: https://docs.bullmq.io/
  - BullMQ Rate Limiting: https://docs.bullmq.io/guide/rate-limiting

#### 9.2.4 结构化日志与崩溃监控
- 先进方案：桌面应用至少要把本地落盘日志、结构化字段、主进程/渲染进程异常捕获、崩溃上报区分开来处理。
- 当前项目差距：当前仓库仍主要依赖 `console.log` / `console.error`，缺少统一落盘日志与崩溃监控方案。
- 候选方案：
  - `electron-log`：最适合当前项目，接入成本低，Electron 亲和度高
  - `@sentry/electron`：适合接住主进程/渲染进程/预加载层异常与 crash 事件
  - `pino` / `winston`：更偏工程化通用日志层，但不如 `electron-log` 适合当前快速落地
- 是否值得引入：值得，高优先级
- 收益：排障效率、生产问题定位能力、崩溃后追因能力都会明显提升
- 成本：低到中
- 风险点：需要先定义脱敏策略，避免把 URL、路径、账号等敏感信息直接打进日志
- 来源：
  - electron-log: https://github.com/megahertz/electron-log
  - Sentry for Electron: https://docs.sentry.io/platforms/javascript/guides/electron/
  - Pino: https://github.com/pinojs/pino
  - Winston: https://github.com/winstonjs/winston

### 9.3 推荐引入/评估的库

#### `electron-log`
- 用途：主进程/渲染进程统一落盘日志
- 收益：排障效率直接提升
- 成本：低
- 风险：低
- 优先级：高
- 更适合当前项目的原因：Electron 亲和度高，能快速替换散落的 `console.log`

#### `@sentry/electron`
- 用途：崩溃与未处理异常监控
- 收益：真正能看生产问题
- 成本：中
- 风险：中（需脱敏）
- 优先级：中高
- 更适合当前项目的原因：当前已接入基础 quit/crash/unhandled exception 治理，但桌面级恢复证据与生产可观测性仍不足

#### `Bottleneck`
- 用途：站点级 API 限流
- 收益：Booru 多站点差异限流更可控
- 成本：中
- 风险：低到中
- 优先级：高
- 更适合当前项目的原因：比 `p-queue` 更适合 Booru 多站点、429、动态退避场景

#### `p-queue`
- 用途：下载执行池
- 收益：比手写循环更稳，更容易做 pause/resume/priority
- 成本：中
- 风险：中
- 优先级：高
- 更适合当前项目的原因：如果希望先少改代码、快速规范化内存队列 API，它比全面重构更容易切入

#### `react-virtuoso` 或 `@tanstack/react-virtual`
- 用途：图库/Booru 大列表虚拟化
- 收益：滚动性能、内存与 DOM 压力显著改善
- 成本：中
- 风险：中
- 优先级：高
- 更适合当前项目的原因：图库和 Booru 页已经明显接近虚拟化收益阈值

### 9.4 当前项目与先进方案差距
- tray / close-to-tray / 通知 / 基础崩溃治理已补齐代码主链路，但真实桌面环境验收证据仍未完备
- 开机自启仍缺失
- 缺稳定队列状态机
- 缺虚拟列表
- 缺结构化日志与更完整的生产级崩溃监控
- webview 集成还停留在“能嵌进去”层面
- 远程内容安全治理未对齐 Electron 官方安全清单

### 9.5 替代建议原则
- 不是“全盘换掉自研”
- 而是把最容易出结构性问题的领域交给成熟库：
  - 执行池
  - 限流器
  - 虚拟列表
  - 日志
  - 崩溃监控
- 更具体地说：
  - 生命周期、安全边界、Tray、Notification 先优先用 Electron 原生能力补齐
  - 队列与限流优先考虑 `Bottleneck` / `p-queue`
  - 列表虚拟化优先考虑 `react-virtuoso` / `@tanstack/react-virtual`
  - 日志与崩溃监控优先考虑 `electron-log` / `@sentry/electron`

## 12. 附：本次审查最重要的判断

如果只保留 8 个最有决策价值的结论：

1. 这个项目的最大风险不在“缺几个按钮”，而在**下载与恢复链路的一致性设计不稳**。
2. 它还没有成为一个合格的桌面下载/图库工具，因为**桌面承载面的代码主链路虽已补齐，但真实桌面环境验收仍未完成，且开机自启等能力仍缺**。
3. 当前的 preload / IPC / shared 协议边界已经开始失控，继续堆功能只会更难收拾。
4. 页面设置项里已经出现“看起来能配，实际上没生效”的伪能力，这是产品可信度问题。
5. 测试全绿不能说明产品合格，因为它没有覆盖桌面生命周期、真实运行入口、后台体验和交互一致性。
6. 大量弹窗和导航行为显得任意化，说明交互规范没有被建立起来。
7. 这个项目已经到了该借力成熟库的时候，再继续扩大自研面，维护成本会急剧上升。
8. 最值得立即投入的不是“再加功能”，而是**先还掉核心技术债**。
